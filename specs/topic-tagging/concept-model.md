# Topic 概念モデル

## Topic とは

> AI が記事ごとに判定する「この記事は何のニュースか」を表すラベル。
> 記事の本質を 2-4 語の英語で表現する。投資家がフィルタに使う分類タグ。

- **生成者**: AI (自由生成 + 正規化)
- **語彙**: 開いている (動的に増加)
- **記事との関係**: 1:1 (記事ごとに 1 Topic)
- **Category との関係**: N:1 (Topic は 1 つの Category に属する)
- **例**: "ai drug discovery", "quantum error correction", "semiconductor trade policy"
- **用途**: 記事カードに表示、サイドバーでフィルタ

## Category との関係

```
Category (人間定義・固定・10個)
  └── Topic (AI生成・正規化済み・動的増加)
        └── Article (記事)
```

- Category は最上位の棚。人間が管理する。変更なし
- Topic は Category の下にぶら下がる
- AI が記事分析時に Category と Topic を同時に判定する
- 記事 → Topic → Category の帰属が一意に決まる
- 同じ Topic 名でも異なる Category に存在しうる ((name, category_id) で UNIQUE)

## 現行 Keyword との対比

| 観点 | 現行 Keyword | 新 Topic |
|---|---|---|
| 生成方式 | 人間が事前定義した 72 語から AI が選択 | AI が自由生成 |
| 精度 | 低い (抽象的すぎて記事の具体性に合わない) | 高い (記事の本質を直接表現) |
| 語彙 | 閉じている (手動追加のみ) | 開いている (AI が必要に応じて拡張) |
| 記事との関係 | M:N (最大 3 個) | 1:1 (記事ごとに 1 つ) |
| Category 紐付け | Keyword.category_id で 1:1 | Topic.category_id で 1:1 |
| 固有名詞の扱い | 分類タグと混在 | Topic には含めない (別機能の対象) |

## 正規化ルール

AI が自由に生成した Topic を以下のパイプラインで正規化する:

```
AI 出力 → 英語小文字化 → 記号統一 → trim → 既存 Topic と完全一致 → 保存
```

### Phase 1 (MVP)
- AI に「英語 2-4 語のラベル」という制約をプロンプトで指定
- 正規化後の完全一致で既存 Topic に収束
- 一致しなければ新規 Topic を自動作成

### Phase 2 (蓄積後)
- Topic の embedding を生成し、類似度で名寄せ候補を検出
- 管理者画面でマージ操作を提供

## 品質ゲート

- AI が直接生成するため、品質はプロンプト設計と正規化に依存
- 不適切な Topic は管理者が後から修正・マージ
- 記事表示を遅延させない (生成即表示)

## 設計判断の記録

### なぜ事前定義リストを廃止するか
- 現行の 72 キーワードは「カテゴリの第2階層」相当の粒度で、記事の具体性に追いつかない
- AI が候補の中から無理やり近いものを選ぶ → 関係ないタグがつく
- 「選択肢の中から選ばせる」アプローチ自体が精度を下げる一因

### なぜ Category と Topic を紐付けるか
- サイドバーの階層構造 (Category 展開 → Topic 一覧) がそのまま使える
- AI に Category と Topic を同時に判定させることで紐付けが自然に出る
- 現行 Keyword の Category 紐付けと同じ構造なので、移行がシンプル

### なぜ記事ごとに 1 Topic か
- 「この記事は何のニュースか」の答えは本質的に 1 つ
- 複数タグをつけると焦点がぼやける (現行の最大 3 個で起きていた問題)
- フィルタとしても 1:1 の方がシンプルで正確

### なぜ (name, category_id) で UNIQUE か
- 同じ Topic 名が異なる Category に存在しうる (例: "supply chain disruption")
- フィルタが Category 内で閉じるので直感的
- 既存 Topic リストをカテゴリ内で照合する設計と噛み合う
