# DB リファクタリング後の品質チェックリスト

> 作成日: 2026-03-24
> 目的: 各 Phase 完了後に確認すべき観点のメモ

## 意識すべき観点

### 1. Fail Fast

- バリデーションは DB 操作より前で行う
- スキーマ（値オブジェクト）→ FK 存在確認 → ドメインルール検証 → DB 操作、の順序を守る
- 不正値を無視（silent skip）せず、ログ出力 + 明示的スキップにする

### 2. 値オブジェクトの振る舞い

- スペック定義の値オブジェクトが `domain/` に実装されているか
- スキーマの入力フィールドで値オブジェクト型を使っているか（`str` のまま残していないか）
- エンティティの状態遷移ルールがドメイン層 or サービス層で強制されているか
- 不変属性（`is_ai_generated` 等）が更新時に保護されているか

### 3. 責務の境界

各レイヤーが自分の責務だけを持ち、他のレイヤーの仕事をしていないか確認する。

| レイヤー | 責務 | やってはいけないこと |
|---|---|---|
| スキーマ（Pydantic） | 入力値の形式検証、型固有の必須フィールド判定 | DB アクセス、HTTP ステータスコード判定 |
| 値オブジェクト（domain/） | 値の不変条件の保証、状態遷移ルールの強制 | 永続化、外部サービス呼び出し |
| ルーター | HTTP の関心事（ステータスコード、レスポンス構築）、FK 存在確認 | ビジネスロジック、文字種バリデーション |
| サービス層 | ビジネスロジックのオーケストレーション、外部サービス連携 | HTTP の関心事、直接的な DB クエリ構築 |
| DB 制約 | 最終防衛線（UNIQUE、CHECK、FK、NOT NULL） | アプリ層のルールの代替 |

#### 境界違反の例

- **スキーマの仕事をルーターでやっている**: `if body.source_type == "rss" and not body.feed_url` → スキーマの `model_validator` が担うべき
- **ドメインの仕事をルーターでやっている**: status 遷移のルールをルーターの `if` 文で書く → ドメインメソッドが担うべき
- **アプリ層の仕事を DB 制約に任せている**: 文字種制約を DB の CHECK 正規表現で書く → 値オブジェクトが担うべき（ロケール依存を避ける）

### 4. AI 出力の扱い

- AI が返すデータは外部入力と同等に扱い、値オブジェクトで検証する
- 検証を通らなかった値はログに記録してスキップする

### 5. 発行される SQL の確認

- N+1 クエリが発生していないか（`selectinload` / `JOIN + GROUP BY` で解決）
- 不要な `SELECT` が走っていないか
- `selectinload` のチェーンが深すぎないか（4段以上は要注意）
- `COUNT` サブクエリが各行ごとに発行されていないか
- フィルタ条件で使う `IN (subquery)` が適切か（`EXISTS` の方が効率的な場合がないか）

### 6. DB 制約との整合

- アプリ層の Enum / 値オブジェクトと DB の CHECK 制約が一致しているか
- FK の ON DELETE（CASCADE / RESTRICT）がスペック通りか
- NOT NULL 制約がモデルの `nullable` 設定と一致しているか

## 現時点の未対応メモ

- `Keyword.status`: `str` のまま → Enum 化 + 状態遷移メソッド未実装
- `Keyword.approved_at`: `status` との整合性を保証するコードがない
- `ai_analyzer.py`: AI が返すキーワードを `KeywordName()` で検証していない
- `KeywordUpdate` スキーマ: status 遷移用のフィールドがない
- `NewsSourceCreate`: 型固有フィールド検証の責務がスキーマとルーターに分散
- `updated_at` の自動更新: 手動で `datetime.now(UTC)` を代入 → ORM イベント or Mixin で共通化すべき
