## 1. アプリケーション定義

### 1.1 このアプリケーションは何か

先端テクノロジー（AI、半導体、バイオ等）に関する英語ニュースをAIで自動収集・翻訳し、
業界の動向やトレンドを継続的に分析した上で日本語で提供する、
個人投資家向けのテック・インテリジェンスWebアプリケーション。
開くだけで先端テック業界の全体像と変化がわかることを目指す。

### 1.2 誰のためのアプリか

- ターゲット: 先端テクノロジーがもっとも資金が集まる領域だと確信している個人投資家
- 投資スタイル（短期/中長期/テーマ型）の分類より、「テクノロジーの動向を深く理解した上で投資判断をしたい」という姿勢が共通点
- 英語の一次ソースに価値を感じているが、言語の壁と分析の手間で情報活用にコストがかかっている人
- 典型的な使い方: 朝一に最新記事を流し読み、気になったときに深掘り、週末にトレンドレポートで全体像を把握

### 1.3 解決する課題

1. 先端テックのニュースは英語記事が多く、日本語で読めるようになるまでにタイムラグがあり、個別に読んで業界動向を把握するのに時間がかかる
2. 1記事ずつ読んでも「この業界で今何が起きているのか」という全体像が見えにくい
3. ニュースに登場する企業の株価・関連ニュースなど、複数ソースを横断しないと得られない情報がある

### 1.4 提供する価値（優先順位）

| 優先度 | 価値 | 説明 | 実装状況 |
|--------|------|------|----------|
| 1 | 分析の深さ | 個別記事の要約だけでなく、業界動向・トレンドを横断的に深く理解できる | 個別記事の分析は稼働中。トレンド分析は未実装 |
| 2 | 速報性 | 英語ニュースをいち早く日本語で読める | 稼働中 |
| 3 | 情報集約 | ニュースと企業・株価を紐づけて一箇所で見られる | 段階A→B→Cで実装（下記「情報集約の段階的実装」参照） |
| 4 | ソース網羅 | 他にはないニュースソースをカバーしている | 稼働中 |

> アプリのアイデンティティ:
> 「英語テックニュースの日本語速報メディア」ではない。
> 「先端テクノロジー業界の動向を深く分析し、投資判断を支援するインテリジェンスツール」。
> 速報性は重要だが、早いけど浅い情報より、少し遅くても深い分析の方が価値がある。

#### 価値を支える「手段」（独立した価値ではないが品質・体験に不可欠）

| 手段 | 支える価値 | 実装状況 |
|------|-----------|----------|
| 重複排除 | 分析の深さ・速報性（重複ニュースがあると分析もノイズが増える） | 稼働中 |
| カテゴリ・タグフィルター | 分析の深さ（関心領域に絞って深掘りしやすくする） | 稼働中 |
| キーワード検索 | 分析の深さ・情報集約（特定の話題や企業名で絞る） | 稼働中 |
| フィルター設定の記憶 | 閲覧体験の快適さ（毎回同じ操作をしなくて済む） | 未実装 |
| お気に入り登録（ウォッチリスト） | 閲覧体験の快適さ（気になった記事を後で読み返せる） | 稼働中 |

> 注: 対象範囲を「先端テクノロジー」に絞って収集している時点で、大きなフィルタリングはすでにかかっている。
> 記事数が将来大幅に増えた場合は、パーソナライズを独立した価値として再検討する。

> 情報集約の段階的実装の詳細は「1.7 情報集約の段階的実装」を参照。

### 1.5 提供形態・技術的前提

- 提供形態: Webアプリケーション
- 開発体制: 1人開発
- 現在の状態: ニュース収集・翻訳・閲覧は稼働中
- 公開状況: 将来的に一般公開予定（現在は非公開）

### 1.6 タグ体系

> 記事に付与されるタグは3種類。それぞれ役割と管理方法が異なる。

| タグ種別 | 役割 | 管理方法 | タップ時の挙動 |
|----------|------|----------|----------------|
| カテゴリ（セクター） | 業界分類の大枠 | 事前定義（管理画面で管理） | 記事一覧フィルタ + トレンドレポート直リンク |
| キーワード（技術タグ） | セクター内の具体的な技術・テーマ | AI自動検出 + 管理者承認（下記参照） | 記事一覧フィルタ |
| 企業タグ | 記事に登場する企業 | AI自動抽出 + 企業マスタで名寄せ | 企業ページへ遷移 |

#### カテゴリとキーワードの関係

- カテゴリ = セクター（AI・ML、半導体、バイオテック等）。10個程度
- キーワード = カテゴリ配下の具体的な技術タグ。セクター内の専門的な粒度で定義する
  - 例: 量子コンピュータ → `超伝導方式`, `イオントラップ方式`, `光量子`, `量子エラー訂正`
  - 例: AI・ML → `large language model`, `AI agent`, `multimodal AI`
- 初期キーワードは事前定義。以降はAI自動検出 + 管理者承認で追加

#### キーワードの自動検出と承認フロー

```
AI分析で未知の技術用語を検出
  → 暫定タグとして記事に付与（ユーザーには控えめなデザインで表示）
  → 管理者に通知

管理者の3アクション:
  A. 正式化 → 正式キーワードとしてマスタに登録（「New」バッジ付きで表示）
  B. マージ → 既存タグの同義語として紐づけ（次回以降は自動変換）
             例: 「Generative AI」→「生成AI」の同義語として登録
  C. 削除 → ブラックリストに追加（次回以降は暫定タグとしても出現しない）

放置時（一定期間未対応）→ 自動非表示（フェイルセーフ）
```

> 設計上の要点:
> - 暫定タグはユーザーに見えるが、正式タグより控えめなデザイン（薄い色、点線ボーダー等）
> - 暫定タグはトレンドレポートの集計対象には含めない
> - 正式化された新キーワードには「New」バッジを表示（管理者承認済み = 品質担保）
> - マージによりAIの表記ゆれを学習的に吸収するシステムに成長する
> - ブラックリストにより誤検出の再出現を防止

#### トレンドレポート内での各タグの扱い

| タグ種別 | トレンドレポート内での表示 |
|----------|--------------------------|
| カテゴリ | レポートの単位そのもの（セクター別レポート） |
| キーワード（正式タグのみ） | 「今週注目された技術」として頻出キーワードを表示 |
| 企業タグ | 「企業別の動き」として言及企業を表示 |

### 1.7 情報集約の段階的実装

| 段階 | 名称 | 内容 | 前提条件 |
|------|------|------|----------|
| **A** | 企業タグ自動抽出 | AI分析時に記事から企業名を抽出。企業マスタと名寄せしてタグ付与。マスタにない企業名は管理者に通知 | 企業マスタの初期構築 |
| **B** | 企業ページ | 企業タグクリック → 関連記事一覧 + 基本情報（正式名称、ティッカー、セクター） | A完了 |
| **C** | 株価データ統合 | 企業ページに終値+変動率+簡易チャート表示。記事一覧では企業タグ横にティッカー+終値の簡易表示 | B完了 + 株価API導入 |

#### 企業マスタ

- 主要企業を事前定義（正式名称、別名/略称、ティッカーシンボル、セクター）
- AIが記事から抽出した企業名をマスタと照合し、名寄せを行う
- マスタにない企業名は暫定タグとして付与し、管理者に通知

管理者の3アクション（キーワードと同じ構造）:
- **正式化**: 新企業としてマスタに登録（正式名称、ティッカー、セクターを設定）
- **マージ**: 既存企業の別名として紐づけ（次回以降は自動変換）
  - 例: 「台湾積体電路」→ TSMCの同義語として登録
  - 例: 「NVIDIA」「Nvidia」「エヌビディア」→ 同一企業
- **削除**: ブラックリストに追加（企業名ではない語の誤検出を防止）

> 放置時は自動非表示（キーワードと同じフェイルセーフ）

#### 株価データソース

- 対象: **米国株のみ**でスタート。日本株は将来有料API導入時に対応
- データソース: Twelve Data（株価）+ Finnhub（企業基本情報）
- 更新頻度: サーバーサイドキャッシュで1日1回更新（リアルタイム不要）
- アトリビューション: 「Powered by Twelve Data」の表示必須
- 商用利用: Twelve Dataは条件付きOK（アトリビューション必須）、Finnhubは明確な禁止なし

#### 表示の2段階

| 場所 | 表示内容 | 例 |
|------|----------|-----|
| 記事一覧（簡易） | ティッカー + 終値 | `NVDA $124.50` |
| 企業ページ（詳細） | 基本情報 + 終値・変動率 + 簡易チャート + 関連記事一覧 | セクター、時価総額、1週〜3ヶ月チャート |

#### 将来構想

- 日本株対応（Twelve Data有料プラン $29/月〜）
- 企業分析レポート（トレンドレポートの企業版）

### 1.8 ユーザー利用フロー

#### パターン1: 朝一チェック（最新記事の流し読み）

```
ログイン
  → ダッシュボード（記事一覧）
    → 最新記事をざっと流し読み
    → 気になるキーワードやカテゴリでフィルタ
    → 気になった記事 → 記事詳細（AI分析・関連記事）
    → 気になった記事をウォッチリストに保存
```

#### パターン2: 随時チェック（特定テーマの深掘り）

```
ログイン
  → ダッシュボード
    → サイドバーでキーワード/カテゴリ絞り込み
    → または検索バーで企業名・技術名を検索
    → 記事一覧 → 記事詳細
      → 企業タグクリック → 企業ページ（関連記事 + 株価）
      → キーワードタグクリック → そのキーワードの記事一覧にフィルタ
```

#### パターン3: 週末の全体把握（トレンドレポート起点）

```
ログイン
  → ヘッダー「Trends」クリック
    → レポート一覧（セクター別の週次レポートが並ぶ）
      → 関心セクターのレポートを選択
        → サマリで「先週何が起きたか」を把握
        → 注目トピックで重要イベント確認
        → 注目された技術キーワード → 記事一覧にフィルタ
        → 企業タグクリック → 企業ページ → 関連記事
```

#### 画面構成と導線

```
ヘッダー: [Dashboard] [Trends] [Watchlist] [Theme] [User]
                         │
         ┌───────────────┼───────────────┐
         │               │               │
    Dashboard        Trends          Watchlist
    (記事一覧)    (レポート一覧)    (保存済み記事)
         │               │
    ┌────┴────┐     ┌────┴────┐
    │         │     │         │
  記事詳細  企業ページ  セクター別レポート
    │         │
    └────┬────┘
    相互リンク
```

- **ダッシュボード → トレンド**: サイドバーのカテゴリ横からセクターレポートに直リンク
- **トレンド → 記事**: レポート内の企業タグ・キーワードから記事一覧にフィルタ遷移
- **記事 → 企業**: 記事詳細・記事一覧の企業タグから企業ページへ遷移

#### 管理者フロー

> 管理画面（要実装）で以下を管理する。

| 機能 | 内容 |
|------|------|
| 暫定タグの承認 | AI検出した暫定キーワード・企業名の一覧。正式化 / マージ（同義語紐づけ） / 削除（ブラックリスト追加）の3アクション |
| キーワード管理 | 正式キーワードの編集・削除・カテゴリ紐づけ・同義語管理 |
| 企業マスタ管理 | 企業の追加・編集（正式名称、別名、ティッカー、セクター） |
| ブラックリスト管理 | 誤検出として除外した語の一覧。再出現を防止 |
| ニュースソース管理 | 収集対象ソースの追加・削除（既存の設定画面を移管） |

### 1.9 代替手段との差別化

> Vectorはニッチなプロダクトである。英語を自力で読める人はこのアプリを必要としない。
> 主な代替手段はAIツール（ChatGPT Deep Research、Perplexity等）で個別に調べること。

#### AIツールとの違い

| | AIツール（Deep Research等） | Vector |
|---|---|---|
| 起動方式 | 能動的（質問しないと始まらない） | 受動的（自動で収集・分析される） |
| 対象範囲 | 聞いたことだけ | 先端テック全域を網羅 |
| 時間軸 | スナップショット（聞いた時点） | 継続的（週次トレンドの変化を追跡） |
| 見落とし | 質問しなかったテーマは見えない | 注目トピックで自動検出 |
| 手間 | 毎回プロンプトを考える必要がある | 開くだけ |

#### Vectorの本質的な価値

**「自分で調べに行かなくても、開けば先端テック業界の全体像と変化がわかる」**

- 能動的に調べなくても情報が集まっている
- 継続的に追跡することで、1回のリサーチでは見えない変化やトレンドが見える
- 関心外の領域でも重要ニュースを見落とさない

### 1.10 収益モデル

- 有料化は検討中だが、有料にすべき機能はまだ未確定
- 現時点では決め打ちせず、後から柔軟に対応できる設計にしておく

### 1.11 対象範囲の境界（このアプリが「やらないこと」）

> ドメインの定義には「何をやるか」と同時に「何をやらないか」が重要。
> 以下は現時点の判断であり、棚卸しを進めながら更新する。

- [ ] 投資助言・推奨銘柄の提示は行わない（情報提供のみ）
- [ ] ニュースの独自取材・執筆は行わない（既存ソースの収集・翻訳・分析）
- [ ] 売買の執行機能は持たない
- [ ] SNS的な機能（コメント、いいね等）は当面持たない
- [ ] （要検討）他に「やらない」と明確に決めたことがあれば追記

#### 「投資支援」と「投資助言」の境界線

> AI分析のプロンプト設計に直結するため、具体例で線引きを明確にする。

**基本原則**:
- 事実の提示 + 「可能性がある」「影響を与えうる」等の示唆 → **OK**
- 「買い」「売り」「割安」「割高」等の売買判断 → **NG**

| 表現例 | 判定 | 理由 |
|--------|------|------|
| 「NVIDIAに関するニュースが今週15件、前週の3倍」 | OK | 事実の提示 |
| 「CHIPS法が半導体サプライチェーンに影響を与える可能性」 | OK | 事実に基づく分析 |
| 「この発表後、NVDA株は5%上昇した」 | OK | 株価変動の事実 |
| 「半導体セクターは今後も成長が期待される」 | OK | セクターの分析 |
| 「規制強化により、TSMCの競争優位性が高まる可能性」 | OK | 事実を踏まえたAI分析 |
| 「NVDA株は割安なので買い時」 | NG | 売買判断の指示 |
| 「今がTSMC株の買い増しチャンス」 | NG | 売買行動の推奨 |

> トレンドレポートの企業別の動きでは、ニュースに関連する株価変動の事実（「発表後に株価が○%変動した」等）を添える。
> これは事実の提示であり、投資助言には該当しない。

### 1.12 成功指標・運用方針

#### 成功指標

- 主要指標: **再訪率**（ユーザーが毎日/毎週戻ってきているか）
- ニッチなプロダクトであり、大量のユーザー獲得は想定しない
- 「使っているユーザーが継続的に価値を感じている」ことが成功の基準

#### 記事の鮮度・アーカイブ方針

| 対象 | 方針 |
|------|------|
| ダッシュボード（記事一覧）のデフォルト表示 | 直近2ヶ月の記事 |
| 検索・企業ページ・キーワードフィルタ | 過去1年分を対象に含める |
| 1年超の記事 | ユーザー表示対象外。データベースには保持し、トレンド分析の素材として活用 |
| トレンドレポート | 過去分も閲覧可能にする（週次レポートのアーカイブ） |

> デフォルト表示は2ヶ月だが、検索や企業ページでは過去の文脈が重要。
> 中長期の投資家にとって、企業や技術テーマの過去の流れが見えることは価値がある。
> 数万件程度ならDBパフォーマンスへの影響は軽微。

### 1.13 速報性とコストのバランス

> 分析の深さが第1の価値であり、速報性は第2。
> 早いけど浅い情報より、少し遅くても深い分析の方が価値がある。
> ただしAPIコストとの兼ね合いも考慮が必要。

#### コスト制約

| 項目 | 方針 |
|------|------|
| 月額コスト上限 | 目安として2,000円/月程度。商業的な可能性が見えれば許容範囲を拡大 |
| 主なコスト要因 | AI分析API（記事ごとの第1層 + 週次の第2層）、株価API（Twelve Data無料枠） |
| 記事数/日 | 速報性を担保できる件数を処理する。ただしコストとのバランスを考慮 |
| ユーザー数 | 目標は未設定。ニッチなプロダクトとして少数ユーザーでも価値提供できる設計にする |
| パイプライン設計 | 過去記事の分析で速報性が詰まらない設計が必要（キューの優先度制御等） |

| 項目 | 基準 |
|------|------|
| 目標時間 | 英語記事の公開から1時間以内に日本語で読めるようにする（個別記事の翻訳・分析） |
| 翻訳品質 | 自然な日本語で読める品質が必要（スピードのために品質を犠牲にしない） |
| 分析品質 | 速報性のために分析の深さを犠牲にしない |
| 記事の優先度 | 重要度が高い記事を優先的に処理する仕組みを導入したい |
| コスト意識 | 速報性を追求して全記事をリアルタイムでAPIに投げるとコストが跳ね上がる。バッチ処理の活用を検討する |

#### 記事の重要度判断（検討中）

重要度の判断軸として以下の3つが考えられる。

1. **ソースの重要度** — 公式発表・大手メディアは個人ブログより優先。NewsSourceの属性として事前設定可能
2. **内容の重要度** — 決算発表・M&A・規制変更などは技術解説より優先。収集時にAIで軽くトリアージする
3. **速報性そのもの** — 「今まさに起きていること」は数日前の記事より優先

> 具体的な実現方法はClaudeCodeとの壁打ちでパイプライン設計と合わせて詰める。

#### AI分析の2層構造

| 層 | 内容 | タイミング | コスト感 |
|----|------|-----------|---------|
| 第1層: 個別記事分析 | 1記事ごとの翻訳・要約・キーワード抽出 | 記事収集後、1時間以内 | 記事ごとにAPI呼び出し |
| 第2層: トレンド分析 | 複数記事を横断した業界動向・トレンドの分析 | バッチ処理（週1回） | まとめて処理でコスト効率が良い |

> 第2層のトレンド分析の例:
> - 「今週、半導体業界では何が起きているか」
> - 「NVIDIAに関するニュースの流れはどう変化しているか」
> - 「AI規制に関する動きのまとめ」
>
> 第2層はバッチ処理にすることで、コストを抑えつつ分析の深さを実現できる。
> これが第1の価値（分析の深さ）の核心部分。

#### 記事の処理パイプライン（想定）

```
【第1層: リアルタイム処理】
収集 → トリアージ（優先度判定） → 翻訳 → 個別記事分析 → 公開
                                  ↑ 品質確保      ↑ 全体で1時間以内

【第2層: バッチ処理】
蓄積された記事群 → トレンド分析 → トレンドレポート公開
                  ↑ 深い分析      ↑ 週1回
```

#### トレンド分析の具体像（第2層の詳細）

##### レポートの形態

- テキストベースのレポート + 客観データ（キーワード頻度・記事数等）のハイブリッド
- セクター（業界）ごとに1レポートを生成

##### レポートの構造（想定）

```
━━━ {セクター名} 週次レポート（{対象期間}） ━━━

■ サマリ（3-5行）
  業界全体の流れをテキストで把握できる要約

■ 注目トピック（見落とし防止）
  株価・業界へのインパクトが大きいイベントを表示（影響度付き）

■ 客観データ
  - 記事数（前週比）
  - 頻出キーワード（出現回数付き）
  - 言及企業（記事数付き）

■ 企業別の動き
  主要企業ごとに何が起きたかを分析

■ 前週からの変化
  トレンドの増減・変化を可視化

■ 関連記事一覧
  レポート対象の全記事へのリンク
```

##### 分析の切り口

| 軸 | 位置づけ | 備考 |
|----|----------|------|
| セクター（業界）別 | 主軸 | 半導体、AI/ML、バイオなど |
| 企業別 | セクター内の要素 | セクターレポート内で企業の動きを分析 |
| テーマ別 | セクター横断の関連 | 「規制動向が半導体とAI両方に影響」等の複数軸分析 |

##### セクター定義

- 事前に固定定義する（AIによる動的クラスタリングは採用しない）
- 現在のカテゴリ分類を流用し、必要に応じて見直す
- 新しいセクター（新興領域）への感度は維持する
  - 既存セクターに分類できない記事が一定数を超えた場合、新セクターの追加を検討する

##### 時間軸

- 週次がベース（日次ではトレンドは把握できない）
- 月次・年次は将来検討（コスト次第）

##### 注目トピックの判定方針

- 判定基準: 「株価・業界へのインパクト」の大きさ
- 判定方法: AI判定 + 定量指標の併用を想定するが、初期はAI判定から開始
- 要件: 記事数が少なくても重要イベント（規制変更、M&A、大型提携等）を見落とさないこと

> 定量指標だけでは「記事1本だが重大」なニュースを見落とすリスクがある。
> AI判定を基本とし、定量指標（記事数急増、キーワード頻度の異常値）は補助的に活用する。

> 画面・導線については「1.8 ユーザー利用フロー」を参照。

### 1.14 セキュリティ上の心配事（CIA-T）

> 第1章で学んだCIA-T（機密性・完全性・可用性・追跡可能性）の観点で整理。

#### 機密性（Confidentiality）

守るべき秘密情報:

| 情報 | 重要度 | 備考 |
|------|--------|------|
| AI分析に使うAPIキー | 高 | 漏洩すると不正利用・課金被害に直結 |
| ユーザーのメールアドレス・パスワード | 高 | 一般公開時に発生。パスワードはハッシュ化必須 |

> 現時点ではニュース記事自体は公開情報なので機密性の対象外。
> ただし、ユーザーの閲覧履歴・関心銘柄は将来的に機密性の対象になりうる。

#### 完全性（Integrity）

投資判断に使われるアプリのため、完全性は最重要。

| 対象 | 重要度 | 具体的な心配事 |
|------|--------|---------------|
| 記事の内容 | 高 | 改ざんされると誤った投資判断につながる |
| 翻訳の正確性 | 高 | 原文の意味を正しく伝えている必要がある |
| AI分析結果 | 高 | 勝手に書き換えられないこと |
| 株価データ | 高 | 正確であること（将来実装時） |

#### 可用性（Availability）

| 対象 | 重要度 | 理由 |
|------|--------|------|
| ニュース収集パイプライン | 致命的 | 収集が止まると分析の元データが途切れ、アプリの第1の価値（分析の深さ）も第2の価値（速報性）も失われる |
| 閲覧サーバー | 致命的 | 収集が動いていても閲覧できなければユーザーにとっては「止まっている」のと同じ |

> 設計上の示唆: 両方とも致命的だからこそ、障害の分離が重要。
> 「片方が落ちてももう片方は動く」構造にすることで、全体がまとめて落ちるリスクを避ける。
> - 収集側が落ちても、既に収集済みの記事は閲覧できる
> - 閲覧側が落ちても、収集は続き、復旧後にすぐ最新記事が読める
>
> 実現方針: 収集・分析処理（クローラー、AI分析バッチ）と閲覧API（FastAPI）のプロセスを分離する。
> 非同期タスクワーカー（Taskiq等）の導入により、重いバッチ処理が閲覧APIのパフォーマンスに影響しない構成にする。

#### 追跡可能性（Traceability）

| 対象 | 必要性 | 理由 |
|------|--------|------|
| 記事がどのソースからいつ収集されたか | 高 | 出所の透明性。投資家は情報源を重視する |
| 翻訳・分析にどのAIモデルを使ったか | 中 | モデル変更時の品質変化を追跡するため |
| 誰がいつどの記事を閲覧したか | 低→高 | 現在は不要だが、有料化時にはアクセスログが必要になる |

### 1.15 AI分析に関する免責方針

| 項目 | 方針 |
|------|------|
| 基本姿勢 | AI分析は誤りがある前提で設計する。確定的な事実ではなく「参考情報」として扱う |
| 責任の所在 | 「AIだから不正確かもしれない」という謝罪ではなく、「判断材料の一つであり、最終的な投資判断はユーザー自身が行う」という責任の所在の明確化 |
| 不正確な分析への対応 | 免責でカバーする。自動再分析や手動修正の仕組みは当面設けない |

#### 免責の3層構造

| 層 | 内容 | 決定状況 |
|----|------|----------|
| 第1層: 利用規約 | 法的な免責条項を記載 | 一般公開時に専門家に相談して策定 |
| 第2層: UI上の免責表示 | アプリのフッターやAI分析結果の近くに簡潔な免責文を配置 | 配置場所は未確定（UI設計時に決める） |
| 第3層: AI出力の表現制御 | 分析結果を「断定」ではなく「示唆」の表現にする | AIのプロンプト設計で制御する |

> 第3層の表現制御の具体的なOK/NG例は「1.11 対象範囲の境界 > 投資支援と投資助言の境界線」を参照。

> 設計上の示唆:
> - AI分析結果のドメイン概念には「これは参考情報である」ことを構造的に表現する
> - 分析結果を「事実」として断定する表現をUIで使わない
> - 将来的に免責表示の具体的な配置を決めた際、表示を強制する仕組みが必要になる可能性がある
> - 法的な免責条項の文言は、一般公開前に専門家に相談すること
