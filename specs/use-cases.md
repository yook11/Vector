# Vector ユースケース・システム振る舞い

> 概念モデル（`db-domain-model.md`）はエンティティの「何であるか」を定義する。
> このドキュメントはシステムの「どう振る舞うか」を定義する。

## 記事の表示・管理

- ダッシュボードのデフォルト表示は直近2ヶ月の記事
- 検索・企業ページ・キーワードフィルタでは過去1年分を対象に含める
- 1年超の記事はユーザー表示対象外。DBには保持しトレンド分析の素材として活用する
- 同一ニュースの重複はシステムが自動検出・排除する（分析品質の担保）

## 記事の処理パイプライン

- 収集 → トリアージ（優先度判定） → 翻訳 → 個別記事分析 → 公開
- トリアージはソース重要度・内容種別・速報性から処理順序を決定する（ユーザーには非公開）
- 目標: 英語記事の公開から1時間以内に日本語で読めるようにする
- 重要度が高い記事（M&A、決算、規制変更等）を優先的に処理する

## 暫定タグの自動非表示

- provisional ステータスのまま一定期間（detectedAt から起算）が経過した Keyword・Company はユーザーへの表示対象から除外する
- ステータスは provisional のまま変更しない
- 管理者が後から承認すれば即座に表示される
- 閾値は運用パラメータとして設定で管理する

## カテゴリの変更運用

- カテゴリは事前定義の固定セクター分類（10個程度）であり、頻繁な変更は想定しない
- カテゴリの変更は「削除」ではなく「改名・統合・分割」で行う
  - 改名: リレーションは維持、名前だけ変わる。影響なし
  - 統合: 2つのカテゴリを1つにまとめる。Keyword・Company・CustomFeed の参照先を管理者が付け替える
  - 分割: 1つのカテゴリを2つに分ける。Keyword・Company を管理者が振り分け直す
- カテゴリを削除すると Keyword（1:N）、Company（M:N）、TrendReport（1:N）、CustomFeed（M:N）に連鎖影響が出る。特に CustomFeed はユーザーのフィルタ設定が壊れるため、削除ではなく統合・分割で対応する
- 既存セクターに分類できない記事が一定数を超えた場合、新セクターの追加を検討する
- 横断的キーワードの出現はカテゴリまたはキーワード粒度の見直しシグナルとして扱い、M:N 化ではなく分類体系の調整で対応する

## AI分析の免責表示

- 全てのAI分析結果は「参考情報」であり事実の断定ではない（ドメイン不変条件）
- UI上でフッターやAI分析結果の近くに簡潔な免責文を配置する
- AI出力の表現は「断定」ではなく「示唆」にする（プロンプト設計で制御）

## 株価データの取得・表示

- データソース: Twelve Data（株価）+ Finnhub（企業基本情報）
- 更新頻度: サーバーサイドキャッシュで1日1回更新（リアルタイム不要）
- 対象: 米国株のみでスタート。日本株は将来有料API導入時に対応
- 記事一覧（簡易）: ティッカー + 終値（例: `NVDA $124.50`）
- 企業ページ（詳細）: 基本情報 + 終値・変動率 + 簡易チャート + 関連記事一覧
- 変動率は closePrice から計算で導出する
- 「Powered by Twelve Data」のアトリビューション表示が必須

## トレンドレポートの生成

- 金曜夜〜土曜朝に自動生成（バッチ処理）
- 過去分も閲覧可能（アーカイブ）
- レポート構造: サマリ → 注目トピック → 客観データ → 企業別の動き → 前週からの変化 → 関連記事一覧

## UI上の導線

- カテゴリタップ → 記事一覧フィルタ + トレンドレポート直リンク
- 企業タグクリック → 企業ページへ遷移（段階B）
- 段階Cでは企業タグ横にティッカー + 終値の簡易表示
- 暫定タグはユーザーに控えめなデザイン（薄い色、点線ボーダー等）で表示

## 検討事項

- importanceLevel（NewsSource）の具体的な値域（3段階 / 5段階等）はAI分析パイプライン設計時に決定する（旧 `impactLevel`（ArticleAnalysis）は 2026-04 に完全廃止）
