## 候補

### 候補1: Keyword の実態化

**概要**: AI が抽出したキーワードの管理を整備する。状態遷移
(PROVISIONAL → OFFICIAL → BLACKLISTED) を実装し、フィルタリングの
精度を上げる。

**Domain_inventory.md との関連**: 「カテゴリ・タグフィルター」
(分析の深さを支える手段)

**やること**:
- Keyword の状態遷移ロジック (repositories/keyword.py, services/keyword.py を新設)
- AI 分析時のキーワード登録フローを状態遷移と統合
- 管理画面でのキーワード承認/拒否 UI

**利点**:
- 既存パイプラインの延長線上。新しい外部 API や大きなアーキテクチャ変更が不要
- 候補2・3 の土台になる (トレンド分析も企業タグ付けもキーワード管理が基盤)
- フィルタリング精度向上でユーザー体験に直結

**規模**: 中

### 候補2: AI トレンド分析 (第2層)

**概要**: 複数記事を横断して「今週のトレンド」「急上昇トピック」を出す。
個別記事を読まなくても全体像が掴める体験。

**Domain_inventory.md との関連**: 提供する価値の第1位「分析の深さ」の核心。
> 「先端テクノロジー業界の動向を深く分析し、投資判断を支援する
> インテリジェンスツール」
> これがアプリのアイデンティティ。

**やること**:
- トレンド分析のドメインモデル設計 (分析対象期間、セクター、要約、根拠記事群)
- バッチ処理の設計 (頻度、コスト見積もり)
- Gemini API のプロンプト設計
- フロントエンドのトレンド表示 UI

**利点**:
- Vector の差別化要素。「ニュース翻訳アプリ」から「インテリジェンスツール」への格上げ
- 個別記事分析パイプラインが安定稼働しているので、その上に乗せる形

**規模**: 大

### 候補3: 企業名タグ付け (レベルA)

**概要**: 記事に登場する企業を AI で自動タグ付けし、企業単位で記事を
フィルタできるようにする。

**Domain_inventory.md との関連**: 提供する価値の第3位「情報集約」の第1段階。
レベル A → B → C の段階的実装の起点。

**やること**:
- AI 分析プロンプトに企業名抽出を追加
- 企業タグの保存 (文字列タグ or 新テーブル)
- フロントエンドのフィルタ UI に企業名を追加

**利点**:
- 投資分析ダッシュボードとしての差別化
- 将来の企業マスタ (レベルB)、株価表示 (レベルC) の土台

**規模**: 中

## 推奨する優先順位

```
候補1 (Keyword)  →  候補3 (企業タグ)  →  候補2 (トレンド分析)
```

**理由**:

1. **候補1 が土台**: Keyword 管理が整わないと、候補2・3 でノイズが増える
2. **候補3 は候補2 の前提**: トレンド分析で「NVIDIA に関するニュースの流れ」を
   出すには、まず企業名タグが必要 (Domain_inventory.md 3.3 節参照)
3. **候補2 が最大の価値だが最大の規模**: 土台を整えてから取り組む方が
   設計判断が確実になる

ただしこれは技術的な推奨であり、**ビジネス価値の観点では候補2 が最も重要**。
Domain_inventory.md は「分析の深さ」を第1の価値と定義しており、
トレンド分析はその核心。「土台を整えてから」と「価値の中心から作る」の
どちらを取るかは、Yk さんの判断による。
