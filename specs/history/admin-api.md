# 管理 API・認可境界の整理 — 思考の推移

> 散在していた複数の設計メモを束ねた歴史記録。現行仕様ではなく、「どう考えが推移したか」を
> 辿るための記録。各節は当時の判断と、なぜそう動いたかを過去形で残す。

## 出発点の問題 — 一覧エンドポイントの用途共有 (2026-04 初旬)

ソース一覧を返すエンドポイントが、「管理 UI のソース一覧」と「記事一覧画面のフィルタ用
ドロップダウン」という性質の異なる2用途で共有されていた。前者は管理者限定であるべきなのに
一般ユーザーでも叩け、スクレイピング先 URL のような管理者しか要らない情報まで返していた
(読み取りだけ認可の穴が空いていた)。後者はソース名しか要らないのに過剰に取っていた。
これを分離し、管理用は管理者限定に締めた。

## 転換 — 管理系を集約し、認可をまとまり単位にする (2026-04 中旬)

個別の役割分離を一般化し、管理者専用のエンドポイントを1つのまとまりに集約、URL を共通
プレフィックス下に統一した。認可チェックを各エンドポイントに個別に書くのをやめ、まとまり
単位で一括してかけることで書き忘れを防ぐ。狙いは認可コードの書き忘れリスク解消と、将来
ネットワーク層(WAF 等)で管理系 API をパスでまとめて識別・制御できる構造の確保。

## 起きた移管(どこからどこへ動いたか)

- 認可が「各エンドポイントに個別」から「まとまり単位で一括」へ移った。
- 管理 API が散在から、共通プレフィックス下への集約へ移った。
