# AI プロバイダ/モデル構成の変遷 — 思考の推移

> 散在していた複数の設計メモを、主題ごとに一本の時系列ナラティブへ束ねた歴史記録。
> 現行仕様ではなく、「どう考えが推移したか」を辿るための記録。各節は当時の判断と、なぜ
> そう動いたかを過去形で残す。

## 出発点 — 埋め込みは分析の産物として置く (2026-03, DB 再設計)

DB 再設計の際、埋め込みベクトルを「記事そのものの事実」ではなく分析の産物と位置づけ、生記事
ではなく分析済み記事の側に置いた。AI が生成する成果物の置き場を、最初に整理した格好。

## 転換1 — レート制限の設定漏れを構造で防ぐ (2026-04 中旬)

AI クライアント層は、同種のリトライ処理が複数メソッドに重複し、レート上限などプロバイダ固有
の知識が散在していて、設定を書き忘れると本番でレート超過が起きうる構造だった。これを、モデルの
差し替え=クラスの差し替えとし、レート上限をそのクラス定義の一部として必須化(無ければ起動前に
検出されて動かない)する形へ整える、と決めた。レート制限の事情がクライアント層より上に漏れない
こと、動作環境を変えても上限が守られることは前提として維持した。

## 転換2 — 埋め込みをローカルへ移し、また戻す (2026-04 中旬)

埋め込みモデルを API から、商用利用可能な日本語向けのローカルモデルへ一括で切り替えることに
した。狙いは API のリクエスト上限と課金の解消、検索クエリの低レイテンシ化。推論は API 呼び出し
からローカル推論サーバーへ置き換える。AI 分析(翻訳・分類)は API のまま据え置き。

このローカル移行は一度実施されたが、後に API の埋め込みへ差し戻した。2つのモデルは次元数こそ
同じでも意味空間が異なり、既存ベクトルと新ベクトルを混在させられないため、移行で生成した
ローカル製ベクトルは破棄して作り直した。上限・課金を避けたい動機は残ったが、ローカル推論へ
切り替える形は残らなかった。なお、このローカル推論サーバーを開発機(Apple Silicon)と本番
(arm64)で同じ構成に揃えるコンテナ整備も併せて計画されていたが、API 埋め込みへ戻したことで
その整備自体が不要になった。

## 転換3 — 分類を別プロバイダへ、切替方式も変える (2026-04 下旬)

記事分類の処理を、コスト削減と方針整合のため別プロバイダのモデルへ乗り換えることにした。送信
するのは翻訳済みの短いテキストのみ、出力は構造を強制、入力長と最大トークンに上限を設ける。
翻訳・要約の工程は据え置き、移行範囲は分類のみ。

途中、当初想定した環境変数によるプロバイダ切替が「分類と翻訳が同じ設定を共有していて、切り替え
ると翻訳側が落ちる」構造的な不具合を生むと判明した。そこで環境変数切替を完全に廃し、起動時の
組み立て箇所(合成ルート)でプロバイダをコードとして固定する方式へ転換した。切替は設定変更では
なく、コード変更とワーカー再起動で行う。

## 起きた移管(どこからどこへ動いたか)

- 埋め込み生成が API → ローカル → API へ往復した。
- 記事分類のプロバイダが切り替わった。
- プロバイダ選択の仕組みが「環境変数での切替」から「合成ルートでのコード固定」へ移った。
- 「プロバイダ固有の事情(レート制限など)を上位に漏らさず、構造で守る」という考えは通底
  していた。
