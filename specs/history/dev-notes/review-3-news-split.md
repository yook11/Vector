# 開発メモ圧縮記録 — レビュー編 3: news の分割とリソース命名 (4/3)

個人開発メモ (Claude.ai 対話ログ、リポジトリ外) の圧縮記録。原本のコードやコマンドは転記せず、
当時考えていたこと・決めたこと・その後の帰結だけを言語化して残す。
層別縦断レビュー (レビュー編、全 6 冊) の第 3 冊。[レビュー編 2](review-2-routers.md) (4/2-4/3) の直後にあたり、
[レビュー編 4](review-4-params.md) (4/4) の ArticleListParams 検討はこの日に作った Depends() 版 Params クラスから始まる。

## 全体像

UI から逆算して news ルーターを見た日。「ユーザー向け read と admin パイプライン操作が混在している」という
診断から、リソース命名の根本——**ユーザーが見ているのは news ではなく分析結果**——に降り、
router / repository / service / schemas の 4 層一括分割をその日のうちに実行した。
あわせて「Claude Code に渡す前に自分で grep で確認する」という事前確認の習慣を自分の弱点として
言語化・習得した日でもある。

## テーマ 1: アクターの混在という診断 (4/3)

- 自分の要約が診断の核: 「ユーザーが使用することと、パイプラインの責務が混在していることが問題」。
  read 3 本 (任意認証・404/503 系) と admin RPC 2 本 (副作用あり・内部処理系) は、
  アクターも認証要件もエラーの性質も違う。
- 個別に挙がっていた問題 (sort 系が任意文字列、クエリパラメータ 9 個の手動詰め替え、
  body 省略=全件という暗黙挙動) は、混在のせいで「どちらの文脈としても設計を詰めきれていない」ことの
  表れだ、と一段抽象して整理した。

## テーマ 2: リソース命名 — ユーザーは news を見ていない (4/3)

- 起点は「分析をするときに news と URL に入っているのは違うんじゃないか」という違和感。
  ユーザーが見ているのは生のニュースではなく AI 翻訳・影響度分析=**分析結果**だから `/articles`、
  admin が操作しているのは「ニュースという資源」ではなく**パイプラインの段階の起動**だから `/pipeline`。
  「news はどちらの文脈にも合っていなかった。中途半端に両方を news でまとめていたことが混在の根本原因」。
- パス構造は flat 案を選択 (段階別のネストは別リソースのパイプラインが実際に現れてから)。
  「実際に必要になるまで先回りしない」の適用。
- AI の「同じテーブルを参照するからリポジトリ分割は慎重に」という指摘を、
  「同じテーブルを触るかどうかは実装の詳細。ドメインの関心が違う以上分離する」と自分で押し切った。

## テーマ 3: 4 層一括分割とリネームの規律 (4/3)

- 段階実行ではなく一括を選んだ: リポジトリだけ先に分けると「ArticleRepository を NewsService が使う」
  という不整合な中間状態が生まれ、import 変更とテスト修正が二度手間になる。
  機械的なリネームと移動で設計判断を伴わないから、まとめて一発+テストで確認が一番安全。
- プラン確認のたびにリネーム漏れ (eager options 関数・Params 名・メソッド名の news 残り) が見つかり、
  「Claude Code に渡す前に、旧名がどこに残っているか自分で grep で一括確認する」を弱点への処方として習得。
  DB モデルの NewsSource は「ニュースソース」の意でリネーム対象外、という判断も grep 結果を見て行う。
- Claude Code の誤提案 (パラメータ定義を別関数に移しただけで二重定義が解消されていない) を
  自分で検証してリジェクトし、「Params クラス自体に Query 定義を持たせる」方向へ修正させた。

## テーマ 4: watchlist 責務の混入 — 認識して先送り (4/3)

- ArticleRepository 内の watched-ids 取得が watchlist のテーブルを参照している責務漏れを自分で発見。
  「記事レスポンスにウォッチ済みフラグを付ける」という利用側の都合で置かれているだけで、
  本来は WatchlistRepository に属すべきと診断した。
- ただし「今のスコープは記事 vs パイプラインの分離。混ぜると変更の意図がぼやける」と先送りを即決。

## テーマ 5: 進め方のメタレビュー (4/3)

- 「縦で見るようになって全体の流れが掴め、おかしいな? という部分がだんだんわかるようになった」という
  自己観察を、対話履歴ごと AI に検証させた。返ってきた評価は、良い兆候 3 つ
  (違和感→なぜ→分離という思考の流れ / AI 提案を検証してリジェクトできる / 今やるか後でやるかの
  スコープ判断の一貫性) と、弱点 1 つ (事前の自己確認=grep の癖がない)。
- 弱点の指摘を「確かに、事前に自分で確認する癖がない」と受け入れ、その場で grep の使い方を習得した。

## 帰結

- **分割は当日 2 コミットで完了した** (午後に 3 層分割→26 分後に schemas 分割+フロント型再生成)。
  以後 News* の残骸を直す修正コミットはゼロで、「渡す前に grep」の規律は初回から機能している。
  NewsSource は意図どおり DB モデルとして据え置かれ現存。`/api/v1/articles` と `/pipeline` の分離は
  現行も維持されている (pipeline はその後 admin 配下へ 2 回移設、embed は red-team 対応で削除——
  [レビュー編 5](review-5-sweep.md) 帰結)。
- PaginatedArticleResponse だけは Response サフィックスのまま現存しており、4/1 に決めた
  サフィックス廃止規約 ([レビュー編 1](review-1-schema.md)) の例外として残っている
  (後のページネーション再設計でも名前は維持)。
- **先送りした watchlist 責務漏れは 3 段階で解消された**: 4/8 に WatchlistRepository へ移動
  (「あるべき場所へ」)、4/10 に表示中記事に限定する watched_among へ置換され空集合の事前条件が
  docstring に明文化 ([structural-guarantees.md](structural-guarantees.md) の「契約による設計」の素材そのもの)、
  そして 4/29 に watchlist メンバーシップが独立リソース化されて watched_among ごと消滅、is_watched フラグも
  API から削除された。現行はフロントが watched ids を独立取得して表示時に合成する。
  責務漏れの最終解は「メソッドの移動」ではなく**リソース境界の再設計**だった——
  メソッドをどの repository に置くかを争っていた問題が、契約の引き直しで問題ごと消えた形。

## 通底していた思考

1. **アクターで切る。** ユーザーと admin の関心の違いが、命名・認証・エラー設計・分割のすべての軸になった。
2. **実装の詳細でドメインの分離を妥協しない。** 「同じテーブルだから」はファイルを分けない理由にならない。
3. **認識と着手を分離する。** watchlist 漏れは発見を記録して先送り——レビュー編 2 のトリアージと同じ規律。
4. **自分の進め方自体をレビュー対象にする。** メタレビューで弱点 (事前 grep) の指摘を受け、
   その場で習慣に変換した。
