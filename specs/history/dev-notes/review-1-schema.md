# 開発メモ圧縮記録 — レビュー編 1: スキーマレビュー (4/1)

個人開発メモ (Claude.ai 対話ログ、リポジトリ外) の圧縮記録。原本のコードやコマンドは転記せず、
当時考えていたこと・決めたこと・その後の帰結だけを言語化して残す。
層別縦断レビュー (レビュー編、全 6 冊) の第 1 冊。全体の地図は [README.md](README.md)。

## 全体像

スキーマ層の縦断レビューを開始した日。出発点の気づきが重要だった:
エンドポイントを眺めながら AI と壁打ちしても解決策が浮かばない。「命名はシンプルに」「前置詞は危険信号」という
知識はあっても活かせない。**結局ユーザーは何を見たいのか、UI で何が使われているのか、に立ち返らないと答えが出ない**。
この「使う側から考える」が、以後のレビュー編全体 (ルーター・リポジトリ・リソース命名) の通底軸になる。

## テーマ 1: ドメインの主従をスキーマ構造で表現する (4/1)

- 主従逆転の発見: アプリの中核は「AI 分析済みニュース」なのに、スキーマはトップレベル=原文情報、
  分析は `analysis: Optional` でぶら下がる構造だった。ストレージのテーブル親子関係が API 表面に漏れていた。
  → 分析フィールドをトップレベルへ昇格、原文情報は `original` embed としてグループ化。
  **「ドメインの主従をスキーマ構造で表現する」「ストレージ構造を API に漏らさない」** という原則として言語化。
- 一覧と詳細で同一レスポンス型を共有し、カードで使わない本文・推論テキストを毎回返していた (over-fetch)。
  → アクセスパターンが明確に異なるため一覧用 / 詳細用 (当時の名は `NewsBrief` / `NewsDetail`) を
  **継承ではなく独立した 2 つの型** に分離。
  継承は「カードが本体で詳細はその拡張」という概念の主従逆転になる、Optional だらけの型は
  「未分析なのか・この endpoint では返さないのか」を読み取れなくなる、が根拠。
- 「一覧は分析済み記事のみ返す (INNER JOIN)」を決めたことで、分析フィールドが required になり
  フォールバック分岐ごと消えた。**ドメインの前提を絞ると型から Optional が消える** という体験。
- 分割の基準も確立: スキーマ分割は「endpoint が異なるデータ形状を返すとき」。
  フロントのコンポーネントごとにフィールドの使用量が違うのは TypeScript の Pick で吸収する領域で、
  「使用箇所の数」で分割しない。

## テーマ 2: 出力スキーマの命名規約 (4/1)

- 命名規約を確定: `Response` サフィックス廃止、`Detail` / `DetailList` / `Embed` (+入力は `Create` / `Update`)。
  Pydantic クラス名がそのまま TypeScript 型名になるため、サフィックス自体が出力であることを示す短い体系を選んだ。
- `KeywordWithCountEmbed` の `With` を「設計の譲歩のパターン」と認識。構造 (何が入っているか) ではなく
  用途 (何に使うか) で命名する——`KeywordStatEmbed` へ。組み合わせ爆発を名前で防ぐ。
- ただし「用途=使用画面」で命名するのも誤り (`CategorySidebar` 用、と名付けると別画面で使えなくなる)。
  **名前が示すのはデータの展開度合いであって、使用場所ではない**。

## 帰結

- このレビューが層別縦断レビューの起点になり、翌日からルーター層へ降りた ([レビュー編 2](review-2-routers.md))。
  「使う側から考える」はルーター・リポジトリ命名・リソース設計まで一貫して効いた。
- 一覧/詳細の独立 2 型は news→articles リネーム ([レビュー編 3](review-3-news-split.md)) を経て
  現行も Brief / Detail の独立型のまま維持されている。`Detail` 体系 (id なしの CategoryDetail 等) も現存。
- 命名規約の例外が 1 つ残っている: PaginatedArticleResponse は Response サフィックスのまま現存
  (レビュー編 3 帰結。後のページネーション再設計でも名前は維持)。
- 「名前と実体の一致」はこの後、テーブル設計 (4/22 article_analyses、[pipeline-failures.md](pipeline-failures.md))
  や結果型 (ArticleReady) まで広がる、月をまたぐ判断軸になった。
