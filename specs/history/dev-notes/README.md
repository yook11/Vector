# 開発メモ圧縮記録 (dev-notes)

個人開発メモ (Claude.ai / Claude Code 対話ログ + ローカルノート、リポジトリ外) の圧縮記録。
原本のコードやコマンドは転記せず、**当時考えていたこと・決めたこと・その後の帰結**だけを言語化して残す。
帰結は現行コード・git 履歴と突き合わせて検証した事実のみを書く。

## 構成 — 月ではなく「考えの変化の筋」で区切る

当初は月次 (2026-03 / 04 / 05) で記録していたが、考えの変化が月をまたいで一本に繋がっていたため、
系列 (変化の筋) ごとのファイルに再編した。各系列は時系列で読め、月は本文中の日付として残る。
月単位の読み口はこの README の「月ごとの変遷」が引き継ぐ。
帰結 (その後どうなったか) の追記先は各系列ファイルのみとし、README には内容を複製しない。

## 月ごとの変遷

### 2026-03 — 責務の所在を一つずつ確定させる

「AI に作らせたものをそのまま使う」段階から「自分が理解して統制する」段階への転換月。
CLAUDE.md vs skills、models vs Alembic、DB vs Redis、VO vs schema vs CHECK 制約、ORM vs トリガー——
ほぼすべてのメモが「この責務は誰が持つべきか」という同型の問いに収束する。
進め方は一貫して **「壁打ちで理解を検証 → 一次情報で裏取り → 判断基準を言語化して記録」**
(「〜という理解で合ってる?」の往復と、ソースコード・公式ドキュメント・deep research による裏付けをセットで行う)。
原本: 約 30 本。主な行き先: structural-guarantees (VO・多層防御) / pipeline-failures (状態の所有権) /
ai-collaboration (CLAUDE.md 精査・提案検査) / misc (DB メンタルモデル・Git)。

### 2026-04 — 価値命題への逆算

判断基準そのものが「このアプリケーションの価値は何か」への逆算に収束した月。
「ユーザー (日本人投資家) は何を見たいのか」が、スキーマの形・機能の存廃・型の保証範囲を決めるようになった。
もう一つの軸は「名前と実体の一致」と「型で段階の前提条件を保証する」——3 月の「値の不変条件」が
「段階・境界の前提条件」へ拡大した。**判断の前に必ず観測を挟む型 (実データ・実挙動で決める)** もこの月に固まった
(空 in_() の SQL 展開、topic 分布の SQL、published_at の NULL 率、再分析のコスト見積もり)。
原本: 約 23 本 + ルーターレビュー系。主な行き先: レビュー編 1-6 / product-value /
structural-guarantees (型で前提条件) / pipeline-failures (テーブル分離・結果型)。

### 2026-05 — 自分が設計する

「はじめて自分の意思で設計をしたのかもしれない」(5/9)、「監査基盤を初めて自分で設計を始められた、
AI ではなく自分で」(5/11) という自覚が言語化された月。4 月の価値命題逆算が失敗系へ拡大し
(「後段に何を保証するか」「監査の読者に何を残すか」)、判断の道具として
**「処理方針と原因の 2 軸分離」「事実と副産物の区別」「シンプルに書けないなら設計が歪んでいる」** の 3 点が加わった。
混ざって見える問題を 2 軸に分けると設計が立つ、という手筋の獲得。
違和感を感覚で終わらせず、依存方向や責務の言葉で診断してから直す。
原本: 9 本 (日付付き 7 + 日付なし 2)。主な行き先: pipeline-failures (取得再設計・分類学・監査・テスト) /
structural-guarantees (AI 境界) / ai-collaboration (主導権の自覚) / misc (運用)。

## 系列を跨ぐ筋

- **名前と実体の一致**: 名前が宣言する内容と実体のズレを設計問題のシグナルとして扱う判断軸。
  スキーマ命名 (レビュー編 1) → リポジトリ命名は呼び出し側の意図 (レビュー編 2 テーマ 7) →
  リソース命名「ユーザーは news を見ていない」(レビュー編 3) → テーブル名 article_analyses (pipeline-failures、4/22) →
  結果型 ArticleReady (同、4/23)。死んだのは個別の規約 (get_/fetch_ 等)、生きたのはその規約を導いた考え方。
- **構造で守る**: 本籍は structural-guarantees。適用例が silent-drop 規律 (レビュー編 4) と
  Unit of Work (レビュー編 6) に埋まっている。
- **認識と着手を分離する**: トリアージ (レビュー編 2 テーマ 5) → 先送りの即決 (レビュー編 3 テーマ 4) →
  スコープ規律の完成形 (レビュー編 6 テーマ 5)。
- **消費側起点の逆算**: スキーマレビューの「結局ユーザーは何を見たいのか」(レビュー編 1) →
  価値命題による取捨選択 (product-value) → 「後段に何を保証するか・監査の読者に何を残すか」(pipeline-failures)。
