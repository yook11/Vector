# 開発メモ圧縮記録 — AI 協働と設計主導権 (3月-5月)

個人開発メモ (Claude.ai / Claude Code 対話ログ + ローカルノート、リポジトリ外) の圧縮記録のうち、
AI との協働の仕方と設計の主導権がどう変わったかを 3 月から 5 月まで一本で追う系列。
原本のコードやコマンドは転記せず、当時考えていたこと・決めたこと・その後の帰結だけを言語化して残す。
全体の地図は [README.md](README.md)。

## 全体像

弧は **「AI に作らせたものをそのまま使う」(3月の自己認識) → 提案を論拠で評価する姿勢の定着 (4月) →
「はじめて自分の意思で設計をしたのかもしれない」(5/9) という主導権の自覚**。
3 月は統制の道具 (CLAUDE.md / skills / 進め方の分割) を作った月、4 月はどの AI の提案も結論ではなく
論拠で検査するようになった月、5 月は設計の主語が自分に移ったことが言語化された月。
docs/ddd-adoption/ の主旋律 (承認するだけ→違和感→問いを握る→道具を統制→規律化) の一次資料群にあたる。

## 1. AI 協働の規律確立 (3/26, 3/28-31)

AI 生成のまま使っていた CLAUDE.md を、リファクタリング経験を踏まえて自分の知見で精査し直した。

- 取捨選択の基準を「消したら Claude が間違えるか」に置いた。公式ドキュメント URL テーブル、コーディング規約の大半、
  linter が検出できるルール、既存コードパターンから学習できるものを削除。
  残したのはバージョン違い防止 (Pydantic v1 / SQLAlchemy 1.x 禁止)・セキュリティ (SQL 文字列結合禁止)・SSoT 宣言・検証コマンド (`uv run` 明記)。
- ドキュメント陳腐化への対策として「コードを正とする (prefer pointers to copies)」を採用し、
  migration / model の手続き知識は skills に分離 (progressive disclosure)。
  一方で **DB 設計の判断基準は意図的にスキル化しなかった**。「プラン作成時に自分が壁打ちで判断する領域」であり、
  AI に自動適用させる知識ではないという線引き。
- AI への指示は「調査 → 判断 → 実装」の 3 分割が効くという運用知見 (3/29)。
  調査結果を見て自分が判断を挟むことで、コンテキスト汚染と AI の勝手な判断の両方を防ぐ。
- AI の失敗パターンを言語化した: 軽い相槌から生まれる YAGNI 違反の抽象化 (3/28)、
  フロントで使わないデータまで返す over-fetch なレスポンススキーマ (3/30)、
  目的を持たない構造分解 (3/31「分解の粒度はユーザーの目的から逆算する」)。
  「AI は構造的に正しい分解はできるが、なぜその部分が存在するかという目的を持たない」という認識にこの時点で到達している。
- 提案を意図とのズレで検査する場面もこの月から繰り返し出てくる: VO 廃止と誤読された件
  ([structural-guarantees.md](structural-guarantees.md))、`is_last_attempt` の必要性を問うた件、
  dedup の本質 (重複か関連か) を問い直した件 ([pipeline-failures.md](pipeline-failures.md))。
  提案の構造ではなく目的との整合で判断する。
- Better Auth で手間取った失敗を「プラン作成前の公式ドキュメント調査不足」と特定したこと ([misc.md](misc.md)) が、
  以後のリサーチ義務化の起点。「理解は対話で検証し、採用は一次情報で裏取りする」が進め方として固まった。

帰結: research スキル・check スキル・スキル分離構成はこのときの方針がそのまま現行運用になった。
CLAUDE.md は約 80 行 → 約 30 行に縮小し、以後も「全セッション展開に値するか」で増減を判断している。

## 2. 提案を論拠で評価する (4月)

- 別 AI (Gemini / 別セッションの Claude) の提案を結論ではなく論拠で評価する場面が定着した:
  snapshot 論を「論拠が誤っている (versioning なしでは成立しない)」と批評した上で自分は別の根拠から
  同じ結論に到達した件 (設計内容は [pipeline-failures.md](pipeline-failures.md) が正本)、
  リアルタイム翻訳 API 案の棄却、ファセット vs リファレンスデータの概念区別。
- レビュー編でも同じ姿勢が機能している: keywords ルーターの保持勧告を「出来が良くないから残すのに違和感がある」と
  覆した件 ([レビュー編 2](review-2-routers.md) テーマ 5)、Claude Code の誤提案 (二重定義が解消されていない) を
  検証してリジェクトした件 ([レビュー編 3](review-3-news-split.md) テーマ 3)、DI パターンの誤った例を
  問い詰めて正させた件 ([レビュー編 4](review-4-params.md) テーマ 4)。
- 自分の進め方自体を AI のレビュー対象にする使い方も始まった ([レビュー編 3](review-3-news-split.md) テーマ 5)。
  メタレビューで弱点 (事前の自己確認=grep の癖がない) の指摘を受け入れ、その場で習慣に変換した。

## 3. 設計の主導権 (5月)

- 「はじめて自分の意思で設計をしたのかもしれない」(5/9)、「監査基盤を初めて自分で設計を始められた、
  AI ではなく自分で」(5/11) というメモタイトルがそのまま証言になっている。
  対象は取得パイプライン再設計とエラー分類学・監査設計 ([pipeline-failures.md](pipeline-failures.md))。
- **設計の主導権は自分、AI は批評・裏取り・命名の相手**、という分担が日常の進め方として定着した。
  AI の提案も既存の分類学との整合を根拠に修正させており、4 月の「論拠で評価する」姿勢の延長にある。
- Claude Code アーキテクチャ調査 (日付なし): CLAUDE.md (静的・常時読込なので最小に) と
  Skills (動的・progressive disclosure) の役割分担、hooks・コンテキストの階層管理。
  3 月の CLAUDE.md 精査路線の続きにあたるリサーチ資料。

## 通底していた思考

1. **提案の構造ではなく、目的との整合で判断する。** 3 月の「意図とのズレで検査」から 4 月の「論拠で評価」まで、
   検査の対象が深くなっただけで同じ問い。
2. **統制は道具と姿勢の両輪。** CLAUDE.md / skills / 調査→判断→実装の 3 分割という道具の整備と、
   論拠を問い詰める姿勢の訓練が並行して進んだ。
3. **弧の到達点は「AI を使わない」ではない。** 設計の主導権を自分が握った上で、批評・裏取り・命名の
   相手として AI を使い倒す、という分担に落ち着いた。
