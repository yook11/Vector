# 開発メモ圧縮記録 — レビュー編 4: パラメータ境界と依存性注入 (4/4)

個人開発メモ (Claude.ai 対話ログ、リポジトリ外) 2 本の圧縮記録。原本のコードやコマンドは転記せず、
当時考えていたこと・決めたこと・その後の帰結だけを言語化して残す。
層別縦断レビュー (レビュー編、全 6 冊) の第 4 冊。[レビュー編 2](review-2-routers.md) (4/2-4/3) の続きで、
検討対象の Params クラスは [レビュー編 3](review-3-news-split.md) (4/3) で作ったもの。

## 全体像

ルーターレビューが最初の本丸である articles 一覧エンドポイントに入った日。
発見したのは「3 責務が混在した Params クラスがリポジトリ層に居る」という多重の歪みで、
これを自分の責務言語化 → 外部リサーチ (FastAPI DI のベストプラクティス) → 既存規律との突き合わせ、の三段で解きほぐし、
「HTTP の形」と「ドメインの形」を型として分離する設計に到達した。
このレビューを通じて「リクエストが何を言っているか」と「リクエストを処理するために何が必要か」という
依存性注入の二分法を自分の言葉で獲得している。設計自体は翌朝さらに進化して畳まれる (帰結参照)。

## テーマ 1: `q` の仲間外れと、リポジトリ層に居る Params (4/4)

- エンドポイントのシグネチャで検索語 `q` だけが Params の外に居る違和感から開始。
  「後から機能を足すとき、既存の型を触らずに引数を追加した」パターンの痕跡と診断し、統合を決めた。
- より根本の発見: `ArticleListParams` がリポジトリ層のファイルに定義されていた。Query や alias という
  HTTP の語彙が DB 層まで貫通している。「リポジトリが知っていいのは検索条件であって、
  クエリパラメータが camelCase で来ることではない」。
- パラメータ定義の場所の一般則も整理: body はスキーマ、path はルーター、query は規模と制約次第
  (複数 + クロスフィールド制約ならモデル化)。

## テーマ 2: 3 責務の混在を自分の言葉で切る (4/4)

- 自分の言語化が起点: 「このクラスは受け取るリクエストの形式を定義しているだけなのに、エラーを吐くようになっている。
  この部分でやることではない」。受け取りの定義 / VO 変換 / HTTP 例外送出、の 3 責務を分離する方針が立った。
- 既存規律「Router で str→VO 早期変換、Service 以降は VO」との矛盾に見えた点を整理し直した:
  方針変更ではなく、**変換の実行場所が Params の `__init__` に埋まっていたことが規律違反**であり、今回はその修正。
  どの層が変換を担うかは変わらない。
- 生の str だった sort 系パラメータが SQL の ORDER BY に届く問題も Enum 化で塞ぐと決めた
  (タイポを型チェッカーが検出する)。

## テーマ 3: HTTP の形とドメインの形 — 二型分離 (4/4)

- `ArticleListParams` (raw 値、HTTP 語彙を持つ) と `ArticleListQuery` (VO 解決済みの frozen dataclass) に分離し、
  境界の変換関数をルーターに置く設計に到達。自分の言語化: 「送られてくるリクエストの形式と、
  実際に処理に進むときの正しい形式を定義する。これでエラーハンドリングが散らばらない」。
  AI の補足で「HTTP への依存も散らばらない」(層の分離) が表裏一体であることも確認した。
- 下流の立場の言語化も自分で行った: サービスとリポジトリは「どんな入力が来たのかも、
  それを変換する作業も知る必要がない」。検証済みの型を受け取った時点で信頼できる。
- 変換ヘルパーの純度も確定: HTTP を知らず ValueError を素通しさせ、422 への変換はルーターに統一。
  途中で合意とドリフトした実装 (ヘルパー内での HTTP 例外送出) を照合で検出して直した。
- 解決済み型の置き場所は「今はスキーマ層に同居 (命名サフィックスで区別)、同種が 3 つ以上になったら
  専用の置き場へ昇格」という段階判断。DTO という概念名もここで得た。

## テーマ 4: 依存性注入の 3 パターン — リサーチとの突き合わせ (4/4)

- リサーチで FastAPI v0.115.0+ の BaseModel + Query() パターンを得て、プランを Plain class + Depends() から切り替えた。
- パターン②の「他の依存性を受け取れる」という利点が腑に落ちず問い詰めた結果、AI の例の誤りを正させた:
  境界は「パラメータをまとめるか、オブジェクトを構築するか」。自分の整理は
  **①=リクエストが何を言っているか / ②=リクエストを処理するために何が必要か** (誰が・どのセッションで・どのサービスで)。
- `_CamelBase` を継承しない判断: alias_generator はクエリパラメータの名前解決には効かない別機構であり、
  そもそも Params はレスポンスに使わないから「**変換する必要がそもそもない**」。
  動くが意図の読めない設定を引き継がない。
- リサーチの使い方の規律も言語化: anti-patterns の一覧は自分のコードベースへ照合して
  初めて行動指針になる。「読んだ→わかった」で終わらせない。

## テーマ 5: 例外変換の散在とスコープの切り出し (4/4)

- EmbeddingError の try/except がエンドポイントの数だけ複製される問題に、グローバル例外ハンドラ
  (ドメイン例外→HTTP 変換をアプリで一箇所に登録) という解を得た。今回のリファクタとはスコープが違うとして別タスク化。
- FastAPI のバージョン更新も「今やると変更が混ざる」として別タスクに切った。
- レビューの進め方も更新: 1 ルーターずつ直すのではなく、全ルーターの問題を先に洗い出して
  共通の問題をまとめて修正する方針へ。

## 帰結

この日の設計は短命だったが、守ろうとしたものはすべて生き残った。

- **二型分離は翌朝 16 分間の 3 連コミットで自分で畳んだ** (4/5)。きっかけは Depends(Model) の
  サイレント破損の発見——VO フィールドが OpenAPI から黙って落ち、常に None になる。
  Annotated[Model, Query()] へ切り替えると Pydantic が parse 時に VO を直接検証して標準 422 を返すため、
  変換ヘルパーも try/except も不要になり、「domain purity のための別 dataclass」という正当化は
  単一エンドポイント用途ではもう成立しない、とコミットメッセージに明記して一型へ統合した。
  silent-drop の知見は PaginationParams の docstring に footgun 警告として恒久化され、
  後の「Annotated[Model, Query()] なら VO 可 / Depends(Model) は禁止」という規律の起点になった。
  型の数は減ったが、「HTTP の形とドメインの形の区別」「変換と検証を一箇所に」は
  VO 直接フィールド + parse 時検証という より単純な形で実現されている。
- 4/8 にはその PaginationParams 自体にも同型の発見が残った: 「HTTP 入力パラメータ」と「ドメイン計算 VO」の
  2 役を 1 つの名前で兼ねているのが違和感の正体。完全分離は現規模では過剰、と判断保留にして言語化だけ残し、
  `Paginated[T]` のジェネリクス化も「2 種類目が出てから」の別タスクへ切った。
- **グローバル例外ハンドラは「別タスク」と言ったその日のうちに実装された** (4/4 朝)。
  当初は NotFound / Duplicate / EmbeddingError(503) の 3 本で、現行は NotFound / Duplicate / InvalidQuery +
  detail の allowlist 化に進化している。
- **`q` は仲間外れ解消の後、機能ごと消えた**。4/6 に検索専用の Params へ分離され、5/28 に任意クエリの
  embedding 検索自体が撤去された (約 2,400 行削除、/similar は残置)。「q の仲間外れ」への違和感は
  最終的にソースフィルタと同じ「機能ごと削る」([product-value.md](product-value.md)) で解消された形。
  embedding は analysis BC の記事ベクトル化だけが残る。
- **FastAPI 更新は約 3 週間後に dependabot 経由で実行された** (0.115 系→0.136 系、以後追従)。
- sort 系の Enum 化 (SortOrder) は現存。extra="forbid" は採用されていない (デフォルトの ignore のまま)。

## 通底していた思考

1. **責務は自分の一文で切る。** 「受け取る形式を定義しているだけなのに、エラーを吐くようになっている」——
   違和感を責務の言葉に直してから設計に入る。5/11 の「シンプルに書けないなら設計が歪んでいる」
   ([pipeline-failures.md](pipeline-failures.md)) の前段。
2. **外部の推奨は自分の問題と突き合わせてから使う。** リサーチの 3 パターンを鵜呑みにせず
   「なぜ②は DB 操作を許容するのか」と問い詰めて境界を確定させ、誤った例を修正させた。
3. **合意と実装のドリフトを照合で検出する。** 決めた方針 (ヘルパーは HTTP を知らない) と
   貼られたコードのズレをレビューの中で拾って直した。
4. **完成した設計も翌日疑える。** 二型分離は到達点ではなく、より単純な機構 (parse 時 VO 検証) が
   見つかった時点で自分のコミットで畳んだ。正当化が成立しなくなった抽象は維持しない。
