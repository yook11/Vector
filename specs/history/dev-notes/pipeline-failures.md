# 開発メモ圧縮記録 — パイプラインと失敗の設計 (3月-5月)

個人開発メモ (Claude.ai 対話ログ + ローカルノート、リポジトリ外) の圧縮記録のうち、
ニュース取得パイプラインの設計と「失敗をどう分類し、どう処理し、どう記録するか」という考えの変化を
3 月から 5 月まで一本で追う系列。原本のコードやコマンドは転記せず、
当時考えていたこと・決めたこと・その後の帰結だけを言語化して残す。全体の地図は [README.md](README.md)。

## 全体像

弧は **状態の所有権 (3月) → テーブルと経路の構造 (4月) → 失敗の語彙 (4-5月) → 記録と検証 (5月)**。
「DB はドメインの事実だけを持つ」(3/27) に始まった思考が、「監査はありのままの事実の記録」(5/11) に
再帰して閉じる。この過程で「処理方針と原因の 2 軸分離」「事実と副産物の区別」
「シンプルに書けないなら設計が歪んでいる」という判断の道具 3 点が加わった。
4 月に確立した「型で段階の前提条件を保証する」([structural-guarantees.md](structural-guarantees.md)) が
失敗系へ拡大した弧でもある。

## 第 1 部: 状態の所有権 (3月)

### パイプライン再設計 — 状態は誰のものか (3/27-28)

巨大モノリスタスク (v1) → DB に status 3 カラム × 5 段階 (v2) → bool 1 本 + Redis (v3) と、
短期間に 2 回設計を覆した。

- 転換点は **「パイプラインの処理状態 (fetching 等) は記事の属性ではない」** という気づき。
  DB はドメインの事実 (本文がある/ない、分析済み/未、永続的に取得不能) だけを持ち、
  キュー・リトライ・クラッシュ復旧は RedisStreamBroker (Ack + xautoclaim) に委譲する。
  v2 で設計した状態遷移ヘルパーや起動時リカバリは丸ごと不要になった。
- エラー分類学の萌芽: 永続的失敗 (403 / robots.txt / Safety Block) と一時的失敗 (5xx / 429 / timeout) を分け、
  **問題が発生した場所で対処する**。本文の品質ゲート (最低文字数、トリミング) は取得段階に置き、
  分析フェーズには「入力が来た時点で分析可能」を保証する。下流での後始末をしない。
- per-article 操作 (取得→分析→埋め込み) と cross-article 操作 (dedup / グルーピング) は粒度が直交するため、
  後者をスコープ外に切り出した。「意味的に似た記事は重複ではなく関連付け」という以前の結論もここで再確認。
- 収集 (fetch_metadata) / 振り分け (dispatch) / 実行 (個別タスク) の責任 3 分割。
- インフラへ委譲してよい根拠を taskiq のソースコードまで読んで検証した
  (Ack → リトライ挿入の順序、xautoclaim の排他ロック、複数ワーカー対応が組み込みであること)。

帰結: 「DB はドメインの事実だけ持つ」原則は、後の監査設計 (consumer-driven の焼き込み判断) や
読取エラー実体化の分類学にも引き継がれている (第 3 部)。

## 第 2 部: 構造と失敗の語彙 (4月)

### テーブル分離と収集戦略 (4/13-29)

- 出発点 (4/13-14): コアロジック (取り込み・分析・AI) が 1 ファイルに集まり、ドメイン概念と整理がついていない。
  「悪い部分を直すというより、やりたいことを実現する中で、いい設計に辿り着く可能性が高い」という方針を明記。
- 3 テーブル分離 (4/22): `article_extractions` (プロセス層・Stage 1 の素材) /
  `article_analyses` (成果物層・ユーザーに見せる分析結果の完全形=Stage 1 データを複製して自己完結) /
  `article_rejections` (対象外判定)。決め手は名前と実体の一致——「`article_analyses` という名前は『分析結果』と
  宣言している。なら分析結果と呼べるもの (翻訳・要約を含む完全形) を格納すべき」。
  別 AI の snapshot 論を「論拠が誤っている (versioning なしでは成立しない)」と批評した上で、
  自分は名前と実体の一致から同じ結論に到達した。
  FK は「AI が判定対象にしているのは extraction であって article ではない」という意味論から
  analyses/rejections → extractions の 1:1 に修正。相互排他はアプリ層+DB トリガーの両層で防御。
- 収集の汎用 vs 専用 (4/17): 実運用で最頻発する変更は「サイト側の仕様変更」であり Vector は制御できない。
  取得失敗は機会損失で許容できないので、**失敗の影響範囲を局所化できる設計を優先**。
  汎用処理ベース+計測に基づき問題のあるソースから順次専用化、という現実解。
- discovery / acquisition の分離 (4/29): 「URL を見つけること」と「本文を確保すること」は別の関心事で、
  後者はソースごとにやり方が違う。`ArticleCandidate` を sum 型 (ContentSupplied / UrlOnly) にし、
  各ソースが自分の経路を宣言、Strategy が型で分岐。**「最終的に同じもの (分析に進める品質) を持つが、
  そこまでの道のりをパターンごとに保証する」**。品質契約は経路ごとに書かず出口の型 1 箇所に集約。

### 結果型と例外の彫琢 (4/14-16, 4/23)

- 「責任を分離してレポジトリ・サービス・ルーターに分ける」は手段であって本質ではなかった、という反省 (4/14)。
  本質は「重要な処理を行うために必要な分離」であり、まず「取り込み処理は何を達成すべきか・どんな例外が起こるか」を
  言語化しないと分離の形は決まらない。
- エラー設計の原則 (4/15-16): 層ごとに適切に変換する。プロバイダ固有のエラー (Gemini) は
  プロバイダ非依存のドメインエラー (RateLimitError 等) に翻訳する。ただし `retry_after` のような属性は
  2 つ目のプロバイダが来るまで足さない——プロバイダ名はエラーメッセージとログの構造化で十分。
- 結果型の再設計 (4/23): `Fetched / AlreadyExists / Skipped` を呼び出し側の関心で見直し、
  **「分析に進めるか」を直接表す `ArticleReady`** に統合 (新規/冪等の区別はログの責任であり型に載せない)。
  「進めない」側も原因の所在で分類: 抽出を試みて失敗 (外部/品質) と、事前判定で対象なし (DB 不整合) を混ぜない。
  さらに後者は「対処できない異常系」なので **sum type から外して例外へ** (sum type は分岐すべき語彙、例外は対処不能な通知)。
  併せて Repository は「見つかった/見つからない」の事実だけ返し、ビジネス解釈は Service が持つ形に整理し、
  lookup 用の中間型を全廃 (約 40 行削減)。Repo=事実、Service=意味づけ、という分担の確立。

帰結: この一連の設計が、後の collection BC の stage 命名 (収集/completion)、Source 集約リファクタ、
extractor の取得/抽出二分割、読取エラー分類学 (reason taxonomy) へと連続していく。

## 第 3 部: 失敗の分類・記録・検証 (5月)

### 設計に丸一日を使う — 取得パイプライン再設計 (5/5)

- 実装前の設計段階で取得部分の問題点を複数発見した。「実装してから気づくと書いたコードを捨てることになる」。
  DDD 本を再読した直後の視点で自分の設計を見直したことが効いた、という自己観察も残している。
- 設計を 4 回作り直して実用に耐える形に到達。経験者との差は「一発で決める精度」ではなく
  「何回目で実用に耐える設計に着地できるか・問題点を発見する速度」だという相対化を得た。
  「なるべく早く動かす」と「設計に時間をかける」は対立しない (まず MVP を出し、運用で見えた問題に本格設計を投資する) という整理。
- 設計判断はその日のうちに短くメモする習慣を決めた。「今日の頭の中の解像度は、明日には半分になる」。
- 取り込み層の論点 4 つを自分で立てた: ソース別の取得形態の違い / 永続化の設計 / 冪等性とリカバリ / 一意性の担保。
- 設計の核は消費側起点だった: **「分析に進む品質を担保したデータを型で定義し、そこから逆算して取得を設計する」**。
  RSS で本文まで揃うソースはそのまま分析へ進め、HTML 抽出が必要なものだけ別テーブルに滞留させて再試行する。
  エラー種別で再試行戦略を分け、処理状態 (二度と処理しない / 処理中 / 処理可能) を設計し、
  後段タスクには「必要な情報がすべて揃っている」ことを保証する。

帰結: この設計はほぼそのまま現行 collection BC になった。収集 (stage1) と本文補完 (stage2) の 2 工程、
未完成記事の滞留テーブルとリース方式の状態管理、ソース宣言としての補完ポリシー、読取エラーの分類学
(AcquisitionReadError 系) まで、5/5 に紙の上で立てた要素が後日すべて実装されている。

### エラー分類学 — 「どう処理すべきか」と「何が起きたか」の 2 軸 (5/8, 5/9)

- 例外設計の起点を「どの分類のエラーか? それに対してどう処理すべきか? を継承で一目で読めるようにする」に置いた。
- 自分で到達した中心整理: **ビジネスロジックが分岐すべきは処理方針 (retryable / drop / keep / unknown) であり、
  原因 (provider 系か stage 固有か) は監査・調査のための従属軸**。「エラーはまず『どう処理されるべきか』という
  大分類にマッピングされるべきだよね?」という自分の問いで軸の主従を確定した。
- AI 呼び出し起因のエラーは stage 非依存 (AIProvider 系)、stage 固有なのは「AI は応答したが、その工程として
  消化できない」失敗だけ、という境界も自分の質問で確定。Stage 4 設計時に Stage 3 の既存パターンを照合し、
  自分の spec が不要な Stage 4 専用派生を量産していたことを発見して畳んだ。
- 同じ原因でも stage によって処理方針が変わることに気づいた: policy block は Stage 3 (本文そのものが拒否された)
  では記事 drop が自然だが、Stage 4 (抽出済み要約の分類で拒否された) では keep が自然。
  原因→処理の固定写像を拒否し、drop/keep の単位 (article / artifact) が stage ごとに違うことも言語化した。
- 段階導入の設計: 例外モジュールの package 化で旧型を同一内容のまま再定義 + 新型を re-export し、
  既存 13 ファイル無変更の互換層を作ってから、骨格→migration→差し替え→旧型撤去の 4 段階に分割。
  Stage 4 は先に「Stage 3 水準とのギャップ監査」を表にしてから段階計画を立てた。

帰結: この骨格は実装後さらに進化した。多重継承 marker と category/code カラムによる表現は、後に
outcome_code / retryability / error_class へ畳み直され (旧カラムは migration で撤去)、現行は AIProviderError 系
(stage 非依存のまま) + Assessment の Recoverable/Terminal 階層に整理されている。legacy 例外 8 種は全廃済み。
表現の語彙は変わったが、「処理方針と原因の 2 軸」という設計判断そのものは現行にそのまま生きている。

### 監査は事実の記録 (5/11)

対象は監査基盤 (pipeline_events) のエラー記録設計。

- 2 つの直感から始まった: 「後から追跡できるようにと ID を無理に焼くのは間違いではないか」
  「Service のメソッドが『永続化して監査テーブルに記録する』とシンプルに書けないなら、何か無理な設計になっているのではないか」。
- 依存方向を確定した: **監査が Service の入出力を観察する**。逆に Service のシグネチャに監査用引数が現れたら、
  横断的関心事がビジネスロジックを侵食している兆候。
- 「ID とは別に一意の値を 1 つ生成して全段に通せばいいのでは」と correlation ID パターンへ自力到達。
  実装は保留したが、観測基盤の trace と同じ識別子で監査を貫く案を得た。
- repository の戻り値を分解して責務を診断した: PK の返却は INSERT の自然な契約 (競合判定・次 stage への連鎖に必要)
  だが、内部で解決した FK 値まで返すのは内部知識の漏洩で、Service を監査への「郵便配達人」にしてしまう。
- さらに核心の問いへ: 「監査はありのままの事実を記録すること。焼き付けの段階で検索時の都合を考えるのは違うのではないか」。
  永続化後に採番される ID や FK 解決値は「AI が応答した」という事象の事実ではなく事後の副産物
  (payload の説明文自身が「ミラー」と自称していた) であり、payload から削除すると決めた。
  記録時の責任と検索時の責任を分離し、自然キーだけで 1-hop 追跡できること、rename 耐性・削除耐性が上がることも確認した。

帰結: 現行コードはこの判断どおりになっている。assessment の監査 payload に ID ミラーは無く、repository は PK のみを返し、
監査メソッドは入力と AI 応答の封筒だけを受ける。correlation_id は実装されず「将来足すなら別列」という設計メモが残り、
観測ポインタとしては trace_id が入った。「監査に何を焼くかは、それを要する consumer がいるかで決める」という
後続の規律 (consumer-driven の焼き込み判断) の起点でもある。なお worker が本体の永続化と失敗監査で
意図的に複数トランザクションを切る構造は、4/11 の見立て ([レビュー編 6](review-6-side-effects.md)) の実装形。

### テストは「分岐の監査への投影」を守る (5/8)

- 監査記録まわりで書く価値が高いテストを 6 種に絞った: 例外→分類 (category/code/error_class) の写像、
  error_chain がその例外の連鎖だけになること、**2 回連続で別例外を記録して前回の値が次の行に混入しないこと**
  (dict 再利用や mutable default という実装事故を狙い撃つ oracle)、監査書込失敗時に業務エラーと監査エラーが
  分離して fallback ログに出ること、リトライの非最終試行では監査行が作られないこと、DROP 経路の行数と記事削除。
- 「リファクタの中核は『分岐が監査へ正しく投影されること』なので、記録の単体テストと dispatch テストが
  いちばん漏れを防ぐ」という選定基準。2 軸設計がそのままテスト観点になっている。

## 通底していた思考

1. **問題が発生した場所で対処し、後段には保証だけを渡す。** 3 月の言語化が、5/5 の消費側起点設計
   (「分析に進む品質」の型から逆算) にそのまま発展した。
2. **軸を分ける。** 処理方針と原因、事実と副産物、記録の責任と検索の責任。混ざって見える問題を 2 軸に
   分けると設計が立つ、という手筋が 5 月に繰り返し使われた。
3. **「事実だけを持つ」が再帰した。** DB はドメインの事実だけ (3月) → 監査はありのままの事実だけ (5月)。
   副産物・検索都合・処理状態を事実から区別する同じ問いが、対象を変えて戻ってきた。
4. **消費側起点の逆算が失敗系へ拡大した。** 4 月の価値命題逆算 ([product-value.md](product-value.md)) が、
   5 月は「後段に何を保証するか」「監査の読者に何を残すか」に適用された。
