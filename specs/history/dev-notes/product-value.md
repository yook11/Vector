# 開発メモ圧縮記録 — プロダクト価値命題による取捨選択 (4月)

個人開発メモ (Claude.ai / Claude Code 対話ログ + ローカルノート、リポジトリ外) の圧縮記録のうち、
「この機能はアプリの価値と整合するか」で機能自体を削る判断が確立した過程を追う系列。
原本のコードやコマンドは転記せず、当時考えていたこと・決めたこと・その後の帰結だけを言語化して残す。
全体の地図は [README.md](README.md)。

## 全体像

4 月、個別の設計判断がすべて「ユーザー (日本人投資家) は何を見たいのか」「この機能はプロダクトの
価値命題と整合するか」への逆算で決まるようになった。
「『どんなことを実現するための機能なのか?』を考えずに実装したものは大体リファクタリングの対象になる」(4/10)。
技術的な改善案を比較していた議論が「機能ごと削る」で全消滅する経験を重ね、
機能を実装するより「なぜ消したか」を語れる方が強い、という整理に到達した月。
前段はレビュー編にある: フィルタ選択肢構想 ([レビュー編 2](review-2-routers.md) テーマ 4)、
keywords 削除 (同テーマ 5)、件数表示への疑問 (同テーマ 6)。

## テーマ 1: ソースフィルタ削除 (4/10)

- 全フィルタ軸 (impact/category/keyword/検索) が「分析結果を起点にした絞り込み」なのに、
  ソースだけが分析前の属性で浮いていた。使用箇所も N=1 で増える兆しがない=ドメインにとって本質的でない軸の傍証。
- 技術的な 3 案 (専用 endpoint / レスポンス同梱 / キャッシュ) を比較していた議論が「機能ごと削る」で全消滅した。
  1 週間前に構想した選択肢集約エンドポイント ([レビュー編 2](review-2-routers.md) テーマ 4) も論点ごと消えた。

## テーマ 2: 記事件数表示の廃止検討 (4/18)

- 件数はあまり有効な情報ではない。「何が盛り上がっているのか?」という情報に変える。
- 帰結: 4/23 に累積件数から直近 24 時間窓 (recentCount) へ切り替えられた。4/3 の「規模によって価値が変わる機能」
  という直感 ([レビュー編 2](review-2-routers.md) テーマ 6) がそのまま実装理由 (「累積は 9999+ で signal を失う」) になった。

## テーマ 3: published_at の代用禁止 (4/19)

- 正常系では分析日時で代用できるが、異常系 (Quota 枯渇で 24h 遅延) でフィールドの意味が嘘になる。
  **正常系で問題ないことは異常系の許容理由にならない**。
- nullable で真実だけ記録し、NULL 率の実データを見てから表示用フィールドの要否を決める。

## テーマ 4: トピック体系のキュレーション (4/23)

- 実データで問題を裏付け (161 topics / 351 analyses、57% が 1 記事のみ、
  `ai agent launch / debugging / acquisition` の細分化暴走、カテゴリ誤分類)。
  event-type で分ける案を退け「主語 (subject) 中心・業界視点」で統合。
- シードは「明らかに存在する代表テーマ」のみ 21 個に絞った——細かく用意しすぎると
  「AI が選択肢から選ぶだけ」になり、打破したかった人間の小分類の限界に戻る、という自分の指摘から。
- 内部キーは英語 (正規化・業界用語の安定・生成精度)、表示は日本語ラベル (シードのみ手動キュレーション、
  動的生成分は英語フォールバック)。「一度やってみてから判断」でタスクを増やさない。
- 既存データは完全リセット&ワンショット再分析 ($1 未満・10 分と見積もって即断)。specs/topic-curation-plan.md に固定。

## 帰結

- 「機能ごと削る」はその後も続いた: keywords は 4/18 に topic タグへ置換され、その topic も翌月には
  全層から撤去された (現行は events 抽出)。任意クエリの embedding 検索も 5/28 に機能ごと削除
  (約 2,400 行、[レビュー編 4](review-4-params.md) 帰結)。
- recentCount (直近 24 時間窓) は現行も稼働している。
- published_at は nullable のまま真実だけを記録する形で現存。

## 通底していた思考

1. **価値命題からの逆算。** スキーマの形・機能の存廃・型の保証範囲が、すべて
   「ユーザーに何を見せるアプリか」という一点から導かれるようになった。
2. **実データ・実挙動で決める。** topic 分布の SQL、published_at の NULL 率、再分析のコスト見積もり——
   判断の前に必ず観測を挟む型がこの月に固まった (実挙動の例は空 `in_()` の SQL 展開、
   [structural-guarantees.md](structural-guarantees.md))。
3. **正常系で問題ないことは異常系の許容理由にならない。**
4. **機能を実装するより「なぜ消したか」を語れる方が強い。**
