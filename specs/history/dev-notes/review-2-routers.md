# 開発メモ圧縮記録 — レビュー編 2: ルーターレビュー縦断 (4/2-4/3)

個人開発メモ (Claude.ai 対話ログ、リポジトリ外) 2 本の圧縮記録。原本のコードやコマンドは転記せず、
当時考えていたこと・決めたこと・その後の帰結だけを言語化して残す。
層別縦断レビュー (レビュー編、全 6 冊) の第 2 冊。全体の地図は [README.md](README.md)。

## 全体像

[レビュー編 1](review-1-schema.md) のスキーマレビュー (4/1) に続き、ルーター層の縦断レビューを開始した 2 日間。
出発点は率直な驚きだった——「まさかルーターが一番てきとうに設計されているとは思わなかった」。
DB には正規化理論のようなある程度の正解があるが、API は「まだ存在しないクライアントの使い方」への賭けであり正解がない。
しかも DB やスキーマは型と制約が強制的に考えさせるのに対し、**ルーターは動いてしまうから粗さが表面化しない**。
この 2 日で「使う側から考える」「内部 ID を漏らさない」「入力の独立で影響を閉じる」「出来の悪いものは消す」という
判断軸を連続的に獲得し、ここで作った Service / Repository の形がそのまま 4/11 のトランザクション境界検討
([レビュー編 6](review-6-side-effects.md)) の出発点コードになる。

## テーマ 1: 三層は積み上げではなく三角形 (4/2)

- 「DB 設計を学び、スキーマ層を直せば、ルーターもスムーズに直せる」という積み上げ観が崩れた。
  DB・スキーマ・ルーターは三角形の関係で、一つを学んでも他の二つとの接合部は別の知識。
  処方は「小さい単位で DB→スキーマ→ルーターを一気通貫で作り切る縦スライスを数本重ねる」×「書籍で判断軸を得る」の掛け算。
- 書籍 (API デザインパターン) から借りた判断軸 3 つ: そのエンティティを親から独立して操作する必要があるか
  (リソースかデータ型か) / 参照かインライン化か / 親子の削除・権限が連動するか (階層か参照か)。
  ただし Vector では「これから設計する」でなく「既存の設計を疑う検証ツール」として使う、と位置づけを修正した。
- スピードと理解の使い分けも確立: 構造を変えるだけで意味が変わらない機械的な分離作業は 1 日で終わらせ、
  理解の投資は設計判断を伴うレビュー本体に充てる。「その切り分け自体が判断力の向上」という自己評価。
- ルーターの仕事の定義: **HTTP の世界とアプリケーションの世界の翻訳**。
  「この関数から Request/Response を取り除いたら残るロジックは何か」が責務チェックの問い。残りがほぼ空なら適切。

帰結: 「1 エンドポイントを縦に完全理解する」方式はその後のレビューの基本形になった
(日付なしの「理解優先順位」メモは [レビュー編 6 の補遺](review-6-side-effects.md) に収録)。
thin router は現行コードの標準形として定着している。

## テーマ 2: internal ID を外に漏らさない (4/3)

- 核心の問いは「**クライアントはこの id を使って何をするのか?**」。`CategoryDetail` の id は使い道を全部洗い出すと
  サイドバーのフィルタ・UI 状態管理・キーワード作成 API への指定で、すべて slug で置き換え可能だった。
- URL に `?kwCategoryId=1` が出る違和感の根っこはフロントではなく API 設計にあった。作成 API が内部 ID を
  要求するから、フロントが ID を持たざるを得ない。「API が上流」を体感した具体例。
- 一律 slug 化もしない: 管理者専用エンドポイントは内部ユーザーしか使わず URL 共有もないので id のままでよい。
  slug 導入のコスト (カラム・migration・ユニーク制約) に見合うのは一般ユーザー向け URL だけ、という使い分け。

帰結: 同日中に実装された (カテゴリの外部識別子を slug へ統一)。現行も id なしの CategoryDetail
(slug / name / recentCount) と CategorySlug VO が生きており、trend_discovery や briefing でも slug VO が使われている。
「管理者用は id でよい」も admin 系 router の現行形。slug 受けのキーワード作成 API は、作成 API 自体の廃止により
実現しないまま消えた (テーマ 5)。

## テーマ 3: バリデーションの層別責務と VO の変換境界 (4/3)

- 「関係ない値を URL に入れたらエラー画面が出た」体験から責務を 3 層に分解: ルーター=形式の検証 (422) /
  サービス=存在の検証 (404) / フロント=エラーの表示。門番不在だと不正値が奥まで侵入して 500 で壊れる。
- VO の変換境界も整理: `str()` で手動ラップするのは TypeDecorator による三層防御のバイパス。
  FastAPI の制約で query param は str で受けるしかないが、ルーター層の出口で即 VO に変換して下流は VO のまま流す。
- フロントの toast 対応は「個別エンドポイントの問題ではなく API 共通のエラーハンドリング方針」として切り分け、
  バックエンドの 422 対応だけを今のスコープにした。

帰結: 進化した形で着地している。query param は str 中継自体が消え、Annotated + Query のモデル受けで
Pydantic が parse 時に VO を検証して 422 を返す (後の「Annotated[Model, Query()] なら VO 可」の規律の起点。
弧の全体は [structural-guarantees.md](structural-guarantees.md))。
例外→HTTP 変換は「ルーターで捕捉」からグローバル exception handler (404/409/422 + detail の sanitize) へ移動し、
「Service が NotFoundError を投げる」側の構図だけが現存する。なお briefing router に HTTPException 直 raise +
SQL 直書きの逸脱が残っており、exceptions の docstring も「Router 層で捕捉」のまま stale。

## テーマ 4: スキーマ共有が生む結合 (4/3)

- ニュース一覧の絞り込みとキーワード作成という無関係な機能が、`CategoryDetail.id` という 1 フィールドを介して
  間接結合していた。**各エンドポイントの入力が独立していれば変更の影響範囲は閉じる**。
  レスポンスの共有はあっても、入力の依存が別ルーター経由で繋がるのは設計の粗さの兆候。
- 管理用途とフィルタ用途の分離: URL の形は所属でなく目的を表す (`/sources` はリソース管理、
  フィルタ選択肢はニュース文脈の付随情報)。フィルタ軸の数だけリソースエンドポイントを叩く現状への対案として
  選択肢集約エンドポイントを構想し、「発見の記録だけして実装はレビュー完了後」と判断した。
- ソース名の重複は DB 制約の変更ではなく API レスポンス側の重複除去で対応すると整理。
  source_type は取得方法=内部実装の都合でありユーザーの関心ではない。(name, source_type) の複合ユニークは
  「同じソースでも取得方法が違えば別レコード」というドメインの宣言として正しいから変えない。
- 定義したが使っていないスキーマは OpenAPI に出力されず、フロントの生成型からも消える。
  定義と使用はセット——孤立した Embed 型は削除した。

帰結: フィルタ選択肢の集約エンドポイントは一度も実装されず構想止まりに終わった。1 週間後 (4/10) に
ソースフィルタが機能ごと削除され ([product-value.md](product-value.md))、論点自体が消滅——
「機能ごと削る」が技術論を全消滅させる価値命題逆算の前段だった。現行のフィルタは category のみで、
選択肢はカテゴリ一覧 API が事実上供給している。

## テーマ 5: 出来の悪い実装は消して、必要になったとき正しく設計する (4/3)

- keywords ルーター 4 エンドポイントは、コンポーネントごとどのページにもマウントされていない「未配線の実装済み機能」。
  AI は「消すと書き直しになる」と保持を勧めたが、「**既存のものがあまり出来が良くないから残すのに違和感がある**」
  と自分で覆し、全削除を決めた (モデル層・リポジトリ層は土台として残す)。やりたいこと (AI 検出→承認待ち→
  管理者承認のワークフロー) が明確なら、中途半端なものに合わせて拡張するより必要なときにゼロから設計する方がよい。
- 「正直かなり問題点が多い」と感じた時点で一度立ち止まり、トリアージした: 今やる (設計判断が不要な修正) /
  レビュー完了後にまとめて (設計判断が伴う) / 記録だけして後回し (将来の機能)。

帰結: 削除は同日実装された (関連スキーマ・フロント込みで約 1,000 行)。承認ワークフローは一度も実装されないまま、
keywords 自体が 4/18 に AI 自由記述の topic タグへ置換され、その topic も翌月には全層から撤去された (現行は
events 抽出)。「必要になったとき設計する」は機能した——必要にならなかったものを作らずに済んだ、という形で。

## テーマ 6: 件数表示の価値は規模で変わる (4/3)

- サイドバーの記事件数に 2 つの疑問を立てた: 件数は記事の増減時しか変わらないのに表示のたびに集計が走る (キャッシュ候補) /
  そもそも 10,000 件を超えたとき件数表示に意味があるのか。**規模によって価値が変わる機能** は、今作り込まず
  「いずれ意味を失う可能性」を認識して外しやすい構造にしておく、と整理した。

帰結: 4/18 の「件数を『何が盛り上がっているか』に変える」([product-value.md](product-value.md)) を経て、
4/23 に累積件数から直近 24 時間窓 (recentCount) へ切り替えられた。実装時の言葉も「累積は 9999+ で signal を失う」で、
この日の直感がそのまま実装理由になっている。

## テーマ 7: リポジトリの命名は呼び出し側の意図で (4/2)

- リポジトリは「倉庫番」: メソッド名は「何が欲しいか・何をしたいか」で考え、「どの SQL が走るか」では考えない。
  結果的に DB 操作と対応する名前になるのは偶然の一致であって設計の出発点ではない。
- URL は外部クライアントに向けた表現、メソッド名は内部の開発者に向けた表現——まったく別の関心事、という区別。
- 規約としては get_=単純な取得 / fetch_=組み立てが必要なクエリ、CRUD は骨格にしつつ読み取り系の
  バリエーションには緩く、と決めた。

帰結: 規約の文字面は生き残らなかった。旧 core 層には fetch_ 系だけが残り、新しい BC のリポジトリは
find_ / exists / save / ドメイン動詞 (claim・sweep 等) の意図ベース語彙へ移行、trend_discovery は組み立てクエリに
get_ を使っている。一方で上位原則「呼び出し側の意図で命名する」は、むしろドメイン動詞への移行として徹底された。
死んだのは接頭辞の規約で、生きたのはその規約を導いた考え方だった。

## 通底していた思考

1. **使う側から考える。** クライアントはこの id で何をするのか / フィルタに本当に要るのは何か /
   呼び出し側にとって意図が伝わる名前か。スキーマレビュー ([レビュー編 1](review-1-schema.md)) の
   「結局ユーザーは何を見たいのか」と同じ構図がルーター層・リポジトリ層へ降りてきた。
2. **変更の影響範囲を閉じる。** 入力の独立、使用箇所の全洗い出しを先にやる (「波及範囲が見えないのが一番怖い」)、
   管理者用と一般用で識別子戦略を分ける。
3. **認識と着手を分離する。** 問題が多いと感じたら手を止めてトリアージし、設計判断の要否で着手時期を分ける。
   「発見だけ記録して実装は後」は 4/11 のスコープ規律 ([レビュー編 6](review-6-side-effects.md)) の前段。
4. **動いてしまう層ほど粗さが隠れる。** 型と制約に守られない層 (ルーター) は意識的にレビューしないと
   一番てきとうなまま残る、という気づきがこのレビュー自体の動機だった。
