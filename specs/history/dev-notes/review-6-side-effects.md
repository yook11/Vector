# 開発メモ圧縮記録 — レビュー編 6: 副作用系レビューとトランザクション境界 (4/11)

個人開発メモ (Claude.ai 対話ログ、リポジトリ外) の圧縮記録。原本のコードやコマンドは転記せず、
当時考えていたこと・決めたこと・その後の帰結だけを言語化して残す。
層別縦断レビュー (レビュー編、全 6 冊) の最終冊。全体の地図は [README.md](README.md)。

## 全体像

横断スイープ ([レビュー編 5](review-5-sweep.md)、4/4) と本冊 (4/11) の間、4/10 に 2 つの自己観察が残っている。
「一回レビューが終わったはずのルーターから問題点がいくつも見つかる。一回では完璧なものは作れない。
繰り返すうちに『もっとこうやりたい』のように肉付けが変わり、目が育つ」——反復前提の自己認識。
そして「名前を決めることには正解がない。バックエンドだけで決まるものではなく、命名は局所的な話ではなく
全体に波及する」——ルーターを睨んでも答えは出ず、使用する側から考える、という
スキーマレビュー ([レビュー編 1](review-1-schema.md)) と同じ構図への回帰。

その目で 4/11、news_sources の delete/activate/deactivate を見た。**初の副作用系レビュー**であり、
読み取り系だけを見ていたことの自覚から、トランザクション境界・振る舞いの所在という新しい層の問いが開いた。
レビュー編の弧はここで「形 (スキーマ) → 入口 (ルーター) → 副作用 (トランザクション)」と最深部に達する。

## テーマ 1: 取得の意味論と存在保証のズレ (4/11)

- `_get_or_raise` への違和感を「メソッドの意味論 (取得) と呼び出し意図 (存在保証) のズレ」と言語化し、
  存在を調べるのは Repository の事実、見つからないを 404 にするのはユースケースのルール (Service)、と分離を確認。
  例外を投げる場所の基準 (エンティティ自身の不変条件なら Model、ユースケース文脈に依存するなら Service) もここで確立した。
- この観点が今まで出なかったのは、取得=目的の読み取り系しかレビューしていなかったからだ、という自己分析も残している。

## テーマ 2: 振る舞いの所在 — 変わる理由が違うものは別の場所に (4/11)

- 「フラグを変えるのは DB 操作ではなくメモリ上の属性代入」という ORM の二段構え (メモリが先に変わり、
  DB は commit 時に追従する) を理解したことで、「activate の意味」を知るべきは Model、永続化の方法を知るべきは
  Repository、**変わる理由が違うものは別の場所に置く**、と振る舞いの所在が確定した。

## テーマ 3: トランザクション境界 — 規約を構造で守る (4/11)

- 自分で書いた 3 行が転換点になった:「画面に返すにはメモリ上に値が要る / DB はメモリの変化を反映する /
  これがセットで 1 つのトランザクションにまとまらないといけない」。ここから
  「1 ユースケース = 1 トランザクション = 1 Service メソッド」と、Repository が commit すると Service が
  境界の主導権を持てないという構造的理由が導かれた。
- 境界の置き場 3 案 (Service 冒頭で明示 / dependency で begin 済み session を配布 / デコレータ) から dependency 案を選択。
  決め手は「書き忘れが構造的に不可能」——**規約をドキュメントではなく構造で守る**。
  VO で不変条件を型で守る ([structural-guarantees.md](structural-guarantees.md)) のと同じ思想。
  flush は「SQL を送るがまだ確定しない」操作で、refresh は commit ではなく flush に依存する、という区別もここで整理した。

## テーマ 4: updated_at の判断軸を自己修正する (4/11)

- 「リアルタイムだからアプリ側」は成立せず、正しい軸は
  「機械的な時刻 (最後の書き換え) = DB トリガー / 業務的な意味を持つ時刻 = アプリ側」。
  `activated_at` の追加は業務要件が出るまで保留とした。

## テーマ 5: worker の複数トランザクションとスコープ規律 (4/11)

- 1 メソッドで複数トランザクションが正当な場面も先回りで整理した: 部分確定を残したいバッチと、
  外部 AI 呼び出し中にトランザクションを握りたくないパイプライン。これは HTTP ではなく worker の話であり、
  session の供給自体を HTTP (begin 済み) と worker (素のまま渡し、各タスクが自分で境界を決める) で分ける、
  という適用範囲の限定まで到達した。
- スコープ規律も働いた: news_sources レビューの中では直さず「発見だけ記録して別ブランチへ」を選び、
  やらないこと (他エンドポイントのレビュー、新規テスト追加、規約の文書化) を列挙してから着手した。
  レビュー編 2 テーマ 5・編 3 テーマ 4 のトリアージ規律の完成形。

## 帰結

翌日 4/12 に一気に実装された。Repository から commit/refresh を剥がし dependency がトランザクションを
所有 (PR #35)、activate/deactivate の Model メソッド化 (PR #36)、規約の ADR-004 (service-layer Unit of Work)
成文化 (PR #37)。現行コードもこの構造のまま: repository 層に commit は 0 件 (flush は PK 採番時のみ)、
worker は自前の session factory で意図的に複数トランザクションを切っており (本体の永続化と失敗監査の分離、
[pipeline-failures.md](pipeline-failures.md) の監査設計に接続)、
「HTTP と worker で供給を分ける」という見立てどおりに着地した。docs/ddd-adoption/ 第 3 幕はこの日の違和感を
冒頭 (3.1) に据えており、rich domain への転換点の一次資料になった。一方 `updated_at` のトリガー統一は
維持されず、トリガーは news_sources の 1 本が残るだけで、以後のテーブルは server_default + アプリ代入へ移行した。
「業務的時刻はアプリで確定」の側は、trends の生成時刻で server_default を意図的に撤去する (時計の源をアプリへ
統一する) 判断として生きている。`activated_at` は予告どおり追加されていない。

## 補遺: レビューの理解優先順位 (日付なし)

データの流れ最優先 (1 エンドポイント完全理解方式) → モデル層 → スキーマ層 → 後回し群、という
コードレビューの理解優先順位メモが残っている。5 月のメモ束に紛れていたが、内容が初期の構成を参照しており、
レビュー編 2 テーマ 1 の縦スライス方式の続きとしてこの時期に書かれたものと推定される。
