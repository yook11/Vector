# 開発メモ圧縮記録 — レビュー編 5: 横断洗い出しと共通パターンの一括解消 (4/4)

個人開発メモ (Claude.ai 対話ログ、リポジトリ外) の圧縮記録。原本のコードやコマンドは転記せず、
当時考えていたこと・決めたこと・その後の帰結だけを言語化して残す。
層別縦断レビュー (レビュー編、全 6 冊) の第 5 冊。[レビュー編 4](review-4-params.md) (同日前半) の直接の続き。

## 全体像

articles ルーターを直した直後、進め方そのものを転換した日の後半。「1 ルーターずつ潰してから次へ」をやめ、
**全ルーターを先に洗い出して共通パターンと個別問題に分類し、共通から一括で対処する**方式に切り替えた。
その場で 5 ルーターを横断スイープし、共通パターン (例外変換の重複・ファクトリ 4 種混在・session 露出・
ページネーションのベタ書き・認証ポリシー不統一) を集計。対話終了から十数分後には
グローバル例外ハンドラとファクトリ統一が連続コミットされており、「洗い出し→分類→一括」がそのまま機能した。

## テーマ 1: 個別撃破から横断洗い出しへ (4/4)

- 転換の理由: ルーター間で共通の問題が見つかる可能性が高く、個別に潰すと後のルーターで
  「さっきと同じ問題だ」となったとき先に直した部分を揃え直す手間が出る。
- 方法はルーターを 1 本見るたびに「共通パターンの蓄積」を更新していく集計方式。
  NotFoundError→404 が 6 箇所、サービス生成パターンが 5 ルーターで 4 種類、session 露出が 4 ルーター——
  と数え上げたことが、そのまま一括対処の根拠と優先順位になった。

## テーマ 2: グローバル例外ハンドラ — 対応表を一箇所に (4/4)

- 自分の要約が設計の核: 「どんなエラーの時にどんな HTTP エラーを返すのかをまとめて定義して、それを使用していく」。
  ドメイン例外→HTTP ステータスの対応表をアプリで一箇所に登録し、サービスは raise するだけ、
  ルーターは catch を書かない。どのステータスになるかはサービスの関心ではなくハンドラの関心。
- **グローバル化の条件は発生源の限定**だと理解した。NotFoundError や DuplicateError は自分が raise した
  場所でしか発生しないから安全。ValueError は Python のあらゆる場所から飛びうる汎用例外で、
  グローバルに 422 へ写すとサーバーのバグ (本来 500) がクライアント入力エラーとして隠れる——だから含めない。
  ルーター固有の変換として残す。
- 「どの値でエラーが起きたのかを解決できないといけない」という要求を先に立て、VO のエラーメッセージの
  十分性を実物で確認 (VO 名+ルール+実際の入力値を含む) してから「文脈付加の個別 catch は不要」と確定。
  実装に入る前に判断ポイントを全部潰す進め方。
- 置き場所の判断: main の責務は「アプリの組み立て」なので、ハンドラ定義は専用ファイル+登録関数に分け、
  main は登録だけを知る。固定メッセージで足りる例外 (embedding 失敗) にメッセージ設定の責務を
  サービスへ持たせない、という複雑さの線引きも同時に行った。

## テーマ 3: ファクトリ統一 — シグネチャを依存関係のドキュメントに (4/4)

- 「明示的に指定することに意味があるのか?」という自分の問いから到達した答え:
  session という**手段**がシグネチャに見える状態をやめ、「このエンドポイントは何のサービスと認証が必要か」を
  宣言的に書く。**エンドポイントの関数定義が、そのまま依存関係のドキュメントになる**。
  エンドポイントはサービスの作り方 (セッション取得・リポジトリ組み立て) を知らない。
- 例外の許容も判断した: キュー投入だけのエンドポイントは DB を使わないので静的呼び出しを維持
  (「必要なものがないなら注入するものもない」)。同一ルーター内の非対称性は理由が明確なら許容し、
  コメント 1 行で意図を残す。対称性そのものは目的ではない。
- DI の一般原則としての理解にも到達: 「動くために必要なものを外から渡す」は言語非依存の設計原則で、
  実現方法がフレームワークごとに違うだけ (型ベースの自動解決か、ミドルウェアでの手動セットか)。
  グローバル例外ハンドラも同型の「共通処理の一箇所集約」。

## 帰結

- **対話終了 (9:09) から十数分で実装された**。ファクトリ統一 (09:16)、グローバル例外ハンドラ (09:17) が
  連続コミットされ、ページネーション共通化も 09:36 に着地——その際 watchlist と articles で
  default が 12 と 20 に割れていた不一致も発見され 20 に統一された。横断洗い出し方式は
  「午前中に共通問題 3 つを一括解消」という形でそのまま機能した。
- ファクトリ統一は現行も標準形 (routers 3 本 + admin 3 feature + trend_discovery が同パターン)。
  逸脱が 2 箇所残っている: admin/pipeline (service なしで session 直注入+エンドポイント内クエリ) と
  briefing (repository 直叩き)。後者は [レビュー編 2](review-2-routers.md) で確認した「briefing router の逸脱」と同一。
- pipeline ルーターはその後 2 回移設された (4/10 に admin 配下 + router-level 管理者認証、
  6/3 に feature-first の縦スライスへ)。「DB 不要だから静的呼び出し」の前提は、fetch がソース名解決で
  DB を使う形になり失効。embed は 5/4 の red-team 対応で「増幅経路を構造的に閉じる」として
  エンドポイントごと削除された——理由つきで許容した非対称性は、前提が変われば消えてよい、の実例。
- 認証ポリシーの不統一 (categories 認証なし vs sources 一覧認証あり) は、当時は「要確認」のまま残り、
  最終的に内部認証 3 層化 (#780、2026-06) で解消された: 共有 read は BFF 経由証明、admin 配下は
  router-level の管理者認証。optional user の仕組み自体が #780 で削除され、
  「user が居れば id を取り出す」というルーター内の分岐パターンも消滅した
  (現行の watchlist は非 Optional の user から id を引数渡しするだけ)。

## 通底していた思考

1. **共通か個別かを先に分類してから手を動かす。** 数え上げ (6 箇所・4 種類・4 ルーター) が
   一括対処の根拠と優先順位を与えた。
2. **グローバル化の条件は発生源の限定。** 便利さではなく「バグが隠れないか」で線を引く。
3. **非対称性は理由が明確なら許容し、意図をコメントで残す。** 揃えること自体を目的にしない。
   そして前提が変われば許容した例外も消えてよい。
4. **実装に入る前に判断ポイントを全部潰す。** メッセージの十分性・スコープ外の確定・置き場所——
   未決を残さず確定させてから書くので、実装はコミット連打で終わる。
