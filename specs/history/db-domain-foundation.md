# DB・ドメインモデル再設計の出発点 — すべての主題が枝分かれした前史

> 散在していた複数の設計メモを束ねた歴史記録。現行仕様ではなく、「どう考えが推移したか」を
> 辿るための記録。これは個別主題の前史にあたり、ここから記事・分類・AI 構成の各スレッドが
> 枝分かれしていった。

## このアプリは何か (2026-03, 最初の整理)

先端テクノロジーの英語ニュースを AI で収集・翻訳し、業界の動向・トレンドを深く分析して日本語で
提供する、個人投資家向けのテック・インテリジェンス Web アプリ。価値の優先順位は (1) 分析の
深さ (2) 速報性 (3) 情報集約 (4) ソース網羅。この定義を、再設計のすべての判断の土台に置いた。

## 現状を棚卸しし、あるべき姿との差分を出す

再設計の出発点として、当時のスキーマを棚卸しし(認証側 + アプリ側の2スキーマ、翻訳テーブルや
中間テーブルを含む構成)、あるべきドメイン概念モデルを描き、その差分(ギャップ)を1つずつ
洗い出して「DB 再設計で何をどう変えるか」の結論に落とし込んだ。

## どう作り直したか — セキュア・バイ・デザイン

再設計の背骨は「不変条件をアプリの約束ではなく、データ構造で担保する」という考えだった。

- 内部マスタ(カテゴリ・キーワード・ソース)は値オブジェクトで文字種を縛り、不正な値が存在
  できないようにする。外部由来のデータ(記事本文など)は制御できないのでサニタイズで守る。
- 一時状態(HTTP キャッシュ、リトライ回数、スケジューラの状態、重複排除キー)は DB から外へ
  追い出し、事実だけを DB に残す。
- 過剰な正規化(AI モデルの別テーブル化、翻訳の別テーブル化)をやめ、文字列カラムや統合属性で
  十分とする。分析結果は記事と1対1の不変エンティティとし、外部データなしでは無意味な感情値
  などは削除した。

## 値オブジェクトを全層で効かせる

当時の ORM の書き方では値オブジェクトをスキーマ層でしか使えず、サービス層で手作業の変換が
必要だった。これを解消するため、対象モデルだけ別の ORM 基盤へ移し、「値オブジェクト定義 /
型変換 / モデル接続」の3層構成でバリデーションを全層に行き渡らせた。

## 値オブジェクトをどこまで使うか — 型に縛るものと DB 制約に委ねるもの (2026-04)

全モデルのフィールドを1枚ずつ見直し、「値オブジェクトにする / 軽い制約付き型にする / DB の
チェック制約だけで足りる / 変更不要」に仕分けた。発見は「値オブジェクトはどこでも有効なわけ
ではない」だった。乱用すると抽象化コストばかり増える。判断の軸を次のように置いた。

- 複数の文脈で同じ正規化・等値判定が要るものだけを値オブジェクトにする。URL がその典型で、
  重複判定・取得・表示で繰り返し使われ、末尾の揺れを吸収して「一致」を意味あるものにしたい。
- 作成経路が1つしかないフィールド(AI 分析パイプラインだけが書くテキスト、内部運用ログの
  数値など)は、空文字列や負数を DB のチェック制約で弾けば十分とし、値オブジェクトは作らない。
- 列挙型・外部キー・真偽値・DB が既定値を入れる時刻などは、それ自体が制約なので手を入れない。
- 値オブジェクトにしたものも、モデル層の型と DB のチェック制約の両方で守る(多層防御)。型を
  信じきらず、最後の砦を DB に残す。

AI が生成する自由テキスト(翻訳タイトル・要約・根拠)には文字数やパターンの制約を課す意味が
ないと判断し、「空であってはならない」だけを DB のチェック制約で守ることにした。

値オブジェクトを実装する側にも発見があった。Python は不変性・等値性・型制約を言語として提供
しないので手で実装する必要があり、その上にランタイム型検査フレームワークとの統合の定型コードも
要る、という二重の手間があった。ただしそれは初回限りの定型コストだとして判断材料から外し、
「コストが高いから避ける」ではなく「この型に値オブジェクトの意味(不変性・等値性、あるいは型を
受け取ること自体のドメイン的な意図)があるか」で使う・使わないを決める、という基準に落ち着いた。

## 当初構想していた、より広い製品像とそこからの後退

再設計の前提として棚卸しした当時の製品像は、いまより広かった。記事に登場する企業を独立した
実体として扱い、ティッカーや株価(外部の株価 API 連携)を持たせ、企業ページや企業名での横断
検索まで提供する構想があった。だが企業実体・株価まわりは未実装のまま落ち、主役は個別記事の
分析と、その先の集約 AI(週次トレンド/ダイジェスト)へ移っていった。

分析モデルも当時はもっと多くを抱えていた。記事ごとにセンチメントや 1〜10 のインパクトスコアを
AI に出させ、翻訳を別テーブルに持ち、重複記事を埋め込みのコサイン距離でグループ化して代表記事
を選ぶ、といった仕組みを構想・一部実装していた。これらは後に「AI に過剰な判断をさせない」方針
と記事モデルの作り直しの中で、廃止・簡素化されていった。振る舞いの面でも、収集記事をトリアージ
で優先度付けし公開1時間以内を狙う、暫定タグを一定期間で自動的に表に出さない、といった運用像が
描かれていた。
