# API・ルーター層の整理 — 思考の推移

> 散在していた複数のレビューメモを束ねた歴史記録。現行仕様ではなく、「どう考えが推移したか」を
> 辿るための記録。各節は、そのレビューで何を発見し、なぜそう動いたかを過去形で残す。
> 管理系の集約と認可境界は隣接する主題として [管理 API・認可境界の整理](admin-api.md) にある。

## 出発点 — モデルごとにスキーマとルーターを1枚ずつ見直す (2026-04)

モデル別(カテゴリ / ニュース / ソース / ウォッチリスト)にスキーマとルーターを読み合わせた。
発見はモデルごとに性質が違った。

- **カテゴリ**: 値オブジェクトがスキーマ層まで一貫して伝わっており、途中の文字列変換も無い理想形
  だった。「修正不要」を確認できたこと自体が、他モデルを測る基準点になった。
- **ニュース(最大の発見)**: API レスポンスの構造がドメインの主従と逆転していた。アプリの中核は
  「AI 分析済み記事」なのに、トップレベルが原文で、分析がサブオブジェクトに沈んでいた。ストレージ
  の親子関係(記事テーブルが親・分析が子)が API 表面にそのまま漏れていた。さらに一覧と詳細が同じ
  型を共有し、カードで使わない本文や推論過程まで一覧で毎回流れ、未分析の記事も混ざっていた。
- **ソース / ウォッチリスト**: 1つのリッチな型をあらゆる用途で使い回していた。フィルタ用の
  ドロップダウンに詳細全部入りの型を渡したり、ウォッチリストが記事一覧型の劣化コピーを手作り
  したりしていた。

発見を一言でまとめると、「保存の形」と「画面が必要とする形」が混ざり、1つの型が全部を兼ねて
いた、ということだった。

## 転換1 — 用途ごとに型を分け、主従を正す

- 分析結果をトップレベルに昇格し、原文を参照情報のサブへ降格させ、スキーマ構造そのものでドメイン
  の主従を表すようにした。
- 一覧用(軽い)と詳細用(全部)を別の型に分け、継承させない。「カードが本体で詳細はその拡張」では
  なく、同一リソースの別ビューだと捉えた。Optional だらけの全部入り型は「この空は未分析だからか、
  一覧だからか」が区別できなくなる。
- 一覧を分析済みのみに絞り、翻訳タイトルや要約を必須にして「型が嘘をつかない」状態にした。
- 使い回していた1つの型を、埋め込み用の軽量型とリッチ型に分けた。ウォッチリスト専用の出力型は
  廃し、記事一覧と同じ型に揃えて、同じ記事が同じ見え方になるようにした。

## 転換2 — ルーターを3層に分ける

レビューを進めるうち、全ルーター共通の構造問題が見えた。1つのエンドポイント関数に、DB クエリ・
データ変換・レスポンス組み立て・HTTP 処理が同居し、関数を開くと SQL が最初に目に入って「何を
しているか」を掴むのにクエリの中身まで読まされる「太ったルーター」になっていた。これを、クエリを
担う層・変換と組み立てを担う層・HTTP だけを担う薄い層の3つに分けた。プロジェクトの規模に対して
中間データ構造は過剰だと判断し、変換層がレスポンス型を直接返す形にした。

## 転換3 — ルーターを消費者の関心で割り、リソース名を消費者語彙へ寄せる

1つのルーターに、ユーザー向けの参照(一覧・詳細・類似)と、管理者向けのパイプライン操作(取得・
埋め込み)が同居していた。認証ポリシーが違うもの(公開ないし任意ログイン vs 管理者必須)が混ざると、
認可の付け忘れが起きやすい。これを参照系とパイプライン操作系に割った。あわせて、ユーザーが消費
するのは「分析済み記事」であってデータの出自(ニュースソースからの取得物)ではない、として参照系の
リソース名を内部語彙から消費者語彙へ寄せた。パイプライン操作は最終的に管理系の集約に合流し、
管理者向けのまとまりの下に収まった([管理 API・認可境界の整理](admin-api.md))。

## 転換4 — エンドポイントを1本ずつ「使いやすさ・効率・堅牢性」で見直す

3層分離が済んだ後、各エンドポイントを1枚ずつ、目的・API 設計・スキーマ・クエリ効率・ロジック・
異常系の観点で点検した。ここで構造的な発見が続いた。

- **一覧取得と意味検索が1つのメソッドに同居していた。** 両者はデータの取り方そのものが違う(片方は
  条件で DB から引き、片方はクエリをベクトル化して全記事と比較する)のに、検索文字列の有無で分岐
  させていた。そのため「検索なし/あり」を表す空値が層をまたいで伝播し、指定しても静かに別のソート
  に上書きされるなどの歪みを生んでいた。別の操作なら全層で分ける、という判断で意味検索を切り出した。
- **記事レスポンスにユーザー固有の「ウォッチ中か」フラグを同梱していた。** そのために記事側の処理が
  ウォッチリスト側のデータへ手を伸ばし、ドメイン境界が曖昧になっていた。これを、記事レスポンスは
  ユーザー非依存にし、ウォッチ中の集合は別途取得してクライアント側で重ねる形へ移した。結果として
  記事レスポンスがユーザーに依存しなくなった。
- **ページネーションの「総ページ数」「オフセット」の計算が、複数の処理に散らばってコピペされて
  いた。** これはビジネスロジックではなくページネーションの定義の一部だとして、ページング値
  オブジェクトとレスポンス生成の1箇所に集約した。
- **使われていないエンドポイントを洗い出して削除した**(別の一括取得で代替できる一覧、UI の無い
  更新操作、需要の無い個別取得など)。
- **アプリの不変条件に照らして余分な防御を外した。** 「ウォッチされた記事は必ず分析済み」が成り
  立つので、起こり得ない事態(未分析記事のウォッチ)に備えて入れていた余計な結合を除いた。

## 起きた移管(どこからどこへ動いたか)

- API レスポンスの主従が「原文が主・分析が従」から「分析が主・原文が参照」へ移った。
- 1つの汎用型の使い回しから、用途別(一覧 / 詳細 / 埋め込み)の型へ移った。
- ルーターが「太った単層」から「クエリ層 / 変換層 / 薄い HTTP 層の3層」へ移った。
- エンドポイントの割り方が「データの出自」から「消費者の関心(参照 / 管理パイプライン)」へ移り、
  リソース名も消費者語彙へ移った。パイプライン操作は管理系のまとまりに合流した。
- ユーザー固有の状態が「記事レスポンスに同梱」から「別取得してクライアントで合成」へ移り、記事
  レスポンスがユーザー非依存になった。
- ページネーションの計算が「各所に散在」から「定義の1箇所」へ移った。
