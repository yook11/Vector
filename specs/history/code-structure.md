# コード/パイプライン構造の再編 — 思考の推移

> 散在していた複数の設計メモを束ねた歴史記録。現行仕様ではなく、「どう考えが推移したか」を
> 辿るための記録。各節は当時の判断と、なぜそう動いたかを過去形で残す。

## 出発点の問題 — 技術軸の水平分割 (2026-04)

コードを技術的関心事(サービス層・リポジトリ層・AI・タスク)で水平に切った結果、1つの業務
フローを理解するのに複数のディレクトリを渡り歩く必要があり、ドメインの繋がりがコード上で
見えなくなっていた。CRUD が整然と並んで「本体」に見える一方、アプリの価値の源泉である知的
処理がバックグラウンドタスクの中に埋もれていた。

## 転換1 — 分類軸をドメインプロセスにする

分類の第一軸をドメインプロセスに変え、「収集(必要な記事を集める)」「分析(記事の意味を理解
する。埋め込み生成も含む)」「検索(分析結果を使って問いに答える)」の3つで分けた。技術的
関心事はドメインの内部か外部インフラ層に置く。

## 転換2 — パイプラインのキューを工程ごとに分ける

収集〜分析〜埋め込みの全タスクが単一キューに同居し、「この記事は次の工程が必要か」の判定が
複数箇所に重複していた。工程完了で次へ進む連鎖が既にあるのに、全件を一括スキャンして振り分ける
中間タスクも残り、複雑さの主因だった。これを、工程ごとにキューを分離し、一括スキャンの中間
タスクを廃して連鎖で進める形に作り直した。

## 起きた移管(どこからどこへ動いたか)

- コードの第一分類軸が「技術的関心事」から「ドメインプロセス(収集/分析/検索)」へ移った。
- パイプラインが「単一キュー + 一括スキャン」から「工程別キュー + 連鎖」へ移った。
