# フロントエンドの API クライアント/型生成の変遷 — 思考の推移

> 古い設計メモを圧縮した歴史記録。現行仕様ではなく、「どう考えが推移したか」を辿るための記録。
> 当時の判断と、なぜそう動いたかを過去形で残す。

## 出発点 — 型だけ生成し、エンドポイントは文字列で引いていた

フロントの型は backend の OpenAPI スキーマから「型のみ」を生成し、API 呼び出しは URL 文字列で
エンドポイントを指すフェッチと、手書きのフェッチラッパに依存していた。文字列キーでの参照や手書き
ラッパは、エラー処理やキャッシュ制御が呼び出し側に散らばる温床だった。

## 転換 — 名前付き SDK 関数中心へ

スキーマから「型 + 名前付きの SDK 関数 + フレームワーク専用の fetch クライアント」をまとめて生成
する方式へ移すことにした。決め手は、フレームワークのキャッシュ拡張オプションが素の fetch 設定
として透過的に渡せる構造で、新しいキャッシュ環境下でも内部で禁止 API を呼ばないと確認できた点。
万一に備え、より低レベルな汎用 fetch クライアントへ一行で差し替えられる逃げ道も用意した。検討
項目を洗い出した結果、構造的なブロッカーは無いと結論し、この方式を採用した。

## 起きた移管(どこからどこへ動いたか)

- API 呼び出しが、URL 文字列での参照から、生成された名前付き SDK 関数の呼び出しへ移った。
- エラー正規化・キャッシュタグ付与・スキーマ参照が、手書きラッパや文字列キーから、生成された
  クライアントとインターセプタ・ランタイム設定へ集約された。
- 型の扱いが「型だけ生成」から「型と呼び出し口を一緒に生成」へ移った。
