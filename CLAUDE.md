# Vector — Agent Working Contract

海外テックニュース収集・AI翻訳・投資分析ダッシュボード。
コミット,PRは日本語で作成する。

## Work Definition

コードの実装前に定義すること。

1. Problem: 今回解く問題を簡潔に定義する。
2. Evidence: 関連する仕様・schema・test・設定・既存実装を確認する。
3. Invariants: 持続的に守るべき振る舞い・制約・境界条件を定義する。
4. Non-goals: 今回やらないことを明確にする。
5. Done: 達成すべき状態と、作業を停止できる条件を定義する。

## Scope Rules

- 既存実装は正解ではなく、仕様・schema・test・設定と並ぶ証拠候補として扱う。
- 変更は Problem / Invariants / Done を満たすために必要十分な範囲に収める。
- 最小実装とは、責務と不変条件を保ったまま Problem を解決する最小の変更を指す。差分や行数の少なさだけを優先しない。
- Done を満たしたら停止し、直接関係しない改善は提案に留める。
- 将来の拡張性や一般論だけを理由に、抽象化・設定・fallback・依存を追加しない。

## Design Rules

- 既存パターンを再利用する前に、責務・境界・契約・失敗条件が今回の実装と一致しているか確認する。
- 実装中に責務の混在を感じたら、名前・責務・境界を説明できる概念に分ける。説明できない概念は導入しない。
- 括り出しは、複数箇所が同じ契約と変更理由を持ち、利用側の理解や変更が単純になる場合のみ行う。見た目の重複だけでは括り出さない。
- 名前は処理や型ではなく、ドメイン上の役割と意図が読める語を選ぶ。
- 同じ概念には既存の名前を使い、同じ名前を別の概念に流用しない。新しい名前は、既存の語彙との違いを説明できる場合のみ導入する。

## Source Of Truth

- API の SSoT は FastAPI の Pydantic schemas。
- DB 変更は Alembic migration 経由のみ。
- 環境変数は設定層経由で扱い、`.env` を読まない・表示しない・編集しない。
- 認証・認可ロジックを簡略化、迂回、無効化しない。

## Public Repository Hygiene

- ポートフォリオの公開 frontend URL は公開導線として commit してよい。
- それ以外の実 production の Fly app 名、internal hostname、deploy / rollback / restore の具体手順は commit しない。
- `fly*.toml` は portfolio 用の構成例として公開し、app 名や URL は placeholder にする。
- 本番 deploy に必要な実値は GitHub Environment secrets / Fly secrets / private runbook 側で管理する。
- docs では設計意図と境界を説明し、運用者だけが使う詳細手順や復旧コマンドは公開しない。

## Research

- ライブラリやフレームワークの使用方法は、憶測で実装せず、`/research` スキルや最新の公式ドキュメントで確認する。
- 設計や実装方針に迷った場合は、既存実装だけで判断せず、現在の Problem / Invariants / Done に近い一次情報・ベストプラクティスを調べる。

## Memory

- メモリは作業ログではなく、今後も使う永続的な情報だけに使う。
- 迷ったら記録せず、必要なら事前に確認する。

## Comments

- コメント・ドックストリングは、コードだけでは読み取りにくい理由・制約・不変条件・外部仕様を補う場合のみ追加する。
- コメントは原則1文で簡潔に日本語で書く。
- 処理内容をなぞるだけのコメントは書かない。
- コメントで実装を正当化しない。説明が長くなる場合は、設計の歪みを疑う。
- 変更していないコードにコメントやドックストリングを追加しない。

## Verification

- 実装変更後は `/check` スキルで検証する。
- テストを通すために機能を削除・無効化しない。
- 検証できなかった場合は、未実行の項目と理由を明記する。

## Failure Learning

- Problem / Done と実際の作業にズレが見えた場合は、失敗から学ぶために `/failure-log` を使う。

## Delegation

- 調査・探索・コードリーディングは、コンテキスト節約のためサブエージェントに委譲してよい。
- 実装系サブエージェントは、その agent の成果物が今回のタスクの成果物に含まれる場合のみ起動する。
  - frontend-ui-builder: 成果物に frontend UI / component / page の production code の変更が含まれる場合。
  - test-writer: 成果物にテストコードの追加・変更が含まれる場合。
- タスクや仕様書に test / frontend という話題が登場するだけでは、起動理由にならない。
  仕様書のテスト項目やUI仕様の記述を変更するタスクの成果物はドキュメントであり、テストコードやUIコードではない。

## Ask First

次は事前確認する。

- DB schema / SQLModel model の変更
- 新規 dependency の追加
- API response shape の破壊的変更
- 認証・認可ロジックの変更
- 複数レイヤーにまたがる再設計
- 大きな構成変更、または既存境界の移動
