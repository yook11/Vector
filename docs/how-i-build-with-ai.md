# AI エージェントとの開発の進め方

> 状態: ドラフト。Vector で AI エージェントをどう開発プロセスに生かしているかを整理するためのメモです。

## この文書の主題

- Vector では、AI を単なるコード生成ツールとしてではなく、開発プロセスの参加者として使っている。
- この文書で書きたいのは、「AI に丸投げしていない」という弁明ではない。
- AI に丸投げしているのか、設計判断にどう向き合っているのかは、主に [docs/ddd-adoption/](ddd-adoption/README.md) 側のドキュメントが語っている。
- ここでは、AI をどのように開発に生かしているのかを書く。
- 特に、仕様書駆動開発、テスト駆動開発への取り組み、検証ハーネス、作業監視の仕組みに焦点を当てる。
- 人間が毎回うまく指示するのではなく、AI が継続的に働ける環境をリポジトリ側に持たせようとしている。
- この文書は、完成した方法論の宣言ではなく、Vector で試している開発プロセスの途中経過として書く。

## 中心文候補

- Vector で試しているのは、AI にコードを書かせることそのものではなく、AI が継続的に開発へ参加できるように、仕様・テスト・検証・監視の仕組みをリポジトリに組み込むことです。
- AI 活用の中心は、プロンプトではなく開発環境の設計です。
- 人間の仕事は、実装の細部を書くことから、仕様・境界・検証・フィードバックループを設計することへ移っています。

## 伝えたい柱

### 仕様書駆動開発

- いきなり AI に実装させるのではなく、実装前に仕様を書く。
- 仕様では、Problem / Evidence / Invariants / Non-goals / Done を先に定義する。
- AI に「いい感じに作って」と頼むのではなく、「この契約の内側で実装して」と渡す。
- 仕様を書くことで、人間側の意図、境界、停止条件を先に固定する。
- 仕様は一時的なプロンプトではなく、[specs/](../specs/) に残る開発資産として扱う。
- 仕様書は、AI に考えさせるための材料であると同時に、人間が受け入れを判断するための基準でもある。
- 仕様書駆動にすることで、AI の出力を「それっぽい実装」ではなく「合意した条件を満たす差分」として見られるようにする。

### テスト駆動開発への取り組み

- AI が書いたコードを信用するには、受け入れ条件としてのテストが必要。
- 完全に全面展開できているわけではないが、red-first や非空虚なテストに取り組んでいる。
- 「通るテスト」ではなく、「壊れた実装なら落ちるテスト」を重視している。
- テストを、後付けの確認ではなく、AI に渡す契約の一部として扱いたい。
- テスト設計もエージェントに任せきりにせず、[backend/tests/AGENTS.md](../backend/tests/AGENTS.md) のように原則を明文化している。
- テストが緑であること自体を仕様の根拠にしない。
- 実装規則をテスト側に複製して、同じ間違いを二重に書くことを避ける。
- 今後は、実装エージェントとテスト設計エージェントを分けることで、より強い検証にしたい。

### 検証ハーネス

- AI の出力品質は、モデル単体ではなく、与える文脈と検証手順に大きく左右される。
- そのため、AI が作業する環境そのものをハーネスとして設計している。
- ハーネスには、[AGENTS.md](../AGENTS.md)、skills、slash commands、subagents、`/check` などが含まれる。
- backend / frontend / tests など、層ごとに異なる契約を持たせている。
- 実装後は、lint、format、typecheck、unit test、integration test、security scan などを通す。
- AI に毎回同じ説明をするのではなく、リポジトリに埋め込まれたハーネスが作業を誘導する形を目指している。
- ハーネスは、AI を一回うまく動かすためではなく、継続的に同じ品質で動かすためのもの。

### 分割と並行実行

- 大きな変更は、AI に一度で任せない。
- 仕様書の段階で PR 単位に分割し、各エージェントには小さな責務を渡す。
- 分割することで、AI の作業範囲、人間のレビュー範囲、検証範囲を揃える。
- worktree を使って、複数のエージェントやツールを並行して走らせる。
- Claude Code と Codex など、複数の実行器を同じ仕様・同じ検証のもとで使う。
- 重要なのは、どの AI を使うかではなく、同じ契約と検証を通せること。

### 作業監視エージェント

- 失敗を記録するために `/failure-log` を作ったが、実際には十分うまく機能しなかった。
- 失敗したあとに、人間や実装エージェントが自発的に記録する運用だと、抜け漏れが起きる。
- 失敗の記録は、作業が終わったあとではなく、作業中に検知できたほうがよい。
- そのため、今後は作業を横から監視するエージェントの導入を検討している。
- 監視エージェントには、仕様からの逸脱を検知させたい。
- Ask First に該当する変更を、実装前または実装中に止められるようにしたい。
- 検証漏れ、テスト未実行、変更範囲と検証範囲の不一致を見つけさせたい。
- 過去の失敗パターンが再発していないかを見させたい。
- failure-log のようにあとから日誌に書く仕組みではなく、作業中に介入する仕組みに寄せたい。
- ここは完成済みの実践ではなく、現在の課題と次に改善したいポイントとして書く。

### 失敗からハーネスを直す

- failure-log は期待ほど機能しなかったが、「失敗を個別対応で終わらせず、仕組みに戻す」という考え方自体は重要。
- AI の失敗を「モデルが悪い」で終わらせない。
- 仕様、テスト、検証、監視のどこで防げたかを見直す。
- 失敗を次のプロンプト改善ではなく、次の開発環境改善につなげる。
- 今後は、記録型よりも監視・介入型の仕組みに寄せたい。

## 強調したいこと

- AI 活用の中心は、プロンプトではなく開発環境の設計。
- 仕様書駆動開発とテスト駆動開発は、AI と開発するための土台になっている。
- ハーネスは、AI の作業を人間の期待に近づけるための実装済みの文脈である。
- うまくいった仕組みだけでなく、failure-log のようにうまく機能しなかった試みも書く。
- うまくいかなかったからこそ、次は作業監視エージェントという形で改善しようとしている。
- Vector は、AI と開発するための仕組みを育てている途中のリポジトリである。

## 書かないこと

- 「AI に丸投げしていない」ことの弁明を主題にしない。
- DDD 採用や撤回の詳細には踏み込まない。
- 設計判断の正しさそのものは、[docs/adr/](adr/README.md) や [docs/ddd-adoption/](ddd-adoption/README.md) に任せる。
- AI 活用の一般論を書かない。
- 生産性が何倍になった、という検証しにくい話にしない。
- 使っているツール一覧の自慢にしない。

## 読者に残したい印象

- AI を使う開発では、人間の仕事がなくなるのではない。
- 人間の仕事は、仕様・検証・監視の設計へ移っていく。
- Vector では、その移動を実際のリポジトリの中で試している。
- 成功例だけでなく、うまく機能しなかった仕組みも含めて、開発プロセスを更新している。
- これは「AI に作らせた話」ではなく、「AI が開発に参加できる環境をどう作っているか」の話。

## あとで表現を育てたいところ

- 仕様書駆動開発を、実際の `specs/` の例と結びつける。
- テスト駆動開発について、できている領域と途上の領域を分けて書く。
- ハーネスという言葉を、初見の読者にも伝わるように言い換える。
- failure-log がなぜうまく機能しなかったのかを、運用の問題として具体化する。
- 作業監視エージェントに何を見させるのかを、実例ベースで書く。
- 「プロンプトではなく開発環境の設計」という主張を、文書全体の背骨にする。
