# AI エージェントとの開発の進め方

開発を進める中で、コーディングエージェントに関しては新しいツールや進め方を試しながら、単なるコード生成ではなく、仕様整理・実装・テスト・レビュー・改善に参加させる方法を探ってきました。

ここでは AI を使った開発プロセスを解説します。

まだ AI エージェントを十分に使いこなしているとは言えませんが、Vector の開発を通じて、開発プロセスにどう組み込むかを試しています。

この文書は、その試行錯誤の途中経過を整理したものです。

## 大事にしていること

AI と開発する中で、最も大事だと感じているのは、実装前に「何をしたいのか」を自分の言葉で明確にすることです。

AI エージェントの性能を引き出すためには、意図を明確に伝えることが重要です。最初は、やりたいことが定まらないまま作業を始めてしまい、実装の途中で目的がずれたり、何度も手戻りしたりすることがありました。

その経験から、実装に入る前に、何を解決したいのか、なぜ必要なのか、何を変えてよくて、何を守るべきなのかを整理するようになりました。

これは AI に正確な指示を出すためだけではなく、自分自身がやりたいことを理解するための作業でもあります。

この考えを実際の開発フローに落とし込むために、取り組んでいることを説明します。


## 取り入れていること

### 1. 仕様書駆動開発

AI エージェントに実装を任せる前に、まず仕様書を書くようにしています。

仕様書には、今回解きたい問題、確認した根拠、守るべき条件、今回やらないこと、完了条件を整理します。これは AI に作業内容を伝えるためだけではなく、自分自身が何を作りたいのか、どこまで変更してよいのかを明確にするための作業でもあります。

以前は、やりたいことが曖昧なまま実装を始めてしまい、結果的に手戻りが発生することがよくありました。そこで、実装前に仕様を言語化し、AI エージェントが判断に迷ったときに戻れる基準を用意するようにしました。

仕様書を作ることで、実装中の判断基準が共有しやすくなります。何を解決するのか、何を守るべきなのか、どこから先は今回の範囲ではないのかが明確になるため、AI エージェントの作業も、人間のレビューもやりやすくなります。

また、仕様書は一時的なプロンプトではなく、[specs/](../specs/) に残る開発資産として扱っています。実装後に見返したときにも、なぜその変更をしたのか、何を完了条件としていたのかを確認できるようにするためです。


### 2. テスト駆動開発

テスト駆動開発については、まだ十分に効果的に使えているとは言えません。

特に足りないと感じているのは、テストをどのように書くか、何を保証するためのテストにするかという視点です。単体テスト、結合テスト、E2E のどれを使うか以前に、その変更で守りたい振る舞いをどう切り出し、テストとして表現するのかがまだ難しいと感じています。

また、テスト駆動開発は毎回使えばよいものではなく、使う場面を見極める必要があることも学びました。

それでも、実装前に「この変更で保証したい条件」をテストとして表現することは、AI エージェントと開発するうえで重要だと感じています。

仕様書を作成したあと、実装に入る前に、守りたい振る舞いや壊れてほしくない条件をテストに落とし込むようにしています。まず失敗する状態を確認してから実装に進むことで、何を満たせばよいのかを具体的にできます。

これは仕様書と同じように、AI エージェントが作業中に目的を見失わないようにするためです。仕様書で作業の方向を定め、テストで満たすべき振る舞いを固定することで、実装中の判断基準を共有しやすくなります。

また、何をテストするのかを考えることは、自分自身が何を実装したいのかを理解するためにも重要な作業だと感じています。


### 3. エージェントの設計

アプリケーションを作っていく中で、自分だけでは判断しきれないことが何度も出てきました。
特にセキュリティについては、十分な知識がなく、どこにリスクがあるのかを判断しきれずに手が止まることがありました。

そこで、アプリケーションを攻撃者の視点でレビューさせるエージェントを作成して、レビューを行わせる取り組みを始めました。

外部 HTML の取得、認証境界、DB / Redis 権限、secret の扱いなどを攻撃者の視点で見直すことで、自分だけでは気づけなかった脆弱性や設計上の弱点を洗い出せるようになりました。

この経験から、わからないことが出てきたときに立ち止まるのではなく、AI を積極的に活用して一緒に考えるという使い方を意識するようになりました。

### 4. 実装フローと検証ハーネス

実装フローについても、固定されたやり方を守るだけではなく、新しい開発手法や AI エージェントの使い方を積極的に取り入れることを意識しています。

たとえば、[AGENTS.md](../AGENTS.md) に作業契約を明文化し、実装前に確認すること、守るべき境界、など毎回プロンプトで説明するのではなく、エージェントが同じ前提で作業できる環境を整えることを重視しています。

最近は、ラバーダックデバッグの考え方を応用し、実装エージェントに自分の変更内容や判断を説明させ、それを別のモデルにレビューさせる流れも試しています。

MCP 経由で Codex と Claude Code を協調させ、実装、自己説明、レビューを分けることで、一つのエージェントだけでは見落としやすい問題を拾えるようにしたいと考えています。



### 取り組んでいきたいこと

今後は、AI エージェントの作業をよりレビューしやすくするために、作業の流れや出力の内容を改善していきたいと考えています。

ただし、何でも AI で自動化すればよいわけではありません。
エージェントを実際に作ったものの、効果的ではなく、使わなくなったものもあります。
AI に任せる部分と、人間が確認すべき部分のバランスを取りながら、実装の意図や問題点が見えやすい開発プロセスにしていく必要があります。

特に課題だと感じているのは、実装中の失敗の扱いと作業中の監視、自己説明とレビューの運用、実装単位の切り方です。

`failure-log` のように、失敗を後から記録する仕組みは試しました。しかし、作業が終わったあとに、実装したエージェント自身が失敗を自発的に記録する運用では、うまく機能しませんでした。失敗を扱うには、作業後の自己申告だけではなく、作業中に仕様からの逸脱や検証漏れに気づける仕組みが必要だと感じています。

そのため、作業するエージェントとは別に、作業を軽量に監視する仕組みを取り入れたいと考えています。仕様から外れていないか、検証が抜けていないか、不要な変更をしていないか、作業中に確認できるようにするためです。これは AI にすべてを任せるためではなく、人間がレビューすべき点を見つけやすくするための補助として位置づけたいです。

MCP サーバーを活用した実装フローはまだ採用したばかりで、どのように運用すれば効果が出るのかについては、十分な情報を集められていません。

特に、AI にレビューをさせる場合、どの観点で確認させるのか、どの程度まで厳しく指摘させるのか、実装者の自己説明をどのような形で残すのかは、まだ検討が必要です。レビューの指摘が多ければよいわけでも、説明が詳しければよいわけでもないため、実装の意図や重要なリスクが伝わる出力の形を探っていきたいです。

また、実装単位の切り方も大きな課題です。エージェントに意図が明確に伝わる作業単位をどう作るか、実装をどこで分割するか、最小実装で一度貫通させる習慣をどう身につけるかは、まだ十分に整理できていません。

そのため、仕様、実装、テスト、レビューを同じ単位で扱えるように、どの粒度で作業を分けるのがよいかを探っていきたいです。

コードレビューを依頼しやすく、AI エージェントにも渡しやすい単位を掴むことが、今後の重要な課題です。
