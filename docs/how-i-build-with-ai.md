# AI エージェントとの開発の進め方

Vector の開発では、新しいコーディングエージェントや手法を試しながら、AI をコード生成だけでなく、仕様整理・実装・テスト・レビュー・改善にどう組み込むかを模索してきました。

まだ AI エージェントを十分に使いこなしているとは言えません。この文書では、現在の開発プロセスと、その形に至るまでの試行錯誤を紹介します。

## 大事にしていること

AI と開発する中で、最も大事だと感じているのは、実装前に「何をしたいのか」を自分の言葉で明確にすることです。
最初は、やりたいことが定まらないまま作業を始めてしまい、実装の途中で目的がずれたり、何度も手戻りしたりすることがありました。

その経験から、実装に入る前に、何を解決したいのか、なぜ必要なのか、実装で守るべき条件を整理するようになりました。

これは AI に正確な指示を出すためだけではなく、自分自身がやりたいことを理解するための作業でもあります。次に、この考えを開発フローへ取り入れるために実践していることを紹介します。

## 取り入れていること

### 1. 仕様書駆動開発

AI エージェントに実装を任せる前に、まず仕様書を書くようにしています。仕様書には、今回解きたい問題、確認した根拠、守るべき条件、今回やらないこと、完了条件を整理します。

エージェントはこの仕様書を確認しながら作業するため、実装中に当初の目的からずれにくくなります。

一方、実装前には全体像を把握しきれず、コードの確認や実装を進める中で、最初の定義では不十分だったと分かることもあります。そのため、自分が何を実現したいのかに、より深く向き合い、仕様書を作るプロセス自体の精度も高めていきたいと考えています。

仕様書は一時的なプロンプトではなく、実装後にも残る開発資産として扱っています。対象範囲に応じて、[specs/](../specs/)、[backend/specs/](../backend/specs/)、[frontend/specs/](../frontend/specs/)に保存し、なぜその変更を行ったのか、何を完了条件としていたのかを後から確認できるようにしています。


### 2. テスト駆動開発への取り組み

テスト駆動開発については、まだ十分に効果的に使えているとは言えません。特に難しいと感じているのは、何を保証するために、どのようなテストを用意するかという判断です。

最近、ある AI エージェントフレームワークの OSS を読み、テストの書き方を学びました。実際の LLM を呼ぶ代わりに、あらかじめ決めた応答を返す fake model を使う方法です。外部 API を呼ばずに特定の応答やエラーを再現できるため、それに対してアプリケーションが正しく動くかを確認できます。ここから、テストでは「何を確認するか」だけでなく、「確認したい状況をどう作るか」まで設計する必要があると学びました。

一方で、まだ判断軸を持てていないこともあります。外部から観測できる振る舞いで確認する範囲と、単体テストで個別に確認する範囲を、どう切り分けるかです。コードリーディングを続けながら、テスト設計の考え方を学んでいきたいと考えています。

また、テスト駆動開発は毎回使えばよいものではなく、使う場面を見極める必要があることも学びました。それでも、実装前に「この変更で保証したい条件」をテストとして表現することは、AI エージェントと開発するうえで重要だと感じています。

テストを書く価値がある変更では、仕様書を作成したあと、守りたい振る舞いや壊してはいけない条件をテストとして書きます。先に失敗する状態を確認してから実装に進むことで、何を満たせば完成なのかが明確になります。

これは仕様書と同じように、AI エージェントが作業中に目的を見失わないようにするためです。仕様書で作業の方向を定め、テストで満たすべき振る舞いを固定することで、実装中の判断基準を共有しやすくなります。

テスト駆動開発を取り入れるにあたって、書籍『テスト駆動開発』を読み始めました。今はまだ、プロセスの形を真似しているだけだと感じています。
読んで、次のようなことを強く感じました。本当に重要なのはテストを書く手順そのものではなく、作業を小さな単位に切り分けること、実装の歩幅を調節すること、そして何をしたいのかを明確にすることです。テストは、それらを支える手段の一つだと考えています。


### 3. 攻撃者視点のレビューエージェント

アプリケーションを作っていく中で、自分だけでは判断しきれないことが何度も出てきました。
特にセキュリティについては、十分な知識がなく、どこにリスクがあるのかを判断しきれずに手が止まることがありました。

そこで、攻撃者の視点でアプリケーションを検証する red-team の仕組みを作りました。1つのエージェントにすべてを調べさせるのではなく、複数のエージェントが3世代にわたって調査を引き継ぐ構成です。各世代は、脆弱性につながりそうな箇所を記録として残し、次の世代はその記録を引き継いで検証を深めます。こうして、一度の文脈では追いきれない範囲まで、少しずつ調査を広げられるようにしました。

外部 HTML の取得、認証境界、DB / Redis 権限、機密情報の扱いなどを見直すことで、自分だけでは気づけなかった脆弱性や設計上の弱点を洗い出せるようになりました。これらに具体的にどう対応したかは、[architecture.md](architecture.md) にまとめています。

この経験から、自分だけでは判断できない問題についても、AI と一緒に仮説を出し、実際のコードや仕様と照らして検証するようになりました。


### 4. 作業契約と複数モデルによる検証

AI エージェントが毎回同じ前提で作業できるように、開発フロー自体も整えています。

たとえば、[AGENTS.md](../AGENTS.md) に、実装前の作業定義や守るべき境界、変更後の検証方法を作業契約としてまとめています。毎回のプロンプトに同じ内容を書かなくても、エージェントが共通の基準に戻れるようにするためです。

最近は、開発フローそのものを複数の役割に分け、一つのエージェントだけで実装を完結させない進め方を試しています。

まず、仕様を成果単位のステップに分け、各ステップでは必要なテストを先に用意してから実装します。ステップごとに、テスト、frontend、production code、レビューといった役割で各エージェントに作業を振り分け、オーケストレーターが一度に一つの作業だけを渡します。オーケストレーターは実際の差分とテスト結果を確認したうえで、PASS・REVISE・BLOCKED のいずれかを判断します。

そのほかにも、ラバーダックデバッグの考え方を応用し、実装したエージェント自身に変更内容と判断理由を説明させ、read-only の別エージェントがその説明を実際のコードやテストと照合して指摘するフローも採用しています。実装・説明・レビュー・最終判断を分けることで、一つのエージェントの思い込みだけで進まないようにしています。

これらのフローはまだ試行段階です。タスクの大きさに応じてどの程度のワークフローを割り当てるかは課題で、役割を分けるほど実行時間が長くなるため、規模に見合ったフローを設計していきたいと考えています。あわせて、MCP を通じて Codex と Claude Code を連携させる際に、どのモデルをどの役割に使うかという選定にも取り組んでいきたいです。


## 今後取り組みたいこと

もっとも、何でも AI で自動化すればよいわけではありません。
エージェントを実際に作ったものの、効果的ではなく、使わなくなったものもあります。
AI に任せる部分と、人間が確認すべき部分のバランスを取りながら、実装の意図や問題点が見えやすい開発プロセスにしていく必要があります。現在は、次の三つを主な課題と考えています。

### 作業中の失敗を検知する

`failure-log` のように、失敗を後から記録する仕組みは試しました。しかし、作業を終えたエージェントが自分の失敗を記録する運用は、うまく機能しませんでした。
もっとも、これまでのモデルでは実行されなかったこの記録を、gpt-5.6 になってからは作業後に自分で実行するようになりました。ただし、この仕組み自体が適切なのかは、まだ判断できていません。

そのため、作業するエージェントとは別に、作業を軽量に監視する仕組みを取り入れたいと考えています。仕様からの逸脱や検証漏れ、不要な変更に作業中に気づき、人間がレビューすべき点を見つけやすくするための補助として位置づけたいです。

### 自己説明とレビューを運用する

MCP サーバーを活用した実装フローはまだ採用したばかりで、効果的に運用するための知見が十分に得られていません。

特に、AI にレビューをさせる場合、どの観点で確認させるのか、どの程度まで厳しく指摘させるのか、実装者の自己説明をどのような形で残すのかは、まだ検討が必要です。レビューの指摘が多ければよいわけでも、説明が詳しければよいわけでもないため、実装の意図や重要なリスクが伝わる出力の形を探っていきたいです。

### 実装単位を整理する

エージェントに意図が伝わり、コードレビューも依頼しやすい実装単位をどう作るかは、まだ十分に整理できていません。仕様、実装、テスト、レビューを同じ単位で扱えるように、実装を分割する粒度と、最小実装で一度処理を貫通させる進め方を探っていきます。
