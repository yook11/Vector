# AI エージェントとの開発の進め方

## この文書について

Vector は、最初から完成した AI 開発フローを持っていたわけではありません。新しいツールや進め方を試しながら、AI を単なるコード生成ではなく、仕様整理・実装・テスト・レビュー・改善に参加させる方法を探ってきました。

設計思想や考え方の変化は [docs/design-journey/](design-journey/README.md) にまとめ、ここでは AI を使った開発プロセスそのものに焦点を当てます。

この文書は、完成した方法論の宣言ではなく、Vector で試している開発プロセスの途中経過です。

## 大事にしていること

Vector で AI と開発する中で、最も意識しているのは、実装前に自分の考えを言語化することです。

AI エージェントの性能を引き出すためには、こちらの意図をできるだけ具体的に伝える必要があります。何を作りたいのか、なぜ必要なのか、何を変えてよくて、何を守るべきなのか。そこが曖昧なままだと、AI は動くコードを書けても、こちらが本当に求めている判断までは代わりにできません。

そのため、実装に入る前に、解く問題、守るべき条件、今回やらないこと、完了条件を自分の言葉で整理することを大事にしています。

AI 活用の中心は、プロンプトだけではなく開発環境の設計だと考えています。この考えを実際の開発フローに落とし込むために、仕様書駆動開発、テスト駆動開発、役割を分けたエージェント、検証ハーネスの整備に取り組んでいます。

## 取り入れていること

### 1. 仕様書駆動開発

AI エージェントに意図を伝えるための取り組みとして、仕様書駆動開発を採用しています。

実装に入る前に Problem / Evidence / Invariants / Non-goals / Done を整理し、仕様書として残すようにしています。AI には、その仕様書をもとに実装プランを作らせ、合意した範囲の中で作業させます。

仕様を書くことで、人間側の意図、境界、停止条件を先に固定できます。AI に「いい感じに作って」と頼むのではなく、「この契約の内側で実装して」と渡すための土台です。

仕様は一時的なプロンプトではなく、[specs/](../specs/) や [backend/specs/](../backend/specs/) に残る開発資産として扱っています。仕様書は、AI に考えさせるための材料であると同時に、人間が受け入れを判断するための基準でもあります。

### 2. テスト駆動開発

テスト駆動開発については、まだ十分に効果的に使えているとは言えません。どの振る舞いをテストで保証するのか、どこまでを単体テストで確認し、どこからを結合テストや E2E に任せるのかは、今も試行錯誤しています。

それでも、実装前に「この変更で保証したい条件」をテストとして表現することは、AI エージェントと開発するうえで重要だと感じています。

Vector では、仕様書を作成したあと、実装に入る前に保証したい条件をテストに落とし込み、まず失敗する状態を確認してから実装する流れを取り入れています。

これは、単にテストを増やすためではありません。何をテストするのかを考えることは、何を実装したいのかを理解することに直結します。また、AI エージェントが実装中に目的を見失わないようにするための、具体的な受け入れ条件にもなります。

目指しているのは「通るテストを書くこと」ではなく、「壊れた実装なら落ちるテスト」を先に置き、その条件を満たすように実装を進めることです。

### 3. エージェントの設計

アプリケーションを作っていく中で、自分だけでは判断しきれない領域が何度も出てきました。特にセキュリティについては、どこにリスクがあるのか、どの対策を優先すべきなのかを十分に判断できず、手が止まることがありました。

そこで、Vector では red team 用のエージェントを作り、アプリケーションを攻撃者の視点でレビューさせる取り組みを始めました。外部 HTML の取得、認証境界、DB / Redis 権限、secret の扱い、公開経路などを攻撃的に見直すことで、自分だけでは気づきにくい脆弱性や設計上の弱点を洗い出せるようになりました。

この経験から、わからないことが出てきたときに立ち止まるのではなく、エージェントに役割を与えて一緒に考えるという使い方を意識するようになりました。

今は、実装、テスト設計、レビュー、調査、red team など、目的に応じてエージェントの役割を分けることに取り組んでいます。重要なのは、AI に何でも任せることではなく、自分に足りない視点を補う役割を設計し、判断材料を増やすことだと考えています。

ただし、エージェントは作ればよいものではありません。実際に作ったものの、うまく運用に乗らず使わなくなったものもあります。役割を分けるだけでなく、何を判断させ、どの成果物を残し、どこで人間が判断するのかまで設計する必要があります。

### 4. 実装フローと検証ハーネス

実装フローについても、固定されたやり方を守るだけではなく、新しい開発手法や AI エージェントの使い方を積極的に取り入れることを意識しています。

たとえば、[AGENTS.md](../AGENTS.md) に作業契約を明文化し、実装前に確認すること、守るべき境界、検証手順をリポジトリ側に持たせるようにしています。毎回プロンプトで説明するのではなく、エージェントが同じ前提で作業できる環境を整えることを重視しています。

最近は、ラバーダックデバッグの考え方を応用し、実装エージェントに自分の変更内容や判断を説明させ、それを別のモデルにレビューさせる流れも試しています。MCP 経由で Codex と Claude Code を協調させ、実装、自己説明、レビューを分けることで、一つのエージェントだけでは見落としやすい問題を拾えるようにしたいと考えています。

一方で、AI にただ多く書かせればよいわけではありません。自己説明や記録を増やしすぎると、冗長な説明や重要でない情報が残り、かえってレビューしづらくなることもあります。

そのため今後は、エージェントの作業を監視し、仕様からの逸脱、検証漏れ、不要な変更、過剰な説明を見つける仕組みを整えたいと考えています。AI の出力を増やすことではなく、必要な判断材料だけを残し、開発ハーネスを継続的に改善していくことが課題です。

## まだ取り組んでいること

AI エージェントと開発するための仕組みは、まだ完成していません。特に課題だと感じているのは、失敗の扱い、作業の監視、実装単位の切り方です。

`failure-log` のように、失敗を後から記録する仕組みは試しましたが、十分には機能しませんでした。作業が終わったあとに人間やエージェントが自発的に記録する運用では、抜け漏れが起きやすいと感じています。

今後は、失敗を後から記録するだけでなく、作業中に仕様からの逸脱、検証漏れ、不要な変更、過剰な説明を見つけられる監視エージェントのような仕組みに取り組みたいと考えています。

また、実装単位の切り方も大きな課題です。変更が大きすぎると、AI エージェントの作業範囲がぼやけ、人間もレビューしづらくなります。一方で、細かく分けすぎると、全体の流れや設計意図が見えにくくなります。

そのため、仕様、実装、テスト、レビューが同じ単位で扱えるように、どの粒度で作業を分けるのがよいかを探っています。コードレビューを依頼しやすく、AI エージェントにも渡しやすい単位を掴むことが、今後の重要な課題です。
