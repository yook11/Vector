# Architecture

このドキュメントは、Vector の全体構成と主要な設計判断をまとめたものです。

ここでは「なぜこの構成にしたのか」「どの代替案を検討したのか」「どのトレードオフを受け入れたのか」に焦点を当てます。

設計に至るまでの試行錯誤や考え方の変化は [docs/design-journey/](design-journey/) に、
AI とどう協働して開発しているかは [how-i-build-with-ai](how-i-build-with-ai.md) に分けています。


## 設計前提

Vectorは、個人で開発・運用している本番アプリケーションです。そのため、運用に使える時間や費用、CIの月間実行枠には限りがあります。
当初はprivate repositoryとして運用しており、GitHub Actionsの月間実行枠を使い切ることがありました。そのため、すべての変更でCIを最後まで実行してからマージできていたわけではありません。
現在は ci-gate の通過をマージの必須条件にしています。ただし、過去の履歴には、自動検証が完了していない状態で取り込んだ変更も含まれています。

このリポジトリでは、アーキテクチャを理解するために必要なコンポーネントの役割や論理構成を公開しています。一方、本番環境のapp名や内部URL、デプロイ・ロールバック・restoreの手順は、非公開の運用情報として扱います。
fly*.toml はportfolio向けの構成例です。本番環境固有の値はリポジトリに含めず、GitHub Environment secretsやFly secretsから渡す前提にしています。


## このプロジェクトで取り組んだこと

Vector では、機能を作るだけでなく、個人で本番運用するアプリとして「どこが危ないか」「どこで壊れやすいか」「後から原因を追えるか」を意識して設計しました。特に次の 3 点に重点的に取り組んでいます。

1. **脅威を前提にしたセキュリティ設計**

    Claude Code で、複数のエージェントが攻撃者の視点でアプリケーションを検証する red-team の仕組みを作りました。外部HTMLの取得、認証境界、DB・Redisの権限、secretの分離、公開経路などを対象に、潜在的なリスクを洗い出しています。
    セキュリティ対策が完成したとは考えていませんが、攻撃者の視点から設計を見直したことで、アプリケーションの分割、最小権限の適用、内部ネットワークへの隔離、BFFによる認証境界の構築など、多くの改善につながりました。

2. **不正な状態を作りにくくする設計**

    ニュース収集からAI分析までの非同期パイプラインには、「この記事は次の工程に進めるか」「この値を保存してよいか」といった多くの前提条件があります。
    そこで、記事の状態や工程の開始条件を型で表し、保存時の整合性をDB制約や責務ごとのテーブル分離で守るようにしました。
    確認のための if を各処理に散らさず、条件を満たしたデータだけを次の工程へ渡すためです。バグが入りやすい確認処理を減らし、壊れにくく変更しやすい構造にすることを目的としました。

3. **失敗を後から追える運用基盤づくり**

   記事を収集してから分析するまでの非同期パイプラインでは、どの工程で処理が失敗しているのかに気づきにくい課題があります。
   そのため、各工程で起きた重要な出来事を監査ログ `pipeline_events` に記録し、どこで何が起きたのかを SQL で追えるようにしました。

   ただし、この仕組みを日常的な監視や復旧判断に十分つなげられているとはまだ言えません。
   今後は、ニュース取得元ごとの成功率、工程ごとの成功率、何をアラート対象にするか、失敗した処理をどう確認・再実行するかを整備していく段階です。

## セキュリティ設計

Vector で最も注意が必要な入口は、外部サイトの HTML を取得して解析する収集処理です。自分たちで制御できない入力を扱うため、設計を誤ると RCE や SSRF の起点になり得ます。

この章では、実際に起きた問題への対応ではなく、想定される脅威に備えて入れている予防的な設計判断をまとめます。基本方針は、どこかが破られても影響をできるだけ狭い範囲に閉じ込めることです。


## 被害を最小限にする

ブラウザから直接到達できるのは、画面の提供、認証、backendへのリクエスト中継を担う公開入口だけです。

アプリケーションデータを扱うAPI、AI分析、scheduler、各種workerは、Fly.ioの内部ネットワークに閉じ、ブラウザから直接呼び出せない構成にしています。

また、外部サイトのHTMLを取得・解析する処理は、信頼できない入力を扱うため、APIやAI分析を担う処理から別appへ分離しています。両者はRedisのキューを介して連携し、収集処理から内部APIへ直接アクセスする経路は設けていません。

| 境界 | 担当する処理 | 狙い |
|----|----|----|
| 公開入口（`vector-frontend`） | UIの配信、認証、内部サービスへの中継 | ブラウザから内部サービスへ直接到達させない |
| 内部サービス（`vector-core`） | APIの提供、AI分析、定期実行 | 重要なデータやsecretを外部入力から隔離する |
| 収集処理（`vector-collect`） | 外部HTMLの取得・解析 | 信頼できない入力を扱う処理の侵害範囲を限定する |

ただし、appを分けるだけでは、同じDB権限やsecretを持っている場合、侵害時の影響を十分に限定できません。そのため、DB、Redis、secret、通信経路についても、処理の責務に応じて境界を分けています。

| 境界 | 採った対策 | 狙い |
|----|----|----|
| DB権限 | app / auth / collect / migration ownerでroleを分離 | アクセスできるtableとデータ、行える操作を必要な範囲へ限定する |
| Redis権限 | 収集処理専用のユーザーを作り、必要なキュー操作だけを許可 | 他工程のキューやAI予算、rate limitなど収集に不要なRedisデータには到達させない |
| secret | 用途ごとにsecretを分け、弱い値や用途間での同一値を起動時に拒否 | 漏洩した場合、別用途の権限へ波及しにくくする、危険な構成のまま運用させない |
| 通信経路 | 公開入口以外を内部ネットワークに閉じ、内部サービスの公開アドレスを持たせない | 外部から直接到達できるのを公開入口だけに限り、内部サービスへの経路を増やさない|


## BFFを認証境界にする

当初は、認証をFastAPI backendの自作JWT（access/refreshトークンのローテーションや失効管理）とfrontendのNextAuthで分担していました。しかし、トークンの有効期限やローテーションの設定が複雑になりやすく、Next.jsのメジャーバージョン更新でNextAuth側のログインが動かなくなったこともあり、認証をBetter Authに寄せてBFFを境界とする構成へ移行しました。

認証の検証をBFFに寄せたことで、backendは記事や分析結果など、アプリケーションデータを扱うAPIに集中できるようになりました。

最初のBFF実装では、認証済みユーザーのIDとroleを、`X-User-ID`や`X-User-Role`といったHTTP headerに設定してbackendへ送っていました。
backendは、同じrequestに含まれる`X-Internal-Secret`が設定値と一致すれば、これらのheaderを信頼する構成でした。

この方式は実装が単純な一方で、backendはユーザー情報の正当性を`X-Internal-Secret`の一致だけで判断していました。
そのため、内部secretを入手してbackendにリクエストを送れる攻撃者は、これらのheaderを偽造し、任意のユーザーやadmin権限になりすませる状態でした。

そこで現在は、BFFがBetter Authのsessionを検証し、認証済みユーザーの`user_id`と`role`を取得します。取得した情報を有効期限の短いJWTに含めて署名し、backendへの内部リクエストに付与する方式へ変更しました。

backendはJWTの署名、有効期限、発行元、宛先を検証し、すべての検証に成功したJWTに含まれる`user_id`と`role`だけを信頼します。


## リクエストに対するレートリミット

短時間に集中するリクエストからアプリケーションを保護し、サービスの安定性を維持するためにレートリミットを設けています。DoS攻撃の軽減に加え、ログイン試行の繰り返しや、誤操作・クライアントの不具合による過剰なリクエストも制限します。
ブラウザからのリクエストに対する判定には、レートリミット専用のRedisを使用しています。本番環境では、このRedisを `vector-redis-rl` として、ジョブ処理用のRedisから分離しています。

ブラウザから送信されたリクエストは、まず `vector-frontend` の Next.js proxy が受け取ります。proxy はリクエストの種類・セッション・IPアドレスをもとに、Redisへ記録されたリクエスト回数を確認し、短時間に制限を超える数のリクエストが送られていないかを判定します。

初期の `vector-frontend` では、静的アセットを除くすべてのリクエストを、IPアドレスごとの共通枠として数えていました。そのため、Next.jsのServer Components、prefetch、APIへのリクエストが同じ枠を消費し、通常の画面遷移だけでも上限に達して429エラーが返ることがありました。

問題は、性質の異なるリクエストを同じ枠で数えていたことでした。現在はprefetch・読み取り・変更系を別々に数え、用途に応じて制限しています。また、認証済みリクエストは、セッション単位とIPアドレス単位の両方で制限しています。


### Redis 障害時の扱い

Redisの障害を理由にすべてのリクエストを拒否すると、backendのAPIが正常でもサービスを利用できなくなります。そこでRedisに接続できない場合は、proxyでのレートリミットを行わず、リクエストを通すfail-open方式にしています。レートリミットの厳格さよりも、画面を閲覧できる状態の維持を優先するためです。


### ログイン試行制限

一方、ログイン試行の制限はRedisに依存させず、Better AuthのDB-backed limiterで行っています。Redis障害時でも制限が維持されるようにするためです。

この構成は、Red teamレビューでの指摘をきっかけに採用しました。Redis障害時にログイン試行制限が外れると、パスワードの総当たりや、他サービスから漏えいした認証情報による不正ログインを防ぎにくくなるためです。

この変更はリスクをなくすものではありません。Redis障害時にもログイン試行制限を維持する代わりに、DBの負荷や行数増加を受け入れる判断です。ログイン試行では、障害時に制限が外れるリスクをより重く見ました。

ただし、古いカウンターを放置すると、`auth."rateLimit"` の行が蓄積し続けます。そのため、maintenance workerで期限切れの行を定期的に削除し、判定に必要な短期間のカウンターだけを残しています。


## 不正な状態を作りにくくする構造

Vector の非同期パイプラインでは、記事は工程ごとに状態を変えます。取得した記事のデータを分析できる形に揃え、二段階の AI 分析を行い、最後にベクトルを生成します。

これを一つの「記事」として表すと、工程によってはまだ存在しない値まで `nullable` で抱えることになります。すると Service や Task の各所で、その値が本当にあるのか、次の工程へ進めてよい状態なのかを繰り返し確認しなければなりません。

そこで、記事が取りうる状態と各工程の開始条件を整理し、型・DB制約・テーブル設計にそれぞれ役割を持たせました。具体的には、次のように分けています。

- ObservedArticle / AnalyzableArticleで、アプリケーションが扱う記事の状態を表す
- ReadyForCuration / ReadyForAssessment / ReadyForEmbeddingで、各工程の開始条件を満たした状態を表す
- DBのテーブル設計、CHECK 制約、composite FKで、保存されるデータの整合性を守る

これにより、状態や開始条件が暗黙の前提として処理ごとに散らばることを避け、必要な条件を満たしたデータだけを各工程で扱えるようにしています。

例えば `AnalyzableArticle` は、記事が分析を開始できる状態にあることを保証する型です。条件を満たす素材からしか生成できないため、後段の処理は、値の有無を確かめ直さずに分析へ進めます。

```python
class AnalyzableArticle(BaseModel):
    """分析工程に進める記事。生成できた時点で、分析に必要な不変条件が揃っている。"""

    model_config = ConfigDict(frozen=True)  # 生成後は変更不可

    title: str = Field(min_length=1, max_length=500)
    body: str = Field(min_length=50, max_length=1_048_576)  # 本文は必須
    published_at: PublishedAt                                # 公開日時は必須
    source_id: int = Field(gt=0)
    source_url: CanonicalArticleUrl

    @classmethod
    def try_build(cls, *, body: str | None, published_at: PublishedAt | None, ...) -> AnalyzableArticle | None:
        # 素材が欠けていれば生成しない = 分析できないデータを型の入口で弾く
        if body is None or published_at is None:
            return None
        return cls(body=body, published_at=published_at, ...)
```

DB 側でも、全工程の結果を 1 つの大きな記事テーブルに詰め込まず、工程ごとにテーブルを分けています。
さらに、投資の対象外と判断した記事は、curation_noises や out_of_scope_articles のように保存先テーブルを分け、対象の記事と構造的に区別しています。

- 分析に進める記事: analyzable_articles
- curation の結果: article_curations（要約）/ curation_noises（ノイズ判定）
- assessment の結果: analyzed_articles（対象）/ out_of_scope_articles（対象外）
- embedding: 専用テーブルを持たず、analyzed_articles の列として分析結果に同居

各工程の結果テーブルは、外部キーと UNIQUE を同じ列に張ることで、「同じ記事に分析結果が二重に作られる」状態を、アプリ側のチェックではなく制約で防ぎます。

```python
class ArticleCuration(Base):
    __tablename__ = "article_curations"
    __table_args__ = (
        # 1 記事につき curation は最大 1 件
        UniqueConstraint("analyzable_article_id"),
    )

    analyzable_article_id: Mapped[int] = mapped_column(
        ForeignKey("analyzable_articles.id", ondelete="CASCADE"),
    )
```
このテーマで重視したのは、抽象化を増やすことではなく、データの正しさを確認すべき場所を減らすことです。

型やDB制約を設計するうえで難しかったのは、制約の強さではなく、アプリケーションがどの状態を正しいとみなすかを決めることでした。
そのためには、扱っている概念や状態の変化、例外となるケースと向き合う必要があります。あらかじめ唯一の正解が用意されているわけではなく、どこまでを保証し、どの層に責任を持たせるかを判断することに難しさを感じました。
この経験から、型やDB制約は単に追加するものではなく、守るべき状態と責任の境界を理解したうえで、その理解を実装に反映するものだと学びました。

この考え方に至るまでの試行錯誤は
- [第2幕: 値オブジェクト](design-journey/02-value-objects.md)では型を取り入れ始めた経緯を、
- [第6幕: ドメインモデル再構築](design-journey/06-domain-model-rebuild.md)では明確な正解がない中でアプリケーション固有の概念と責任の境界に向き合っていく過程を紹介しています。


## 黙って消える失敗を見える形にする

非同期パイプラインの障害は画面にすぐ表示されないため、同期APIの障害より発見が遅れやすいです。記事が次工程へ進まない、週次処理が実行されない、キューにタスクが滞留するといった形で表れます。実際、過去には収集工程の停止を長時間検知できなかったことがありました。

そこでVectorでは、パイプラインの各工程で発生した重要な処理結果や失敗理由を `pipeline_events` に記録しています。外部データをアプリケーション内の形式に変換できなかったケースや、AIの応答が期待する形式・内容を満たさなかったケースも記録し、原因の調査や改善に活用できるようにしています。

Logfireのspanとmetricでも、パイプラインの工程ごとの成否や処理結果を確認できるようにしています。

ただし、監査ログやメトリクスを日常的な監視と復旧判断に活用するための運用手順は、まだ十分に整備できていません。障害時に何を確認し、どの条件で再実行または処理対象からの除外を判断するかも、明確には定まっていません。
現時点では、完成した障害対応の仕組みではなく、workerの失敗を発見し、原因調査と復旧につなげるための基盤と位置づけています。

## 非同期パイプライン

Vector は、ニュースサイトや外部 AI API など、自分では応答時間や稼働状況を制御できないサービスに依存しています。

そのため、ニュース収集から AI 分析までの処理を複数の非同期 stage に分け、時間のかかる処理や一時的に失敗する可能性がある処理をバックグラウンドで実行しています。


### 処理の流れ

記事は、収集、本文補完、翻訳・要約、投資分析、ベクトル生成の順に処理されます。

| 分類 | stage | 役割 |
|----|----|----|
| 起点 | dispatch | DBから処理対象を選び、収集・本文補完のタスクを投入する |
| 収集 | acquisition | ニュースソースから新しい記事候補を取得する |
| 収集 | completion | 本文が不足している記事をHTMLから補完する |
| AI分析 | curation | 記事を日本語化・要約し、分析しやすい形に整える |
| AI分析 | assessment | 重要度、カテゴリ、投資文脈を分析する |
| AI分析 | embedding | 分析済み記事のベクトルを生成する |
| 派生処理 | trend discovery | 一定期間の記事から注目トレンドを抽出する |
| 派生処理 | briefing | カテゴリごとに記事を分析して週次ブリーフィングを生成する |
| 復旧・保守 | maintenance / backfill | 未完了工程の再投入やキューの監視を行う |

schedulerがcron形式のスケジュールで最初のタスクを投入し、各stageは処理結果をDBへ保存してから、次のstageのタスクを明示的に投入します。


### キューとworkerの境界

タスクキューにはRedis Streamsを使い、stageごとにキューを分けています。一方、worker processは処理の性質が近い工程で共有し、収集系のacquisitionとcompletion、AI分析系のcurationとassessmentがそれぞれ同じプロセスで動きます。

キューを分けているのは、各工程にどれだけタスクが滞留しているかを個別に把握するためです。それ以外のworker、同時実行数、DB接続、障害の影響範囲は共有し、プロセスを増やしすぎないことでメモリの使用量などを抑えています。

そのため、キューが分かれていても処理能力や障害の影響範囲までは分離されていません。特定のstageの遅延が続く場合は、観測結果をもとにworkerを分けるか判断します。


### 設計上の要点

外部サービスの遅延やworkerの停止を前提に、次の設計で整合性を保っています。

- タスクにはDBレコードのIDだけを渡し、workerが最新状態を読み直して前提条件を検証してから処理する
- 未完了工程の復旧はPending再配送とbackfillの2系統。タスクが重複することを前提に、DB側の制約で整合性を守る
- 再投入は件数・日次使用量・対象期間に上限を設け、hold gateやkill switchで抑制する

移行の経緯（Redis List → Streams）、Pending再配送、backfill、タスク重複実行への対処は、Zennの記事で詳しく解説しています。

## 運用して分かったこと / 取り組み中の課題

実際にデプロイしてみると、ローカル開発では見えていなかった問題がいくつも出ました。コードとして正しく動くことと、ユーザーが触ったときに安心して使えること、本番環境の制約の中で安定して動くことは別の問題だと学びました。

**非 AI worker のメモリ逼迫** — デプロイ後、AI を実行しない scheduler / collect / maintenance のプロセスが、起動直後にメモリ制限へ達することがありました。原因は、すべての worker が起動時に AI provider の重い SDK まで読み込んでいたことです。現在は、AI を使う worker だけが必要になったタイミングで SDK を読み込むようにし、非 AI プロセスの常駐メモリを抑えています。

**フロントエンドの使用感** — フロントエンドにはまだ改善の余地がありますが、実際に触る中で、ロード中なのか分からない、ボタンを押せたのか分からない、サーバーのコールドスタートで timeout する、といった問題に気づきました。API が正しいレスポンスを返すだけでは不十分で、待ち時間や操作中の状態を画面でどう伝えるかも含めてユーザー体験であり、それは実際に触ってみないと分からないことだと痛感しました。

**外部サービスの実行枠の制約** — DB、CI、AI provider などには実行枠や課金上限があり、使い切ると処理が止まることがあります。止まってから気づくのでは遅いため、使用量、キューの滞留、失敗件数を可視化し、どこで詰まっているのかを追える仕組みが必要だと分かりました。監査ログやメトリクスは入れ始めていますが、日常的な alert や復旧手順としてはまだ整備途中です。

**安全なデプロイ手順** — デプロイして初めて、DB migration だけでなく、キューに残っているタスク、処理中の worker、新旧コードの互換性まで確認する必要があると学びました。
新しい実装へ置き換えるときに古い経路をすぐ消すと、本番で問題が出たときに戻しにくくなります。そのため、新旧の経路を一時的に共存させ、デプロイ後に動作確認してから古い経路を削除する進め方が必要だと感じています。
まだ十分に体系化できてはいませんが、変更を小さく出し、確認してから片付ける意識を持つようになりました。
