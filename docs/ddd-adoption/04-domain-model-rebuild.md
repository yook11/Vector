# 第4幕 — ドメインモデルの作り替え:設計の問いと目標を自分が握る

第3幕までは、新しいドメイン観のもとで「型と構造を整える」段階でした。値を VO に閉じ込め、振る舞いをモデルへ戻し、モジュールをドメイン軸で切り、不変条件を構造と型で守る——どれも、すでにある形を正しく置き直す作業です。

第4幕(2026年5月の前半)は、その観点で**既存のドメインモデルを大きく作り替える**時期です。型の粒度を作り直し、入れたばかりの中間概念を三日で畳み、前日の自分の判断すら条件付きで巻き戻す——という試行錯誤が続きます。

そしてこの幕には、もう一つの主旋律があります。**何を解くか(問い)と、どういう状態を正解とするか(目標)を、AI ではなく自分が決めるようになった**ことです。当時の開発メモには「はじめて自分の意思で設計をしたのかもしれない」(5/9)という一行が残っていて、その対象がまさにこの取得パイプラインの再設計でした。3月の「AI に作らせたものをそのまま使う」から、4月の「提案を論拠で評価する」を経て、5月に設計の主語が自分に移った——その移り変わりが、以下のエピソードの底に流れています。先に分担だけ書いておくと、ここで落ち着いたのは「**設計の主導権は自分、AI は批評・裏取り・命名の相手**」という形でした。

## 4.1 型が「保証しているふり」をしていた — 厚い Ready

第3幕で立てた指針の一つに、「型 = データの形ではなく、operation の前提条件」というものがありました。「Stage X に進める条件」が `ReadyForX` という型に閉じ、Task や Service に散らない——そう書いて、各段階の入口に Ready 型を置いていました(4/28 の設計メモ、実装は #227 / #238 ほか)。当時の作り方は、上流の Task が下流のために Ready を構築して次段へ渡す形(メモでの呼び名は Pattern A')でした。

ところが 5/11 に再評価して、この型が**保証しているふりをしていただけ**だと気づきました。当時のメモは、保証の実体が無い理由を三点で挙げています。

1. 「`int` の nominal wrapper で中身に invariant が無い」——名前は付いているが、中身は ID 一個で、守っている不変条件が無い。
2. 「kiq enqueue → 実行までに DB 状態が変わる」ため、「Service は到着した Ready が表す precondition を真実とみなせず、DB fetch + None チェックで実質的に再検証している」。
3. 「precondition 検査が『`RuntimeError` fail-fast』に名前を変えただけで責務は消えていない」。

つまり、型を受け取った側が結局もう一度確かめている。これでは「型に入れた時点で前提条件が満たされている」という当初の狙いが成立していません。再評価の場で自分が書いた指摘が、問題の核でした——責務の主語が「**処理を開始するときに、処理ができることを保証する**」ではなく「上流が下流のために次に進めることを保証する」になっている。「『上流が下流のために』次に進めることを保証する構造 → 責務の主語が間違っている」とメモに残しています。

ここで Ready 型そのものを廃止する案(案2)まで検討しました。けれど却下しています——「ready のようにその処理に進めるという型は必要」だからです。型が要らないのではなく、構築の主語が逆だっただけでした。確定した形(案3)は、「Ready 型は**処理に必要な内容をすべて値として運ぶ厚い型**であり、**下流 Stage 自身が処理開始時に DB から内容を fetch して構築する**」。薄い Ready との本質的な差分は、構築のタイミングを処理開始時に固定したこと(取得から処理までの時間ずれをゼロにしたこと)にあります。

実装は Stage5 から入れ(#466, `87d73c08`, 5/12)、収集側へも同じ作り(house style)として展開しました(#542, `564d6234`, 5/16)。いまのコードでは、`ReadyForArticleCompletion` は frozen dataclass(kiq の payload にはせず in-process で受け渡すため。BaseModel にはしていません)、他段の `ReadyForAssessment` などは将来の手動 enqueue 経路で taskiq formatter と整合させる必要から Pydantic の frozen モデル、という使い分けになっています。いずれも payload には ID だけを運ぶ Trigger を渡し、Ready 自体は下流が処理開始時に DB から構築します。

後日談として、厚くした field の一部はさらに見直しました。#467 で監査の都合から足した `source_name` を、「読む consumer も存在しない」ことを確認して撤去しています(#683, `b209c562`, 5/30)。厚くするのも、厚くしたものを削るのも、同じ「いま要るか」の問いでした。

## 4.2 議論の単位が間違っていた — per-article から per-source へ

次に作り替えたのは、型の中身ではなく**議論の粒度そのもの**でした。

当時、本文の補完ロジックは「ソースが増えるたびに新しいフラグを足し、`complete_with_html` の中の分岐を増やす」形で書いていました。記事ごとに「HTML を取りに行くか」「どちらを正本にするか」を boolean や None で持たせていたわけです。5/17 の設計メモは、この置き方を「ソースが増えるたびに新フラグ＋ `complete_with_html` 内の分岐が線形に腐る。多源ニュースアグリゲータでパターン増加はほぼ確実なので、**これは投機ではなく既に発生している負債**」と書いています。第3幕で置いた歯止め(「変更の蓋然性が具体的に見えているか」で投資を決める)に照らして、ここは将来の先取りではなく、もう起きている負債だから手を入れる、という判断でした。

核心の気づきはこうです。「どのフィールドを fetch する必要があるか／どちらを正本とするかは、記事が下す判断ではなく**ソースの出自(provenance)で構造的に決まる**。…これは**ソース種別の能力(capability)**であって記事ごとの自由状態ではない。それを全記事インスタンスに booleans/None で複写するのは Minsky の『実態と食い違える per-instance optional』アンチパターン」。補完方針はソースの性質なのに、記事一件ごとに同じ方針を複写していた——議論の単位を間違えていた、というわけです。

そこで「補完ポリシーは per-source、取得事実は per-article。取れた事実は全部保存し、方針(profile)が利用可否を決める」(#544, `7a91be16`)へ作り替えました。いまのコードでは、方針は `ArticleCompletionPolicy`(frozen dataclass)としてソース側に ClassVar で持たせ、`title` / `body` / `published_at` ごとに「HTML を正本にする / HTML を優先する / 観測値を優先する」のルールを宣言します。記事の側(`AnalyzableArticle` / `ObservedArticle`)は素の属性だけを持ち、補完方針は一切持っていません。取れた事実は per-article に全部残し、それをどう使うかは per-source の方針が決める——責務の置き場が、ようやく出自の所在と一致しました。

## 4.3 入れた中間概念を、三日で畳む — Adapter が消えるまで

この作り替えのなかで、入れてすぐに畳んだ抽象があります。`SourceAdapter` です。

5/15、全ソースの取得処理を統一する型基盤として導入しました。当時の PR 背景にはこう書いています——「現状 32 fetcher のうち 26 個が `passport_builder` 経由・残り 13 箇所が `ReadyForArticle` / `IncompleteArticle` を直接構築している。後続 Phase で全 fetcher を『Adapter が `FetchedArticle` を yield → 共通 `ArticleFetcher` が passport に変換』に統一するための型基盤を先行確立する」(#522, `910d1f15`)。段階置換(strangler)で 45 ソースを一括で新経路へ移し、旧 Fetcher を全削除するところまで、二日で走り切りました(#532 / #533)。

撤回の引き金は、その二日後でした。per-source の知識をどこに置くか調べ直すと、「共有基底 4 + 継承 13 の階層」が見えてきて、「rename 単独だと is-a 違反、正すには継承→委譲が不可分」だと分かりました(5/17, spec §4.1)。Adapter という名前を付け替えるだけでは済まず、継承で組んだ構造そのものが間違っていたわけです。

そこで「『Fetcher/Adapter 中心』から『Source 中心』へ転換する原子的切替」を行いました(#545, `5dc5f1a3`)。このとき `ArticleSource` は `SourceAdapter` Protocol の改名ではなく、新規の集約として作り直しています——「改名は責務(identity vs 取得)を曖昧化、新規集約が DDD 上正」と判断したからです。構造の移送は byte 不変で検証できる形に保ち、汎用化しすぎないために「33 standalone は machinery として残置(over-generalize しない)」としました。

そして翌日、Adapter という中間概念そのものを除去しました(#546, `dd6ede07`)。「P2 で `ArticleSource` 集約 + `adapter_factory` 遅延 callable を導入したが、**Adapter という中間概念**が残存していた」からです。判断の軸は担保の強さでした——「『profile を読むのに machinery を作る』経路が**構造的に不能**」になるよう、契約をクラス参照の構造保証へ(設計の約束から構造へ)昇格させた。このとき目指していたのは、自分の言葉で「**Adapter という中間概念が消えている**」という到達状態でした。誕生(5/15)から概念除去(5/18)まで、三日です。

ただし、これは全面破棄ではありません。ソース固有の解析(bespoke parser)は「個別 `XxxSource` クラスで保持(汎用化しない)」と決め、消すものと残すものを選り分けています。仕上げに、93 ファイルの docstring から Adapter の語彙と当時の設計史を一掃しました——「設計が流動的な段階で docstring に強い契約・設計史を書くと、コードを読んで設計を再考する際に過去の文章へ引っ張られる。docstring を『地図』ではなく『名札』に最小化する」(#557, `5af69005`)。入れた抽象を、価値の問いで測り直して畳む——「投資する価値があるか」という問いが、ここでは最も短いサイクルで回りました。

## 4.4 赤子を、産湯ごと流さない — 解体と回復

作り替えが進むと、今度は取得処理の名前と実体のズレが気になり始めました。きっかけは、姉妹にあたる本文補完(stage2)を「acquire → complete → persist」に純化したときに、違和感の正体が分かったことです(#598, `88251b35`, 5/23)。

そこで取得側を解体しました(#608, `7e6bfff6`, 5/24)。「`ArticleFetcher` が collect+convert を一つの runner に畳み込み、service から変換工程が消えていた」——「名前は『Fetcher』(取得)なのに実体は取得+変換 — **名が体の一部しか表していない**」状態だったからです。狙いは「service を読むだけで収集の全体フローが分かる」ことでした。

ところが、この解体は行きすぎていました。約四時間後、条件付きで巻き戻しています(#609, `d15efb4d`, 5/24)。spec の言葉では、先行リファクタは「次の二つを**同時に**実施した: 1. **convert を runner から切り離す** — 正しい。2. **engine そのものを消す** — 過剰だった」。本質は「**engine は要る。癒着していたのは convert だけ**」で、「解体は赤子(engine)を産湯(convert 癒着)ごと流した」と書いています。

ここで大事にしたのは、これがフリップフロップ(やっては戻すの繰り返し)ではないと示すことでした。「先行 spec を否定するのではなく、その一決定を**条件付きで巻き戻す**」とし、「返り値が `FetchedArticle` で convert を含まない ── この一点が『フリップフロップでない』構造的証拠」だと明記しています。前日の自分の判断でも、間違っていた一点だけを構造の根拠つきで戻す——問いを自分が握っていると、自分の決定すら測り直せる、ということだったと思います。

確定した形は、責務を三つに分けたものです。**Source は宣言**(`read` / `in_scope` / `select` / `map_entry` の四つの classmethod)、**Fetcher は汎用の engine**(`fetch_articles` がソースの宣言を駆動して `FetchedArticle` を yield する。変換は含まない)、**Reader は道具**(各ソースが取得時に呼ぶ I/O クライアントの束)。変換は別の関数 `convert_fetched_article` が担い、一件の `FetchedArticle` を「分析可能 / 観測どまり / 棄却」の三つに値で振り分けます(想定内は例外を投げない total な分類)。`map`(写像・失敗なし)と `convert`(漏斗・分類)という動詞を対比させ、どこが素通りでどこが絞り込みかを名前から読めるようにしています。

採らなかった案も残しておきます。ソースごとの差異(Reader の異種性)を engine の中の巨大な `match` に集約する god-dispatch 案は却下しました——「異種性は各 source に局在させるのが正」だからです。共通の engine は薄く保ち、違いは各ソースの宣言に閉じ込める、という線引きでした。

## 4.5 構造が決まってから、名前が来る

最後は、語彙の収束です。作り替えで構造が固まったあとに、名前を合わせていきました。順序が逆でなかったこと——**語彙は構造の確定に従属させた**こと——が、振り返ると一つの規律になっていました。

stage1 のディレクトリは、一度 `source_fetch` から `article_collection` へ改名しています(#607)。本当は `acquisition` にしたかったのですが、当時その語は stage2 が使っていて衝突していました。このとき「`acquisition` の概念は stage2 のもの。名前は accurate なので維持」と据え置いたのですが、翌日それを覆します——「Html 限定子で非衝突とした過去の据え置き根拠は、限定子が…脱落するため成立していなかった」(#612)。stage2 から acquisition を剥がす(scrape へ移す)ことで語が空き、ようやく統一できました。

確定した語彙はこうです(#613, `12000376`, 5/25)。「acquisition = 『記事(article)そのものをシステムに獲得する』工程 / BC / 監査 identity」「fetch = HTTP/transport/read の I/O 動詞。工程語彙ではない。共用・凍結」。結果は綺麗な二層で、「**acquisition 工程が、汎用 fetch I/O 基層の上に立つ**」形になりました。いまのコードでも、工程は `article_acquisition/`、I/O 基層は `fetcher.py` の `fetch_articles` として分かれています。

改名の通底動機は、機構ではなくドメイン状態を名前に語らせることでした。たとえば「『HTML 取得待ち』という**機構**を語っており、『本文未取得の未完成記事が入っている』という**ドメイン状態**を表せていなかった」(#606)という直しです。構造が決まる前に良い名前を急いで付けると、その名前に引っ張られて構造を見直しにくくなる——だから名前は最後に、というのが、この幕で身についた手順でした。

## 4.6 第4幕の終わりに

ここまで作り替えてこられたのは、無防備に大手術をしたからではありません。45 ソースを一括で移すような変更は、段階置換(strangler)と「**発見オラクル先行**」で安全網を張りました。普遍的な契約テストを先に書いて現行コードに走らせ、赤(失敗)で問題を発見してから直す——「実装してからテスト」ではなく「テストを発見の道具にする」順序です(#558〜#564、production コード変更ゼロで赤を立てる回もありました)。このテストとの付き合い方そのものを「道具を統制する」観点で扱うのは、第7幕に譲ります。

考え方の面で、この幕で変わったのは**設計の問いと目標を自分が握った**ことだったと思います。第3幕で「違和感を信じる」ようになり、その違和感を言葉にして設計を動かしてきましたが、ここではもう一歩進んで、目指す到達状態を「Adapter という中間概念が消えている」のように自分の言葉で持ち、AI の提案も自分の言葉で測り直すようになりました。早い時期のメモに「AI は構造的に正しい分解はできるが、なぜその部分が存在するかという目的を持たない」と書いていたのが、ここで「主導権は自分、AI は批評・裏取り・命名の相手」という分担に落ち着いています。到達点は「AI を使わない」ではなく、握る場所を変えた、ということでした。

そしてこの作り替えのなかで、「失敗を表す型」がいくつも増えました。変換の棄却、観測どまり、品質不足——成功と失敗を別の型に分けるほど、構造は安全に見えます。次の第5幕では、その**失敗の型に何を語らせるか**を問い直します。「型を増やすほど安全」という、私自身の信念のほうを測り直す話です。
