# 第7幕 — 失敗の定義が、工程の理解を問うてきた

第6幕で取得パイプラインを作り替えるうちに、失敗を表す型がいくつも増えました。変換の棄却、観測どまり、品質の不足。それらを監査に残そうとしたとき、一番難しかったのは、監査を書くことではありませんでした。そもそも、その失敗をどう定義するか、でした。

やってみて分かったのは、これが単なるラベル付けではないということです。ある失敗は、ある工程では「もう一度試す」が正しく、別の工程では「諦める」が正しい。同じ失敗でも、捨てるべき工程と、残すべき工程がある。なぜそうなるのかを言葉にしようとするたびに、足りていないのは型ではなく、その工程への理解なのだと気づきました。うまく定義できない失敗は、たいてい、まだ分かっていない工程でした。第7幕は、失敗を定義しようとして、一つひとつの工程を理解していった話です。

## 7.1 「リトライできるか」を、何回目かで決めて、嘘をついた

一番はっきり表れたのが、「もう一度試せば直るか」をどう定義するかでした。最初は、何回目の試行かで決めようとしました。途中の試行なら「まだ直るかもしれない」、最後の試行なら「もう直らない」。一見もっともらしい定義です。

けれど、これは嘘を書いていました。たとえば API キーが設定されていない失敗は、一回目から直りません。なのにこの定義では、一回目・二回目は「直るかもしれない」と記録され、最後の試行でやっと「直らない」に変わる。すると、「直らない失敗」を拾って人に知らせるはずの仕組みが、最後の一回まで黙ってしまう。事実と食い違うラベルを、自分で焼いていたのです。

直し方は、ラベルの付け方を工夫することではありませんでした。「もう一度試せば直るか」に見えていたものが、実は三つの別のことだと理解することでした。一つは、失敗そのものの性質。アクセスを拒否された失敗は何度試しても拒否されるし、通信が時間切れになった失敗は次は通るかもしれない。これは工程に関係なく、失敗が起きた瞬間に決まっています。二つめは、その工程がどう扱うか。同じ失敗でも、ある工程は「ここでは諦める」と決める。三つめは、実際に試行回数を使い切ったか。これは、その実行が最後にどうなったかという、後からの事実です。

この三つは、各工程が失敗をどう扱っているかを分かって初めて、切り分けられました。性質は失敗自身に、扱いはその工程に、使い切りは実行の記録に。一つの「リトライできるか」に融かしていたものを、それぞれの居場所へ戻したのです。最初の嘘は、定義をしくじったというより、工程をまだ分かっていなかったことの表れでした。

## 7.2 「値か、例外か」も、工程の理解が決めていた

失敗を「値として返すか、例外として投げるか」も、最初は書き方の好みの問題だと思っていました。けれど、これも工程を理解しないと決められないものでした。

分かれ目は、たった一つの問いでした。この失敗のあとに、続けるものがあるか。記事を一件ずつ処理していく工程で、ある記事が条件を満たさなかった——これは、次の記事へ進めばいい。だから失敗は、処理を止める例外ではなく、理由を載せた値として返す。一方、外部のサイトへデータを取りに行って、そもそも繋がらなかった——これは、そのソースについて続けるものがありません。だから、処理を中断する例外として投げる。

同じ「失敗」でも、片方は値、片方は例外。その違いは、工程が何を、どんな単位で処理しているかを分かっていないと出てきません。値か例外かは、エラーの見た目ではなく、工程の進み方が決めていました。

## 7.3 「捨てるか、残すか」は、工程ごとに答えが違った

もっとはっきり工程の理解を問われたのが、失敗した記事を「捨てるか、残すか」でした。

たとえば、内容が方針に触れて拒否される、という同じ原因の失敗があります。ある工程では、これは記事の本文そのものが拒否されたということなので、その記事は捨てるのが自然です。けれど別の工程では、すでに取り出した要約を分類しようとして拒否されただけで、要約そのものは残っています。だから、ここでは残すのが自然です。同じ原因なのに、捨てると残すに分かれる。

この違いは、各工程が何を作り、何を守っているのかを分かって初めて言えます。だから、失敗をどう扱うかを、原因から機械的に決めることはできませんでした。一時は、その「どうするか」を失敗の型の名前そのものに書き込んでいました。けれど名前が処理方針を語ると、肝心の「何が起きたか」を語る場所がなくなる。そこで名前からは処理方針を外し、原因を語らせるようにしました(#741)。

ここで一つ、軸が定まりました。コードが分かれ道で見るべきなのは「次にどうするか」(再試行・捨てる・残す)で、「何が起きたか」(原因)は、後から読む人に伝えるために監査へ残す。この二つを別の軸として扱うことになったのも、もとをたどれば、同じ原因でも工程ごとに扱いが違うと理解したからでした。

## 7.4 だから、見つけた場所が定義する

失敗を定義するのに工程の理解が要るなら、定義は、その理解がある場所に置くしかありません。だから、失敗を見つけた場所が、その失敗に名前を付けることにしました。記事が条件を満たさないという失敗は、その条件を持つドメインが。外部から取れないという失敗は、取りに行った層が。AI の応答がおかしいという失敗は、応答の中身を点検した側と、接続を扱う側が、それぞれ自分の見えた違反に名前を付ける。どの工程のことも分からない一つの万能分類器に集めて判定させることは、しませんでした。

外から来る失敗は、最初は細かく型に分けていました。繋がらない失敗、読めない失敗、と別々の入れ物を作って。けれど、元のエラーが既に「自分は何の失敗か」を語っていて、私が作った型は、それを写しているだけでした。だから入れ物は一つに畳み、元のエラーを抱えたまま、それ自身に語らせました(#689)。あわせて、外部や AI だけの事情——通信の番号、業者ごとの言い回し——は、工程の奥まで持ち込むと振り回されるので、入口で一度、自分たちの工程の言葉へ翻訳してから中へ通しました。

AI の失敗には、もう一つ特徴がありました。どの工程で呼んでも、出てくる失敗は同じ顔をしている。回数制限、認証、応答切れ。だから工程ごとに定義し直すのではなく、「立ち直り方」——待てば直る、人が直すしかない、この記事は諦める——で一度括り、細かい事情は調査用に別に持たせました。失敗を定義する場所を、それを一番分かっている場所へ寄せていく。それが、ばらばらに見えた整理に通っていた、一つの筋でした。

## 7.5 別々に定義して、記録の一枚でだけ揃える

こうして、ドメインの失敗、外部取得の失敗、AI の失敗は、それぞれ別の工程の理解から定義されました。だから、無理に一本の大きな型の系統へまとめることはしませんでした。三つは、出どころごとに別々のままです。

合流するのは、たった一箇所——監査の台帳です。そこで各失敗が、自分の「種類・もう一度試せるか・記事を消すか・理由」を、同じ書式に差し出す。記録する側は、どの工程のことも分かっていないのだから、もう一度判定し直すことはせず、各失敗が差し出したものを、そのまま転記するだけにしました。理解している者が定義し、記録する者は転記する。役割を分けたのです。

## 7.6 定義を間違えたとき、それは故障を隠した

工程を分からないまま定義すると、どうなるか。それも、はっきり出ました。

あるとき、AI が返す分類名が、こちらの想定の一覧と食い違う失敗を、「想定内の、よくある停止」として並べていました。けれど実際には、これは一覧とデータがずれているという、こちら側のバグでした。バグを「想定内」と定義していたために、直すべき故障が、平常運転の記録に紛れていたのです。これは、起動時に気づいて即座に止める形へ変え、想定外として扱うようにしました(#736)。

もう一つ。再試行の待ち時間を決める表を、すっきり一つの形に畳んだとき、その表が、混雑を知らせる応答に含まれる「これだけ待ってほしい」という指示を、黙って捨てていたことが分かりました(#758)。きれいに見えていた定義が、実は故障を黙らせていた。畳んで定義し直した瞬間に、隠れていたものが出てきたのです。

誤った定義は、理解が足りていないサインであると同時に、失敗を見えなくします。だから、定義は、外から覗いて確かめられる形にしておく。整理とは、隠さないことでもありました。

## 7.7 第7幕の終わりに

振り返ると、難しかったのは、型をいくつ作るかではありませんでした。「ここでの、この失敗とは何か」を言えるかどうか。そしてそれは、その工程を本当に分かって初めて言える、ということでした。

失敗の性質と、工程の事情と、実行の結果を分ける。続けられるかで、値と例外を分ける。原因は伝えるために、処理方針は分岐のために、別の軸に置く。出どころごとに定義して、記録の上でだけ揃える。どれも最初から分かっていたわけではなく、定義しようとして詰まり、その詰まりが「まだ工程を分かっていない」と教えてくれて、理解してから定義し直す——その繰り返しでした。

エラーを定義する作業は、結果として、一つひとつの工程を自分の言葉で理解し直す装置になりました。第6幕では「監査が書きづらい場所には設計の歪みがある」と気づきましたが、第7幕で分かったのは、「失敗がうまく定義できない場所には、まだ分かっていない工程がある」ということでした。次の第8幕では、こうして設計を支えてきた道具——テスト、ドキュメント、そして AI——の側に、同じ「分かってから使う」という規律を向けます。
