# 第5幕 — 監査が、責任分離を本物にする

第4幕で、価値の中心が AI 分析だと分かり、そこへ設計を投資し始めました。けれど、作ったものを実際に動かしてみると、別の壁にぶつかりました。「いま、どこで、何が起きているのか」が分からないのです。記事が分析まで進まない。けれど、取得で落ちたのか、本文補完で落ちたのか、AI 呼び出しで落ちたのか——どこで止まったのかが見えない。直すべき場所が、見えませんでした。

第5幕は、その「見えなさ」を埋めようとして監査(pipeline_events)を入れ始め、そこから責任の分け方を本気で考えるようになっていく話です。書き始める前の私は、責任分離は「ある程度はできているつもり」でした。ところが監査を書こうとした途端、それが本物でなかったことが次々に露わになります。そして、その過程で一つの線引きを得ました——**監査は探針(probe)であって、設計者ではない**。監査は歪みを照らし出すけれど、設計の答えはいつもドメインの側から出す。この章で扱う監査基盤は、振り返ると、私が初めて「AI ではなく自分で」設計を主導したものでもありました。

## 5.1 ログは、二度失敗していた

監査をゼロから考えたわけではありません。「失敗を記録に残したい」という気持ちは、実は4月の時点で一度立ち止まって、あえて見送っていました。当時のトランザクション境界の設計メモ([ADR-004](../adr/004_unit_of_work_service_convention.md) の母体)に、こう書いています——「失敗記録を DB に残したいというニーズは rollback と相性が悪い。当面はアプリログで代替し、本当に必要になったら別セッション・別トランザクションの専用仕組みを用意する」。必要性は感じつつ、まだ具体像が見えないので先取りしない、という判断でした。

そして、素朴なログ(`fetch_logs`)は、二度失敗していました。一つは、**書いたのに誰も読まなかった**こと。後の監査 ADR でも「`fetch_logs` テーブルは現状読み手が居ない(API / UI / agent / 診断 SQL いずれも参照ゼロ)」と確認しています(`11f03c29`, 2026-05-04)。もう一つは、**運用ログを業務状態に流用した**ことでした。HN の増分取得カーソルを `fetch_logs` の `MAX(fetched_at)` から算出していたのを、専用の置き場へ移しています——「FetchLog is an ops log and should not drive business logic」(`6c68d7ec`, 2026-04-21)。記録のための場所を、処理を駆動する状態に使ってしまっていた。この二度の失敗が、監査を「読み手は誰か」「履歴と状態は分けるべきだ」という二点から設計し直す動機になりました。

監査 ADR の最小目標は、いま読み返しても自分の出発点をよく表しています——「過去には Stage 2 が17時間停止していた事実をリアルタイムに検知できず、再現にも数時間を要した」「**3 ヶ月後の自分が、何が起きていたかを SQL 1 本で再構成できる**ことを最小目標とする」。非同期パイプラインの怖さは、HTTP と違って**失敗が黙って消える**ことにあります。worker の失敗は、ユーザーに 500 を返してくれません。

ひとつ正直に書いておくと、この移行は今も完了していません。失敗と変換棄却は pipeline_events へ移しましたが、成功イベントの記録はまだ途中で、`fetch_logs` も実測を確かめてから畳む予定のままです。「監査を本物にする」は、達成済みの過去形ではなく、いまも続いている現在進行です。

## 5.2 監査を書こうとして、責任の混線が見えた

監査を書き始めて最初に手が止まったのは、「うまく書けない」という感覚でした。当時の記録に、その違和感がそのまま残っています——「Service のメソッドが『永続化する/監査テーブルに記録する』のようにシンプルに書けないなら、それは何か無理な設計になっているのではないか」(2026-05-11)。これは後から振り返ると、「シンプルに書けないなら、設計のどこかが歪んでいる」という、設計の臭いを嗅ぎ取るセンサーでした。

このセンサーが、まず収集の2段目で響きました。監査が無かった Stage 2 に監査を入れようとした(`1e869b1d`, #369, 2026-05-05)ところ、「そもそも再試行すべきか否かを区別する仕組みが無い」ことに気づきます。翌日、そこから芋づる式に歪みが言語化されました(`508d843e`, #370, 2026-05-06)——URL を見た事実と取得状態が同じテーブルに同居しているために「永続失敗の理由を sparse カラムとして `discovered_articles` に持たせる悪臭」が出ている、と。そして結論はこうでした——「**監査基盤を正しく設計するには、まず先にデータモデルを正常化する必要がある**」。監査を真っ当に書こうとしただけなのに、その手前にあったデータモデルの責務混在が、先に照らし出されたのです。

似たことは、失敗の分類でも起きました。当初は retry の状態で失敗の種類(category)を決めようとしていたのですが、これだと監査が嘘をつきます。設計メモに「決定的だったのは、**アラートが事実と矛盾して黙る**問題」と書いています——直らない例外(API キー欠落など)が、attempt 1〜2 では「retryable」と焼かれ、最終 attempt まで誰も気づけない。「直らないのに直る扱い」という、ラベルの嘘でした。監査に正しい事実を焼こうとすると、失敗の分類のしかた自体を作り直す必要が出てきた、ということです。

戻り値の設計も同じでした。Service の戻り値に失敗カウンタや内訳といった観測値が混ざっていたのを、監査 payload 側へ寄せています(`25b117c5`, #353, 2026-05-04)。Service が「次の段階に渡す価値があるもの」だけを返すように痩せたのは、監査が「観測値はこっちが持つ」と引き取ったからでした。

## 5.3 監査は、観察者であって設計者ではない

混線が次々に出てくるなかで、監査と業務コードの関係について一つの向きを定めました。当時の言葉では「**監査が Service の入出力を観察する**」、逆に「Service のシグネチャに監査用の引数が現れたら、横断的関心事がビジネスロジックを侵食している兆候」です(2026-05-11)。監査が業務に依存するのではなく、業務の入出力を監査が外から観察する。依存の向きを、はっきり片側に倒しました。

失敗の扱いでも、同じ向きを引いています——「ビジネスロジックが分岐すべきは**処理方針**(retryable / drop / keep)であり、原因(provider 由来か stage 固有か)は監査・調査のための**従属軸**」(2026-05-08)。処理方針が主で、原因は従。原因を細かく知りたいのは監査の都合であって、その都合で業務の分岐を膨らませない、という線引きでした。

この向きを、後から自分なりに「監査は probe(露呈させる)であり witness(事実を残す)であって、設計者ではない。設計の答えは常にドメインから」と言葉にしました。さきほどの #369 → #370 が、まさにその実例です。監査(探針)が「データモデルが歪んでいる」と照らし出した。けれど、それを解いたのはドメイン側の正規化(取得対象の台帳・完成結果・作業領域への分割)であって、監査ではありません。「監査を正しく設計するにはデータモデルを直す必要がある」は、一見すると「監査の都合でドメインを変えた」ように読めますが、実際は逆で、**監査という探針がドメインの歪みを先に照らし、ドメイン自身の規律でそれを正した**、ということでした。

## 5.4 監査は、事実だけを記録する

向きを定めると、次は「では監査は何を焼くべきか」でした。ここでも、出発点は当時の自分の問いです——「監査というのは、ありのままの事実を記録すること。後から追跡できるようにと id などを無理に焼こうとするのは、間違いではないか。焼き付けの段階で、後で使うときのことを考えるのは違うのではないか」(2026-05-11)。

この問いで、監査の payload が「証人(witness)」ではなく「鏡(mirror)」になっていたことに気づきました。実際、当時の payload のコメントは、AI 分析の結果に紐づく id を「成功時の永続化結果ミラー」と自称していました。設計者である自分が、役割の二重化を白状していたわけです。採番された主キーや、その時点でたまたま FK が指していた id は、「AI が応答した」という事象の事実ではなく、事後の操作的な副産物です。そこで、それらを payload から落とし、Service が監査のために値を運ぶだけの「郵便配達人」になっていた経路も解消しました。

この「証人は他のテーブルを辿らない」という形は、別の場所でも効いてきました。監査に必要な値を Repository が2ホップの逆引きで集めていたのを、処理開始時に必要な値を全部そろえて運ぶ「厚い」前提条件の型(Ready)に持たせることで、逆引きそのものを消しています([stage5-ready-thick-pattern](../../specs/backlog/stage5-ready-thick-pattern.md) ほか、2026-05-12)。証人が事実を残すために他所を探し回る、という構造自体を、型に値を載せることで無くした、という整理でした。

監査の哲学は、二つの誘惑を断ることでもありました([ADR-008](../adr/008_pipeline_events_audit.md))——一つは「業務テーブルに reason 列を生やす」誘惑(状態テーブルが時系列の事実を兼ねると、最新状態しか残らず『何度失敗したか』が消える)、もう一つは「監査を制御状態テーブルとして使う」誘惑(監査が制御に使われると、書き換え圧力がかかって immutable でなくなる)。この「監査を制御に使わない」は、実は5月に突然得たものではなく、4月に「FetchLog を業務ロジックから参照させない」と決めたところから続く、同じ規律の延長でした。後の救済機構の設計でも、「監査を read して制御フローを分岐させるのは witness 哲学のグレーゾーン」として退け、分類の効果は読むときではなく書くときに置く方式を選んでいます。

向きとしては、二方向あります。**失敗を隠さない**こと。そして、その裏返しで**非失敗を失敗に偽装しない**ことです。収集対象外のデータを変換失敗として扱うと、失敗監査を非失敗で汚してしまう——「失敗を隠さないのと同様、非失敗を失敗に偽装しない」と書いています。対象外かどうかの宣言は、写像でも変換失敗でもない第三の責務として、別に扱いました。

そして、何を焼くかは「読み手がいるか」で決める、という規律に落ち着いていきました。読み手のいない `fetch_logs` は退役方針にし(`dfe61932`, #637)、どの consumer も使わない値(分析側で参照されない source 名など)は payload から外し(`b209c562`, #683)、件数のような集計値は監査ではなく metric の領分だと切り分けています(`2206ed8d`, #776)。この監査と Logfire の役割分担は、別途明文化しました——「『source A の parse_failed は何件か』は監査、『P99 レイテンシは』『いまどこで詰まっているか』は Logfire、例外は両方」([logfire-integration](../../specs/logfire-integration.md))。監査は which / why、metric は rate / trend、という分担です。さらに後には、それぞれの型が「監査に焼く情報を自分で持ち、監査は転写するだけ」という形まで進めました(`b97787e6`, #682 ほか)。最終的に、この監査は実際の読み手も得ています——管理画面のソース健全性表示は、すべて pipeline_events を集計源にしています(payload の詳細や生のエラーメッセージは外に出さない、という線も引いた上で)。

## 5.5 入れて、撤去して、建て直した

監査の歩みは、一直線ではありませんでした。一度入れた成功側の監査を、「中途半端な構造」としてまるごと撤去したことがあります(`68038d1f` / `f9eff594`, #515/#516, 2026-05-14)。コミットには「成功側 audit は中途半端な構造として今回まるごと外し、後続で proper な audit subsystem を設計し直して再導入する」「Service の主語を『補完する』に縮めた」と書いています。中途半端なまま置いておくより、一度外して設計し直す、という判断でした。その後、失敗パスを3層構造で建て直し(`7b916ef0` / `5b325b04`, #534/#535, 2026-05-16)、成功の証跡もきちんとした形で入れ直しています(`377fbbd5` / `e8fd8978`, #621/#624, 2026-05-25)。

この「入れて、撤去して、建て直す」は、この記録全体で繰り返し出てくる、あの問いの現れでした——「ここに監査を焼く価値があるか」。監査だから無条件に増やす、のではありません。読み手のいない値は焼かないし、中途半端な監査はいったん畳む。先ほどの「件数は metric へ移す」(#776)も、一度作った監査から「監査の固有価値(消えた失敗の why を SQL で再構成する)」に寄与しない値を剥がした例です。第2幕や第3幕で型や抽象に向けた「投資する価値があるか」という問いを、ここでは監査そのものに向けていた、ということになります。

## 5.6 第5幕の終わりに

この章で監査がしてくれたのは、責任分離を「本物にする」ことでした。書き始める前は、分離は「したつもり」でした。けれど、監査を真っ当に書こうとした瞬間に、データモデルの責務混在も、失敗分類の嘘ラベルも、Service に漏れていた観測の関心も、payload の鏡化も、次々に歪みとして表に出てきました。「シンプルに書けないなら設計が歪んでいる」というセンサーが、分離の甘いところを片端から照らしてくれた、という感覚です。

ただ、監査はあくまで探針であって、設計者ではありませんでした。歪みを照らしはしても、解いたのはいつもドメインの側です。そして、この「読み手がいるかで残すものを決める」という規律は、後から見ると監査だけの話ではありませんでした——同じ判断軸を、DB のカラムを足すかどうかや、監視を作るかどうかにも向けていました。監査で得た規律が、設計全体の物差しの一つになっていった、ということだと思います。

最後に、個人的なことを一つ。この監査基盤は、振り返ると、私が初めて「AI ではなく自分で」設計を主導したものでした。当時のメモにも「監査基盤を初めて自分で設計を始められた」「はじめて自分の意思で設計をしたのかもしれない」(2026-05-09 / 11)と残っています。AI は批評と裏取りと命名の相手で、判断は自分が握っていた。プロローグで書いた「最初は、ただ承認していただけ」から、ここでようやく一歩、抜け出せた時期でした。だからこそ、誇張せずに書いておきます——監査は焼けた事実が正しいことは保証しても、焼き漏らしがゼロであることまでは保証しません。本番に出してからの運用数値も、まだ手元にありません。ここで主張できるのは、計測の成果ではなく、設計の判断です。

監査が照らし出した責務の混線——とりわけ収集まわりの Fetcher が抱え込んでいた仕事は、まだ言葉にして見えるようになっただけで、作り替えてはいません。次の第6幕は、その炙り出された責務を、ドメインの論理で一つずつ解き直していく話です。
