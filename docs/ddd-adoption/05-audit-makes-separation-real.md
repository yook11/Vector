# 第5幕 — 監査が、責任分離を本物にする

第4幕で価値の中心に手を入れ始めると、次に問題になったのは、パイプラインで何が起きているのかを後から追えないことでした。記事を取得してから分析まで完了していないとき、取得で落ちたのか、本文補完で止まったのか、
AI 呼び出しで失敗したのかが分からない。

第5幕は、その「見えなさ」を埋めようとして監査(pipeline_events)を入れ始めたことをきっかけに、責任の分け方を本気で考えるようになっていく話です。それまでも責任分離は「ある程度はできているつもり」でした。ところが監査を書こうとした途端、それが本物でなかったことが次々に露わになります。監査を素直に書けない場所には、決まって何かが混ざっていました——状態と理由、業務結果と観測値、原因と処理方針、永続化と監査記録。この章で扱う監査基盤は、振り返ると、私が初めて「AI ではなく自分で」設計をする、考えるということに取り組んだように思います。

## 5.1 記録はあったが、監査にはなっていなかった

監査の必要性は、5月になって突然出てきたものではありません。4月の時点でも、失敗した記録を DB に残したいという考えはありました。
また、取得処理には、`fetch_logs` という記録テーブルがありました。どのソースを取得したか、成功したか、何件取れたか、エラー内容や処理時間を残すテーブルです。

ただ、今振り返るとこれはかなり場当たり的な設計でした。何を後から知りたいのか？何を記録するべきなのか？が曖昧なまま、「とりあえず実行結果を残す」ために作っていたからです。
実際分かるのは、ソース単位の成功・失敗と件数くらいでした。失敗したとしても、ネットワーク接続で落ちたのか、接続はできたがレスポンスを解釈できなかったのか、取得した記事を保存する段階で落ちたのかを、追うことはできませんでした。

もう一つの問題は、前回いつ取得したかを記録テーブルから調べ、それをもとに次の取得時間を決めていたことです。
記録はあくまで「過去に何が起きたか」を残すもので、業務のロジックを算出するものではありません。

「記録を残すテーブル」はあっても、監査の役割は果たせていませんでした。過去に何が起きたかを、後から調べ直せる形になっていなかったからです。

必要だったのは、失敗したときに、どの工程で、何が起き、なぜ止まったのかを後から追えることでした。特に非同期 worker の失敗は、HTTP の 500 のようにユーザーへ返ってきません。どこかで止まっても、そのまま見えなくなります。

そこで、目標を置き直しました。3ヶ月後の自分が「何が起きていたか」を SQL 1本で再構成できること。そのために、取得・補完・AI 分析・埋め込みまで、パイプライン全体の出来事を残す `pipeline_events` の設計を開始しました。

## 5.2 監査を広げるほど、責任の混線が見えてきた

`pipeline_events` は、段ごとに少しずつ入れていきました。そして下流へ広げるたびに、別々の責任の混ざり方が、順に表へ出てきました。

最初に入れたのは Stage 1(取得)です(`25b117c5`, #353, 2026-05-04)。ここはまだ素直でした。Service の戻り値に失敗カウンタや内訳といった観測値が混ざっていたのを、監査の payload 側へ寄せています(戻り型の整理と観測値の活性化は `26b12c0a`, #367)。Service は「次の段階に渡す価値があるもの」だけを返すように痩せ、観測値は監査が「こちらで持つ」と引き取りました。**業務結果と観測値**を分けた、最初の一手です。けれど、段を下流へ広げるほど、もっと根の深い混線が出てきました。

二段目で、最初の大きな引っかかりが出ます。監査が無かった Stage 2(本文補完)に監査を入れようとした(`1e869b1d`, #369, 2026-05-05)ところ、「そもそも再試行すべき失敗と、もう諦めるべき失敗を区別する仕組みが無い」と気づきました。404 で死んだ URL も、cron のたびに再試行されていたのです。失敗を監査に残そうとして初めて、**状態**——もう一度試せるのか、ここで打ち切るのか——を表す術が無いことが見えました。

翌日、その根がデータモデルにあると言語化します(`508d843e`, #370, 2026-05-06)。「URL を見つけた事実」と「本文取得の作業状態」が同じテーブルに同居していたために、永続失敗の理由までそこへ、sparse なカラムとして持たせようとしていました。当時の spec は、これを「永続失敗の理由を sparse カラムとして `discovered_articles` に持たせる悪臭」と呼び、結論をこう書いています——「**監査基盤を正しく設計するには、まず先にデータモデルを正常化する必要がある**」。ここで、**状態は状態テーブルが持ち、理由は監査(`pipeline_events.payload`)に残す**、という分離へ進みました。取得対象の台帳・作業領域・完成記事へとテーブルを割ったのも、このときです。監査を真っ当に書こうとしただけなのに、その手前のデータモデルの責務混在が、先に照らし出された、ということでした。

AI 分析の側でも、同じ書きづらさが出ます(Stage 3 / 4 の監査統合、`cf65d2ee`, #417 ほか、2026-05-08)。Task には原因別の except が縦に並んでいたのですが、「次にどう扱うか」——再試行するのか、記事を捨てるのか、残すのか——という処理方針が、型として表れていませんでした。後から ADR にまとめた言葉では、決定的だったのは「アラートが事実と矛盾して黙る」問題です([ADR-008](../adr/008_pipeline_events_audit.md))。retry の状態で失敗の種類を決めていると、直らない例外(API キー欠落など)が、最初の数回は「retryable」と焼かれ、最終試行まで誰も気づけない。ここで見えたのは、**原因と処理方針**が混ざっている、という気づきまででした。処理方針を主軸に分岐させ、原因を従属軸として型と値で表し直すのは、もう少し先——第7幕で扱います。第5幕の段階では、監査を広げる過程でこの混線に気づいたことに、意味がありました。

そして、いちばん見えにくい混線が、Stage 4 で出ました(`9226458b`, #465, 2026-05-10〜11)。AI 分析の監査で、Service が、自分では使わない値(分類カテゴリの id など)を、監査へ渡すためだけに持ち回っていたのです。当時のメモの言い方では、Service が監査の「郵便配達人」になっていた。監査の payload は、その時点で起きた事実を残す証人ではなく、業務テーブルの状態を写す「鏡」になりかけていました。**永続化の結果と、監査の記録**が混ざっていた、ということです。

4つの場所に共通していたのは、同じ書きづらさでした。後から振り返ると、監査を素直に書けない場所には、決まって責任の混線がありました。「永続化して、監査に記録する」と一行で言えないとき、その裏では、状態と理由、業務結果と観測値、原因と処理方針、永続化と監査記録のどれかが混ざっていた。書きづらさそのものが、設計の歪みを嗅ぎ分けるセンサーになっていきました。

## 5.3 監査は歪みを照らすが、答えはドメインが出す

混線が次々に出てくるなかで、監査と業務コードの関係について、一つの向きを定めました。当時の言葉では「**監査が Service の入出力を観察する**」、逆に「Service のシグネチャに監査用の引数が現れたら、横断的関心事がビジネスロジックを侵食している兆候」です(2026-05-11)。監査が業務に依存するのではなく、業務の入出力を監査が外から観察する。依存の向きを、はっきり片側に倒しました。

この向きを、後から自分なりに「監査は probe(歪みを露呈させる)であり witness(事実を残す)であって、設計者ではない。設計の答えは常にドメインから」と言葉にしました。5.2 で並べた出来事は、どれもこの形をしていました。いちばん分かりやすいのは Stage 2 の #369 → #370 です。監査(探針)が「データモデルが歪んでいる」と照らし出した。けれど、それを解いたのは監査ではなく、ドメイン側の正規化(取得対象の台帳・作業領域・完成記事への分割)でした。「監査を正しく設計するにはデータモデルを直す必要がある」は、一見「監査の都合でドメインを変えた」ように読めますが、実際は逆で、探針がドメインの歪みを先に照らし、ドメイン自身の規律でそれを正した、ということです。原因より処理方針を主軸に分岐させる、という 5.2 の気づきも、同じ向き——監査・調査の都合(原因を細かく知りたい)で業務の分岐を膨らませない——の現れでした。

## 5.4 監査は、事実だけを記録する

向きを定めると、次は「では監査は何を焼くべきか」でした。ここでも、出発点は当時の自分の問いです——「監査というのは、ありのままの事実を記録すること。後から追跡できるようにと id などを無理に焼こうとするのは、間違いではないか。焼き付けの段階で、後で使うときのことを考えるのは違うのではないか」(2026-05-11)。

5.2 の最後にふれた、payload が「鏡」になりかけていた問題は、この問いで解けました。実際、当時の payload のコメントは、AI 分析の結果に紐づく id を「成功時の永続化結果ミラー」と自称していて、設計者である自分が役割の二重化を白状していたわけです。採番された主キーや、その時点でたまたま FK が指していた id は、「AI が応答した」という事象の事実ではなく、事後の操作的な副産物です。そこで、それらを payload から落とし、監査の payload を、業務の状態を写す鏡ではなく、その時刻に起きたことを残す証人に戻しました。

この「証人は他のテーブルを辿らない」という形は、別の場所でも効いてきました。監査に必要な値を Repository が2ホップの逆引きで集めていたのを、処理開始時に必要な値を全部そろえて運ぶ「厚い」前提条件の型(Ready)に持たせることで、逆引きそのものを消しています([stage5-ready-thick-pattern](../../specs/backlog/stage5-ready-thick-pattern.md) ほか、2026-05-12)。証人が事実を残すために他所を探し回る、という構造自体を、型に値を載せることで無くした、という整理でした。

監査の哲学は、二つの誘惑を断ることでもありました([ADR-008](../adr/008_pipeline_events_audit.md))——一つは「業務テーブルに reason 列を生やす」誘惑(状態テーブルが時系列の事実を兼ねると、最新状態しか残らず『何度失敗したか』が消える)、もう一つは「監査を制御状態テーブルとして使う」誘惑(監査が制御に使われると、書き換え圧力がかかって immutable でなくなる)。この「監査を制御に使わない」は、実は5月に突然得たものではなく、4月に「FetchLog を業務ロジックから参照させない」と決めたところから続く、同じ規律の延長でした。後の救済機構の設計でも、「監査を read して制御フローを分岐させるのは witness 哲学のグレーゾーン」として退け、分類の効果は読むときではなく書くときに置く方式を選んでいます。

向きとしては、二方向あります。**失敗を隠さない**こと。そして、その裏返しで**非失敗を失敗に偽装しない**ことです。収集対象外のデータを変換失敗として扱うと、失敗監査を非失敗で汚してしまう——「失敗を隠さないのと同様、非失敗を失敗に偽装しない」と書いています。対象外かどうかの宣言は、写像でも変換失敗でもない第三の責務として、別に扱いました。

そして、何を焼くかは「読み手がいるか」で決める、という規律に落ち着いていきました。これは少し後、運用しながら刈り込んでいった話です。読み手のいない `fetch_logs` は退役方針にし(`dfe61932`, #637)、どの consumer も使わない値(分析側で参照されない source 名など)は payload から外し(`b209c562`, #683)、件数のような集計値は監査ではなく metric の領分だと切り分けています(`2206ed8d`, #776)。この監査と Logfire の役割分担は、別途明文化しました——「『source A の parse_failed は何件か』は監査、『P99 レイテンシは』『いまどこで詰まっているか』は Logfire、例外は両方」([logfire-integration](../../specs/logfire-integration.md))。監査は which / why、metric は rate / trend、という分担です。さらに後には、それぞれの型が「監査に焼く情報を自分で持ち、監査は転写するだけ」という形まで進めました(`b97787e6`, #682 ほか)。最終的に、この監査は実際の読み手も得ています——管理画面のソース健全性表示は、すべて pipeline_events を集計源にしています(payload の詳細や生のエラーメッセージは外に出さない、という線も引いた上で)。

## 5.5 入れて、撤去して、建て直した

監査の歩みは、一直線ではありませんでした。一度入れた成功側の監査を、「中途半端な構造」としてまるごと撤去したことがあります(`68038d1f` / `f9eff594`, #515/#516, 2026-05-14)。コミットには「成功側 audit は中途半端な構造として今回まるごと外し、後続で proper な audit subsystem を設計し直して再導入する」「Service の主語を『補完する』に縮めた」と書いています。中途半端なまま置いておくより、一度外して設計し直す、という判断でした。その後、失敗パスを3層構造で建て直し(`7b916ef0` / `5b325b04`, #534/#535, 2026-05-16)、成功の証跡もきちんとした形で入れ直しています(`377fbbd5` / `e8fd8978`, #621/#624, 2026-05-25)。

この「入れて、撤去して、建て直す」は、この記録全体で繰り返し出てくる、あの問いの現れでした——「ここに監査を焼く価値があるか」。監査だから無条件に増やす、のではありません。読み手のいない値は焼かないし、中途半端な監査はいったん畳む。先ほどの「件数は metric へ移す」(#776)も、一度作った監査から「監査の固有価値(消えた失敗の why を SQL で再構成する)」に寄与しない値を剥がした例です。第2幕や第3幕で型や抽象に向けた「投資する価値があるか」という問いを、ここでは監査そのものに向けていた、ということになります。

## 5.6 第5幕の終わりに

この章で監査がしてくれたのは、責任分離を「本物にする」ことでした。書き始める前は、分離は「したつもり」でした。けれど、監査を真っ当に書こうとした瞬間に、データモデルの責務混在も、失敗分類の嘘ラベルも、Service に漏れていた観測の関心も、payload の鏡化も、次々に歪みとして表に出てきました。「シンプルに書けないなら設計が歪んでいる」というセンサーが、分離の甘いところを片端から照らしてくれた、という感覚です。

ただ、監査はあくまで探針であって、設計者ではありませんでした。歪みを照らしはしても、解いたのはいつもドメインの側です。そして、この「読み手がいるかで残すものを決める」という規律は、後から見ると監査だけの話ではありませんでした——同じ判断軸を、DB のカラムを足すかどうかや、監視を作るかどうかにも向けていました。監査で得た規律が、設計全体の物差しの一つになっていった、ということだと思います。

最後に、個人的なことを一つ。この監査基盤は、振り返ると、私が初めて「AI ではなく自分で」設計を主導したものでした。当時のメモにも「監査基盤を初めて自分で設計を始められた」「はじめて自分の意思で設計をしたのかもしれない」(2026-05-09 / 11)と残っています。AI は批評と裏取りと命名の相手で、判断は自分が握っていた。プロローグで書いた「最初は、ただ承認していただけ」から、ここでようやく一歩、抜け出せた時期でした。だからこそ、誇張せずに書いておきます——監査は焼けた事実が正しいことは保証しても、焼き漏らしがゼロであることまでは保証しません。本番に出してからの運用数値も、まだ手元にありません。ここで主張できるのは、計測の成果ではなく、設計の判断です。

監査が照らし出した責務の混線——とりわけ収集まわりの Fetcher が抱え込んでいた仕事は、まだ言葉にして見えるようになっただけで、作り替えてはいません。次の第6幕は、その炙り出された責務を、ドメインの論理で一つずつ解き直していく話です。
