# 第6幕 — 炙り出された責務を、ドメインモデルへ解き直す

第5幕で、監査という探針が、収集まわりの責務の混線を照らし出しました。Fetcher が取得と変換を抱え込んでいたこと、失敗が黙って捨てられていたこと、保存の形が歪んでいたこと——けれど、そこではまだ「言葉にして見えるようになった」だけで、作り替えてはいません。第6幕(2026年5月、第5幕と並行して走った時期)は、その炙り出された責務を、ドメインの論理で一つずつ解き直していく話です。

作り替えは、一直線ではありませんでした。型の粒度を作り直し、入れたばかりの中間概念を三日で畳み、同じ日の自分の判断すら条件付きで巻き戻す——という試行錯誤が続きます。そしてこの幕には、もう一つの主旋律があります。**何を解くか(問い)と、どういう状態を正解とするか(目標)を、AI ではなく自分が決めるようになった**ことです。第5幕でも触れた「はじめて自分の意思で設計をしたのかもしれない」(5/9)というメモは、監査基盤と並んで、この取得パイプラインの再設計を指していました。3月の「AI に作らせたものをそのまま使う」から、4月の「提案を論拠で評価する」を経て、5月に設計の主語が自分に移った——その移り変わりが、以下のエピソードの底に流れています。先に分担だけ書いておくと、ここで落ち着いたのは「**設計の主導権は自分、AI は批評・裏取り・命名の相手**」という形でした。

なお、この時期は語彙の統一や改名が並行して走っていたため、当時の名前と現在の名前がずれる箇所があります。逐語を引くときは、その都度ことわります。

## 6.1 型が「保証しているふり」をしていた — 厚い Ready

第3幕で立てた指針の一つに、「型 = データの形ではなく、operation の前提条件」というものがありました。「Stage X に進める条件」が `ReadyForX` という型に閉じ、Task や Service に散らない——そう書いて、各段階の入口に Ready 型を置いていました(4/28 の設計メモ、実装は #227 / #238 ほか)。当時の作り方は、上流の Task が下流のために Ready を構築して次段へ渡す形(メモでの呼び名は Pattern A')でした。

ところが 5/11 に再評価して、この型が**保証しているふりをしていただけ**だと気づきました。当時のメモは、保証の実体が無い理由を三点で挙げています。

1. 「`int` の nominal wrapper で中身に invariant が無い」——名前は付いているが、中身は ID 一個で、守っている不変条件が無い。
2. 「kiq enqueue → 実行までに DB 状態が変わる」ため、「Service は到着した Ready が表す precondition を真実とみなせず、DB fetch + None チェックで実質的に再検証している」。
3. 「precondition 検査が『`RuntimeError` fail-fast』に名前を変えただけで責務は消えていない」。

つまり、型を受け取った側が結局もう一度確かめている。これでは「型に入れた時点で前提条件が満たされている」という当初の狙いが成立していません。再評価の場で自分が書いた指摘が、問題の核でした——責務の主語が「**処理を開始するときに、処理ができることを保証する**」ではなく「上流が下流のために次に進めることを保証する」になっている。「『上流が下流のために』次に進めることを保証する構造 → 責務の主語が間違っている」とメモに残しています。

ここで Ready 型そのものを廃止する案(案2)まで検討しました。けれど却下しています——「ready のようにその処理に進めるという型は必要」だからです。型が要らないのではなく、構築の主語が逆だっただけでした。確定した形(案3)は、「Ready 型は**処理に必要な内容をすべて値として運ぶ厚い型**であり、**下流 Stage 自身が処理開始時に DB から内容を fetch して構築する**」。薄い Ready との本質的な差分は、構築のタイミングを処理開始時に固定したこと(取得から処理までの時間ずれをゼロにしたこと)にあります。実装は Stage5 から入れ(#466, `87d73c08`, 5/12)、収集側へも同じ作り(house style)として展開しました(#542, `564d6234`, 5/16。こちらは補完段固有の解き直しで、中間型 `ClaimedPendingHtmlArticle` を `ReadyForArticleCompletion` に統合し、Service を「Ready を完成させるだけ」に縮めています)。

「型に保証させる」という同じ規律は、Ready 以外の型にも広げました。本文や抽出結果のように「満たすべき条件がある値」では、その条件の知識が converter や `_parse_html` のような変換コードに二重に実装されていることがありました——本文の長さやタイトルの長さの閾値が、複数箇所に literal で散っている、という形です。これを、型の構築境界(smart constructor)に畳みました。Stage 1 では `AnalyzableArticle.try_build`(#572, `9f36e404`, 5/21)、Stage 2 では `AcquiredContent.try_create`(#591, `373bfc72`, 5/22)——どちらも「条件を満たさなければ構築できない」factory で、不変条件の SSoT を変換コードから型の側へ移しています。二つの段で対称に進めたのは、同じ「保証しているふり」を、両方で本物の型保証へ解き直したかったからでした。あわせて、述語の名前も実体に寄せています——たとえば「Stage1 に進めることを妨げる」という結果を語る名前から、`requires_html_completion`(何を要するか=理由を語る名前)へ。

後日談として、厚くした field の一部はさらに見直しました。#467 で監査の都合から足した `source_name` を、「読む consumer も存在しない」ことを確認して撤去しています(#683, `b209c562`, 5/30。監査がなぜそれを欲しがり、なぜ要らなくなったかは第5幕で書きました)。厚くするのも、厚くしたものを削るのも、同じ「いま要るか」の問いでした。

## 6.2 議論の単位が間違っていた — per-article から per-source へ

次に作り替えたのは、型の中身ではなく**議論の粒度そのもの**でした。

当時、本文の補完ロジックは「ソースが増えるたびに新しいフラグを足し、`complete_with_html` の中の分岐を増やす」形で書いていました。記事ごとに「HTML を取りに行くか」「どちらを正本にするか」を boolean や None で持たせていたわけです。5/17 の設計メモは、この置き方を「ソースが増えるたびに新フラグ＋ `complete_with_html` 内の分岐が線形に腐る。多源ニュースアグリゲータでパターン増加はほぼ確実なので、**これは投機ではなく既に発生している負債**」と書いています。第3幕で置いた歯止め(「変更の蓋然性が具体的に見えているか」で投資を決める)に照らして、ここは将来の先取りではなく、もう起きている負債だから手を入れる、という判断でした。

核心の気づきはこうです。「どのフィールドを fetch する必要があるか／どちらを正本とするかは、記事が下す判断ではなく**ソースの出自(provenance)で構造的に決まる**。…これは**ソース種別の能力(capability)**であって記事ごとの自由状態ではない。それを全記事インスタンスに booleans/None で複写するのは Minsky の『実態と食い違える per-instance optional』アンチパターン」。補完方針はソースの性質なのに、記事一件ごとに同じ方針を複写していた——議論の単位を間違えていた、というわけです。

そこで「補完ポリシーは per-source、取得事実は per-article。取れた事実は全部保存し、方針(profile)が利用可否を決める」(#544, `7a91be16`)へ作り替えました。いまのコードでは、方針は `ArticleCompletionPolicy`(frozen dataclass)としてソース側に ClassVar で持たせ、`title` / `body` / `published_at` ごとに「HTML を正本にする / HTML を優先する / 観測値を優先する」のルールを宣言します。記事の側(`AnalyzableArticle` / `ObservedArticle`)は素の属性だけを持ち、補完方針は一切持っていません。取れた事実は per-article に全部残し、それをどう使うかは per-source の方針が決める——責務の置き場が、ようやく出自の所在と一致しました(なお、ここで削除した `IncompleteArticle` は当時のドメインの値型です。後にこの名前は別レイヤーの ORM テーブル名として再利用されますが、別物です)。

のちに、この方針には「写像」の責務も持たせています(#594, `0455c34a`, 5/23)。観測値と HTML をフィールドごとのルールで束ねる `resolve` を方針側に置き、ただし完成記事そのものを組み立てる構築は、出口の契約型 `AnalyzableArticle` に残しました——方針が完成型まで作ってしまうと、依存の向きが倒立するからです。束ねが漏れなく全フィールドを埋めることは、写像が total であることを確かめる専用テストで担保しました(この「写像の totality をテストで証明する」やり方は、6.6 で収集スコープを扱うときにも出てきます)。

## 6.3 入れた中間概念を、三日で畳む — Adapter が消えるまで

この作り替えのなかで、入れてすぐに畳んだ抽象があります。`SourceAdapter` です。

5/15、全ソースの取得処理を統一する型基盤として導入しました。当時の PR 背景にはこう書いています——「現状 32 fetcher のうち 26 個が `passport_builder` 経由・残り 13 箇所が `ReadyForArticle` / `IncompleteArticle` を直接構築している。後続 Phase で全 fetcher を『Adapter が `FetchedArticle` を yield → 共通 `ArticleFetcher` が passport に変換』に統一するための型基盤を先行確立する」(#522, `910d1f15`)。段階置換(strangler)で 45 ソースを一括で新経路へ移し、旧 Fetcher を全削除するところまで、二日で走り切りました。

撤回の引き金は、その二日後でした。per-source の知識をどこに置くか調べ直すと、「共有基底 4 + 継承 13 の階層」が見えてきて、「rename 単独だと is-a 違反、正すには継承→委譲が不可分」だと分かりました(5/17, spec §4.1)。Adapter という名前を付け替えるだけでは済まず、継承で組んだ構造そのものが間違っていたわけです。

そこで「『Fetcher/Adapter 中心』から『Source 中心』へ転換する原子的切替」を行いました(#545, `5dc5f1a3`)。このとき `ArticleSource` は `SourceAdapter` Protocol の改名ではなく、新規の集約として作り直しています——「改名は責務(identity vs 取得)を曖昧化、新規集約が DDD 上正」と判断したからです。構造の移送は byte 不変で検証できる形に保ち、汎用化しすぎないために「33 standalone は machinery として残置(over-generalize しない)」としました。

そして翌日、Adapter という中間概念そのものを除去しました(#546, `dd6ede07`)。「P2 で `ArticleSource` 集約 + `adapter_factory` 遅延 callable を導入したが、**Adapter という中間概念**が残存していた」からです。判断の軸は担保の強さでした——「『profile を読むのに machinery を作る』経路が**構造的に不能**」になるよう、契約をクラス参照の構造保証へ(設計の約束から構造へ)昇格させた。このとき目指していたのは、自分の言葉で「**Adapter という中間概念が消えている**」という到達状態でした。誕生(5/15)から概念除去(5/18)まで、三日です。

ただし、これは全面破棄ではありません。ソース固有の解析(bespoke parser)は「個別 `XxxSource` クラスで保持(汎用化しない)」と決め、消すものと残すものを選り分けています。仕上げに、93 ファイルの docstring から Adapter の語彙と当時の設計史を一掃しました——「設計が流動的な段階で docstring に強い契約・設計史を書くと、コードを読んで設計を再考する際に過去の文章へ引っ張られる。docstring を『地図』ではなく『名札』に最小化する」(#557, `5af69005`)。

同じ「一度入れて作り直す」は、モジュールの切り方でも起きていました。collection BC を Aggregate 軸(article / incomplete_article)で分ける案を実装し、main まで入れています(#510, `77e2a0bb`, 5/14)。けれど二日後、その aggregate のパッケージを解体し、ドメインの型だけを集めた層と、工程ごとの操作を置く層へ作り直しました(#540, `e75bc635`, 5/16)。採らなかった案というより、一度採って、到達した後に畳んだ案でした。入れた抽象を価値の問いで測り直して畳む——「投資する価値があるか」という問いが、この幕では最も短いサイクルで回りました。

## 6.4 赤子を、産湯ごと流さない — engine の解体と回復

作り替えが進むと、今度は取得処理の名前と実体のズレが気になり始めました。きっかけは、姉妹にあたる本文補完(stage2)を「acquire → complete → persist」に純化したときに、違和感の正体が分かったことです(#598, `88251b35`, 5/23)。その spec 自身が「stage 1(収集)も同じ3工程パイプラインであり、それを覆う Fetcher が不要だと気づいた」「article_completion と同型の問題」と、姉妹リファクタを相互に参照しています。

そこで取得側を解体しました(#608, `7e6bfff6`, 5/24)。「`ArticleFetcher` が collect+convert を一つの runner に畳み込み、service から変換工程が消えていた」——「名前は『Fetcher』(取得)なのに実体は取得+変換 — **名が体の一部しか表していない**」状態だったからです。狙いは「service を読むだけで収集の全体フローが分かる」ことでした。

ところが、この解体は行きすぎていました。約四時間後、条件付きで巻き戻しています(#609, `d15efb4d`, 5/24)。spec の言葉では、先行リファクタは「次の二つを**同時に**実施した: 1. **convert を runner から切り離す** — 正しい。2. **engine そのものを消す** — 過剰だった」。本質は「**engine は要る。癒着していたのは convert だけ**」で、「解体は赤子(engine)を産湯(convert 癒着)ごと流した」と書いています。

ここで大事にしたのは、これがフリップフロップ(やっては戻すの繰り返し)ではないと示すことでした。「先行 spec を否定するのではなく、その一決定を**条件付きで巻き戻す**」とし、「返り値が `FetchedArticle` で convert を含まない ── この一点が『フリップフロップでない』構造的証拠」だと明記しています。同じ日の自分の判断でも、間違っていた一点だけを構造の根拠つきで戻す——問いを自分が握っていると、自分の決定すら測り直せる、ということだったと思います。

確定した形は、責務を三つに分けたものです。**Source は宣言**(`read` / `in_scope` / `select` / `map_entry` の四つの classmethod)、**Fetcher は汎用の engine**(`fetch_articles` がソースの宣言を駆動して `FetchedArticle` を yield する。変換は含まない)、**Reader は道具**(各ソースが取得時に呼ぶ I/O クライアントの束)。変換は別の関数 `convert_fetched_article` が担い、一件の `FetchedArticle` を「分析可能 / 観測どまり / 棄却」の三つに値で振り分けます(想定内は例外を投げない total な分類)。`map`(写像・失敗なし)と `convert`(漏斗・分類)という動詞を対比させ、どこが素通りでどこが絞り込みかを名前から読めるようにしています。複数フィードを束ねるソース(NASA など)では、フィードごとの失敗を隔離する取得を `MultiFeedRssReader` という道具に切り出し、横断的な重複排除は `select` へ移しました——素朴に書くと、一つのフィードの失敗でソース全体が落ちてしまうからです。いまのコードでも、engine は各ソースの `in_scope` で絞って `map_entry` で写すだけのループになっていて、ソースごとの違いは各 `XxxSource` の宣言に閉じています。

採らなかった案も残しておきます。ソースごとの差異(Reader の異種性)を engine の中の巨大な `match` に集約する god-dispatch 案は却下しました——「異種性は各 source に局在させるのが正」だからです。共通の engine は薄く保ち、違いは各ソースの宣言に閉じ込める、という線引きでした。

## 6.5 保存と identity を、ドメインの語に正規化する

監査を真っ当に書こうとして第5幕で露わになった歪みの一つに、保存の責務混在がありました。「URL を見たという事実」と「本文取得の状態」が同じテーブルに同居し、永続失敗の理由を sparse なカラムとして抱え込んでいた——その混線を、保存ドメインの作り替えとして解いたのが、この節です。

まず保存を正規化しました(#370, `508d843e`, 5/6)。指導原則は第5幕で得た監査哲学のドメイン側への翻訳でした——「理由は監査、状態は state table」。永続失敗の reason を状態テーブルの sparse 列に持たせるのをやめ、理由は監査(`pipeline_events.payload`)へ、状態テーブルはそのときの状態だけを表す、と分けています。保存の形そのものも、一度の答えで決まったわけではありません。記事を分ける形を一度実装し、のちに覆して別の形へ作り直しています。「行が存在すること自体を『分析可能』の担保にする」という核は引き継ぎ、テーブルの割り方という形のほうは捨てました(現在の最終的なテーブル構成はその後さらに変わっているので、ここでは当時の正規化として書きます)。

次に identity の倒立を正しました(#567, `009287c4`, 5/20)。本文取得待ちの記事が「自分がどのソース由来か」を、表面のカラムではなく JSONB の中に押し込めていて、代わりに表層には内部 ID だけが乗っている——という倒立した形でした。これを、`source_name` を表層の NOT NULL 列に昇格し、`(source_id, source_name)` の複合 FK で `news_sources(id, name)` を指す形にしています(いまの ORM でも、この複合 FK が `RESTRICT` で張られています)。三つの層(DB 列・JSONB・registry)に散っていた drift を、一箇所に畳んだわけです。

同じ PR で、中間翻訳層も畳んでいます。`CompletionProfileResolver` は、当時の設計メモの言葉では「domain service を装った 3 層(DB 列 / JSONB / registry)翻訳の adapter」で、「補完 policy をどう決めるか」という domain decision は何もしていませんでした。これを解体し、呼び出し元を「ソース定義から方針を直接読む」一行へ縮めています。第5幕で、監査が Service を「郵便配達人」にしていた——本来の仕事をせず値を運ぶだけになっていた——のと同型の歪みが、保存側にもあった、ということです。

一つ、隠していたものを表に出した判断も残しておきます。registry に未登録のソースが来たとき、当時は既定の方針(`DEFAULT_PROFILE`)へ fall back する形になっていました。これは registry と `news_sources` の食い違い(drift)を黙って覆い隠す経路です。これをやめ、未登録なら `KeyError` で即座に止まるようにしました。安全網のように見えて、実は故障を隠す経路だったものを、drift を可視化する信号へ変えた——第5幕で書いた「故障を隠さない」を、保存側でもう一度やった格好です。

## 6.6 対象外は、失敗ではない — 収集スコープを第三の責務として析出する

第5幕で、もう一つ概念だけ予告したことがありました。「対象外かどうかの宣言は、写像でも変換失敗でもない第三の責務として別に扱う」。この節は、その第三の責務を実際に**ソースの述語として析出していく**過程です。通底する原理は、failure-visibility の裏返しでした——「**失敗を隠さないのと同様、非失敗を失敗に偽装しない**」。対象外の記事を変換失敗(棄却)として落とすと、失敗の監査を「本当は失敗していないもの」で汚してしまう。だから、対象外の宣言を失敗から切り離す必要がありました。

作業は、失敗の値化から始まりました(#556, `0af3867f`, 5/18)。それまで黙って `None` を返して捨てていた変換不能の entry を、理由付きの棄却(`AcquisitionConversionRejection`)という値に置き換え、監査に貫通させています。TechCrunch の空タイトルが「タイトル欠落」として初めて表に出たのも、このときでした。

ここで一つ、テストの作り方そのものに踏み込んでいます。各ソースの写像が漏れなく全件を写し切る(total である)ことを、どう確かめるか。ふつうの converter / fetcher の単体テストでは、これは**構造的に証明できません**。`FetchedArticle` を直接与えるテストは、各ソース固有の写像を通らないからです。そこで、ソースごとの継ぎ目(per-seam)で写像の totality を確かめる「発見オラクル」を先に立てました(#560, `16770a43`, 5/19)。production のコードを一行も変えずに、現行コードへ走らせて赤(失敗)を出す——「テストを実装の後に書く」のではなく、「テストを問題発見の道具にする」順序です。実際に五件の赤が出て、それを直していきました。面白かったのは、スコープ述語が見ていないフィールドを意図的に壊した標本(たとえば Crossref の述語は type・license・本文・公開日は見るが DOI は見ない——だから DOI が欠けていてもスコープは通る)で写像の穴をあぶり出せたことです。**責務の分け方の根拠を、DDD の綺麗さではなく、テストで確かめられるか(発見オラクルが空虚にならないか)から導いた**——この幕で得た、もう一つの手応えでした。

そのうえで、スコープの述語を「ソースの公開された判定」として名前を与えました(#561, `76800900`, 5/19。たとえば Meta AI の対象判定)。判定式そのものは一文字も変えず、挙動を保ったまま、隠れていた述語を表に出した格好です。残っていた沈黙の drop——The Register、Frontiers、複数フィードのソース——も、順に除去して各ソースの写像を total にしました(#558 / #562 / #564, 5/19〜20)。

対象外をどこで宣言するかには、もう一段の判断がありました。スコープが確定しているものは、ソースに述語として宣言する(MDPI の CC BY 4.0 のように、法的に対象だと確定しているもの)。けれど、まだ確信が持てないものは、ルールを急いで作らず、いったん通常の棄却経路に流して「どんな理由で何件落ちているか」を観察してから決める——実データで確かめてから線を引く、という使い分けです。なお、それぞれの棄却に**何を語らせるか**(理由のための型をどう設計するか)は、ここでは値にしたところまでに留め、次の第7幕で扱います。この節で確立したのは、失敗を値にしたこと、非失敗を失敗に偽装しないこと、そして「素通り(写像)」と「絞り込み(変換)」の境界をどこに引くか、までです。

## 6.7 構造が決まってから、名前が来る

最後は、語彙の収束です。作り替えで構造が固まったあとに、名前を合わせていきました。順序が逆でなかったこと——**語彙は構造の確定に従属させた**こと——が、振り返ると一つの規律になっていました。

stage1 のディレクトリは、一度 `source_fetch` から `article_collection` へ改名しています(#607)。本当は `acquisition` にしたかったのですが、当時その語は stage2 が使っていて衝突していました。このとき「`acquisition` の概念は stage2 のもの。名前は accurate なので維持」と据え置いたのですが、翌日それを覆します——「Html 限定子で非衝突とした過去の据え置き根拠は、限定子が…脱落するため成立していなかった」(#612)。stage2 から acquisition を剥がして scrape へ移すことで語が空き、ようやく統一できました。

確定した語彙はこうです(#613, `12000376`, 5/25)。stage1 の工程は `acquisition`(記事をシステムに獲得する工程)、stage2 の工程は `scrape`、その下で共用される低レベルの I/O 動詞が `fetch`(HTTP / transport / read。工程語彙ではなく、共用・凍結)。工程の語彙と、I/O の語彙を、二層に分けた格好です。この二層は、コミット本文に「`fetch` 系は基層として据え置く」という保持リストが具体的に残っていて、当時のメモだけでなく実装の記録としても確かめられます。改名の通底動機は、機構ではなくドメイン状態を名前に語らせることでした——たとえば「『HTML 取得待ち』という**機構**を語っており、『本文未取得の未完成記事が入っている』という**ドメイン状態**を表せていなかった」(#606)という直しです。

同じ「構造の確定に名前を従わせる」は、分析側でも起きていました。Stage 4 を、「分類された(Classified)」「却下(Rejection)」のような**行為や能動を含む名前**から、「対象範囲内(InScope)」「対象範囲外(OutOfScope)」という**観察された属性・状態**を表す名前へ寄せ、工程全体も「分類」から「評価(Assessment)」へ統一しました(#427 `54c4ecd5` / #428 `d90d24f7`, 5/9)。AI の判定を受け取って永続化しているだけなのに「却下」という能動の名前を付けていた——名前が実体より一段強いことを言っていたわけです。ここでもう一つ規律になったのは、**工程の語彙と永続化契約の語彙を分けた**ことでした。工程としての `assessment` は工程に残しつつ、DB のテーブルやレコードは「分析済み記事(analyzed_articles)」「対象外記事(out_of_scope_articles)」という記事の状態で呼ぶ。この境界別の語彙分離は、その後の一連の改名(#798〜#802)としてすべて反映済みです。

構造が決まる前に良い名前を急いで付けると、その名前に引っ張られて構造を見直しにくくなる——だから名前は最後に、というのが、この幕で身についた手順でした。名前を扱う理屈そのもの(「ドメインの言葉を使い、ストレージの言葉を避ける」「機械的な一貫性より意味的な正確さ」)は、4月のレビュー期に文献を足場に固めていたもので、この幕はその適用編にあたります。

## 6.8 第6幕の終わりに

ここまで作り替えてこられたのは、無防備に大手術をしたからではありません。45 ソースを一括で移すような変更は、段階置換(strangler)と、6.6 で触れた「発見オラクル先行」で安全網を張りました。普遍的な契約テストを先に書いて現行コードに走らせ、赤で問題を発見してから直す——production のコードを一行も変えずに赤を立てる回もありました。このテストとの付き合い方そのものを「道具を統制する」観点で正面から扱うのは、第7幕・第8幕に譲ります。

考え方の面で、この幕で変わったのは**設計の問いと目標を自分が握った**ことだったと思います。第3幕で「違和感を信じる」ようになり、その違和感を言葉にして設計を動かしてきましたが、ここではもう一歩進んで、目指す到達状態を「Adapter という中間概念が消えている」のように自分の言葉で持ち、AI の提案も自分の言葉で測り直すようになりました。早い時期のメモに「AI は構造的に正しい分解はできるが、なぜその部分が存在するかという目的を持たない」と書いていたのが、ここで「主導権は自分、AI は批評・裏取り・命名の相手」という分担に落ち着いています。到達点は「AI を使わない」ではなく、握る場所を変えた、ということでした。

同時に、隠さず書いておきたいことがあります。この幕の作り替えは、一度で正解に着いたものはほとんどありません。Adapter は三日で畳み、engine は同じ日に解体して回復し、保存の形は一度作って覆しました。当時のメモにも「設計を4回作り直して実用に耐える形に到達した」と残っていて、自分でも「経験者との差は『一発で決める精度』ではなく、『何回目で実用に耐える設計に着地できるか、問題点を発見する速度』だ」と書いています。まだ理解の途上だったことも含めて、巻き戻しを厭わずに測り直し続けた——それがこの幕の実際でした。

そしてこの作り替えのなかで、「失敗を表す型」がいくつも増えました。変換の棄却、観測どまり、品質不足——成功と失敗を別の型に分けるほど、構造は安全に見えます。第6幕では、失敗を値にし、非失敗を失敗に偽装せず、素通りと絞り込みの境界を引くところまでをやりました。けれど、その一つひとつの失敗の型に**何を語らせるか**——理由をどう型で表すか——には、まだ踏み込んでいません。次の第7幕は、その失敗の型を測り直し、「型を増やすほど安全」という、私自身の信念のほうを測り直す話です。
