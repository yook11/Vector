# 第6幕 — 炙り出された責務を、ドメインモデルへ解き直す

第5幕で監査を入れようとして見えたのは、処理・状態・例外・記録の責務がまだ混ざっているということでした。ただ、それはまだ「混ざっている」と気づいた段階にすぎません。

第6幕では一つの概念として扱っていたものを紐解いていきます。
中心になるのは、それまで AI の提案を受けながら作ってきた型や抽象を、「この型は何を保証するのか？」を問いながら、自分の理解で測り直すことでした。

なお、この時期は語彙の統一や改名が並行して走っていたため、当時の名前と現在の名前がずれる箇所があります。逐語を引くときは、その都度ことわります。

## 6.1 Ready を、「次へ進む合図」から「処理開始時点の保証」へ作り直す

第3幕で、私は「型はただのデータの形ではなく、処理を開始できる条件を表せるものだ」と考えるようになりました。その考えを、4月末にパイプラインへ広げようとしていました。Stage X に進める条件を `ReadyForX` という型に閉じ込めれば、その型を受け取った Service は、前提条件を確かめ直さずに処理へ進める。そう考えて、各工程の入口に Ready 型を置き始めました。

その考えをもとに、各工程の入口に Ready 型を置き始めました。記事を分類できる状態、埋め込みを生成できる状態、本文を抽出できる状態——それぞれの工程に進める条件を、型として表そうとしたのです。名前だけを見ると、パイプラインの各段階に「処理できる状態」が用意されているように見えました。

けれど、実際には Ready はまだ「前提条件の型」になっていませんでした。最初の Ready は、次の工程に渡す ID を包んだだけの型で、
一つ前の処理が終わったことを伝えるだけの意味のないものになっていました。

本来 Ready が保証すべきだったのは、実際にその処理を始める時点で、処理ごとの前提条件を満たしていることでした。まだ処理済みではないこと、処理に必要な値が揃っていることなど、そうした条件を満たして初めて、Ready と呼べるはずでした。

当時の私は、そこを十分に考えられていませんでした。型を置くこと自体に意識が向いていて、その型が何を、いつ、誰の責任で保証するのかまでは詰められていなかった。見ていたポイントが、「次へ進めることを型で表す」という漠然としたものになっていて、「その型を受け取れば、その処理の前提条件が満たされている」と言えるところまでは届いていませんでした。

作り直した後は、前の工程から Ready を渡すのをやめました。渡すのは ID だけです。次の工程は、その ID を受け取ってから、処理開始時点の DB 事実を読み直す。対象が存在し、まだ未処理で、必要な値が揃っていると確認できたときだけ Ready を構築する。

これで Ready は、「次へ進めるはず」という合図ではなくなりました。処理する側が、自分の開始条件を満たしていると確認した証明になったのです。Ready が作れなければ処理しない。Ready が作れたら、Service は状態確認に戻らず、その Ready を使って本来の処理に集中できる。

この形を後段の工程から入れ、同じ考え方を本文補完にも広げていきました。本文補完では、補完待ちの記事が本当に処理できる状態にあるかを Task が開始直前に確認し、Ready を作る。Service はキューや claim の状態を知らず、渡された記事を完成させる責務だけを持つようになりました。

ここで学んだのは、型に名前を付けても、それだけでは保証にはならないということです。`Ready` と呼ぶなら、その型が何を、いつ、誰の責任で保証しているのかが必要です。最初の Ready は、「次へ進む合図」を型で包んでいただけでした。作り直した後の Ready は、処理する側が、処理開始時点の DB 事実から構築する前提条件の証明になりました。

つまりこの節で作り直したのは、Ready 型そのものというより、保証の所在でした。前の工程が次の工程の状態まで保証するのではなく、各工程が自分の開始条件を自分で保証する。その境界を引き直したことで、Ready はようやく「処理できる状態」を表す型になりました。

## 6.2 記事を取得する工程は、ただ取ってくるだけではなかった

最初は、記事を取得する工程をとても単純に考えていました。まず記事の URL を集める。あとからその URL を開いて、本文や公開日時やタイトルを取りに行く。どのソースから来た記事でも、同じ流れで処理していました。

流れだけを見れば、分かりやすい設計です。手がかりとなる URL を集め、本文を取りに行き、揃ったものを分析へ渡す。けれど運用してみると、この単純さの裏で、ソースが最初から提供してくれている情報を捨てていることに気づきました。

ソースによって提供している情報は違います。本文まで揃っているものもあれば、タイトルと URL だけのものもある。公開日時は信頼できるけれど、本文だけは後から取りに行くべきものもある。それを全部、同じものとして扱っていたのです。

捨てていたのは、情報だけではありませんでした。すべてをページを開いて取りに行く形に寄せると、成功率も下がります。ページは構造が変わることもあれば、アクセスを制限されることも、うまく解析できないこともある。一方で、配信データや API が最初から正確な情報を持っているなら、それを使ったほうがいい。後から全部を取りに行って完成させるのではなく、最初に取れているものを活かす必要がありました。

ここで、見方が変わりました。記事を取得する工程は、分析の前に置かれた雑な前準備ではなく、分析に渡す材料の品質を決める境界だったのです。

そこで、まず「何が揃っていれば、そのまま分析に回してよいか」を先に決めました。タイトル、本文、公開日時、取得元、元記事の URL が揃っていて、本文の長さや URL の妥当性も満たしている。そこまで揃ったものを、分析に回してよい記事とする。この合格ラインを、取得元ごとの処理に散らすのではなく、出口の条件として一か所に集めました。

ただし、取得した一件一件は、単純に「成功／失敗」で判定できるものではありません。提供されたデータだけで、そのまま分析に進められる記事もあります。一方で、タイトルや公開日時などの基本情報は揃っているものの、本文は後続処理で取得する必要がある記事もあります。さらに追加で取得を試みても、分析対象にできないものもあります。
そこで各記事を、初期データだけで分析可能なもの、追加取得によって記事として完成するもの、追加取得しても分析対象にならないもの、の三つに分類しました。

変わったのは、コードの形だけではありません。記事を取得する工程を、分析の品質を支える工程として設計し直しました。取得元が最初から渡してくれるデータは捨てない。使えるものは使い、足りないところだけ後から補う。「取ってくる」だけに見えていた工程は、実は、取れた事実を見て扱いを決める工程だったのです。

## 6.3 ひとつの言葉が、三つの仕事を隠していた

最初は、追加取得によって記事を完成させることは単純な作業だと考えていました。「完成させる」というメソッド一行で済むように見えていました。

けれど、その一語の中で、様々なことが起きていました。まず、追加取得待ちの記事の URL を開いて、データを取りに行く。次に、最初に取れていたデータと合わせて、分析に回せる記事の形に組み立てる。最後に、それを保存して、追加取得待ちの記事ではなくなったことを表す。「完成させる」は、取りに行く・組み立てる・保存するという、三つの仕事を抱えた言葉だったのです。

この混ざりが一番表に出たのは、失敗処理を書こうとしたときでした。
同じ「記事が完成しなかった」という結果でも、失敗した段階は異なります。取得段階の失敗は、相手サーバーの停止、タイムアウト、HTML の解析不能などにより、必要な外部データを取り込めなかった失敗。一方、組み立て段階の失敗は、取得できた情報と保存済みの情報を合わせても、分析に回せる記事として成立しなかった失敗です。
ところが当時は、この区別がなく、失敗の扱い方も工程ごとに揃っていませんでした。ある失敗は戻り値として返り、例外として投げられているものもあったため、呼び出し側から見ると、どの工程がなぜ失敗したのかが分かりにくくなっていました。

そこで、「完成させる」という処理の中身を分解し、不足データを外部から取得する処理、取得済みデータと保存済みの事実から記事を組み立てる処理、完成した記事を保存する処理、の三つとして明確にしました。

三つの工程を名前で分けたことで、失敗もそれぞれの工程に結びつけて扱えるようになりました。取得に失敗した場合は、取得工程がその理由を返すというように、
失敗した場所と理由がずれないように、各工程が自分の結果として失敗理由を返す形にしました。

今回分かったのは、失敗処理の書きづらさは、責任が混ざっているサインになりうるということでした。「完成させる」という一つの言葉の中に、取得、組み立て、保存が混ざっていたため、どこで失敗し、誰がその理由を持つべきかも曖昧になっていました。工程を分け、それぞれが自分の失敗理由を返す形にしたことで、失敗の扱い方も自然に整理できました。

## 6.4 ソースごとの違いを扱う難しさ

記事を完成させる工程と同じように、外部ソースから記事候補を取り込む処理でも責任が混ざっていました。以前は、候補を取得することと、その候補を6章2で述べた三つの状態に振り分けることが同じ処理に入っていました。そこで、候補を集める処理と、候補の扱いを決める処理を分けました。

ただ、難しかったのはここからです。取得した記事候補をどう扱うべきかは、外部ソースごとの事情に左右されました。6.2で見たように、候補の時点でどこまで情報が揃っているかが、ソースごとに大きく違うからです。

しかも、その違いはこちらでコントロールできるものではありません。外部ソースが何を提供し、どんな形式で配信するかは相手側の都合に依存します。本文を含めていた配信が要約だけになることもあれば、HTML の構造が変わることもある。新しいソースを追加するたびに、ソースごとの事情に合わせた個別対応が増えていきます。
だから、ソースごとの違いに毎回その場で対応するのではなく、どこまでを共通の流れとして保ち、どこからをソースごとの性質として切り出すかを決める必要がありました。

何度も作り直しました。
はじめは、ソースごとの差を扱うために、一段薄いラッパーをかませていました。けれど、その層は、どの候補をどう扱うかを判断していたわけではありませんでした。実際の取得処理は各ソース側にあり、それを呼び出しているだけでした。不要だと判断して畳むと、Service からは「収集、変換、永続化」の流れが見えるようになりました。

ただ、その形では、「ソースがニュースを取得する」という不自然な振る舞いになってしまいました。ソースは本来、「この URL を読む」「この項目は信じてよい」「本文は後から補う」といった性質を持つものです。にもかかわらず、外部データを読み、対象を選び、重複を落とし、記事候補へ写すところまでを 持たせていたため、ソース自身が動いて記事を取りに行く主体のように見えていました。

この違和感を測り直すと、問題はソースごとの差そのものではなく、その差を「ソース自身の振る舞い」として持たせていたことだと分かりました。ソースごとに違うのは、どの URL を読むか、どの項目を信じるか、どれを対象にするか、本文を後から補うか、といった性質です。一方で、外部データを読み、対象を選び、記事候補へ写し、その後の扱いを決める流れまで、すべてソースの仕事にする必要はありません。
そこで、ソースには「このソースはこう読める」という宣言を持たせ、取得を進める流れは外に出しました。取得した候補を三つの状態に振り分ける判断も、ソースの中には戻しませんでした。ソースごとの差は宣言として残し、候補をどう扱うかは別の責務として分けたのです。

何度も動かしていたのは、コードそのものではなく、「ソースごとの判断を、どこに置くか」でした。どう読むかも、どの情報を信じて、どれを後から補うかも、記事一件ごとに毎回その場で決めることではありません。ソースの性質として、はじめから決まっている。だから、それらをソースの宣言として持たせ、共通の流れはその宣言に沿って動くだけにする。記事側には、実際に取れた事実をそのまま残す。こうして、ソースごとの違いは、条件分岐の山ではなく、ソースごとの宣言として積み上がるようになりました。

## 6.5 第6幕の終わりに

ここまで作り替えてこられたのは、無防備に大手術をしたからではありません。多くのソースを一気に移すような変更には、段階置換で安全網を張り、普遍的なテストを先に書いて現行のコードに走らせ、問題を見つけてから直す、という手順を踏みました。production のコードを一行も変えずに、テストだけで赤を立てて問題をあぶり出す回もありました。このテストとの付き合い方そのものを「道具を統制する」観点で正面から扱うのは、第7幕・第8幕に譲ります。

考え方の面でこの幕で変わったのは、設計の問いと目標を自分が握ったことだったと思います。第3幕で違和感を信じるようになり、その違和感を言葉にして設計を動かしてきましたが、ここではもう一歩進んで、目指す到達状態を「Adapter という中間概念が消えている」のように自分の言葉で持ち、AI の提案も自分の言葉で測り直すようになりました。早い時期のメモに「AI は構造的に正しい分解はできるが、なぜその部分が存在するかという目的を持たない」と書いていたのが、ここで「主導権は自分、AI は批評・裏取り・命名の相手」という分担に落ち着いています。到達点は「AI を使わない」ではなく、握る場所を変えた、ということでした。

同時に、隠さず書いておきます。この幕の作り替えは、一度で正解に着いたものはほとんどありません。Adapter は三日で畳み、取得を進める共通の流れは同じ日に消して戻し、保存の形は一度作って覆しました。当時のメモにも「設計を四回作り直して実用に耐える形に到達した」と残っていて、自分でも「経験者との差は、一発で決める精度ではなく、何回目で実用に耐える設計に着地できるか、問題点を発見する速度だ」と書いています。まだ理解の途上だったことも含めて、巻き戻しを厭わずに測り直し続けた——それがこの幕の実際でした。

名前は、いつも最後に来ました。構造が固まる前に良い名前を急いで付けると、その名前に引っ張られて構造を見直しにくくなる。だから構造を先に動かし、名前は構造の確定に従わせる。「本文取得待ち」という機構を語っていた名前を、「本文をまだ持たない未完成の記事が入っている」というドメインの状態を語る名前へ改めた(`f5d9dd5b`、#606)のも、その一例です。

結局この幕で分かったのは、ニュースを取得するというのは、外から来たデータをそのまま保存する工程ではなかった、ということです。それは、玉石混淆の材料を、分析に進める事実へ変換する境界でした。だから本文が取れるかどうかが境界の判断になり、取れた一件を三つに仕分ける必要があり、その仕分けの責務をどこに置くかが問題になった。第6幕は、その境界をドメインモデルとして引き直していく幕だったのだと思います。

そしてこの作り替えのなかで、失敗を表す型がいくつも増えました。変換の棄却、観測どまり、品質の不足——成功と失敗を別の型に分けるほど、構造は安全に見えます。第6幕では、失敗を値にし、失敗していないものを失敗のふりさせず、素通りと絞り込みの境界を引くところまでをやりました。けれど、その一つひとつの失敗の型に何を語らせるか——理由をどう型で表すか——には、まだ踏み込んでいません。次の第7幕は、その失敗の型を測り直し、「型を増やすほど安全」という、私自身の信念のほうを測り直す話です。
