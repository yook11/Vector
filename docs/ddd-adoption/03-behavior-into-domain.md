# 第3幕 — 値の次は、振る舞いをドメインに引き戻す

第2幕で、一つひとつの「値」は型に意味を持たせられました。けれど「振る舞い(ドメインのルール)」は、まだコードのあちこちに散らばったままでした。

当時の置き方は、フレームワークの既定にほぼ沿ったものでした。`routers/` と `models/`、せいぜい `services/` と `repositories/` があるくらいで、ロジックは router や service に手続き的に書いていました。この構成のままでよいのか、既定の枠に収めるべきか、それともドメインごとに切り出すべきか——正直、試行錯誤していました。第3幕は、第1幕のレビューの最終日が開いた問いから始まる、「振る舞いと構造を、ドメインの側に引き戻す」という転換の話です。

## 3.1 違和感 — ドメインのルールが Repository に貼り付いている

きっかけは小さな実装でした。ニュースソースの有効化 / 無効化(`activate` / `deactivate`)を書いていたとき、その処理が `Repository` のメソッドに収まっていることに引っかかりました。当時の設計メモにそのまま残しています——「`news_source` の activate/deactivate を実装するなかで、Repository メソッドが『ドメインルール(`is_active` を書き換える)』と『永続化(commit する)』を同一メソッドに抱えていることへの違和感が出発点となった」。

掘ってみると、これは置き場所の問題ではありませんでした。メモはこう続きます——「単なる命名や置き場所の問題ではなく、(1) Transaction Boundary をどこに置くか (2) Repository の責務をどこまで狭めるか (3) ドメイン状態遷移を Model に持たせるか(Rich Domain Model 化)、の3つが絡み合った構造的な論点」。

答えとして、状態遷移そのものを Model のメソッドに移しました(`bf885469`, 2026-04-12)。`is_active` を書き換えるのはエンティティ自身の振る舞いで、`Repository` はそれを永続化するだけ、トランザクションの commit は依存(FastAPI dependency)が一括で持つ——という分担です(この約束は [ADR-004](../adr/004_unit_of_work_service_convention.md) に記録しました)。`Repository` から commit / refresh と状態変更が剥がれ、手続きを束ねるだけだった Service が痩せていきました。振る舞いの無い anemic なモデル + 太った Service から、振る舞いをモデル自身が持つ rich domain への、最初の一歩です。

この議論でもう一つ、後々まで効く原則を置きました——「トランザクションの粒度は『何を原子操作とみなすか』で決まる。HTTP 経路では『1 ユーザー操作 = 1 原子操作』、Worker 経路では『1 パイプライン段階 = 1 原子操作』。両者は性質が異なるため、同じ設計で揃えない」。メモには「揃えることを美徳としない」とも書いています。同じに揃える心地よさより、文脈に合うことを優先する、という線引きでした。

## 3.2 ならば、モジュールの切り方もドメインに合わせる

振る舞いをモデルへ戻すと、今度は「そもそもコードをどんな単位で分けているのか」が気になりました。既定の構成は技術的な関心で水平に切る形——`services/` `repositories/` `ai/` `tasks/`——で、当時の設計メモはその不満をこう言語化しています。「技術軸で水平に切った結果、1つの業務フローを理解するために4つ以上のディレクトリを渡り歩く必要がある」「ビジネス重要度とコード構造が逆転している(価値の源泉である知的処理が `tasks/` の中に埋もれ、ただのバックグラウンド処理に見える)」。

そこで、モジュールの第一の分類軸を技術からドメインへ反転させました(`c9a7d735` ほか, 2026-04-15)。収集(`collection`)・分析(`analysis`)・検索(`search`)というドメインの意味で境界を引き、技術的な関心はその内側か、共有のインフラ層へ置く。設計原則として「ドメインプロセスでモジュールを切る — 技術的関心事ではなく、ドメインの意味で境界を引く」「コード構造はビジネス重要度を表現する」を据えました。「この枠に収めるか、切り出すか」と迷っていた試行錯誤に、〈ドメインという軸〉という答えが出た瞬間でした。

同時に、切り出しすぎないための歯止めも同じメモに書いています——「現状追従も将来先取りもしない。『変更の蓋然性が具体的に見えているか』で判断する」。たとえば AI 分析の内部をさらに細かく割る案は、「分割する蓋然性はあるが、具体的なモデル選定・実行順序が未定」として見送りました。ドメインで切る勢いのまま全部を刻むのではなく、いまの構造を根拠にもせず、見えていない将来を先取りもしない——これは「伝えたいこと」で書いた〈ここに投資する価値があるか〉という問いそのものでした。

## 3.3 ドメインの不変条件を、構造と型で強制する

振る舞いをモデルへ、モジュールをドメインへ移したら、次は「そのドメインが満たすべき条件を、どう守るか」です。第2幕で値に対してやったこと——不正な値を型として書けなくする——を、今度は構造と状態に広げていきました。

まず構造で。記事テーブルを `DiscoveredArticle`(収集しただけのメタデータ)と `Article`(本文が `NOT NULL`)の二つに割りました(`376c12ec`, 2026-04-20)。コミットの言葉では「articles 行が存在する = 分析可能(analysis-ready)という構造的不変条件を確立する」。「本文があるか」を実行時に確かめるのではなく、〈`articles` に行がある〉という事実そのものを不変条件にした——値の検証から、テーブルの形による保証への一歩です。

次に状態で。処理結果を「タグ用の列挙 + nullable」で表すのをやめ、判別可能な合併型(sum type)に割りました。本文取得の結果を `status: Literal[...]` + `article_id: int | None` から `Fetched | AlreadyExists | Skipped` に分け、`Fetched` は `article_id` を必ず持つようにしています(`526e4da4`, 2026-04-21。コミットの言葉では「the article_id invariant is structural rather than runtime」)。`(None, None, None)` のような「墓標」インスタンスも、成功と失敗を別の型に分けることで消しました(`aae9fc68`)。

そして「ありえない状態」を型で書けなくする、まで進めました。AI 分析では、対象範囲内を表す `InScope` の `category` が、対象外(`OUT_OF_SCOPE`)まで含む列挙を使っていて、「対象範囲内なのにカテゴリは対象外」という矛盾が型の上で許されていました。当時の spec も「`OUT_OF_SCOPE` も型上 valid(型の弱さ)」と認めています。これを、対象外を除いた `InScopeCategory` を新設して塞ぎ、「『対象範囲内なのに OUT_OF_SCOPE』という矛盾状態を型システムで排除する」ようにしました(`c165b04f` / #435, 2026-05-10)。分岐は `match` + `assert_never` で書き、union に枝を足したら網羅漏れが型チェックで弾かれるようにしています。

ここで一つ、意図して「型を要求しない」判断もしました。AI(Gemini / DeepSeek)には判別可能 union の形では返させず、平らな応答スキーマのまま受けて、コード側で振り分けています。理由は spec に明記したとおり「構造化出力で discriminated union を要求すると AI 精度が落ちるため」。型の安全さ(内側)と AI の精度(境界)を両立させるために、境界では平ら・内側でだけ union、という非対称をあえて選びました。

## 3.4 第3幕の終わりに

振り返ると、ここまでの三つの幕は、「不正なものを型・構造として書けなくする」という同じ一つの発想が、対象を広げていった時期でした——**値(VO)→ 振る舞い(モデルへ)→ モジュール境界(ドメイン軸)→ 構造(テーブル)→ 状態(union)**。判別可能 union や `assert_never` は、その一番外側で使った道具で、それ自体が目的だったわけではありません。

考え方の面では、この時期に変わったのは「違和感を信じる」ことだったと思います。第1幕のレビューで違和感を拾う目が育ち、この幕の転換はどれも、自分のコードを読んで引っかかった違和感を言葉にするところから始まっています。

ただ、ここまではまだ「新しいドメイン観のもとで、型と構造を整える」段階でした。次の第4幕では、その観点のもとで既存コードを大きく作り替えます——ドメインモデルの形と粒度を作り直し、入れたばかりの中間概念を3日で畳む。その試行錯誤のなかで、設計の問いと目標を AI ではなく自分が握るようになっていく過程です。
