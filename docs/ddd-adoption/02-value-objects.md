# 第2幕 — 値オブジェクトを入れる

時間は少し戻ります。第1幕のレビューがまだモデルと DB に居た、3月の話です。

## 入口はセキュリティでした

モデルと DB のレビューを進めながら、並行して読んでいた本のもう一つの系統がセキュリティでした。最初から DDD に憧れていたわけではありません。DDD への入口は、こちらから開きました。

最初に読んだのは、『体系的に学ぶ 安全な Web アプリケーションの作り方』(徳丸本)です。XSS、SQL インジェクション、オープンリダイレクト、SSRF——読み進めるほど、Web アプリケーションで対策しなければならないことの多さに驚きました。

当時の自分は、まず「危険な入力を受け取る場所で、バリデーションをたくさん書かなければいけない」と考えました。SQL インジェクションを防ぐ、URL を検証する、HTML に出す値を扱う、外部リクエスト先を制限する。けれどそう考えると、検証を書く場所はアプリケーション中に広がっていきます。どこか一箇所で書き忘れれば、そこが穴になります。

「すべての入口で漏れなく検証する」ことは現実的ではないのではないな。
そう感じていたときに読んだのが、『セキュア・バイ・デザイン』でした。

この本は、セキュリティを「あとから足す対策」ではなく、「正しいドメイン設計から自然に生まれるもの」として捉え直します。中心にあるのが domain primitive です。これは、構築の瞬間に不変条件を強制し、不正な状態をそもそも持てなくする小さな値の型で、値オブジェクト(Value Object)とほぼ同じものでした。

ビジネスロジックを型として正確に表せば、検証は値を作る一箇所に集まり、不正・危険な状態は「書けなく」なる。
散らばっていた `if` の検証が、型の境界へ畳み込まれていく。脆弱性が「気をつけて潰すもの」から、「そもそも作れないもの」へ近づく。この見方に、素直に感動しました。

早速自分のアプリケーションにもこの概念を取り入れることにしました、値オブジェクトの導入から始まります。

## 2.1 生の `str` には意味がない

最初に手をつけたのは、カテゴリの `slug` と `name` でした。当時はどちらも `str` として定義されており、slug が満たすべき形式も、表示名に許される文字も、型のどこにも書かれていませんでした。コードを読む側から見ると、この `str` が未検証の入力なのか、すでに安全な値なのかを型から判断できません。

そこで、最初の値オブジェクトとして `CategorySlug` と `CategoryName` を導入しました。
slug は URL や識別子として扱うため、小文字英数字とアンダースコアだけを許す値にしました。一方で name は画面に表示する名前なので、日本語や中黒、空白を許します。同じ `str` でも、使われ方が違えば満たすべき条件も違う。その違いを、別々の型として表すようにしました。

今振り返ると、これは少し手段が先に立った導入でもありました。それでも、この経験によって「どの値を VO にすべきか」を考えるものさしが生まれます。実装が面倒かどうかでも、VO を入れるとかっこいいかどうかでもなく、**その型を受け取ること自体に意味があるか**。この問いは、のちに型を増やすときにも、減らすときにも効いていきます。

## 2.2 ライブラリが設計の邪魔をする — SQLModel から SQLAlchemy へ

値オブジェクトを入れた時点では、なぜ実装がここまで重くなるのか、まだよく分かっていませんでした。slug や name に意味を持たせたいだけなのに、不変性、等値性、Pydantic との接続まで自分で抱え込む必要があり、1 VO あたり約 90 行、3 つで 270 行ほどのボイラープレートになっていました。

最初は、自分の実装力が足りないだけだと思っていました。けれど調べていくと、問題は書き方だけではなく、使っていた SQLModel の設計にもありました。SQLModel は、Pydantic の schema と SQLAlchemy の ORM model を一つのクラスで書ける手軽さがあります。ただ、その手軽さは、ドメインの値を独立した型として扱いたい場面では足かせになりました。ORM のカラムには `CategorySlug` のような VO をそのまま載せづらく、結局 `str` などの基本型に寄せるしかない。すると models 層では「これは検証済みのカテゴリ slug である」という型情報が失われ、service 層で手動変換を書くことになります。ドメインの値を型で表したいのに、永続化の都合で型が崩れてしまうことが、SQLModel との一番大きな噛み合わなさでした。

ここで初めて、ライブラリ選定の重みを意識しました。SQLModel は素早く書き始めるにはよくできています。ただ、自分がやりたかったのは「ドメインの値を型で表す」ことで、その目的に対しては SQLModel の手軽さがかえって足かせになっていました。**「速く書けるか」ではなく「自分が表現したいものを表現できるか」でライブラリを選ぶ**、という当たり前のことを、痛みを通して教わったように思います。

最初の VO からおよそ 5 日後、全モデルを SQLModel から SQLAlchemy 2.0 の Declarative(`DeclarativeBase`)へ移しました。移行後は、VO を Pydantic の `RootModel` として軽く定義し、SQLAlchemy 側で `str ↔ VO` の変換をまとめて扱えるようになりました。これにより、重いボイラープレートや service 層での手動変換を減らせました。

この経験から、ライブラリは「速く書けるか」だけでなく、「自分が表現したい設計を無理なく表現できるか」で選ぶ必要があると学びました。判断の経緯と具体は [docs/adr/value_objects_sqlalchemy_migration.md](../adr/value_objects_sqlalchemy_migration.md) に残しています。

## 2.3 セキュリティとドメインモデリングが重なるところ

この幕の冒頭で書いたセキュリティの動機が、最初にはっきりコードに出たのは、URL の値オブジェクト `SafeUrl` でした(`6f33d56a`, 2026-03-30)。

記事の元 URL は、それまで `Mapped[str]` でした。空文字でも、HTTP/HTTPS 以外でも、型の上では同じ文字列です。最初の `SafeUrl` がやったのは、ここに「URL として成立している値」という境界を置くことでした。空でないこと、長すぎないこと、HTTP/HTTPS の URL であることを、値を作る時点で検証する。あわせて ORM の保存・読み出しでは必ず `SafeUrl` を通し、DB 側にも最低限の CHECK 制約を置きました。まずは、URL の基本形を型・ORM 境界・DB 制約で重ねて守り始めた、という段階です。

ただし、この時点で SSRF まで防げていたわけではありません。実際に外部 URL を fetch する経路を見直す中で、内部ネットワークへ向いた URL や DNS 解決後の到達先も問題になると分かり、4/27 に `SafeUrl` と fetch 層へ SSRF 防御を追加しました。その後も、共通の安全な HTTP client を通す形にし、5/9 には DNS rebind 対策として検証済み IP に接続先を固定するところまで強化しています。さらに 5/29 には、`host_not_public_ip` のような失敗理由を enum として監査に残せる形にしました。

つまり、`SafeUrl` は最初から完成形だったわけではありません。最初は URL の形を型で守るところから始まり、実際に fetch する入力としての危険性が見えてから、防御を段階的に厚くしていきました。この経験から、検証はなるべくランタイムの `if` として散らばらせるのではなく、型や制約や境界に寄せて、構造として強制するものだと考えるようになりました。

## 2.4 第2幕の終わりに

この時期の自分は、値オブジェクトにかなり強い可能性を感じていました。入力を毎回 `if` で検証するのではなく、不正な値をそもそも作れない形にする。セキュリティで感じていた「検証を書き忘れたら穴になる」という不安に対して、型で構造的に守るという考え方は、とても大きな発見でした。

一方で、今振り返ると、まだ「必要だから入れる」というより、「学んだパターンを自分のアプリに当てはめてみる」段階でもありました。`app/domain/` や `shared/value_objects/` のような置き場所も、明確な境界から導いたというより、DDD らしい形を先に作っていた部分があります。これは後に「VO パターン学習期の遺産」として整理し直すことになります(`#635`)。

それでも、この幕で得た「不変条件は、できるだけ構造で守る」という感覚は、その後の設計判断の土台になりました。ただし、型を増やせば常に安全になるわけではありません。その素朴な確信は、のちの幕で失敗の分類にまで広げたときに、一度大きく見直すことになります。

次の第3幕では、関心が「値」から「振る舞い」へ移ります。4月のレビューで層を上がっていく中で、Repository に置かれていたルールや状態変更を、ドメイン側へ引き戻す問いが開いていきます。
