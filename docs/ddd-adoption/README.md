# 型に意味を載せ、載せすぎて、削る — Vector における DDD 体得の記録

> 状態: ドラフト。プロローグ + 第1幕〜第4幕 まで。第5幕以降は順次追記します。

## この文書について

これは Vector の設計を時系列で振り返る読み物です。「何を作ったか」を並べたカタログというより、**ある設計判断にたどり着き、ときにそれを撤回するまでの過程**を、当時の git 履歴と凍結された spec をもとに書き起こしたものです。

役割を先に分けておきます。

- [docs/adr/](../adr/) の各 ADR には、**決定そのもの**(最終的にこう設計した、その根拠はこう、という結論)を記録しています。
- [docs/dev-notes/](../dev-notes/README.md) には、当時の個人開発メモ(AI との対話ログ+ローカルノート)の圧縮記録を置いています。この文書が「当時こう考えていた」と書くときの一次資料です。
- この文書には、ADR には書かない**そこへ至る / それを畳むまでの過程**(なぜそう作り、何をきっかけに過剰だと気づき、どう直したか)を残しています。

本文の内容は、できるだけ commit hash と PR 番号をたどれる形にしました。`git show <hash>` で当時のコードや判断の言葉をそのまま確認できます。設計論として読むよりも、手元で確かめられる記録として読んでいただければ幸いです。

## この記録で伝えたいこと

この記録でいちばん伝えたいのは、ドメイン駆動設計(DDD)の知識ではありません。最初は AI が生成したコードをただ承認していただけだったところから、「コードを理解できるようになりたい」「自分で意思決定をしていきたい」と思って少しずつコードレビューを始め、**「よりわかりやすいコードとは、持続可能なコードとは、何か」**を問い続けるうちに、設計に対する考え方そのものが変わっていった——その過程です。

その問いを続けるなかで少しずつ身についたのが、DDD という知識そのものというより、**「ここに設計を投資する価値があるか」を毎回立ち止まって問う**やり方だったように思います。

型やドメインモデルを足すか迷ったとき、見ていたのは主に次のようなことでした。

- **守るべき不変条件があるか** — 「不正な値がそもそも存在できない」形にしたいか。初期の設計メモには「プリミティブ型への執着(Primitive Obsession)を避け、値オブジェクトで『不正な値がそもそも存在できない』設計にする」と書いていました。
- **段階(境界)を跨ぐとき、その段階の責任を果たし切っているか** — ある段階で守るべき条件は、次の処理や永続化に渡す前に、その段階のうちに満たしておきたいと考えていました。型に値を入れられた時点で前提条件が満たされている形にできれば、その型を受け取った側は確かめ直さずに処理を進められます。別の設計メモでは「型 = データの形ではなく、operation の前提条件」「『Stage X に進める条件』が `ReadyForX` という型に閉じ、Task や Service に散らない」と書いていました。
- **さもなくば、バリデーションが複数箇所に散るか** — 同じ検証をスキーマ層・service 層・models 層に書き散らさずに済むか。これは第2幕冒頭のセキュリティ動機とそのまま地続きでした。
- **そもそも、その変更の蓋然性が具体的に見えているか** — 別の設計メモに残した言い方では「現状追従も将来先取りもしない。『変更の蓋然性が具体的に見えているか』で判断する」。いまの都合で決めず、まだ具体像の見えない将来を先取りもしない、という線引きです。

逆に、この問いに「ここは投資に見合わない」と答えが出たときは、たとえ一度入れた抽象でも畳むようにしてきました。翌日や、ときには同じ日に、コミットメッセージへ撤回の理由を書き添えながら。

ですので、この記録で並ぶのは成功談の連続ではありません。**「投資する価値があるか」という一つの問いが、VO・型・集約・監査と形を変えて何度も出てくる**、その繰り返しです。over-engineering をしなかったという話ではなく、価値を見誤って入れてしまった過剰を、同じ問いで測り直して畳む——そのやり方を少しずつ手に入れていった過程を、ここに残したいと思っています。

## この記録の歩き方 — 考え方の変化を追う

この記録には、二つの弧が流れています。

一つは技術の弧です。不変条件を閉じ込める対象が、**値 → 振る舞い → モジュール境界 → 構造 → 状態**へと広がっていきます。各幕の技術的な中身は、この弧の上に並んでいます。

もう一つは、こちらが主旋律なのですが、**考え方の弧**です。この記録は「何を作れるようになったか」のカタログではなく、**設計に対する考え方がどう変わっていったか**を追えるように書いています。

| 幕 | 時期 | このときの考え方 |
|---|---|---|
| プロローグ | 2026-02〜03 | AI の出力を、ただ承認していた。「コードを理解できるようになりたい」「自分で意思決定をしたい」と思い、層を一枚ずつレビューすることから始めた |
| 第1幕 層を一枚ずつレビューする | 03〜04 上旬 | 足場のあるモデルと DB から読み始め、「これはどこが持つべきか」を一つずつ決めていった。スキーマ・ルーターへ層を上がるなかで「使う側から考える」という自分の判断軸が生まれ、AI の提案を検証して覆すことを覚えた |
| 第2幕 値オブジェクト | 03 下旬 | レビューと並行して読んだセキュリティの本が、DDD への入口になった。本のパターンを当てはめれば良い設計になる、と考えていた時期で、のちに「学習期の遺産」と呼んで畳む箱も作った |
| 第3幕 振る舞いをドメインへ | 04 中旬〜 | 「違和感」を、設計を見直すシグナルとして信じるようになった。「揃えることを美徳としない」 |
| 第4幕 ドメインモデルの作り替え | 05 前半 | 設計の問いと目標を自分が握り、AI の設計を自分の言葉で測り直すようになった |
| 第5幕 失敗に何を語らせるか | 05 | 「型を増やすほど安全」という自分自身の信念も、同じ問いで測り直した |
| 第6幕 監査は探針 | 05 | 観測装置に設計の歪みを炙り出させる。ただし、観測装置に設計はさせない、と線を引いた |
| 第7幕 削る・畳む・作らない | 05 後半〜06 | テスト・ドキュメント・AI という「道具」の側を統制する設計に回った |
| エピローグ | 06 | 変化した考え方を、再現できる規律として書き残した |

なお、幕は「取り組みを始めた順」に並べていて、時間としては重なります。とくに第1幕(層別レビュー)と第2幕(値オブジェクト)は、どちらも 3月後半に走り出した二本の線です——セキュリティの本から DDD と secure-by-design の考え方が入ってきたのもこの頃で、4月のレビューはそれを前提に進んでいます。面白いのは、二本が逆を向いていたこと。第1幕は「判断軸を本から自分へ移す」方向、第2幕は「本の DDD パターンを当てはめる」方向で、この緊張は後に、自分の判断が育って当てはめた抽象すら畳めるようになる形で解けていきます。

各エピソードは、できるだけ同じ順序で書きます——**当時こう考えていた → 何が起きた(何に気づいた)→ どう考え直した → いまのコードはこうなっている**。最初の「当時こう考えていた」は、後知恵で賢く書き直さず、当時の spec やコミットの言葉のまま残すようにしました。また、採用した設計だけでなく、**採らなかった案・一度入れて畳んだもの**を各幕に必ず残します。試行錯誤は、捨てた道を見せないと伝わらないと思うからです。

## プロローグ — 最初は、承認していただけでした

Vector は、最初の行から AI(コーディングエージェント)と作ってきたアプリケーションです。これは隠すことではなく、この記録の前提です。

立ち上げ期(2026年2月)の考えは「とにかく動くものを作ろう」でした。認証・収集・AI 分析・ダッシュボードという骨格は、AI の力で数日のうちに立ち上がっています。ただ、当時の私は AI の出力に向き合えていませんでした。振り回された、というのとも少し違います。**ただ、承認していただけ**でした。差分を眺め、動くことを確かめ、マージする。設計の判断は、実質的に AI がしていました。

それを変えたい——**自分でコードを理解できるようになりたい**、そして**自分で意思決定をしていきたい**。この記録は、そう思って始めた取り組みが、設計に対する自分の考えをどう変えていったかの記録です。

最初の一歩は、コードベースを層ごとに区切って、一枚ずつ読んでいくことでした。当時読んでいた『達人に学ぶ DB設計 徹底指南書』と『SQLアンチパターン』を足場に、自分が良し悪しを判断できるデータベース側——モデルと DB のレビューから始めています(3月上旬の DB 設計の棚卸し、`f8b41373`)。**本で足場を作り、足場の届く層からレビューする**——振り返ると、このパターンをその後も繰り返すことになります。

ここから先は、幕ごとに追っていきます。当時何を考えていたかの一次資料として、個人開発メモの圧縮記録([docs/dev-notes/](../dev-notes/README.md))を随所で参照します。
