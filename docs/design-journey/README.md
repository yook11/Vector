# — Vector で、設計の考え方が変わっていった記録


## プロローグ：最初は、承認していただけでした

Vector は、最初の一行から AI コーディングエージェントと一緒に作ってきたアプリケーションです。

立ち上げ期だった 2026 年 2 月、考えていたことはシンプルでした。とにかく、動くものを作ること。認証、ニュース収集、AI 分析、ダッシュボード。アプリケーションの骨格は、AI の力を借りて数日のうちに立ち上がりました。

ただ、当時の私はコードのことをほとんど理解していませんでした。AI が作業する様子を眺め、動くことを確認し、問題がなさそうなら承認してマージする。設計の判断は、実質的に AI に任せて、自分はただ承認していただけでした。

その状態を変えたいと思いました。自分でコードを理解できるようになりたい。自分で設計の良し悪しを判断できるようになりたい。
AI に作ってもらうだけではなく、AI と一緒に考え、自分の意思決定として開発を進められるようになりたい。

この記録は、そう思って始めた取り組みが、私の設計に対する考え方をどう変えていったかの記録です。

最初の一歩は、処理の流れを一つずつ読み解くことでした。当時読んでいた『達人に学ぶ DB設計 徹底指南書』と『SQLアンチパターン』を頼りに、
まずは自分がいちばん理解しやすかったデータベース側、つまりモデルと DB 設計のレビューから始めました。

## この記録で伝えたいこと

この振り返りで一番強く感じたのは、最初の段階から「何を作りたいのか」「何を大事にするのか」を明確にしておくことの重要さでした。

自分が何も考えずに付けていた命名や、AI の提案をそのまま受け入れていた設計は、後から何度も修正することになりました。今でも、当時の曖昧さが残っている箇所はあると思います。最初に意図を持たずに作ったものを、後から理解し直し、設計し直すのはとても難しい。これは、この取り組みを通じて痛感したことです。

もちろん、今でも難しいと感じることはたくさんあります。責任の分け方、型や制約の使い方、命名、ライブラリ選定、デプロイや監視。
進めば進むほど、学ばなければならないことが増えていく感覚があります。

実装の進め方にも、まだ課題があります。もっと早く動くものを作ること、作業単位を小さく切り、最小実装で実現すること。
AI の使い方についても、エージェントの作業をどう見守り、どう改善につなげるのか、まだ取り組むべきことが残っています。

それでも、この過程で、自分の中に大事だと思える考え方が二つできました。最初から持っていたものではありません。レビューと作り直しを繰り返す中で、少しずつ形になってきたものです。

### 一つめは、価値があるのかどうかを常に考えることです。

初めの頃は、どこに時間をかけるべきかという判断軸を持っていませんでした。重要度の低い部分にも手を入れ、設計を整えること自体が目的になっていたことがあります。

今でも、気を抜くと必要以上に作り込んでしまいそうになります。だからこそ、「このリファクタリングは今必要なのか」「この機能は、このアプリケーションに本当に必要なのか」「求められている範囲を超えていないか」と、自分に問い直すようにしています。

すべてを同じ重さで設計するのではなく、そこに時間をかけることで、ユーザーに届く価値が増えるのかを考える。どこに設計の力を注ぎ、どこを最小限に留めるのか。その判断軸を常に意識しながら、見極める力をさらに身につけていきたいと思っています。


### 二つめは、実装する部分の責任を見極め、わかりやすい形に落とすことです。

責任が曖昧なままでは、コードは動いていても、どこで何が起きているのか分かりにくくなります。
「この処理は何をしているのか」「この振る舞いはどこが持つべきなのか」を一つずつ考える必要があると感じました。

それを強く感じたのは、監査ログとエラーハンドリングを入れようとしたときでした。責任が混ざっている場所では、どこで何が失敗したのかを記録することすら難しかったからです。

責任や前提を型や構造として表せると、コードは読みやすく、壊れにくくなります。型は単にデータの形を示すだけではなく、その処理に進める条件が満たされていることを伝えてくれるものでもありました。

チーム開発では、他の人が読んだときに意図が伝わることがより重要になると考えています。
さらに、コーディングエージェントの性能を引き出すうえでも、こちらの意図をどれだけ正確に伝えられるかが大きく影響すると感じました。

だからこそ、最初の段階から、実装する部分が何を担うのかを明確にしておく必要があります。その視点は、これからもっと磨いていきたい部分です。
