# — Vector で、設計の考え方が変わっていった記録

> 状態: ドラフト。プロローグ + 第1幕〜第7幕 + エピローグ。

## プロローグ — 最初は、承認していただけでした

Vector は、最初の行から AI(コーディングエージェント)と作ってきたアプリケーションです。これは隠すことではなく、この記録の前提です。

立ち上げ期(2026年2月)の考えは「とにかく動くものを作ろう」でした。認証・収集・AI 分析・ダッシュボードという骨格は、AI の力で数日のうちに立ち上がっています。ただ、当時の私は 何もわかっていなかったので**ただ、承認していただけ**でした。ただAIの作業を眺め、動くことを確かめ、マージする。設計の判断は、実質的に AI がしていました。

それを変えたい——**自分でコードを理解できるようになりたい**、そして**自分で意思決定をしていきたい**。この記録は、そう思って始めた取り組みが、設計に対する自分の考えをどう変えていったかの記録です。

最初の一歩は、処理の流れを、一つずつ読み解いていくことでした。当時読んでいた『達人に学ぶ DB設計 徹底指南書』と『SQLアンチパターン』を足場に、自分が良し悪しを判断できるデータベース側——モデルと DB のレビューから始めています。**本で足場を作り、足場の届く層からレビューする**——振り返ると、このパターンをその後も繰り返すことになります。

なお、前半(第1〜3幕)の幕は「取り組みを始めた順」に並べていて、時間としては重なります。とくに第1幕(層別レビュー)と第2幕(値オブジェクト)は、どちらも 3月後半に走り出した二本の線です——セキュリティの本から DDD と secure-by-design の考え方が入ってきたのもこの頃で、4月のレビューはそれを前提に進んでいます。面白いのは、二本が逆を向いていたこと。第1幕は「判断軸を本から自分へ移す」方向、第2幕は「本の DDD パターンを当てはめる」方向で、この緊張は後に、自分の判断が育って当てはめた抽象すら畳めるようになる形で解けていきます。

後半(第4幕以降)は、時系列そのものより**考え方が変わっていった順**で並べています——価値の中心に設計を投資し、監査で責務の混線に気づき、それをドメインの論理で解き直し、最後に失敗の型を測り直す、という流れです。ここでも時間は重なっていて、とくに第5幕(監査)と第6幕(作り替え)は 5月のあいだ並行して進みました。

各エピソードは、できるだけ同じ順序で書きます——**当時こう考えていた → 何が起きた(何に気づいた)→ どう考え直した → いまのコードはこうなっている**。最初の「当時こう考えていた」は、後知恵で賢く書き直さず、当時の spec やコミットの言葉のまま残すようにしました。また、採用した設計だけでなく、**採らなかった案・一度入れて畳んだもの**を各幕に必ず残します。試行錯誤は、捨てた道を見せないと伝わらないと思うからです。


## この記録で伝えたいこと

自分は今でもこのコードベースをコントロールし切っていると感じることはできていません。
自分が何も考えずに付けていた命名などに関して、何度も修正を繰り返しました。今でも残っている箇所があるかもしれません。
痛感したのは、最初から意図を持って設計をしないと後から取り戻すことは、とても難しいということです。

今でもわからないと感じることはたくさんあります。責任の分け方、型で保証するものと DB 制約に任せるもののバランス、監視についてメトリクスで何を出すのか？何をDBに保存するのか？どのようなことが起こった時に気づけるようにするべきなのか？——進めば進むほど、学ばなければいけないことが増えていく感覚があります。

それでもこの過程で、自分の中で大事だと思っている考え方が三つあります。最初から持っていたものではなく、レビューと作り直しを繰り返すなかで、少しずつ形になってきたものです。

一つめは、価値ある部分の設計に投資する、ということ。

すべてを丁寧に設計するのではなく、「ここに設計を投資する価値があるか」を毎回立ち止まって問う。守るべき不変条件があるか、変更の蓋然性が具体的に見えているか——そういう問いに「見合わない」と答えが出たときは、たとえ一度入れた抽象でも畳むようにしてきました。投資する場所を選ぶことは、投資しない場所を決めることでもありました。

二つめは、その部分の責任を見極める、ということ。

投資すると決めた場所では、まず「その部分は何をしているのか」を紐解き、「この振る舞いは、どこが持つべきか」を一つずつ決めていきました。その処理が守るべき責任を、その処理のなかで守り切る。そうすれば、それを使う側は中身を疑わずに済みます。

責任の境界がいかに大事かが本当に分かったのは、監視とエラーハンドリングを入れようとしたときでした。「どこで何が失敗したのか」を記録しようとすると、責務が混ざっている場所では、そもそも失敗を定義することすらできない。監査を書こうとして初めて、責務の混線がはっきり見えたのです。境界がはっきりすれば、受け取る側は前提を確かめ直さずに処理を進められるし、失敗したときも「どの段階の責任か」を言い切れます。

三つめは、型を効果的に使って、持続可能なコードベースにする、ということ。

見極めた責任を、忘れた頃にも壊れない形に落とすのが型と制約の役割でした。「型 = データの形ではなく、operation の前提条件」と設計メモに書いていたように、型に値を入れられた時点で前提が満たされている形にできれば、同じ検証をスキーマ層・service 層・models 層に書き散らさずに済みます。ただし型を増やすほど安全になるわけではなく、その信念自体も後で測り直すことになります。

ですので、この記録で並ぶのは成功談の連続ではありません。この三つの考え方が、VO・型・集約・監査と形を変えて何度も出てくる、その繰り返しです。over-engineering をしなかったという話ではなく、価値を見誤って入れてしまった過剰を、同じ問いで測り直して畳む——そのやり方を少しずつ手に入れていった過程を、ここに残したいと思っています。
