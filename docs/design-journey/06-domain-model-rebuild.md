# 第6幕 — 炙り出された責務を、ドメインモデルへ解き直す

第6幕では一つの概念として扱っていたものを紐解いていきます。
中心になるのは、それまで AI の提案を受けながら作ってきた型や抽象を、「この型は何を保証するのか？」を問いながら、自分の理解で設計し直していきます。

なお、この時期は語彙の統一や改名が並行して走っていたため、当時の名前と現在の名前がずれる箇所があります。

## 6.1 Ready を、「次へ進む合図」から「処理開始時点の保証」へ作り直す

第3幕で、私は型がただのデータの形ではなく、満たさなければならない条件を表せるものだと知りました。

その考えを、4月末に非同期パイプラインへ広げようとしていました。パイプラインでは、ある工程から次のタスクへ記事を渡すときに、満たしていなければならない条件があります。
そうした条件を `ReadyFor {ステージ名}` という型として定義すれば、次の工程に渡せるものだけを明確にでき、より堅牢な設計になると考えました。

タスクに投入する前に「次の処理に進めること」を確認できれば、それで十分だと考えていました。上流の Task が DB を見て、未処理であることや必要な値が揃っていることを確認し、ReadyForX を作ってキューに積む。そうすれば、下流の Task はその型を受け取った時点で、前提条件を満たしているものとして処理へ進めるはずだと考えていました。

```python
class ReadyForClassification(BaseModel):
    extraction_id: int
    translated_title: str
    summary: str

class ReadyForExtraction(BaseModel):
    article_id: int
    original_title: str
    original_content: str
```

しかし、ここで見落としていることがありました。前の工程が終わると、その結果は DB に永続化されます。一方で、次の工程には、型に詰めた値を渡していました。

つまり、同じ事実が DB の中にも、キューに積まれた値の中にも存在する状態になっていたのです。
私は条件を型で表しているつもりでしたが、実際には、DB に保存された事実と Ready にコピーした値の整合性をどう保つかまでは考えられていませんでした。

そうなると、処理に使う値の正本をどちらに置くのかを決める必要があります。DB に永続化した値を正本とするなら、次の工程へ値そのものを渡すのではなく、ID だけを渡して、次の工程が DB から読み直せばよい。そう考えて、一度は ID だけを次の処理へ渡す発想に向かいました。

```python
class ReadyForEmbedding(BaseModel):
    model_config = ConfigDict(frozen=True)

    analysis_id: int = Field(gt=0)
```
次の処理では、この ID を使って DB からデータを読み直し、処理を開始するようにしました
。DB を正本にするという意味では、こちらのほうが筋が通っているように見えました。

けれど、この型を見たときに、別の違和感が残りました。
実際には ID を運んでいるだけの薄いラッパーになっていたのです。

条件を保証するための方のはずなのに、IDを運んでいるだけになった。

そこで気づきました。前の工程が「次の処理に進める」と保証するのではなく、その処理を始める工程自身が、開始直前に「いま処理できる」と保証するべきだったのではないか、と。

```python
class ReadyForEmbedding(BaseModel):
    model_config = ConfigDict(frozen=True)

    analyzed_article_id: int = Field(gt=0)
    text_for_embedding: str = Field(min_length=1)
    analyzable_article_id: int = Field(gt=0)
```

本来 Ready が保証すべきだったのは、実際にその処理を始める時点で、処理ごとの前提条件を満たしていることでした。

前の工程から渡すのは ID だけにする。現在はトリガーという型で表しています。
次の工程は、その ID を受け取ってから、 Ready を構築する。
こうして Ready は、「次へ進めるはず」という合図ではなく、処理する側が開始条件を満たしていると確認した証明になりました。

大事なのは、その型が何を、いつ、誰の責任で保証するのかを決めることです。守りたい条件が曖昧なまま型を作ると、名前だけが強くて、実際には何も守れていない型になってしまう。だからこそ、どこで何を保証する必要があるのかを考えてから、型を設計する必要があると学びました。


## 6.2 記事を取得する

このとき考え直したのは、記事を取得する工程が「何を返すべきか」でした。

特に記事を取得する工程では、後から記事を補完するという工程があったので、何を満たすべきなのか？曖昧になっていました、

それまでは、取得できたもの、追加取得が必要なもの、失敗したものを表す型が増えていきました。
けれど型が増えても、工程の意味が明確になるわけではありませんでした。必要だったのは、途中状態を細かく名づけることではなく、最終的にどの条件を満たせば分析へ進めるのかを決めることでした。

そこで、分析に進める記事の条件を先に定義しました。タイトル、本文、公開日時、取得元、元記事の URL が揃っていて、本文の長さや URL の妥当性も満たしていること。そこまで満たしたものだけを、分析に回してよい記事として扱う。

この出口を決めたことで、途中の型の役割も整理できました。すぐ分析に進めるものは `AnalyzableArticle`。まだ補完が必要なものは、取得できた事実として保持する。条件を満たせないものは、理由を持って落とす。型を増やして工程を説明するのではなく、工程が返すべきものを問い直した結果、必要な型だけが残っていきました。

ここでは３幕で以前どうなっていて、そこからどう変わったのか？
特にAIが作成した不要な抽象を畳んでいく
真に必要なものだけを残していくというものにするといいよね？


以前からどのような設計にするといいのか？悩み続けていた箇所がありました。

この節で向き合ったのは、記事を取得する処理そのものです。
これまでは、記事取得を「URL を集めて、その URL を開き、データを取って分析へ渡す」くらいの単純な前処理として見ていました。

けれど実際には、ソースごとに最初から提供している情報が違うこと、その情報をどこまで信じてよいかも違います。記事取得とは、ただ外部からデータを取ることではなく、取れた事実を見て、分析に渡せる記事として成立するかを判断する工程だったのです。

ソースが公式で提供している情報は拾うべき、この設計では、そのようなデータを全て捨てていた
ソースごとの違いを考えていない設計になっていた

すべてをページを開いて取りに行く形に寄せると、成功率も下がります。ページは構造が変わることもあれば、アクセスを制限されることも、うまく解析できないこともある。一方で、配信データや API が最初から正確な情報を持っているなら、それを使ったほうがいい。後から全部を取りに行って完成させるのではなく、最初に取れているものを活かす必要がありました。

分析の前に置かれた雑な前準備ではなく、分析に渡す材料の品質を決める境界だったのです。

そこで、まず「何が揃っていれば、そのまま分析に回してよいか」を先に決めました。タイトル、本文、公開日時、取得元、元記事の URL が揃っていて、本文の長さや URL の妥当性も満たしている。そこまで揃ったものを、分析に回してよい記事とする。この合格ラインを、取得元ごとの処理に散らすのではなく、出口の条件として一か所に集めました。

ただし、取得した一件一件は、単純に「成功／失敗」で判定できるものではありません。提供されたデータだけで、そのまま分析に進められる記事もあります。一方で、タイトルや公開日時などの基本情報は揃っているものの、本文は後続処理で取得する必要がある記事もあります。さらに追加で取得を試みても、分析対象にできないものもあります。
そこで各記事を、初期データだけで分析可能なもの、追加取得によって記事として完成するもの、追加取得しても分析対象にならないもの、の三つに分類しました。

変わったのは、コードの形だけではありません。記事を取得する工程を、分析の品質を支える工程として設計し直しました。取得元が最初から渡してくれるデータは捨てない。使えるものは使い、足りないところだけ後から補う。「取ってくる」だけに見えていた工程は、実は、取れた事実を見て扱いを決める工程だったのです。

ここで初めて、記事取得という工程をドメインとして見始めました。URL を集める、HTML を読む、本文を保存する、という実装上の手順ではなく、「どの情報が揃えば分析に進めるのか」「どの情報はソース由来の事実として信じてよいのか」「足りないものは後から補えるのか」という問いとして捉え直したのです。

## 6.3 ひとつの言葉が、三つの仕事を隠していた

最初は、追加取得によって記事を完成させることは単純な作業だと考えていました。「完成させる」というメソッド一行で済むように見えていました。

けれど、その一語の中で、様々なことが起きていました。まず、追加取得待ちの記事の URL を開いて、データを取りに行く。次に、最初に取れていたデータと合わせて、分析に回せる記事の形に組み立てる。最後に、それを保存して、追加取得待ちの記事ではなくなったことを表す。「完成させる」は、取りに行く・組み立てる・保存するという、三つの仕事を抱えた言葉だったのです。

この混ざりが一番表に出たのは、失敗処理を書こうとしたときでした。
同じ「記事が完成しなかった」という結果でも、失敗した段階は異なります。取得段階の失敗は、相手サーバーの停止、タイムアウト、HTML の解析不能などにより、必要な外部データを取り込めなかった失敗。一方、組み立て段階の失敗は、取得できた情報と保存済みの情報を合わせても、分析に回せる記事として成立しなかった失敗です。
ところが当時は、この区別がなく、失敗の扱い方も工程ごとに揃っていませんでした。ある失敗は戻り値として返り、例外として投げられているものもあったため、呼び出し側から見ると、どの工程がなぜ失敗したのかが分かりにくくなっていました。

そこで、「完成させる」という処理の中身を分解し、不足データを外部から取得する処理、取得済みデータと保存済みの事実から記事を組み立てる処理、完成した記事を保存する処理、の三つとして明確にしました。

三つの工程を名前で分けたことで、失敗もそれぞれの工程に結びつけて扱えるようになりました。取得に失敗した場合は、取得工程がその理由を返すというように、
失敗した場所と理由がずれないように、各工程が自分の結果として失敗理由を返す形にしました。

今回分かったのは、失敗処理の書きづらさは、責任が混ざっているサインになりうるということでした。「完成させる」という一つの言葉の中に、取得、組み立て、保存が混ざっていたため、どこで失敗し、誰がその理由を持つべきかも曖昧になっていました。工程を分け、それぞれが自分の失敗理由を返す形にしたことで、失敗の扱い方も自然に整理できました。

## 6.4 ソースごとの違いを扱う難しさ

記事を完成させる工程と同じように、外部ソースから記事候補を取り込む処理でも責任が混ざっていました。以前は、候補を取得することと、その候補を6章2で述べた三つの状態に振り分けることが同じ処理に入っていました。そこで、候補を集める処理と、候補の扱いを決める処理を分けました。

ただ、難しかったのはここからです。取得した記事候補をどう扱うべきかは、外部ソースごとの事情に左右されました。6.2で見たように、候補の時点でどこまで情報が揃っているかが、ソースごとに大きく違うからです。

しかも、その違いはこちらでコントロールできるものではありません。外部ソースが何を提供し、どんな形式で配信するかは相手側の都合に依存します。本文を含めていた配信が要約だけになることもあれば、HTML の構造が変わることもある。新しいソースを追加するたびに、ソースごとの事情に合わせた個別対応が増えていきます。
だから、ソースごとの違いに毎回その場で対応するのではなく、どこまでを共通の流れとして保ち、どこからをソースごとの性質として切り出すかを決める必要がありました。

何度も作り直しました。
はじめは、ソースごとの差を扱うために、一段薄いラッパーをかませていました。けれど、その層は、どの候補をどう扱うかを判断していたわけではありませんでした。実際の取得処理は各ソース側にあり、それを呼び出しているだけでした。不要だと判断して畳むと、Service からは「収集、変換、永続化」の流れが見えるようになりました。

ただ、その形では、「ソースがニュースを取得する」という不自然な振る舞いになってしまいました。ソースは本来、「この URL を読む」「この項目は信じてよい」「本文は後から補う」といった性質を持つものです。にもかかわらず、外部データを読み、対象を選び、重複を落とし、記事候補へ写すところまでを 持たせていたため、ソース自身が動いて記事を取りに行く主体のように見えていました。

この違和感を測り直すと、問題はソースごとの差そのものではなく、その差を「ソース自身の振る舞い」として持たせていたことだと分かりました。ソースごとに違うのは、どの URL を読むか、どの項目を信じるか、どれを対象にするか、本文を後から補うか、といった性質です。一方で、外部データを読み、対象を選び、記事候補へ写し、その後の扱いを決める流れまで、すべてソースの仕事にする必要はありません。
そこで、ソースには「このソースはこう読める」という宣言を持たせ、取得を進める流れは外に出しました。取得した候補を三つの状態に振り分ける判断も、ソースの中には戻しませんでした。ソースごとの差は宣言として残し、候補をどう扱うかは別の責務として分けたのです。

何度も動かしていたのは、コードそのものではなく、「ソースごとの判断を、どこに置くか」でした。どう読むかも、どの情報を信じて、どれを後から補うかも、記事一件ごとに毎回その場で決めることではありません。ソースの性質として、はじめから決まっている。だから、それらをソースの宣言として持たせ、共通の流れはその宣言に沿って動くだけにする。記事側には、実際に取れた事実をそのまま残す。こうして、ソースごとの違いは、条件分岐の山ではなく、ソースごとの宣言として積み上がるようになりました。

## 6.5 第6幕の終わりに

この幕で一番大きかったのは、正しい設計に一度でたどり着いたことではありません。むしろ、正解がはっきり見えない中で、「この処理は何をしているのか」「その責任はどこに置くべきなのか」を問い続けたことでした。
Ready は何を保証するのか。「完成させる」とは、実際には何をしているのか。ソースは記事を取りに行く主体なのか、それとも読み方を表すものなのか。そうした問いを立て直すたびに、名前も型も置き場所も変わっていきました。
今の形が最終的な正解だと断言できるわけではありません。それでも、違和感をそのまま流さず、責任の境界を言葉にし直し、テストで確かめながら作り替えていく。その姿勢を持てるようになったことが、第6幕の一番大きな変化でした。

そして、その問いは失敗の型にも向かっていきます。失敗を値にするだけで十分なのか。その値は何を語るべきなのか。第7幕では、今度は失敗の表し方そのものを測り直していきます。
