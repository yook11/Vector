# 第5幕 — 監査をきっかけに

第4幕で価値の中心に手を入れ始めると、次に問題になったのは、パイプラインで何が起きているのかを後から追えないことでした。
記事を取得してから分析まで完了していないとき、取得で落ちたのか、本文補完で止まったのか、AI 呼び出しで失敗したのかが分からない。

第5幕は、その「見えなさ」を埋めようとして監査ログ(pipeline_events)を入れ始めたことをきっかけに、責任の分け方を本気で考えるようになっていく話です。
それまでも責任分離は「ある程度はできているつもり」でした。ところが監査ログを導入する過程で、それが十分でなかったことが次々に露わになります。

素直に導入できない場所には、決まって何かが混ざっていました——記録する場所、失敗の種類と扱い、何を記録するかの基準。この章で扱う監査基盤は、振り返ると、私が初めて「AI ではなく自分で」設計をする、自分が主導で考えるということに取り組んだように思います。

## 5.1 意図のないテーブル

監査の必要性は、5月になって突然出てきたものではありません。4月の時点でも、失敗した記録を DB に残したいという考えはありました。
外部から記事を取得する処理を記録する `fetch_logs` というテーブルがありました。どのソースを取得したか、成功したか、何件取れたか、エラー内容や処理時間を残すテーブルです。

```text
fetch_logs
+----------------+----------------------------------+
| column         | meaning                          |
+----------------+----------------------------------+
| id             | ログID                           |
| source_id      | 取得対象のニュースソース          |
| status         | success / error                  |
| articles_count | 取得できた記事数                  |
| error_message  | エラー内容                        |
| duration_ms    | 処理時間                          |
| fetched_at     | 取得処理を実行した時刻            |
+----------------+----------------------------------+
```

ただ、今振り返ると、これはかなり場当たり的な設計でした。何を後から知りたいのか、何を記録すべきなのかが曖昧なまま、「とりあえず実行結果を残す」ために作っていたからです。

そのため、失敗したことは分かっても、ネットワーク接続、レスポンスの解釈、記事の保存のどこで失敗したのかまでは追えませんでした。後から「なぜ失敗したのか」を調べ直せる設計にはなっていませんでした。

もう一つの問題は、`fetch_logs` を処理の記録としてだけでなく、次回取得の判断にも使っていたことです。前回いつ取得に成功したかを `fetch_logs` から調べ、その時刻をもとに次の取得範囲を決めていました。

過去の出来事を残すための記録テーブルに、次に何を取得するかを決める状態管理の責任まで混ざっていたのです。

`fetch_logs` を拡張する案もありましたが、責任が曖昧なまま継ぎ足すだけになると判断し、設計し直すことにしました。

必要だったのは、失敗したときに、どの工程で、何が起き、なぜ止まったのかを後から追えることでした。
特に非同期 worker の失敗は、どこかで止まっても、そのまま見えなくなります。

そこで、目標を設定しました。未来の自分が「何が起きていたか」を SQL 1本で再構成できること。
そのために、取得・補完・AI 分析・埋め込みまで、パイプライン全体の出来事を残す `pipeline_events` の設計を開始しました。


## 5.2 監査を広げるほど、問題点が見えてくる

`pipeline_events` は、最初から完成した監査基盤として入れたわけではありません。順に各工程へ広げていきました。その過程で、失敗を表現することができていないことに気づいていきます。

外部ページから本文を補完する処理に監査ログを導入しようとした時、すぐに問題に気づきました。

失敗を再試行すべきかどうかの判断が曖昧だったことです。404 のように何度試しても変わりにくい失敗も、通信エラーのように時間を置けば成功する可能性がある失敗も、同じ処理をしていることに気づきました。

そこで、リトライできない失敗と、リトライできる失敗を分けました。

```python
class PermanentFetchError(Exception):
    """リトライ不可のフェッチ失敗（403 / 404 / robots.txt で拒否）。"""

class TemporaryFetchError(Exception):
    """リトライ可能なフェッチ失敗（5xx / タイムアウト / 429）。"""
```

この分類によって、「もう一度試す意味があるか」は表せるようになりました。けれど、監査ログに残す分類としてはまだ粒度が粗すぎました。

404、403、robots.txt による拒否は、どれも PermanentFetchError です。
5xx、429、タイムアウトも、どれも TemporaryFetchError です。実際に監査ログへ書き込むときも、具体的な原因が読み取れないものになってしまいます。

```python
except PermanentFetchError as e:
    await self._audit_terminal(
        ...,
        # 404 なのか、403 なのか、robots.txt 拒否なのかは
        # reason_code だけでは区別できない
        reason_code="permanent_fetch_error",
        exc=e,
    )
    return TerminallyDropped(reason_code="permanent_fetch_error")
```


### AI分析側にも

AI分析側は、失敗をまったく分類していなかったわけではありません。

```python
class ConfigurationError(AnalysisDomainError): ...
# API key やモデル名など、設定・認証の問題

class ProviderError(AnalysisDomainError): ...
# provider 側の障害や、期待した応答が返らない問題

class NetworkError(AnalysisDomainError): ...
# タイムアウトや接続失敗などの通信問題

class RateLimitError(AnalysisDomainError): ...
# レートリミット制限

class InsufficientBalanceError(AnalysisDomainError): ...
# 残高不足による停止
```

大まかな例外を task 側で捕まえ、その分岐の中で「これはリトライする」「これは記事を残して止める」と判断していました。

```python
except ConfigurationError as exc:
    await _audit_extraction_failure(..., outcome_code="ai_error_config", ...)
    return

except InsufficientBalanceError as exc:
    await _audit_extraction_failure(..., outcome_code="ai_error_insufficient_balance", ...)
    return

except NetworkError as exc:
    if is_last_attempt(ctx):
        await _audit_extraction_failure(..., outcome_code="ai_error_network", ...)
        return
    raise
```
ハンドリングはできているつもりでした。けれど監査ログを書こうとすると、まず AI のエラー分類の粒度が粗すぎることに気づきました。

`ProviderError` のような大きな分類では、provider 側の一時障害なのか、AI の応答形式が壊れていたのか、安全性の理由でブロックされたのかを後から追いにくくなります。

もう一つの問題は、リトライするかどうか、記事を残すかどうかといった判断を、すべて task 側の条件分岐で書いていたことです。

そのため、監査ログを書いても、「なぜ失敗したのか」と「その失敗をどう扱ったのか」を一貫して記録することが難しくなっていました。失敗の原因と扱いが、型ではなく、その場の分岐に依存していたのです。

そこで、AI を呼び出したときに起こる共通のエラーを定義し直す必要があると考えました。

例外型には二つの意味を持たせていました。一つは、監査ログに残すための `CODE` です。もう一つは、その失敗をどう扱うかを表す marker です。

```python
class AIProviderNetworkError(AIProviderError, RetryableError):
    CODE = "ai_error_network"
    INLINE_RETRY = True

class AIProviderInsufficientBalanceError(AIProviderError, NonRetryableKeepArticle):
    CODE = "ai_error_insufficient_balance"
```

この形にすると、失敗の種類と処理方針を例外型に持たせることができます。
けれど設計を進めるうちに、同じ AI provider の失敗でも、工程によって扱いが変わることに気づきました。
そこで一度、marker を廃止して、工程ごとに扱いを tuple で分ける方向へ進みます。

```python
ASSESSMENT_RECOVERABLE_PROVIDER_ERRORS = (
    AIProviderRateLimitedError,
    AIProviderServiceUnavailableError,
    AIProviderNetworkError,
)

ASSESSMENT_TERMINAL_SKIP_PROVIDER_ERRORS = (
    AIProviderConfigurationError,
    AIProviderInsufficientBalanceError,
    AIProviderInputRejectedError,
)
```

監査を入れようとしたことで、まず「再試行する価値があるか」を分ける必要があること、そして工程ごとに扱いが違うということに気づきました。けれど設計を進める中で、この設計でも、「なぜ失敗したのか」を後から十分に追えないことが見えてきます。

当時はまだ、例外型に持たせた `CODE` をそのまま監査ログへ記録するだけでした。そのため、`ai_error_network` や `ai_error_insufficient_balance` のような大まかな分類は残せても、本当に何が起きたのかまでは十分に記録できていませんでした。

この違和感が、後に失敗の型や reason code を見直すきっかけになりました。


## 5.3 何でも記録すればいいわけではない

監査ログを導入し始めた頃は、複数の工程をまたぐ処理を後から追跡できるようにするため、関連するテーブルを横断して、できるだけ多くの情報を監査に残すべきだと考えていました。

そのため、AI 分析の結果を保存したあとに採番される `assessment_id` や、AI が返したカテゴリを DB 上の `category_id` に解決した結果まで、監査に含めようとしていました。Service が本来の処理では使わない値を、監査に渡すためだけに JOIN して取りに行くような形になりかけていたのです。

当時は、カテゴリの構成が後から変わっても、その時点でどのカテゴリとして扱ったのかを追えるようにしたいという意図もありました。けれど、その目的を `category_id` のような後続の lookup 結果に頼って満たそうとしたことで、監査に残すべき事実の境界が曖昧になっていました。

その不自然さを辿っていくと、監査の目的がずれていることに気づきました。

監査に残すべきなのは、後から便利に追跡するために集めた値ではなく、その時点で実際に起きたことです。工程をまたいで追跡したいのであれば、本来は監査 payload に無理に情報を焼き込むのではなく、相関 ID のような仕組みでつなぐべきこともこの時点で学びました。

この気づきは、失敗の記録を見直すきっかけにもなりました。

「その時点で起きたことを自然に記録する」という視点で見ると、当時の監査ログは、失敗したこと自体は残せても、そこで本当に何が起きたのかまでは十分に記録できていませんでした。監査を整えることは、単に記録する項目を増やすことではなく、出来事を正しく表せる設計にすることなのだと考えるようになりました。


## 5.4 第5幕の終わりに

監査ログを記録する、という目的から、たくさんのことを学びました。

この取り組みを始める前にも、ログに出す、例外を定義する、エラーハンドリングを書く、という発想自体はありました。けれど「失敗とは何か」を十分に定義できていませんでした。

どの工程で、何に対して、なぜ失敗し、その結果どう扱われたのか。後から読み返せる出来事として記録するには、失敗をもっと明確に言葉にする必要がありました。

このとき、失敗の性質を見極めることはとても難しいと感じました。どの部分で起きて、どのように扱われるのか。その例外単体で見たときに分かるようにしなければいけません。処理としては同じ「失敗」に見えても、それぞれの持っている意味は違っていました。

その中で、「その時点で起きたことを正確に、自然に記録する」という感覚が少しずつ自分の基準になっていきました。この感覚は、後に何度も設計へ違和感を持ち、修正していくときの助けになりました。

個人的には、この監査基盤が、AI の提案をただ受け入れる段階から抜けて、自分で問いと到達状態を握り始めた最初の設計だったと思います。失敗とは何かを考え続けることは、このあとも簡単には終わりません。何度も違和感を覚え、そのたびに設計を直していくことになります。

次の第6幕では、まず取得・補完・保存の責務をドメインモデルとして捉え直していきます。そして、失敗そのものの分類は、後の第7幕で改めて向き合うことになります。
