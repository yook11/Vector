# 第5幕 — 監査が、責任分離を本物にする

第4幕で価値の中心に手を入れ始めると、次に問題になったのは、パイプラインで何が起きているのかを後から追えないことでした。記事を取得してから分析まで完了していないとき、取得で落ちたのか、本文補完で止まったのか、
AI 呼び出しで失敗したのかが分からない。

第5幕は、その「見えなさ」を埋めようとして監査(pipeline_events)を入れ始めたことをきっかけに、責任の分け方を本気で考えるようになっていく話です。それまでも責任分離は「ある程度はできているつもり」でした。ところが監査を書こうとした途端、それが本物でなかったことが次々に露わになります。監査を素直に書けない場所には、決まって何かが混ざっていました——記録する場所、失敗の種類と扱い、何を記録するかの基準。この章で扱う監査基盤は、振り返ると、私が初めて「AI ではなく自分で」設計をする、考えるということに取り組んだように思います。

## 5.1 記録はあったが、監査にはなっていなかった

監査の必要性は、5月になって突然出てきたものではありません。4月の時点でも、失敗した記録を DB に残したいという考えはありました。
また、取得処理には、`fetch_logs` という記録テーブルがありました。どのソースを取得したか、成功したか、何件取れたか、エラー内容や処理時間を残すテーブルです。

ただ、今振り返るとこれはかなり場当たり的な設計でした。何を後から知りたいのか？何を記録するべきなのか？が曖昧なまま、「とりあえず実行結果を残す」ために作っていたからです。
実際分かるのは、ソース単位の成功・失敗と件数くらいでした。失敗したとしても、ネットワーク接続で落ちたのか、接続はできたがレスポンスを解釈できなかったのか、取得した記事を保存する段階で落ちたのかを、追うことはできませんでした。

もう一つの問題は、前回いつ取得したかを記録テーブルから調べ、それをもとに次の取得時間を決めていたことです。
記録はあくまで「過去に何が起きたか」を残すもので、業務のロジックを算出するものではありません。

「記録を残すテーブル」はあっても、監査の役割は果たせていませんでした。過去に何が起きたかを、後から調べ直せる形になっていなかったからです。

必要だったのは、失敗したときに、どの工程で、何が起き、なぜ止まったのかを後から追えることでした。特に非同期 worker の失敗は、HTTP の 500 のようにユーザーへ返ってきません。どこかで止まっても、そのまま見えなくなります。

そこで、目標を置き直しました。3ヶ月後の自分が「何が起きていたか」を SQL 1本で再構成できること。そのために、取得・補完・AI 分析・埋め込みまで、パイプライン全体の出来事を残す `pipeline_events` の設計を開始しました。

## 5.2 監査を広げるほど、責任の混線が見えてきた

`pipeline_events` は、最初から完成した監査基盤として入れたわけではありません。Stage 1 から順に各工程へ広げていきました。その過程で、監査を素直に書けない場所には、責任分離の甘さが残っていることに気づき始めました。

最初に監査を入れたのは、ニュースを取得する工程です。ここで行ったのは、記録する場所の整理でした。それまでも取得の実行結果をまったく残していなかったわけではありません。ソース単位の成功・失敗は `fetch_logs` に書き、個々の記事で何件失敗したか、何件処理できなかったかは取得処理の中で数え、呼び出し側のログや実行結果に載せていました。けれど、それらは一つの監査としてまとまっておらず、「あとから何が起きたかを追うための情報」が、処理本体・タスク・ログの間に散っていました。

`pipeline_events` を入れたことで、こうした情報を監査の場所に集められるようになりました。

二段目で、監査が無かった Stage 2(本文補完)に監査を入れようとしたところ、失敗をすべて同じものとして扱っていることに気づきました。一時的な通信エラーなら、時間を置いて再試行すれば成功する可能性があります。一方で、404 のように何度試しても結果が変わらない失敗もあります。にもかかわらず、当時はどちらも同じように再試行していました。監査を書こうとしたことで、失敗には種類があり、その後の扱いも分ける必要があると見えてきました。

AI 分析の側でも、失敗の種類を考えることができていませんでした。当時は、通信エラー、設定ミス、残高不足、AI の応答不正といった失敗原因ごとに分岐を書き、その場で扱いを決めていました。失敗の種類が増えるほど、分岐も増えていく構造になっていたのです。

ここで気づいたのは、失敗原因ごとに個別対応を書くのではなく、その工程で必要になる処理の大枠を先に作ればよいのではないか? AI 分析であれば、もう一度試す、記事を処理対象から外す、記事は残して止める、といった扱いです。個別のエラーは、その大枠のどれかへ結びつける。そうすれば、失敗の種類が増えても、分岐を原因ごとに増やし続けなくて済みます。

ただし、この分類の仕方自体は、後にもう一度見直します。第7幕で改めて扱います。

ここで学んだのは、書きづらさは設計の違和感だということでした。監査を素直に書けない場所には、たいてい責任の混線がある。書きづらさが、設計の歪みに気づく手がかりになっていきました。

## 5.3 何でも記録すればいいわけではない

もう一つ見えてきたのは、監査は「できるだけ多く記録すればよい」わけではない、ということでした。
当時の自分の問いにも、その違和感が残っています——「監査というのは、ありのままの事実を記録すること。後から追跡できるようにと id などを無理に焼こうとするのは、間違いではないか。」

監査を入れ始めた頃は、後から追えるようにするために、取れる情報をなるべく多く残すべきだと考えていました。その結果、AI 分析の結果を保存したあとに採番された id や、カテゴリを DB 上の id に解決した結果まで、監査に残そうとしていました。Service は自分では使わない値を、監査へ渡すためだけに持ち回ることになります。

けれど、それらは「AI が応答した」という時点で起きていた事実ではありません。監査に残すべきなのは、後で便利そうな情報を全部集めた写しではなく、その時点で実際に起きた事実だと気づきました。

次に、「次に何をするか」を監査に焼かないことです。404 だった、本文を抽出できなかった、AI の応答が不正だった、という事実は監査に残します。一方で、その URL を次回も試すのか、もう処理対象から外すのか、一定時間止めておくのかは、次の実行時に参照する現在状態です。

監査をその判断に使い始めると、「起こったことを記録する」という責任に、「次に何をするかを決める」という責任が混ざってしまいます。だから、監査には起きた事実を残し、次の処理を動かすための状態は別に持つ必要がありました。

## 5.4 監査をきっかけに、例外処理の責務を分け直す

監査を入れる過程で、Service の中に残していた例外処理の責務が曖昧になっていることに気づきました。

それまでは、失敗したときにログへ出すだけで済んでいました。ログであれば、エラーが起きたことと最低限の文脈を残せば十分でした。けれども監査として残すとなると、話は変わります。どの処理が失敗したのかだけでなく、その失敗が一時的なものなのか、競合なのか、永続的な失敗なのか、あとから読む人にとって意味のある形で記録する必要がありました。

そこで見えてきたのは、エラーごとに記録すべき内容が違うということでした。本文補完であれば、成功、競合、一時失敗、永続失敗は同じ「失敗」ではありません。それぞれ、次に再試行すべきか、既に処理済みとして扱うべきか、データとして欠落しているのかが違います。にもかかわらず、それらを Service の中で直接組み立て始めると、Service の主語が「補完する」から「補完しながら監査用の事情も全部判断する」に広がってしまいます。

Service は、そのユースケースで何が起きるのかをパッと読める場所であるべきです。本文補完の Service なら、HTML を補って記事を完成させる流れがまず読める必要があります。例外ごとに監査 payload を組み立てる処理まで入り込むと、本来の業務ロジックが見えづらくなります。

そのため、失敗したときに何を監査へ残すかは Service 本体から切り出し、専用の handler と audit repository に持たせる形にしました。Service は補完の流れを担い、handler は例外の種類ごとに監査として残す意味を整える。そう分けることで、処理本体と監査記録の責務を混ぜずに済むようになりました。

ここで学んだのは、監査は単にログを置き換えるものではないということです。ログでは同じように見えていた失敗も、監査として残すなら意味が分かれます。その違いを Service の中に押し込むのではなく、例外処理と監査記録の責務として分けて扱う必要がありました。

## 5.5 第5幕の終わりに

この章で監査が明らかにしたのは、責任分離が「できているつもり」のままでは、あとから出来事を読み返せないということでした。ログや実行結果はありました。けれど、どの工程で何が起き、なぜ次へ進まなかったのかを再構成するには足りませんでした。監査を入れようとして初めて、記録する場所、失敗の扱い、状態と出来事の境界が曖昧だったことが表に出てきました。

特に大きかったのは、失敗を監査に残そうとした瞬間に、「失敗」を一括りにしてはいけないことがわかったことです。ログならエラーを出すだけだったものの、監査では失敗ごとに後から欲しい情報も違います。その違いを Service の中で組み立て始めると、Service の主語はすぐに広がってしまう。だから、Service は取得する、補完する、分析するという流れに戻し、例外ごとに何を監査へ残すかは handler と audit repository 側へ分ける必要がありました。

監査は、分離が甘い場所を見つける探針でした。素直に書けないところには、記録と状態、例外処理と業務処理、写像と判断のようなものが混ざっている。見つけたあとにどう分けるかは、各工程の責務へ戻って考えなければいけません。

個人的には、この監査基盤が、AI の提案をただ受け入れる段階から抜けて、自分で問いと到達状態を握り始めた最初の設計だったと思います。

監査が照らし出した責務の混線は、まだ言葉にして見えるようになっただけでした。次の第6幕では、その混線を、取得・補完・保存のドメインモデルとして一つずつ解き直していきます。
