# 第1幕 — レビューを始める

「自分でコードを理解できるようになりたい」と思い、最初に始めたのがこのレビューでした。

当時はまだ、ユーザーからリクエストが来て、どのようにデータが返っていくのか？その全体像すら掴めずにいました。

コードベースをみてもどこで何をしているのか全くわからないところからのスタートでした。

当時、自分が一番理解できる可能性がある部分はどこなんだろう？と考えて、『達人に学ぶ DB設計 徹底指南書』と『SQLアンチパターン』の 2 冊を読んでいた経験から、
データベース。ORMの部分から見ていこうと決めました。

## 1.1 まず、モデルと DB から

最初に、現状を棚卸ししてみることにしました。Claude Code に現在のテーブル設計を書き出してもらい、それを一つひとつ見ていくところから始めました。

そこでまず分かったのは、設計の良し悪しを判断する以前に、自分は ORM、つまり Python 側の models と、実際のデータベースが別物なのだと分かっていなかった、ということです。
models を書き換えれば、そのままデータベースも変わる。そう思い込んでいました。実際には、models は「データベースはこうあるべき」という宣言にすぎず、現状との差分を実際のデータベースへ反映するのは Alembic の migration でした。レビューを始めるまで、そうしたことを何ひとつ意識していませんでした。

ただ、本当に向き合うことになったのは、知識不足そのものではありませんでした。

棚卸しした内容を確認していく中で、このテーブル構成は自分が設計したものではない、と気づいたのです。
入れた覚えのないテーブル。本来必要な数をはるかに超える列を抱えたテーブル。そこで思い知らされました。

コーディングエージェントを使えば、提案を承認していくだけで、それらしいものが出来上がっていきます。
それまでは AI が主導し、自分はその出力を承認してきただけでした。提示された形を受け入れ、並べられた選択肢から選んでいただけで、「自分はこのアプリケーションのデータをどのように扱うべきなのか」を一度も考えていなかったのです。

この気づきが、出発点でした。ここから初めて、自分の意図で一つずつ決め直していくことになります。

## 1.2 「結局ユーザーは何を見たいのか」 — スキーマレビュー(4/1)

API が返すデータをどう直すべきか、最初は正しい設計が何かも、どう見ていけばいいのかさえ分かりませんでした。

とりあえず手を動かそうと、DB のときと同じく、まず技術書に頼りました。『API デザインパターン』(まだ全ては読めていません)を読み始め、あわせて設計のベストプラクティスを AI に調べてもらうようにもなりました。「命名はシンプルに」「前置詞は危険信号」「用途が名前から読み取れるように」——知識は少しずつ入りました。

ただ、それで進められたのは、命名の方向性を揃えることくらいでした。肝心の部分には、どう手をつければいいのか分からないままでした。
DB のときは、現状を棚卸しして差分を見る、という道筋がありました。今回はその道筋が、自分の中にありませんでした。
知識は入れた。それなのに、コードを読んでも「ここで定義している型が、何に使われているのか、正直まったく分からない」。エンドポイントを眺めて壁打ちするだけでは、答えが出ませんでした。

そんな中、ふと考えが浮かびました。「実際にそのデータが使われている箇所から確認してみるのがいいのでは？」
本当に欲しい情報は何か。結局ユーザーは何を見たいのか。その視点から見直してみることにしました。

その目で見ると、問題点が見えるようになってきました。

このアプリの中核は「AI 分析済みニュース」なのに、レスポンスは原文のニュースが主役になっていました。型の名前は NewsResponse——「返すのはニュースだ」という前提で、最上位に原文タイトルや元記事の URL が並び、AI 分析はその中の省略可能な一フィールドとしてぶら下がっている。ところが実際の画面でユーザーが見ている見出しも本文も、主役は分析でした。型の上では原文ニュースが主役で、分析が付属している——主従が逆さまだったのです。

さらに、一覧画面と詳細画面が同じ型を共有していて、一覧カードでは表示しないデータを毎回返していました。
返すデータの形を決めるときに、それが画面でどう使われるのかまで考えが及んでいませんでした。

そこで、API で返すデータを、ユーザーが実際に見るものに合わせて、AI 分析を主軸に設計し直しました。さらに、一覧と詳細では必要な情報が違うため、用途に合わせて型を分け、不要なデータを返さない形にしました。
「実際に使われている部分から逆算する」という考え方に切り替えたことで、何を直すべきかが少しずつ見えるようになりました。この経験は、その後のレビューでも判断の軸になっていきます。

## 1.3 一番てきとうな層 — エンドポイントレビュー、最初のリジェクト(4/2-3)

レビューを始めてすぐに、一番大きな粗さが表面化しました。エンドポイントが叩かれたときの処理、DB クエリの組み立ても、データの変換も、レスポンスの組み立ても、すべて一つのハンドラに定義されていました。今まで見てきた中で一番大きな違和感で、当時のメモにも「まさか router が一番てきとうに設計されているとは思わなかった」と残しています。
この時はまだ、プロジェクトのデフォルトのフォルダ設定でした、自分で新しいものを作るという発想はありませんでした。

とても読みにくいものになっていたのですが、どう直すべきかが分からず、FastAPI のレイヤー分離とファイル構成のベストプラクティスをリサーチしてみることにしました。
そこで出てきた記事で、Router / Service / Repository に役割で分ける型を知ります。
HTTP の受け渡しは router、ビジネスロジックは Service、DB アクセスは Repository。フォルダで役割を分けるという考えが生まれたのは、このときです。最初に見ていた categories から始めて、他のルーターにも同じ形を広げていきました。

整理を行ったことで、今度はエンドポイントそのものの粗さが見えてきました。
最初に引っかかったのが /news です。分析済み記事を読むための GET /news・GET /news/{id}・GET /news/{id}/similar と、裏側の処理を動かす POST /news/fetch・POST /news/embed が同居していました。前者はユーザーが画面で記事を読むため、後者は管理者がニュース取得や AI 処理をやり直すための API です。目的が全く違うものが同じ /news 配下にあるせいで、「これは何のための API なのか」が見えにくくなっていました。

そこで、境界を「ニュースというデータ名」ではなく「誰が何のために使うのか」で切り直しました。ユーザーが扱うものは /articles、管理者が操作する処理工程は /pipeline。さきほど切り出した Service / Repository もこの境界に合わせて下ろし、記事参照とパイプライン操作を別のファイルに分けました。URI からコードの各層まで、一つの軸が通った形です。

次に、各エンドポイントが実際に使われているかを洗い出すと、どこからも呼ばれていないものがいくつも見つかりました。特に keywords の CRUD は、必要だから設計したのではなく、いつか使うかもしれないものとして残っていた機能でした。画面で何をするのか、ユーザーや管理者が本当にその操作を必要とするのかを考えないまま、API と UI 部品だけが先に用意されていたのです。ここでも、自分が考えて設計していなかったことを痛感しました。使われていないものは負債になるので、削除しました。

一方で、AI のベクトルを生成するエンドポイントも、どこからも呼ばれていない未使用のエンドポイントでした。ただ、生成に失敗した記事を手動で生成し直す場面があるかもしれない、と考えて残しました。いま振り返ると、この判断にはまだ「必要になったときに設計する」ではなく、「使うかもしれないから残す」という迷いが残っていました。未使用コードを削るという原則に気づき始めてはいたものの、それを最後まで貫けるほどには、まだ判断軸が固まっていなかったのだと思います。

このレビューで感じたのは、API 設計の難しさでした。DB には正規化理論のような手がかりがあって、方向性を決めることができていました。でも API は、「これがどのように使われるのか」を想像しながら設計するしかなく、明確な正解がありません。

そして、もっと根本的なことに気づきました。DB の設計を固めて、API で返すデータの形式を直して、最後にエンドポイントを整える ―― DB を起点に順番に決まっていくのだと考えていました。
しかし実際にはこの三つは切り離して進められるものではなく、互いに関係し合っていました。どれか一つを先に固めるのではなく、三つを行き来しながら一貫して考える必要がある、と学びました。

客観的な正解がない中で頼りになったのは、前章で掴んだ「使う側から考える」という視点でした。「このエンドポイントを誰が、何のために叩くのか」を軸に、エンドポイントを一本ずつ追っていきました。


## 1.4 抽象の価値を、守る条件で測る(4/4-5)

router / service / repository に分けたことで、エンドポイントが叩かれてから下の層へどう流れるかを、一本の線で追えるようになりました。そこで、エンドポイントを一本ずつ、「このエンドポイントは何をすべきか」から確認し直していきました。

この見直しの途中で、自分のレビューの仕方そのものに引っかかりを覚えました。コードを読むとき、自分は「これは何をしているか」を理解することに気を取られていて、「これは何をすべきか、そしてそれを達成しているか」という問いが抜けていたのです。前者はコードに引っ張られ、後者は目的から逆算する。効率や書き方の良し悪しは、その後の話でした。この一つの軸 ―― 表面の形ではなく、何のために何をすべきか ―― で見ると、単体では見えなかった粗さが次々に表面化しました。

最初は記事一覧の list_articles でした。受け取るパラメータの定義が router の関数引数と ArticleListParams の二箇所に分かれていたので、一箇所にまとめようとしました。けれど、まとめる先のその ArticleListParams は、repositories(DB を読む層)に置かれていました。HTTP の入口で扱うべき Query や alias の情報が、DB アクセスのための層に入り込み、そのファイルは不正なカテゴリに 422 を返すところまで抱えていた。ここで分かったのは、コードを一箇所にまとめることと、責務を正しい層に置くことは別だ、ということでした。どの層が何を知るべきかは、「その層が何のためにあるか」で決まります。

同じ list_articles を、今度は「このパラメータは何をすべきか」で見ていくと、フィルタとソートの区別が曖昧になっていました。impact_level は「この重要度のものだけ見たい」というフィルタのはずが、>= で「その重要度以上」を返していた。さらに sort_by には、フィルタであるはずの impact_level が並び順の選択肢として紛れ込み、検索(q)は本来フィルタなのにソートまで上書きする特別扱いをしていました。整理すると、フィルタは対象を絞るもの、ソートは公開日順だけ。そう割り切ると、impact_level は完全一致に直り、sort_by というパラメータも、検索時のソート分岐も丸ごと不要になりました。概念を区別したことが、そのまま実装の単純さになった。コードを読んで「何をしているか」を追うだけでは、この >= の誤りには気づけません。「何をすべきか」から見たから、ずれが見えたのです。

次は /articles/{id}/similar の get_similar です。コサイン距離で関連記事を返す処理で、同じくベクトル検索を使うセマンティック検索(ArticleSearchService)に寄せてはどうか、という案が出ました。同じ技術を使っているのだから、まとめたほうがきれいに見える。けれど、get_similar がしているのは記事詳細画面で関連記事を数件出すことで、ユーザーが能動的に情報を探すセマンティック検索とは目的が違います。大事なのは、受け取るものや使う技術が似ているかではなく、何のための処理か。同じコサイン距離でも目的が違う以上、get_similar は記事を扱う ArticleService に残しました。これは、/news を「誰が何のために使うか」で分けたのと、リポジトリを「何のために読むか」で分けたのと、同じ判断でした。

その get_similar の異常系も、同じ軸で決めました。対象の記事が無い、embedding が未生成、件数が limit に満たない ―― こうした場合に 404 を返すか、状態を区別すべきか、という話も出ましたが、自分の考えは「見つからないなら、見つからない。その事実を返せばいい」でした。すべて 200 と空配列で統一する。理由は、このエンドポイントが返すのは「類似記事のコレクション」であって「特定の記事」ではないからです。記事そのものを返す /articles/{id} なら、無ければ 404 が正しい。けれど類似記事のコレクションは、空でも 200 + [] が筋で、検索のヒット 0 件が 404 にならないのと同じです。{id} がパスにあると 404 にしたくなりますが、返すものの性質で考えれば 200。余計な解釈を足さず、事実を返す。この割り切りで、Service は数行、Repository は 1 クエリに削れました。

この一連で繰り返し効いたのは、一つの軸でした。データ名でも、使っている技術でも、コードの薄さでもなく、何のために何をすべきか。それで見ると、コードがどの層にあるべきか、何を返すべきか、どこまでが一つの責務かが決まる。エンドポイントは、入口の形だけを眺めても良し悪しは分かりません。叩かれてから何をするのかを追い、それが目的に適っているかを問うて、初めて見えるものでした。

## 1.5 第1幕の終わりに

この幕で取り組んでいたのは、突き詰めると「自分でコードを理解する」ための練習でした。

自分は最初、コードを理解するために、正直に上から順番に処理を追っていきました。なんとか全体像を掴もうとする。けれど、何も分かっていない状態でそれを続けるのは、とても疲れる作業でした。
そこから少しずつ、「誰が何のために使うのか」「何を達成すべきなのか」から逆算して見るようになりました。その視点に変わったことで、ただ処理を追っているだけでは気づけなかったことが見えるようになっていきました。目的から見直して初めて違和感として捉えられたものでした。

一方で、同じ時期にはもう一本の流れも始まっていました。より良い設計を求めて書籍を読み、セキュリティや secure-by-design、値オブジェクトといった新しい概念を、自分のコードに取り入れようとしていたのです。次の幕では、少し時間を戻して、その試行錯誤を振り返ります。
