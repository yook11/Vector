# 第1幕 — 層を一枚ずつレビューする

プロローグに書いた「自分でコードを理解できるようになりたい」の、最初の一歩がこのレビューでした。コードベースを層ごとに区切り、一枚ずつ読んでいく——3月のモデルと DB に始まり、4月にはその外側へ:スキーマ(4/1)、ルーター(4/2-3)、パラメータ境界(4/4)、横断洗い出し(4/4)、そして副作用とトランザクション境界(4/11)。

ただ、この時点で「設計の判断軸を持とう」と意気込んでいたわけではありません。あったのは、もっと素朴な動機——**自分でコードを理解できるようになりたい**、という思いだけでした。当時はまだ、データがどう流れ、データベースの値がどんな形に変わってフロントエンドへ返っていくのか、その全体像すら掴めずにいました。だから自分の中で無いなりに、一番知識のあった——バックエンド、それもデータベースの側から見ていこう、と考えたのです。

なお、こうして進めるうちに、ちょうど 3月後半あたりから、セキュリティの本をきっかけに DDD や secure-by-design の考え方も入ってきています(その入口の話は第2幕)。

## 1.1 まず、モデルと DB から(3月)

足場になったのは、『達人に学ぶ DB設計 徹底指南書』と『SQLアンチパターン』の 2 冊でした。最初にやったのは、コードを書き換えることではなく、現状を棚卸しすることです。当時の DB 設計を一枚ずつ読み、本で学んだ「あるべき設計」と照らして、そのズレ(差分)をドキュメントに書き出していきました。

この差分を書き出しながら、初めて気づきました。**それまで自分は、自分の意図で設計をしていなかった**のだ、と。AI の出した形をただ承認してきただけで、読み直してみれば、スキーマは自分が思い描いていたものとは別物でした。自分の意図が、どこにも入っていなかったのです。

ところが、その差分を埋めようにも、最初はやり方すら分かっていませんでした。**models を書き換えれば DB も変わる**と思っていたのです。実際には、models は「あるべき姿」を宣言するだけで、現状の DB から差分を当てて近づけるのが Alembic の migration——この基本の構図を掴むこと自体が、この時期の作業の一部でした。

そこからの差分の解消は、本当に手探りでした。自分の意図とずれていた部分を見つけ、その理由を確かめながら直していく作業でした。

まず見えてきたのは、必要としていない概念が入り込んでいたことです。カテゴリには、本来の分類であるセクター別キーワードのほかに、AI の出力に含まれていた「投資観点タグ」(`competitive_edge` や `growth_catalyst` といった主観的なラベル)が別体系として入っていました。頼んだ覚えはなく、要らないと判断して削除しました。キーワードが複数のカテゴリに属せる形になっていたのも意図と違ったので、一つに絞りました。

多言語対応の翻訳テーブルも、よかれと思って入れたものの一つでした。しかし読者は日本の投資家であり、このアプリで届けたい価値には必要ないと判断して削除しました。

もう一つは、**性質の違うものが混じっていた**こと。`news_sources` には、ソースの正体や設定だけでなく、取得の実行状態(`etag`、`last_fetched_at`、連続エラー数など)まで列として溜め込まれていました。「そのソースが何か」と「いま取得がどう進んでいるか」は別物なので、実行状態の列を外し、テーブルにはソースの正体だけを残しました(Phase 3)。外した状態の置き場所——Redis や `fetch_logs`——はまだ手探りで、`fetch_logs` は後で畳むことになりますが、`news_sources` 自体は今も正体だけのテーブルのままです。

こうした作業を言葉にすると、**それぞれのものが、どんな属性を持ち、それをテーブルとカラムでどう表すべきか**——そもそも持たせる必要があるかも含めて——を、一つずつ決めていくことでした(この時期のメモは [dev-notes の 3月](../dev-notes/README.md))。進め方もまだ固まってはいませんが、壁打ちで理解を確かめ、一次情報で裏を取り、判断の理由を言葉に残す、という後の型の芽は、この時期に出はじめていました。

なお、見ていたのはテーブルとモデルの構造だけで、データアクセスの層ではありません。リポジトリという独立した層は、この時点ではまだ存在せず、クエリは router や service に直書きされていました。

## 1.2 「結局ユーザーは何を見たいのか」 — スキーマレビュー(4/1)

スキーマをどう直すべきか、最初は正しい設計が何か分からず、どのように見ていけばいいのかさえ全くわかりませんでした。足場として『API デザインパターン』(６月時点で全てはまだ読めていません)を読み始めます——3月の DB レビューで本を足場にしたのと同じやり方です。ただ今回は、思ったような成果が得られませんでした。本から得た「命名はシンプルに」「前置詞は危険信号」といった知識はあっても、エンドポイントを眺めて AI と壁打ちするだけでは答えが出ない。突破口は、「**結局ユーザーは何を見たいのか、UI で何が使われているのか、に立ち返らないと答えが出ない**」
という気づきでした。

その目で見ると、スキーマは UI から切り離されて設計されていました。アプリの中核は「AI 分析済みニュース」なのに、レスポンスは原文情報が主役で、分析が付属物のようにぶら下がっている——「分析が主役」という意図とは、主従が逆さまだったのです。さらに一覧と詳細が同じ型を共有していて、一覧カードでは使わない本文や推論テキストまで毎回返していました。**画面が何を表示し、ユーザーがそこで何をするか**から逆算すれば、一覧と詳細は必要とするデータが違う——そこで分析をトップレベルへ昇格させ、一覧用と詳細用を継承ではなく独立した 2 つの型に分けました。そしてもう一つ、より根本的な矛盾に気づきます——**このアプリの価値は AI が分析したニュースを見せることなのに、一覧は未分析の記事も通してしまう設計だった**。ユーザーは分析を見に来ているのに、設計はそれを保証していなかったのです。そこで「一覧は分析済みだけを返す」と決め、それを INNER JOIN で構造的に保証しました([レビュー編 1](../dev-notes/review-1-schema.md))。

この「使う側から考える」が、以後のすべてのレビューの通底軸になりました。

## 1.3 一番てきとうな層 — ルーター縦断と、最初のリジェクト(4/2-3)

ルーター層のレビューは、率直な驚きから始まりました。当時のメモには、「まさかルーターが一番てきとうに設計されているとは思わなかった」と残しています。実際に読んでみると、エンドポイントの分け方、責務の置き方、認証やエラーの扱いに一貫した判断軸がなく、この層に一番強い違和感を覚えました。

ただ、その違和感は同時に、API 設計の難しさを知る入口でもありました。DB には正規化理論のような手がかりがありますが、API は「まだ存在しないクライアントが、これをどう使うか」を想像しながら設計しなければならない。だからこそ、前章で掴んだ「使う側から考える」という視点を、ここでも判断の軸にしました。

当時の設計で最初に違和感があったのは、`news` ルーターが性質の異なる二つの責務を抱えていたことでした。任意認証で記事を読むユーザー向け API と、副作用を伴う admin 向けのパイプライン操作 API が同じ `/news` 配下にあり、利用者、権限、エラー設計が混在していました。

そこで、API の境界を「ニュースというデータ名」ではなく、**「誰が何を操作するのか」**を基準に切り直しました。ユーザーが扱うのは分析済みの記事なので `/articles`、admin が操作するのは処理工程なので `/pipeline` とし、router / repository / service / schema の 4 層を同じ境界で再編成しました。

その結果、URI とコード構造の両方から責務が読み取れるようになり、ユーザー向け機能と運用向け機能を独立して変更しやすくなりました。

この分割の過程では、AI の提案をそのまま採用するのではなく、自分の判断軸で検証し直す場面も増えていきました。

　一つ目は、keywords の CRUD ルーターです。キーワード自体は AI 分析の結果として生成され、記事やカテゴリの画面で表示されるため必要でした。
しかし、ユーザーや admin が直接作成・更新・削除する対象ではありません。表示に使うだけのデータに対して、未使用の CRUD API が用意されていたことが問題でした。

AI は「消すと書き直しになる」と保持を勧めましたが、今使っていないものを、将来使うかもしれないという理由だけで残すこと自体が負債になると判断しました。必要になったときに、その時点の要件に合わせて作り直すほうがよいと考え、土台になる model と VO だけを残して、ルーター、サービス、リポジトリ、スキーマ、フロントまで一式を削除しました。

　二つ目は、リポジトリ層の分割です。リポジトリは DB クエリを置く層ですが、当時の NewsRepository には、ユーザーに記事を表示するための読み取りメソッドと、裏側のパイプライン処理で使うメソッドが混在していました。

AI は、どちらも同じ記事・分析テーブルを参照するため、分けることに慎重でした。しかし、ここで重要なのは、どのテーブルを触るかではなく、何のためにそのデータを読むのかです。ユーザーに表示するための読み取りと、パイプラインを進めるための読み取りでは、使われ方も変わる理由も違います。そこで、読み取り系は ArticleRepository、パイプライン系は PipelineRepository に分割しました。

　三つ目は、記事一覧 API のパラメータ定義です。list_articles は 9 個のクエリパラメータを受け取っていましたが、当時はそれをルーター関数の引数に書き、さらに ArticleListParams に手で詰め替えていました。「どんなパラメータを受け取るか」という定義が二箇所に分かれていたのです。

Claude Code は、これを依存関数に切り出す案を出しました。ルーター本体は薄くなりますが、9 個の Query 定義と詰め替え処理が別の関数に移っただけで、二重定義は残ったままでした。そこで、定義を ArticleListParams クラス自身に持たせ、ルーターは Depends() で受け取るだけの形に直しました。

見た目には整理されたように見えても、重複の原因が残っているなら、問題は本質的には解消していません。
ここでは、コードが薄くなったかではなく、「定義を一箇所に集める」という目的を満たしているかを見ていました。

## 1.4 抽象の価値を、守る条件で測る(4/4-5)

前節の最後で、記事一覧 API のパラメータ定義を `ArticleListParams` クラスに集約しました。関数シグネチャと詰め替え処理に分かれていた定義を一箇所に寄せたことで、いったんはルーターを薄くできたように見えました。

ところが翌日、その `ArticleListParams` を見直すと、別の違和感が出てきました。本来このクラスは、「この API は `keywordId`、`category`、`sortBy` などを受け取る」というリクエストの形を宣言するためのものです。にもかかわらず、内部では、受け取った `category` が有効なカテゴリかを判定し、失敗すれば HTTP 422 を返すところまで担っていました。さらに、`sortBy` は生の文字列のまま下流へ渡っており、並び替え条件として安全に扱える形にもなっていませんでした。

この違和感を、自分の中では「受け取る形式を定義しているだけなのに、エラーを吐くようになっている」と言葉にしました。そこで、外から来た HTTP の入力をそのまま表す `Params` と、検証後にアプリ内部で安全に使える値を表す `Query` を分けました。カテゴリや並び替え条件の変換、失敗時の 422 の判断は API の入口であるルーター側に置き、下流には検証済みの `Query` だけを渡す形にしました。

ただし翌朝、この二型分離を維持する理由がなくなりました。きっかけは、パラメータ型(Pydantic のモデル)を FastAPI の `Depends()` で受け取ると、VO のフィールドが OpenAPI から警告なく落ちてしまう、という落とし穴の発見でした。調べ直すと、`Depends(Model)` ではなく `Annotated[Model, Query()]` で受ければ、Pydantic が parse 時に VO を直接検証してくれると分かりました。

これなら、外から何を受け取るかを明示し、受け取った値を安全な形に検証し、その変換と検証を一箇所にまとめられます。つまり、前日に分けた `Params` と `Query` の目的は、単一の型でも満たせるようになった。そこで、別の `Query` 型や変換ヘルパーは不要だと判断し、単一の型へ統合しました。

型の数は減りましたが、守りたかった条件は残りました。抽象の価値は、型を増やすことではなく、守りたい条件をどれだけ単純に満たせるかで決まるのだと、このとき実感しました。

同じ日の後半には、複数のルーターに散らばった共通処理を、どこまで一箇所に集めてよいかを見極めました。
全ルーターを確認すると、同じ種類の処理があちこちに散らばっていることが分かりました。
`NotFound` を 404 に変換する処理が 6 箇所に重複し、サービスの組み立て方もルーターごとに 4 通りに割れていました。同じことがこれだけ繰り返されているなら、1 箇所ずつ直すより共通の仕組みにまとめたほうがいい——数えること自体が、その判断の根拠になりました。例外を HTTP ステータスへ変換する対応表はアプリに一箇所だけ置いてルーターから `try/except` を消し、サービスの組み立ても `Depends()` のファクトリに集約しました。

ただし、何でもグローバルに寄せたわけではありません。共通化してよいのは、`NotFoundError` や `DuplicateError` のように、アプリ側で定義し、意図した場所でだけ投げる例外に限りました。`ValueError` のような汎用例外まで 422 に変換してしまうと、サーバー側のバグが入力エラーに見えてしまうからです。だから、共通化する範囲にも線を引きました([レビュー編 5](../dev-notes/review-5-sweep.md))。

## 1.5 副作用の層へ — トランザクション境界が見える(4/10-11)

4/10 のメモに、こんな自己観察が残っています——
「一度レビューしたはずのルーターから、また問題が見つかる。一回では完璧にならない」。

その翌日、初めて副作用系の API をレビューしました。対象は `news_sources` の `delete` / `activate` / `deactivate` です。それまで見ていたのは、記事やカテゴリを取得する読み取り系の API でした。状態を変える操作に踏み込んだことで、読み取り系では出てこなかった問いが見えてきます。

中心になったのは、書き込みの境界です。当時のメモには、「画面に返すにはメモリ上に値が要る / DB はその変化を反映する / これがセットで 1 つのトランザクションにまとまらないといけない」と書いています。途中で失敗したときに、一部の変更だけが DB に残ってはいけません。だから、1 つのユーザー操作は「全部成功するか、全部なかったことにするか」の単位として扱う必要がある。ここから、**1 ユースケース = 1 トランザクション**という考えにたどり着きました。

重要だったのは、その境界を Service や Repository の `commit()` に任せなかったことです。リクエストごとに渡される session の dependency がトランザクションを持ち、正常終了なら commit、例外なら rollback する。Service も Repository も `commit()` を書かないので、書き忘れる対象そのものがなくなります。VO で不正な値を存在できない形にするのと同じく、規約を人の注意ではなく構造で守る設計でした。

ただし、この形は HTTP 経路に向いたものです。worker 側の具体的な transaction 設計や監査の焼き方は、後の pipeline_events 設計で具体化していきます。

## 1.6 第1幕の終わりに

この幕で一番大きく変わったのは、既存実装の形をそのまま受け入れるのではなく、自分で考えて判断し始めたことでした。必要な概念か、使う側に合っているか、責務が混ざっていないか、抽象化や共通化が本当に問題を単純にしているかを、自分の言葉で確かめるようになっていきました。

その変化は、AI への向き合い方にも表れています。提案を結論として受け取るのではなく、判断材料として扱い、論拠を見て採否を決めるようになりました。

いま振り返ると、この幕で探していた答えは「よりわかりやすいコードとは何か」「持続可能なコードとは何か」に集約されます。当時その言葉で言えていたわけではありませんが、以後の判断は、この問いの延長線上にありました。

この幕は、二つの次の問いへつながります。一つは、3月のモデルレビューで出会った値オブジェクトです。これは第2幕で扱います。もう一つは、4/11 の副作用系レビューで開いた「書き込みをどこで確定するか」という問いです。これは第3幕へ続きます。
