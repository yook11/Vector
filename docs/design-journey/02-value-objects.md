# 第2幕 — 新しい概念に出会い、取り入れる

少し時間を巻き戻して、第1幕のレビュー対象がまだモデルとDBだった3月の話です。

私がセキュリティを学び始めたきっかけは、たまたま見た動画で、通称「徳丸本」——『体系的に学ぶ 安全なWebアプリケーションの作り方』——が紹介されていたことでした。何気なく読んでみようと思っただけで、特別な動機があったわけではありません。それでも読み進めるうちに、本に書かれていることが、次第に他人事ではなくなっていきました。

XSS、SQLインジェクション、オープンリダイレクト、入力値の検証、URLの扱い、外部サイトへのリクエスト。どれも、自分が書いたばかりのコードに心当たりのあるものばかりでした。

「こう書くと危ない」という具体例に出会うたびに、自分の実装を開き、同じ問題がないか確かめるようになりました。
そうして読み進めるうちに、Webアプリケーションで考慮すべき攻撃面の多さと、それらを意識せずにコードを書いていたという事実に、圧倒されていきました。


## 2.1 セキュリティから、新しい概念へ

当時の私は、まず「外部から入力を受け取る場所ごとに、バリデーションを書かなければならない」と考えました。
けれど、守るべきなのは入力を受け取る場所だけではありません。DBに問い合わせるところ、画面に表示するところ、外部APIにリクエストを送るところ。
SQLインジェクションを防ぎ、URLを検証し、HTMLに出力する値を安全に扱おうとすると、対策が必要な場所はアプリケーション全体へと広がっていきました。

もちろん、入力値の検証も、エスケープも、認可チェックも、URLの検証も欠かせません。問題は、それらの対策がコードのあちこちに散らばっていることでした。どこか一か所でも書き忘れれば、そこがそのまま穴になる。私が不安を感じたのは、対策を正しく実装しなければならない場所が増え続ける、その構造自体でした。

「本当に、このやり方でよいのだろうか」。そう考えていたときに出会ったのが、書籍『セキュア・バイ・デザイン』でした。
この本を読んで特に衝撃を受けたのは、ビジネス上のルールをコードで正確に表現することが、そのまま脆弱性を減らすことにつながる、という考え方でした。

それまで私にとってセキュリティは、危険な箇所を見つけるたびに、対策を後から足していくものに見えていました。しかし『セキュア・バイ・デザイン』を読み、設計そのものによって安全性を支えられるのだと知りました。
型によって、そもそも不正な値を表現できないようにする。それは単にコードを整理するためではなく、脆弱性を減らすことに直接つながる。この考え方に、当時の私は素直に感動しました。

早速自分のアプリケーションにもこの考え方を取り入れてみたくなりました。当時はまだ十分に理解できていたわけではありません。それでも、本に出てきた概念を自分のコードで試してみたいという気持ちが強くありました。

最初に手をつけたのは、値オブジェクトです。

URL、メールアドレス、金額、ステータスのような値を、ただの文字列や数値として扱うのではなく、意味を持つ型として表す。まずはそこから始めれば、自分のアプリケーションにもこの考え方を持ち込めるのではないか。そう思って、見よう見まねで導入を始めました。


## 2.2 値オブジェクトを導入する

最初に値オブジェクトとして切り出そうとしたのは、ニュースカテゴリを表す二つの値でした。URLに使う短い識別子と、画面に表示する名前です。

この二つは、当時の私がアプリケーションの中でも特に慎重に扱いたいと感じていた値でした。カテゴリの識別子はURLの一部に組み込まれ、表示名はユーザーが画面上で直接目にします。どちらもデータとしては文字列ですが、どんな文字列でもよいわけではありません。

振り返ってみると、この時点ですでに「この値には守るべき条件がある」と感じていたのだと思います。URLに使える文字の種類や長さ、表示名に許される文字数、空文字を許すかどうか。当時は、こうした最低限の制約をDBモデルに定義していました。

```python
class Category(SQLModel, table=True):
    slug: str = Field(max_length=50, unique=True, nullable=False, index=True)
    name: str = Field(max_length=50, unique=True, nullable=False)
```

ただ、DBモデルに制約を置いただけでは、アプリケーションの中で扱う slug と name は、どちらも str のままでした。
その値が必要な条件を満たしているかどうかも、コード上では判断できない状態でした。

そこで、値を作る時点で検証し、ルールを満たした値だけを slug や name として扱えるようにしようと考えました。一度作られた値は正しいものとして扱える。そうすれば、同じ条件を使う場所ごとに繰り返し書かずに済みます。

```python
# 小文字または数字で始まり、英小文字・数字・アンダースコアの1〜50文字に制限する。
class CategorySlug:
    _PATTERN = re.compile(r"^[a-z0-9][a-z0-9_]{0,49}$")

    def __init__(self, value: str) -> None:
        if not isinstance(value, str):
            raise TypeError(...)

        v = value.strip()

        if not self._PATTERN.fullmatch(v):
            raise ValueError(...)

        object.__setattr__(self, "_value", v)

#文字・数字・中黒・空白・ハイフンなどからなる1〜50文字に制限する。
class CategoryName:
    _PATTERN = re.compile(r"^[\w・ \-]+$", re.UNICODE)
    _MAX_LENGTH = 50

    def __init__(self, value: str) -> None:
        if not isinstance(value, str):
            raise TypeError(...)

        v = value.strip()

        if not v:
            raise ValueError(...)

        if len(v) > self._MAX_LENGTH:
            raise ValueError(...)

        if not self._PATTERN.fullmatch(v):
            raise ValueError(...)

        object.__setattr__(self, "_value", v)

# After
class Category:
    slug: CategorySlug
    name: CategoryName
```

値オブジェクトを導入したことで、Category 自身が文字列の形式を毎回確認する必要はなくなりました。
CategorySlug と CategoryName が作られている時点で、それぞれのルールを満たしていると判断できるからです。


### 安全なURL

なかでも、URLの扱いには強い問題意識を持ちました。このアプリケーションは外部サイトからニュースを取得するため、URLを扱う場面が数多くあります。本を読み進めるうちに、URLを単なる文字列として扱うことの危うさが、特に印象に残りました。
しかし当時は、ニュースを取得するためにサーバーが実際に接続するURLさえ、事前に検証していませんでした。

そこで導入したのが、URLを表す値オブジェクト SafeUrl です。
URLに求める条件をこの型に閉じ込め、SafeUrl として生成できた値だけ扱えば、安全になると考えました。

```python
_url_adapter = TypeAdapter(AnyHttpUrl)
_MAX_LENGTH = 2048


class SafeUrl(RootModel[str]):

    model_config = ConfigDict(frozen=True)

    @field_validator("root", mode="before")
    @classmethod
    def _validate(cls, v: Any) -> str:
        if not isinstance(v, str):
            msg = f"Expected str, got {type(v).__name__}"
            raise ValueError(msg)

        v = v.strip()

        if not v:
            msg = "SafeUrl must not be empty"
            raise ValueError(msg)

        if len(v) > _MAX_LENGTH:
            msg = f"SafeUrl must be at most {_MAX_LENGTH} chars"
            raise ValueError(msg)

        try:
            _url_adapter.validate_python(v)
        except ValidationError:
            msg = "SafeUrl must be a valid http or https URL"
            raise ValueError(msg) from None

        return v
```

SafeUrl は、値が文字列であること、前後の空白を除いて空でないこと、2048文字以内であることを確認します。
さらに、Pydanticの AnyHttpUrl を使い、HTTPまたはHTTPSのURLとして解析できることを検証します。これらの条件を満たさない値からは、SafeUrl を作れません。


## 2.3 ライブラリにも向き不向きがある

しかし、これらの値オブジェクトを実際に導入しようとすると、思っていたほど簡単にはいきませんでした。

書籍に登場する値オブジェクトは、クラスとして表現されていました。そのため当時の私は、`slug: CategorySlug` のように型ヒントを書けば、その値を受け取るときに検証が行われるのだと思っていました。型を指定すれば、それくらい自然に実現できるものだと思っていました。

ところが、型ヒントに自作したクラスを書くだけでは、必要な変換は自動で行われませんでした。

APIリクエストや外部から取得したデータは、文字列から値オブジェクトへ変換して検証する。APIレスポンスでは、値オブジェクトを再び文字列に戻す。DBへの保存時には文字列へ変換し、読み出すときには値オブジェクトへ復元する。アプリケーションの境界をまたぐたびに変換処理を実装する必要があり、想像していた以上に膨らんでいきました。

たとえば DB との境界では、その変換を値オブジェクトごとに専用の型として書きます。

```python
# SafeUrl を DB に保存・復元するための変換（SQLAlchemy の TypeDecorator）
class SafeUrlType(TypeDecorator[SafeUrl]):
    impl = String(2048)

    def process_bind_param(self, value, dialect):    # Python → DB
        ...  # None / SafeUrl / str を判定して文字列を返す（約 8 行）

    def process_result_value(self, value, dialect):  # DB → Python
        ...  # 文字列を読み出し、値オブジェクトに戻す（約 3 行）

type_annotation_map = {
    SafeUrl: SafeUrlType,
    # 値オブジェクトを増やすたびに、この変換型（約 20 行）を 1 つずつ足していく
}
```

しかも、これは DB との境界だけの話です。入力を受け取る側でも、API へ文字列で返す側でも、同じように変換を書く必要がありました。値オブジェクトを一つ「型」として通すだけで、これだけの実装が付いてきたのです。

この違和感の理由を調べるうちに、私は初めてSQLModelの仕組みを意識するようになりました。SQLModelは、データの検証や変換を担うPydanticと、PythonのオブジェクトをDBへ対応づけるSQLAlchemyを組み合わせたライブラリです。この二つをつなぐことで、型ヒントを中心に、少ないコードでモデルを定義できるようになっています。

この抽象化は、とても便利です。str や int といった標準的な型を使う限り、一つの型ヒントで、入力値の検証、Python上での値の扱い、DBカラムの定義までをまとめて扱えます。
ところが、独自の値オブジェクトをその流れに組み込もうとすると、普段はSQLModelに任せていたPydanticとSQLAlchemyの接続部分を、独自の型に合わせて自分で実装する必要が出てきたのです。

この経験から、ライブラリを選ぶ前に、まず自分が実現したい設計を明確にする必要があると学びました。
ライブラリにはそれぞれ異なる目的や設計思想があり、自分の設計に合ったものを選ぶことが、実装のスムーズさにつながります。


## 2.4 第2幕の終わりに

今振り返ると、この時期の私は、解決したい問題よりも、値オブジェクトを導入すること自体に意識が向いていました。

値にルールがあるという理由だけで、値オブジェクトにする必要はありません。その値がアプリケーションにとって重要な概念を表し、複数の場所で同じルールを守る必要があるなら、独立した型として定義する意味があります。生成時に検証を済ませれば、以降は正しい値として扱えるため、検証の重複や書き忘れを減らせるからです。

一方、特定の入力を受け取る場面だけで必要な検証なら、入力スキーマの field_validator で十分な場合もあります。当時の私は、この違いを判断する軸を持たないまま、本で知った値オブジェクトを「制約のある文字列」に当てはめようとしていました。

値オブジェクトを作る前に、まず考えるべきだったのは、「その型によって何を保証したいのか」でした。値オブジェクトが引き受けるのは、アプリケーションに必要な保証の一部です。

この切り分けができていなかったことは、SafeUrlに最もよく表れていました。当時の私は、「安全なURL」という型を一つ作れば、URLを安全に扱えるようになると考えていました。しかし、実際にこの型が保証していたのは、URLの形式上の正しさだけでした。

私は「URLを扱う」ということを、文字列がURLとして正しいか確認する問題としてしか捉えていませんでした。しかし、このアプリケーションは、そのURLを使ってサーバーから外部へ実際に通信します。URLという値ばかりに目を向け、その先で発生する通信について考えられていなかったのです。

URLとして正しいことと、そのURLへ通信してよいことは別の問題です。形式として正しくても、内部ネットワークを指しているかもしれません。最初の接続先に問題がなくても、リダイレクト先が安全とは限りません。URLの形式は型で保証できても、通信先の安全性は、実際に通信する境界で確認する必要があります。

この経験から、まず「何を保証しようとしているのか」を明確にしたうえで、何を型の不変条件とし、どの層に何を任せるのかを切り分ける必要があると学びました。その境界をどこに引くかは、今でも難しいと感じています。

それでも、この試みには大きな意味がありました。本で読んだ「制約を型に閉じ込める」という考え方を、自分のアプリケーションで初めて試すことができたからです。実装には悩みましたが、正しさを処理ごとの注意に任せるのではなく、型によって構造的に守るという視点を得られました。

第2幕では一つひとつの「値」に目を向けましたが、次第に、その値を使って何を行うのか、どの振る舞いをどこに置くのかという疑問が生まれていきます。
