# 第2幕 — 新しい概念に出会い、取り入れる

時間は少し戻ります。第1幕のレビューがまだモデルと DB に居た、3月の話です。

## セキュリティから、新しい概念と出会う

セキュリティに関しても重要だと考えるようになり、『体系的に学ぶ 安全な Web アプリケーションの作り方』、いわゆる徳丸本を読んでみることにしました。XSS、SQL インジェクション、オープンリダイレクト、SSRF。読み進めるほど、Web アプリケーションで考えなければならない攻撃面の多さに圧倒されました。

当時の自分は、まず「危険な入力を受け取る場所で、バリデーションをたくさん書かなければいけない」と考えました。SQL インジェクションを防ぐ、URL を検証する、HTML に出す値を扱う。けれどそう考えると、検証を書く場所はアプリケーション中に広がっていきます。

フォームの入力を受け取るところ、DB に問い合わせるところ、画面に表示するところ、外部 API にリクエストを送るところ。守るべき場所が増えるほど、「どこか一箇所で書き忘れれば、そこが穴になる」という不安が強くなりました。

もちろん、入力値検証、エスケープ、認可チェック、URL の検証はどれも重要です。ただ、それらがコードのあちこちに散らばっている状態では、一箇所忘れると脆弱性に繋がる。自分が不安だったのが、攻撃手法そのものだけではなく、対策の置き場所が増え続けることでした。

本当にこれでいいのだろうか？と思っていた、そのタイミングで出会ったのが、『セキュア・バイ・デザイン』でした。

この本で特に衝撃を受けたのは、ビジネスロジックを正確に表そうとすることが、そのまま脆弱性を減らすことにつながる、という点でした。

それまで自分にとってセキュリティは、危険な箇所に対策を足していくものに見えていました。しかし『セキュア・バイ・デザイン』を読んで、設計そのものが安全性を支えるのだと知りました。型でそもそも不正な値が入らないことを保証をする、単にコードを整理するためではなく、脆弱性を減らすことにつながる。この見方に、当時の自分は素直に感動しました。

そう考えると、早速自分のアプリケーションにもこの考え方を取り入れてみたくなりました。正直に言えば、当時はまだ十分に理解できていたわけではありません。それでも、本に出てきた概念を自分のコードで試してみたいという気持ちが強くありました。

最初に手をつけたのは、値オブジェクトです。

URL、メールアドレス、金額、ステータスのような値を、ただの文字列や数値として扱うのではなく、意味を持つ型として表す。まずはそこから始めれば、自分のアプリケーションにもこの考え方を持ち込めるのではないか。そう思って、見よう見まねで導入を始めました。

## 最初の値オブジェクト

最初に値オブジェクトとして切り出そうとしたのは、ニュースカテゴリの URL に使う短い識別子と、画面に表示する名前でした。

この二つは、当時の自分にはアプリケーションの中でも大事な値に見えていました。URL の一部になる値は外から直接見えるし、表示名はユーザーが画面で目にする名前です。どちらもただの文字列ではありますが、何でも入ってよい値ではありません。

だから、値オブジェクトという概念に出会う前から、この部分にはバリデーションを書いていました。URL に使える文字や長さのルール、表示名として許せる長さ、空文字をどう扱うか。そうした条件を持つ値だと、当時の自分は感じていたのだと思います。

そのうえで値オブジェクトという考え方に出会い、「このルールを値そのものに閉じ込められるのではないか」と考えました。使う場所ごとに `if` を書くのではなく、値を作るタイミングで検証する。一度作れた値は、ルールを満たしたものとして扱える。そうすれば、今後この値を使う場所が増えても、検証があちこちに散らばらずに済むと思いました。

今振り返ると、この判断は技術先行だったと思います。値オブジェクトを「どこに入れるとアプリケーションの重要なルールを守れるか」から考えるのではなく、「本で読んだ値オブジェクトをどこに入れられるか」という目でコードを見ていました。

そのため、制約のある文字列を見つけると、まず値オブジェクトとして切り出す候補にしていました。URL に使う識別子や表示名にルールがあることは確かですが、それを独立した型にする必要があるのか、それとも入力スキーマの `field_validator` で十分なのか。当時はまだ、その判断軸を持てていませんでした。

それでも、当時の自分にとっては、値オブジェクトを導入すること自体に意味がありました。本で読んだ「制約を型に閉じ込める」という考え方を、初めて自分のアプリケーションで試す入口だったからです。

## ライブラリにも向き不向きがある

しかし実際に書き始めると、すぐに手が止まりました。

書籍で読んだ値オブジェクトは、クラスとして表現されていました。なので自分も、型ヒントにそのクラスを書けばよいのだと思っていました。URL 用の識別子を表すクラスを作り、表示名を表すクラスを作る。そして、その型を指定すれば、値を受け取るときに検証が走り、「この値はただの `str` ではなく、意味を持つ値である」と表現できるはずだ。当時の自分には、それくらい素直にできるものだと思えていました。

けれど実際には、それだけでは済みませんでした。自分で定義したクラスをアプリケーションの中で使おうとすると、すぐにいくつもの変換を考える必要が出てきました。

外から入力される値は、最初はただの文字列です。それをどのタイミングで自作のクラスに変換し、どこで検証するのか。逆に API のレスポンスでは、自作クラスのまま返すのではなく、普通の文字列として返したい。その変換はどこに書くのか。

さらに DB に保存するときは、アプリケーション内では値オブジェクトとして扱いたい一方で、DB には文字列として保存する必要があります。DB から読み出した文字列を、もう一度値オブジェクトに戻す処理も必要になります。

入力、出力、DB 保存、DB 読み出しのそれぞれで、文字列と自作クラスを行き来させる必要がありました。自分がやりたいことは「型ヒントに自分のクラスを書く」だけのはずなのに、そのために必要なコードがどんどん増えていきました。

この違和感を調べていく中で、初めて SQLModel がどういうライブラリなのかを理解し始めました。SQLModel は、入力値の検証を担う Pydantic と、DB との対応づけを担う SQLAlchemy を重ねて、少ないコードでデータ定義を書けるようにしてくれるライブラリです。

その抽象化はとても便利です。普通に `str` や `int` を使っている限り、入力の検証、Python の型、DB のカラム定義が自然につながっているように見えます。けれど、自分で定義した値オブジェクトをその流れに差し込もうとすると、隠れていた接続部分を自分でつなぎ直す必要が出てきます。

このとき初めて、ライブラリには向き不向きがあるのだと実感しました。選んだ道具がその設計に向いているかどうかも、実装のスムーズさを大きく左右するのだと知りました。

## 2.3 URL の型と、通信先の安全性

URL に関しては、特に対応が必要だと考えていました。このアプリケーションでは、外部サイトからニュースを取得するため、URL を外部から受け取る場面が多くあります。そこで最初に導入したのが、URL の値オブジェクト `SafeUrl` でした。

記事の元 URL は、それまでただの文字列として扱われていました。空文字でも、`ftp:` でも、`javascript:` でも、型の上では同じ文字列です。しかしこのアプリケーションにとって、URL は単に画面に表示するだけの値ではありません。実際に外部へ通信するための入力でもあります。

だからこそ、URL をただの文字列として流し続けるのではなく、「安全な URL」として扱える境界を作りたいと考えました。この値にルールを持たせることは、セキュリティの学習で得た問題意識を、自分のコードに落とし込む最初の具体的な試みでした。

ただし、この時点の `SafeUrl` は完成形ではありませんでした。守れていたのは、主に URL の基本的な形です。HTTP/HTTPS であることや、URL として成立していることは確認できましたが、実際にその URL へ通信して安全かどうかまでは、まだ十分に扱えていませんでした。

後から追加したのは、URL の形ではなく通信先そのものを見る防御でした。IP リテラルが private IP や loopback を指していないか。ドメイン名を DNS 解決した結果が内部ネットワークに向いていないか。リダイレクトで別の危険な URL に飛ばされないか。さらに、検証時と接続時で DNS の解決結果が変わる DNS rebind を防ぐため、検証済み IP に接続先を固定する処理も入れました。

ただ、このあたりをどこまで `SafeUrl` の責務に含めるべきなのかは、今でも少し悩んでいます。URL の文字列だけを見て判断できることと、実際に通信する瞬間でなければ判断できないことは違います。HTTP/HTTPS であることや、IP リテラルが明らかに private IP であることは、値オブジェクトの中で扱いやすい。一方で、DNS 解決後の IP やリダイレクト、DNS rebind のような問題は、fetch する境界で扱う方が自然にも見えます。

それでも、この経験から学んだのは、「安全な URL」という一つの型を作ればすべて終わるわけではない、ということでした。型に閉じ込められる安全性と、実行時の境界でしか守れない安全性がある。その責任分離をどこに置くかは、今でも考え続けているテーマです。

### 2.4 第2幕の終わりに

この時期の自分は、値オブジェクトにかなり強い可能性を感じていました。入力を受け取るたびに `if` で検証するのではなく、不正な値をそもそも作れない形にする。セキュリティを学ぶ中で感じていた「検証を書き忘れたら穴になる」という不安に対して、型で構造的に守るという考え方は、とても大きな発見でした。

ただし、実際に手を動かしてみると、その理解はまだかなり素朴でした。ニュースカテゴリの識別子や表示名に値オブジェクトを導入したとき、自分は「制約がある値なら型にすべきではないか」と考えていました。けれど今振り返ると、利用箇所が限られているなら `field_validator` でも十分だったかもしれません。必要な抽象を選んだというより、本で学んだパターンを自分のアプリケーションに当てはめてみる段階だったのだと思います。

その一方で、実装してみたからこそ見えたものもありました。自分で定義した型を使おうとすると、入力、出力、DB 保存、DB 読み出しのそれぞれで文字列と値オブジェクトを行き来させる必要がある。そこで初めて、ライブラリには向き不向きがあり、速く書けることと、自分が表現したい設計を無理なく表現できることは同じではないのだと知りました。

`SafeUrl` の経験も同じでした。URL をただの文字列として流さず、「安全な URL」という境界を置いたことは、セキュリティの問題意識をコードに落とし込む大きな一歩でした。ただ、URL の形を検証することと、実際に通信して安全かどうかを保証することは別です。IP アドレス、DNS 解決、リダイレクト、DNS rebind のような問題は、型だけでは閉じきれず、実際に fetch する境界でも守る必要がありました。

この幕で得たのは、「値オブジェクトを使えば安全になる」という単純な確信ではありませんでした。むしろ、不変条件を構造で守ることの力と同時に、どの値を型にするのか、どこまでを型の責務にするのか、どこから先を実行時の境界で守るのかを考える必要がある、という感覚でした。

次の第3幕では、関心が「値」から「振る舞い」へ移ります。4月のレビューで層を上がっていく中で、Repository に置かれていたルールや状態変更を、ドメイン側へ引き戻す問いが開いていきます。
