# 第2幕 — 新しい概念に出会い、取り入れる

時間は少し戻ります。第1幕のレビューがまだモデルと DB に向かっていた、3月の話です。

セキュリティを学び始めたきっかけは、正直に言えば特別なものではありませんでした。たまたま見た動画で、徳丸本——『体系的に学ぶ 安全な Web アプリケーションの作り方』——が紹介されていて、読んでみようと思った。それだけです。

ところが読み進めるうちに、本の中の話が他人事ではなくなっていきました。XSS、SQL インジェクション、オープンリダイレクト、入力値の検証、URL の扱い、外部サイトへのリクエスト。どれも、自分が書いたばかりのコードに心当たりのある話でした。

「こう書くと危ない」と具体的に突きつけられるたびに、自分のコードはどうなっているのかを確認するようになりました。読み進めるほど、Web アプリケーションで考えなければならない攻撃面の多さに圧倒されていきました。

## 2.1 セキュリティから、新しい概念へ

当時の自分は、まず「危険な入力を受け取る場所で、バリデーションを書かなければいけない」と考えました。SQL インジェクションを防ぐ、URL を検証する、HTML に出す値を扱う。けれどそう考えると、検証を書く場所はアプリケーション中に広がっていきます。

ユーザーの入力を受け取るところ、DB に問い合わせるところ、画面に表示するところ、外部 API にリクエストを送るところ。守るべき場所はどんどん増えていきます。

もちろん、入力値検証、エスケープ、認可チェック、URL の検証はどれも重要です。ただ、それらがコードのあちこちに散らばっている状態では「どこか一箇所で書き忘れれば、そこが穴になる」。自分が不安だったのは、攻撃手法そのものだけではなく、対策の置き場所が増え続けることでした。

本当にこれでいいのだろうか？と思っていた、そのタイミングで出会ったのが、『セキュア・バイ・デザイン』でした。

この本で特に衝撃を受けたのは、ビジネスロジックを正確に表そうとすることが、そのまま脆弱性を減らすことにつながる、という点でした。

それまで自分にとってセキュリティは、危険な箇所に対策を足していくものに見えていました。しかし『セキュア・バイ・デザイン』を読んで、設計そのものが安全性を支えるのだと知りました。
「型で、そもそも不正な値が入らないことを保証する。それは単にコードを整理するためではなく、脆弱性を減らすことに直接つながる」このことに当時の自分は素直に感動しました。

早速自分のアプリケーションにもこの考え方を取り入れてみたくなりました。当時はまだ十分に理解できていたわけではありません。それでも、本に出てきた概念を自分のコードで試してみたいという気持ちが強くありました。

最初に手をつけたのは、値オブジェクトです。

URL、メールアドレス、金額、ステータスのような値を、ただの文字列や数値として扱うのではなく、意味を持つ型として表す。まずはそこから始めれば、自分のアプリケーションにもこの考え方を持ち込めるのではないか。そう思って、見よう見まねで導入を始めました。

## 2.2 値オブジェクトを導入する

最初に値オブジェクトとして切り出そうとしたのは、ニュースカテゴリの URL に使う短い識別子と、画面に表示する名前でした。

この二つは、当時の自分にはアプリケーションの中でも特に気になる値でした。カテゴリの識別子は URL の一部として表示される値であり、表示名はユーザーが画面で直接目にする名前です。どちらもただの文字列ではありますが、何でも入ってよい値ではありません。

今振り返ると、この時点ですでに「この値には守るべき条件がある」と感じていたのだと思います。URL に使える文字や長さ、表示名として許せる文字数、空文字をどう扱うか。最低限の制約は DB モデル側に置いていました。

```python
class Category(SQLModel, table=True):
    slug: str = Field(max_length=50, unique=True, nullable=False, index=True)
    name: str = Field(max_length=50, unique=True, nullable=False)
```

ただ、その条件をどこに置けばよいのかは、まだ分かっていませんでした。slug と name はどちらも str のままで、型としては「カテゴリの URL 識別子」や「画面に表示する名前」ではなく、ただの文字列として扱われていました。

そのうえで値オブジェクトという考え方に出会い、「このルールを値そのものに閉じ込められるのではないか」と考えました。値を作る時点で検証し、一度作れた値はルールを満たしたものとして扱う。そうすれば、同じ条件を使う場所ごとに書かずに済むと思ったのです。

```python
class CategorySlug:
    _PATTERN = re.compile(r"^[a-z0-9][a-z0-9_]{0,49}$")

    def __init__(self, value: str) -> None:
        if not isinstance(value, str):
            raise TypeError(...)

        v = value.strip()

        if not self._PATTERN.fullmatch(v):
            raise ValueError(...)

        object.__setattr__(self, "_value", v)
```

ただ、今振り返ると、この判断は少し技術先行でした。カテゴリの slug や表示名にルールがあることは確かです。けれど、それを独立した型として守るべきなのか、入力スキーマの field_validator で十分なのか。当時はまだ、その判断軸を持てていませんでした。本で読んだ値オブジェクトを、制約のある文字列に当てはめようとしていたのだと思います。

それでも、当時の自分にとっては、値オブジェクトを導入すること自体に意味があったと思います。本で読んだ「制約を型に閉じ込める」という考え方を、初めて自分のアプリケーションで試す入口だったからです。


### 安全なURL

URL に関しては特に強い問題意識を持っていました。本を読むなかでも、URL の扱いは強く印象に残った部分で、対策をしなければいけないと感じていました。
このアプリケーションは外部サイトからニュースを取得するため、URL を扱う場面が多くあります。

当時はニュースを取得するために、サーバーから実際に外部へ通信する URL に対しても検証はありませんでした。

そこで導入したのが、URL の値オブジェクト `SafeUrl` です。
正しい URL の形式を型として定義し、その型を使えば安全に扱えるようになると考えました。

```python
_url_adapter = TypeAdapter(AnyHttpUrl)
_MAX_LENGTH = 2048


class SafeUrl(RootModel[str]):

    model_config = ConfigDict(frozen=True)

    @field_validator("root", mode="before")
    @classmethod
    def _validate(cls, v: Any) -> str:
        if not isinstance(v, str):
            msg = f"Expected str, got {type(v).__name__}"
            raise ValueError(msg)

        v = v.strip()

        if not v:
            msg = "SafeUrl must not be empty"
            raise ValueError(msg)

        if len(v) > _MAX_LENGTH:
            msg = f"SafeUrl must be at most {_MAX_LENGTH} chars"
            raise ValueError(msg)

        try:
            _url_adapter.validate_python(v)
        except ValidationError:
            msg = "SafeUrl must be a valid http or https URL"
            raise ValueError(msg) from None

        return v
```

この SafeUrl で、URL の「形」——HTTP/HTTPS であること、URL として構文的に成立していること——を、型として保証できるようになりました。


## 2.3 ライブラリにも向き不向きがある

しかし、これらの値オブジェクトを導入しようとした時に、ある問題が起きました。

書籍で読んだ値オブジェクトは、クラスとして表現されていました。なので自分も、型ヒントにそのクラスを書けばよいのだと思っていました。型を指定すれば、その値を受け取るときに検証され、「この値はただの文字列ではなく、意味を持つ値である」と表現できるはずだ。当時の自分には、それくらい自然にできるものだと思えていました。

それなのに、単純に型ヒントへ自分のクラスを書くだけではうまくいきませんでした。

外から値を受け取るとき（API リクエストや、外部から取得したデータ）は、文字列を値オブジェクトに変換して検証し、API レスポンスでは文字列に戻して返し、DB とのあいだでも保存と読み出しで変換する。この境界を跨ぐたびに変換コードが必要になり、実装は思っていた以上に膨らんでいきました。

たとえば DB との境界では、その変換を値オブジェクトごとに専用の型として書きます。

```python
# SafeUrl を DB に保存・復元するための変換（SQLAlchemy の TypeDecorator）
class SafeUrlType(TypeDecorator[SafeUrl]):
    impl = String(2048)

    def process_bind_param(self, value, dialect):    # Python → DB
        ...  # None / SafeUrl / str を判定して文字列を返す（約 8 行）

    def process_result_value(self, value, dialect):  # DB → Python
        ...  # 文字列を読み出し、値オブジェクトに戻す（約 3 行）

type_annotation_map = {
    SafeUrl: SafeUrlType,
    # 値オブジェクトを増やすたびに、この変換型（約 20 行）を 1 つずつ足していく
}
```

しかも、これは DB との境界だけの話です。入力を受け取る側でも、API へ文字列で返す側でも、同じように変換を書く必要がありました。値オブジェクトを一つ「型」として通すだけで、これだけの実装が付いてきたのです。

この違和感を調べていく中で、初めて SQLModel がどういうライブラリなのかを理解し始めました。SQLModel は、入力値の検証を担う Pydantic と、DB との対応づけを担う SQLAlchemy を重ねることで、少ないコードでデータ定義を書けるようにしてくれるライブラリです。

その抽象化はとても便利です。普通に str や int を使っている限り、入力の検証、Python の型、DB のカラム定義が自然につながっているように見えます。けれど、自分で定義した値オブジェクトをその流れに差し込もうとすると、普段は SQLModel が隠してくれていた接続部分を、自分でつなぎ直す必要が出てきました。

このとき初めて、ライブラリには向き不向きがあるのだと実感しました。便利な抽象化でも、自分が表現したい設計と相性が悪い場合があります。選んだライブラリがその設計に向いているかどうかは、実装のしやすさを大きく左右するのだと知りました。


## 2.4 第2幕の終わりに

今振り返ると、値オブジェクトは「その値にルールがあるか」で決めるものではありませんでした。ルールがあること自体は、多くの値に当てはまります。大事なのは、それがアプリケーションにとって重要な概念なのかを考えることでした。

`SafeUrl` がそうでした。「安全な URL」という型を一つ作れば、それで安全になると思っていました。でも実際に保証できていたのは、URL の「形」——HTTP/HTTPS であることや、構文として正しいこと——だけでした。その URL へ実際に通信して安全かどうかは、別の話です。後から、「形として正しいか」と「サーバーが本当に通信してよい相手か」は、層の違う別の問題だと分かりました。

大事だったのは、まず「何を保証したいのか」を考えることでした。値オブジェクトは、その保証の一部を引き受けるものでしかありません。どこまでを型の不変条件にし、どこから先を別の境界に任せるのか。型に何を保証させるのかは、今でも難しいと感じています。

その一方で、まず手を動かしてみたからこそ得られたものもありました。特に大きかったのは、「型で構造的に守る」という考え方に触れられたことです。
実装には悩みましたが、この視点を得られたこと自体が、自分にとって大きな発見でした。

次の第3幕では、関心が「値」から「振る舞い」へ移っていきます。

値オブジェクトを扱う中で、ただの文字列や数値だと思っていたものにも、アプリケーション固有の意味や守るべきルールがあることを知りました。その感覚を入口にして、少しずつ「ドメイン」という考え方に触れていくことになります。
