# 第3幕 — 小さな違和感をきっかけに、振る舞いの置き場を探す

第2幕で、形だけではあったかもしれませんが、値オブジェクトを導入しました。その値が存在する時点で「検証済みである」ことを型が保証してくれる。不正な値を、少なくともアプリケーションの中では作りにくくできる。その手応えがありました。

その流れで、次は「振る舞い」もドメインの中に置いてみたい、と考えるようになります。ただ、どの振る舞いをどこに置くべきかまでは、まだ見えていませんでした。新しく知った考え方に惹かれながら、自分のコードを読み返しては、router や service、repository に処理を押し込めていることに、少しずつ違和感を覚え始めていた段階です。

コードを読んで引っかかった小さな違和感に対して、一つずつ「なぜこの処理は分かりにくいのか」「この部分は何を表しているのか」と問い直し、その違和感を、メソッドを置く場所、データの型として、一つずつ直していった時期でした。

その最初の入口になったのが、管理画面でニュースソースを有効化・無効化する、ごく小さな処理でした。管理者しか触らない設定変更で、ここに大きな設計投資をする必要は今考えるとなかったように思います。それでも、「状態を変える」という処理をどこに置くのかを、初めて考えることができました。ここで身についた「これはどこに置くべき処理なのか」という問いの立て方が、その後の設計でも、繰り返し効いてくることになります。

## 3.1 小さな違和感

最初に引っかかったのは、有効化・無効化という操作が、値の代入と保存として書かれていたことでした。

ニュースソースを有効化する処理では、Service が `source.is_active = True` のように値を決め、そのオブジェクトを Repository の `save` に渡していました。動かすだけなら、それで十分です。けれどコードを読んでも、「有効化する」という意図は見えてきません。`is_active` に `True` を入れて保存する、という手順を読んで初めて、これは有効化なのだと理解する必要がありました。無効化も同じで、`False` を入れて保存する、という形になっていました。

そこで、`save` を直接呼ぶのをやめ、`activate()` / `deactivate()` という名前のメソッドを作りました。最初にその動詞を置いた場所は Repository です。呼ぶ側は「有効化しろ」「無効化しろ」と言えばよく、読む側も、代入と保存の組み合わせから意図を推測するのではなく、メソッド名から操作の意味を読めるようになります。

名前を与えたことで、操作の意図は見えるようになりました。けれど次に引っかかったのは、その Repository の `activate()` が、状態を変えるだけでなく `commit` まで行っていたことでした。

最初は、トランザクションも Repository が管理するものだと思っていました。けれど考えていくと、Repository がその場で `commit` してしまうと、同じユースケースの中で複数の変更を一つの操作として扱いにくくなります。片方の変更だけが確定し、もう片方が失敗するような中途半端な状態が残るかもしれない。そこで、Repository から `commit` を外し、確定の境界はリクエスト単位の transaction に寄せました。

`commit` を外すと、Repository の `activate()` に残るのは「`is_active` を変える」という処理だけになります。すると今度は、その残った処理そのものをどこに置くのか、が次の問いになりました。確定の手続きと切り離して初めて、`is_active` を変えることが、保存とは別の何かに見えてきたからです。

このとき、自分は DB の値を変更するときの流れをまだうまくイメージできていませんでした。DB に保存されている値を変えるのだから、それは Repository の仕事ではないかと思っていました。

けれど実際には、更新処理は DB の値を直接その場で書き換えているわけではありません。まず DB から `NewsSource` を取得し、アプリケーションのメモリ上にオブジェクトとして展開する。そのオブジェクトの属性を変更し、最後にトランザクションが確定されるタイミングで DB に反映されます。この流れを理解して、`source.activate()` は保存の手続きではなく、取得済みの `NewsSource` オブジェクトの状態を変える処理なのだと分かりました。

保存ではなくオブジェクトの状態を変えることなら、それは「どこに保存するか」ではなく「`NewsSource` がどう振る舞うか」の話です。だとすれば、この処理を Repository ではなく `NewsSource` 自身に持たせることもできる——その見方を、ここで知りました。「有効化される」「無効化される」という状態遷移として表せる、ということです。

ただ、今振り返ると、この判断には動機が先行していたと思います。「振る舞い」と呼べそうなものを見つけると、とにかく Model に置いてみたかった。実際にやっているのは `is_active` と `updated_at` を変えるだけで、この小ささなら Repository に `activate()` / `deactivate()` を置き、Service から呼ぶ形でも十分だったかもしれません。むしろ Model に移したことで、処理を追うには Service から Model へ一度見に行く必要が生まれました。振る舞いを持たせること自体が目的になっていて、あまり時間をかけるべきところではなかったと感じています。

それでも、メモリ上の値と DB の値の違いを区別できたこと、そして DB に保存される値を変える処理でも必ず Repository に置くわけではないと分かったことが、この小さな実装の収穫でした。

## 3.2 コードの置き場所を、作業の単位で考える

この頃、コードを直すたびに引っかかっていたことがありました。たとえば「記事を分析する処理」を追おうとすると、`services/`、`ai/`、`tasks/`、`infra/` と複数のフォルダを行き来する必要があります。どのファイルも、置き場所として間違っているわけではありません。けれど、やりたいこと（記事を分析する、という一つの作業）と置き場所の名前（技術の種類）が一致しておらず、処理の流れが直感的に分かりにくくなっていました。

当時の自分は、それをまだ「価値のある部分に設計投資する」とまでは言語化できていなかったと思います。もっと素朴に、このアプリでよく編集する場所、これから機能を追加したい場所、それらが散らばっていることに引っかかっていました。

最初に手をつけたのは、AI 分析まわりでした。当時は `services/ai_analyzer.py` や `services/gemini_analyzer.py`、`ai/embedding/`、`tasks/analysis_tasks.py` のように、分析に関わる処理が技術名ごとの場所に散っていました。ここを最初に選んだ理由は、AI モデルの差し替え、プロンプトの見直しのように、今後変更することが明確だったからです。作業の単位でまとめて、分かりやすくしたいと考えました。

最初は `ai/` というフォルダに整理しようとしましたが、すぐにそれだけでは足りないと気づきます。ここで扱っていたのは、単に AI API を呼ぶ処理ではなく、記事に意味を与える「分析」という業務上のまとまりでした。そこで `app/analysis/` を作り、整理を行いました。

その過程で、`analysis/` の内部をどこまで分けるかには迷いました。最初は、分析の中でやっていたこと ―― 要約・翻訳・分類・キーワード抽出 ―― を、それぞれ別ファイルに分ければ構造が理解しやすくなると考えていました。けれど当時の分析は、要約も翻訳もキーワードも一つのプロンプトでまとめて処理していました。操作ごとにファイルを割ると、ファイルの分かれ方と実際の処理の単位がズレて、一つの呼び出しをまたいで状態を共有する必要が出てきます。結局、内部は AI 呼び出しと埋め込みの二つに分けるに留め、それ以上の細分化は、プロンプトを分ける理由が見えてからにすることにしました。

この時点での分け方の軸は、価値の中核というより、作業の単位・変更の単位でした。あるドメインが、ほかとは別の理由で変わっていくか ―― つまり独立した変更の軸を持っているか。そして、その変更がこれから具体的に起きると見えているか。この二つを見ていました。「いつか大きくなるかもしれない」ではなく、「もう触る理由が見えている」ことを基準にしていました。

## 3.3 ドメインの不変条件を、構造と型で強制する

次に気になったのは、「このデータは次の工程に進めるのか」をどう表すかでした。

当時、収集した記事はすべて `NewsArticle` という一つのテーブルで扱っていました。記事を見つけた段階で行を作り、本文を取得できたら同じ行の `original_content` に書き込み、データが足りないものは、同じ行にフラグを立てて「分析に進めない記事」として区別していました。

この構造では、記事そのもののデータがどこまで揃っているのかと、システムがその記事をどの段階まで処理したかが、同じ場所に置かれます。URL、タイトル、公開日、本文のような記事の属性と、本文取得待ち、取得スキップ、分析可能といったパイプライン上の状態が、一つの `NewsArticle` の中で混ざっていました。

ただ、この違和感には一度、手を入れたつもりでいました。少し前にパイプラインを設計し直したとき、「記事がいまどの処理段階なのか」は `NewsArticle` の属性ではないと考え、試行回数や進行中の段階を表す列を外しました。その時点では、記事そのものと、パイプラインの制御に必要な状態を切り離せたつもりでいました。

でも外せていたのは進行を管理する情報だけで、取得できなかった・スキップした、という結果は、まだフラグとして残ったままでした。

この混在は、本文を取得する処理やリトライ処理、あるいは「どの記事を分析に回すか」を選ぶ処理で、具体的な読みにくさとして表れました。同じ `NewsArticle` を見ながら、本文の有無やフラグの値を手がかりに、それが本文取得待ちなのか、取得をスキップしたものなのか、もう分析に回せるものなのかを毎回読み解く必要がありました。

最初にやったのは、「この記事は分析に進めるか」を判定するメソッドを一つ用意することでした。それまでは「本文があって、スキップされていない」という同じ条件式が、一覧・検索・ウォッチリストといった各所に散らばっていました。それを一箇所にまとめました。

ただ、条件をまとめていくうちに、もう一つ気づいたことがありました。判定を書く代わりに、「分析対象だけが入る場所」を作れば、その判定自体が要らなくなるのではないか。行があること自体が「分析に回せる記事だ」という意味になるなら、`if` で確かめる必要がなくなる。状態を後から判定するのではなく、構造そのもので分ける。

そこで、一つのテーブルにすべてを載せるのをやめ、記事の段階ごとに保存先を分けることにしました。ひとつは、記事を見つけた時点の URL やタイトルなど、まだ本文がないメタデータを置く場所。もうひとつは、本文の取得に成功し、分析対象になったものだけを置く場所です。

分析対象になった記事では本文を必須にしました。本文を取得できて初めて行が作られるので、その行が存在すること自体が「本文があり、分析へ回せる」という意味になります。

この設計で、状態を後から確かめる必要がなくなりました。「この記事は本文があるか」を処理のたびに見ていたのが、「行があるか」だけで分析に回せるか決まります。状態を管理していたフラグも、行があるかないかという構造に置き換わりました。

本文取得処理の戻り値でも、同じ問題が見えてきました。

当時結果は `ContentFetchResult` という型で表していました。`status` には「取得できた」「すでに取得済みだった」「スキップした」のどれかが入り、`article_id` には分析に回す記事の ID が入る、という形です。

ただ、この型では `status` と `article_id` の関係までは保証できていませんでした。型の上では、「取得成功なのに ID がない」「スキップなのに ID がある」という組み合わせを作れてしまいます。

ここで気づいたのは、結果を型で表すだけでは足りない、ということでした。結果の種類と、その結果が持てるデータの組み合わせまで、型の形にする必要がある。そこで `Fetched`、`AlreadyExists`、`Skipped` に分け、分析に回せる結果だけが ID を持つようにしました。

共通していたのは、「実行時に確かめる」を「構造で表す」に置き換える、という発想でした。テーブルの形でも、戻り値の型でも、あとから条件分岐で読み解いていたものを、最初から間違った形で表せないようにしていったのです。

## 3.4 第3幕の終わりに

ここで触れた具体的な設計は、今の形にはほとんど残っていません。`NewsArticle` はなくなり、記事テーブルの分割や `Fetched | AlreadyExists | Skipped` という戻り値の型も、その後のリファクタリングで作り直されました。

今は、外部ソースから取れた内容を、まず `AnalyzableArticle` か `ObservedArticle` に変換します。本文や公開日時などが揃っていれば、そのまま分析に進める `AnalyzableArticle` になる。足りないものは、取れた事実だけを持つ `ObservedArticle` として残し、HTML から補完できる段階で `ReadyForArticleCompletion` として取り出します。そこで条件を満たしたものだけが、改めて `AnalyzableArticle` として保存され、次の分析工程へ渡されます。

つまり、この章は正しい設計に到達した話ではありません。当時の設計にはまだ問題が多く、処理の状態と記事の概念を分けようとしながら、今度は本文取得の結果を型に背負わせすぎていました。リトライ、監査、補完ポリシー、完成した記事の条件も、まだ十分には分かれていませんでした。

それでも、この時期に得たものは大きかったと思います。毎回 `if` で読み解いていた状態を、テーブルの形や型の形に移す。記事を「いまどの工程にいるか」で読むのではなく、その段階にふさわしい名前と構造を与える。その発想は、後の設計にも残っていきました。

共通していたのは、「実行時に確かめる」を「構造で表す」に置き換える、という向きでした。ただし、この時点ではまだ、その構造をどこまで細かく分けるべきか、どの責任を型に持たせるべきかまでは測れていませんでした。次の第4幕では、設計の形そのものではなく、「このアプリで価値を生むものは何か」から考え直していきます。
