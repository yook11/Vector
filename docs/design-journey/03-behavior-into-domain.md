# 第3幕 — 小さな違和感をきっかけに、振る舞いの置き場を探す

第2幕で、形だけではあったかもしれませんが、値オブジェクトを導入しました。その値が存在する時点で「検証済みである」ことを型が保証してくれる。不正な値を、少なくともアプリケーションの中では作りにくくできる。その手応えがありました。

その流れで、次は「振る舞い」もドメインの中に置いてみたい、と考えるようになります。ただ、どの振る舞いをどこに置くべきかまでは、まだ見えていませんでした。新しく知った考え方に惹かれながら、自分のコードを読み返しては、router や service、repository に処理を押し込めていることに、少しずつ違和感を覚え始めていた段階です。

コードを読んで引っかかった小さな違和感に対して、一つずつ「なぜこの処理は分かりにくいのか」「この部分は何を表しているのか」と問い直し、その違和感を、メソッドを置く場所、データの型として、一つずつ直していった時期でした。

その最初の入口になったのが、管理画面でニュースソースを有効化・無効化する、ごく小さな処理でした。管理者しか触らない設定変更で、ここに大きな設計投資をする必要は今考えるとなかったように思います。それでも、「状態を変える」という処理をどこに置くのかを、初めて考えることができました。ここで身についた「これはどこに置くべき処理なのか」という問いの立て方が、その後の設計でも、繰り返し効いてくることになります。

## 3.1 小さな違和感

3.1 小さな違和感
最初に引っかかったのは、命名でした。ニュースソースを有効化する処理は、Service が is_active を書き換え、Repository の save に渡す形で書かれていました。動かすだけなら、それで十分です。ただ、save という名前は「保存する」としか言っていません。そこに「有効化する」「無効化する」という意図が埋もれていることが、最初の違和感でした。
そこで、toggle をやめて activate() / deactivate() に分けました。最初にその動詞を置いた場所は、Repository です。save の汎用性に意図を埋もれさせるのではなく、「有効化する」「無効化する」という操作そのものに名前を与える。呼ぶ側は「有効化しろ」とだけ言えばよく、読む側も名前だけで意図が分かる。これが最初の一手でした。
ただ、この時点の Repository の activate() は、状態を変えるだけでなく、その変更を DB に確定させ、更新後のオブジェクトを返すところまで担っていました。is_active を True にするという NewsSource 自身の状態変化と、それを保存して確定させるという永続化の手続きが、同じメソッドの中に同居していたわけです。
先に手をつけたのは、後者の方でした。Repository から commit と refresh を外し、変更を確定させる境界を、リクエスト単位のトランザクション(get_session())に寄せました。Repository は保存はするが、いつ確定させるかは持たない。この時点で activate() は、外部の状態確定から切り離され、「is_active を変える」という純粋な状態変化だけを残したメソaddになりました。
確定の手続きを剥がしてみると、残った activate() の中身が、もう Repository のものではないことがはっきりしました。is_active を True にするのは、データを保存する操作ではなく、NewsSource というエンティティが「有効化される」という状態遷移そのものです。そこで activate() / deactivate() を NewsSource 自身に移しました。is_active の変更は、その Model メソッドの中だけで起きる。Repository は対象を取り出して保存する境界に徹し、状態を変える意味には関与しない。Service は対象を取得して source.activate() を呼び、レスポンスを組み立てるだけになりました。

## 3.2 コードの置き場所を、作業の単位で考える

この頃に気になり始めたのは、コードを直したいときの見通しの悪さでした。

たとえば「記事を分析する処理」を追おうとすると、`services/`、`ai/`、`tasks/`、`infra/` を行き来する必要がありました。どのファイルも間違った場所にあるわけではありません。けれど、やりたいことと置き場所の名前が一致しておらず、よく触る処理ほど、どこから読めばいいのか直感的に分かりにくくなっていました。

当時の自分は、それをまだ「価値のある部分に設計投資する」とまでは言語化できていなかったと思います。もっと素朴に、このアプリでよく編集する場所、これから機能を追加したい場所、これらが散らばっていることに引っかかっていました。

最初に手をつけたのは、AI 分析まわりでした。当時は `services/ai_analyzer.py` や `services/gemini_analyzer.py`、`ai/embedding/`、`tasks/analysis_tasks.py` のように、分析に関わる処理が技術名ごとの場所に散っていました。

最初は `ai/` という箱に整理しようとしましたが、すぐにそれだけでは足りないと気づきます。ここで扱っていたのは、単に AI API を呼ぶ処理ではなく、記事に意味を与える「分析」という業務上のまとまりでした。そこで `app/analysis/` を作り整理を行いました。

この時点での分け方は、まだ「価値の中核に設計投資する」というより、作業単位・変更単位に近いものでした。分析を直すときに見る場所、収集を直すときに見る場所、検索を直すときに見る場所を分けたい。さらに、それぞれが今後どのくらい大きくなりそうか、変更理由が別々になりそうかを見ていました。

だから、`analysis/` は切り出す価値があると感じました。AI モデルの差し替え、埋め込み、プロンプトの見直しなど、今後も触る理由が具体的に見えていたからです。一方で、分析の内部を翻訳・要約・分類のようにさらに細かく分けるところまでは、まだ根拠が足りませんでした。分けられそうだから分けるのではなく、変更の理由が見えてから分ける。そのくらいの感覚でした。

## 3.3 ドメインの不変条件を、構造と型で強制する

次に気になったのは「このデータは次の工程に進めるのか」をどう表すかでした。

きっかけは抽象的なものではなく、本文取得まわりのコードを書いている最中の、具体的な手触りでした。`ContentFetchService` やパイプラインのエラー処理・リトライ処理を書いていると、同じ `NewsArticle` を収集・本文取得・分析の各工程で「これはいま何なのか」と毎回読み解く必要がありました。RSS で見つけただけなのか、本文取得を待っているのか、取得に失敗したのか、もう分析に進めるのか。すべて同じ `NewsArticle` として扱い、その違いを `original_content` の有無や `skip_content_fetch` という真偽値で判断していました。その読み解きが、あちこちの `if` になっていました。

引っかかっていたのは `if` の数そのものではなく、その奥にあるものでした。一つの `NewsArticle` に、記事そのものの事実(URL・タイトル・本文)と、パイプラインの処理結果(取得したか・スキップしたか・失敗したか)という、性質の違う二つの軸が同居していた。コードが読みにくいというより、状態の違いが構造に表れていなかった、というのが近いです。

今振り返ると、この違和感は二段階で来ていました。原因の側は、もっと早く言葉になっていました。少し前にパイプラインを設計し直したとき、「処理の途中の状態を DB に持たせるのは、そのエンティティの関心ごとではない」と考え、過渡的な状態を追っていた列(取得時刻や試行回数など)を落とし、途中の状態は別の仕組みへ逃がしていました(`c4c586b2`, 2026-03-28)。ただ、このとき結果(取得失敗・スキップ)だけは `skip_content_fetch` という真偽値として `NewsArticle` の行に残してしまいました。あとで違和感の正体になるこのフラグは、当時の自分が「正しい原則」から入れたものでした。原則は合っていたのに、適用が途中状態の軸で止まっていたのだと思います。

数週間後、本文取得を独立させる作業の中で(`07cadde2`, 2026-04-16)、同じ原則が今度は結果の軸で戻ってきます。「記事とは、分析する価値のある記事だと定義してしまえば、あちこちの条件分岐を書かずに済むのではないか」と気づいたのがこのときでした。同じ気づきが二度来て、二度目でようやく構造まで降ろした、という感覚に近いです。

そこから、状態を後から判定するのではなく、構造そのもので分ける方向へ進みます。

まず構造で。記事テーブルを `DiscoveredArticle`(収集しただけのメタデータ)と `Article`(本文が `NOT NULL`)の二つに割りました(`376c12ec`, 2026-04-20)。コミットの言葉では「articles 行が存在する = 分析可能(analysis-ready)という構造的不変条件を確立する」。「本文があるか」を実行時に確かめるのではなく、〈`articles` に行がある〉という事実そのものを不変条件にした——値の検証から、テーブルの形による保証への一歩です。この分割で、三週間前に自分が入れた `skip_content_fetch` も、行が存在するかどうかという構造に置き換わって消えました。

次に状態で。処理結果を「タグ用の列挙 + nullable」で表すのをやめ、判別可能な合併型(sum type)に割りました。本文取得の結果を `status: Literal[...]` + `article_id: int | None` から `Fetched | AlreadyExists | Skipped` に分け、`Fetched` は `article_id` を必ず持つようにしています(`526e4da4`, 2026-04-21。コミットの言葉では「the article_id invariant is structural rather than runtime」)。`(None, None, None)` のような「墓標」インスタンスも、成功と失敗を別の型に分けることで消しました(`aae9fc68`)。行のレベルで `skip_content_fetch` を消したのと同じことを、今度は結果型のレベルでやっていた、とも言えます。

そして「ありえない状態」を型で書けなくする、まで進めました。AI 分析では、対象範囲内を表す `InScope` の `category` が、対象外(`OUT_OF_SCOPE`)まで含む列挙を使っていて、「対象範囲内なのにカテゴリは対象外」という矛盾が型の上で許されていました。当時の spec も「`OUT_OF_SCOPE` も型上 valid(型の弱さ)」と認めています。これを、対象外を除いた `InScopeCategory` を新設して塞ぎ、「『対象範囲内なのに OUT_OF_SCOPE』という矛盾状態を型システムで排除する」ようにしました(`c165b04f` / #435, 2026-05-10)。分岐は `match` + `assert_never` で書き、union に枝を足したら網羅漏れが型チェックで弾かれるようにしています。

ここで一つ、意図して「型を要求しない」判断もしました。AI(Gemini / DeepSeek)には判別可能 union の形では返させず、平らな応答スキーマのまま受けて、コード側で振り分けています。理由は spec に明記したとおり「構造化出力で discriminated union を要求すると AI 精度が落ちるため」。型の安全さ(内側)と AI の精度(境界)を両立させるために、境界では平ら・内側でだけ union、という非対称をあえて選びました。

## 3.4 第3幕の終わりに

振り返ると、ここまでの三つの幕は、「不正なものを型・構造として書けなくする」という同じ一つの発想が、対象を広げていった時期でした——**値(VO)→ 振る舞い(モデルへ)→ モジュール境界(ドメイン軸)→ 構造(テーブル)→ 状態(union)**。判別可能 union や `assert_never` は、その一番外側で使った道具で、それ自体が目的だったわけではありません。

考え方の面では、この時期に変わったのは「違和感を信じる」ことだったと思います。第1幕のレビューで違和感を拾う目が育ち、この幕の転換はどれも、自分のコードを読んで引っかかった違和感を言葉にするところから始まっています。

ただ、ここまではまだ「新しいドメイン観のもとで、型と構造を整える」段階でした。次の第4幕では、設計の手法そのものを追うのをやめ、「このアプリで価値を生んでいるのは何か」「そこをより良くするにはどうするか」を先に問うようになります。設計が、綺麗にするための目的から、価値を良くするための手段へと変わっていく時期の話です。
