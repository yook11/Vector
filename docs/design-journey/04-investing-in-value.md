# 第4幕 — 価値の中核に、設計を集中する

第3幕までで、値・振る舞い・構造に意味を持たせる、少しずつ、設計の手段が揃ってきていました。

けれど、この頃の私は、それらをまだ本質的には使えていませんでした。

「いい設計にしたい」
「コードを綺麗にしたい」
「ベストプラクティスの形に寄せたい」

そうやって手は動かしていたものの、その設計がこのアプリケーションにどのような価値を与えるのかを、まだ十分には掴めていなかったのです。

もちろん、漠然とした構想はありました。海外の先端テックニュースを集め、翻訳し、AI で分析して、日本の投資家が読める形にする。けれど、その中のどの部分が本当に使ってもらう理由になるのか。ユーザーが価値を感じるものはなんなのか？そこを正面から考えきれていませんでした。
当時の開発メモにも、その癖が残っています——「どんなことを実現するための機能なのかを考えずに実装したものは、大体リファクタリングの対象になる」。

第4幕は、その問いが変わった時期の記録です。「どうすれば綺麗になるか」ではなく、「このアプリで価値を生んでいるのは何か」「それをどう良くするか」を先に考えるようになりました。価値の中心は、AI 分析でした。海外ニュースをただ翻訳することではなく、投資家が読むべき情報として整理し、判断材料に変えること。そこが価値の中核だと見て、逆算して設計していきました。

設計は目的ではなく、価値を高めるためのもの。不思議なもので、このように意識を変えてからの方が、漠然と綺麗にしようとしていた頃よりも、
開発ははるかに前に進むようになりました。

ここでやったことは、大きく4つの動きに分けられます。

## 4.1 小さなきっかけ

最初の大きな転換は、このアプリケーションが提供している価値とは何かを考え直したことでした。

「中核にあるのは、AI によって翻訳・要約・分析された記事を提供すること」この前提に立ってから、これまで見えていなかった設計上のズレが見えるようになりました。

ユーザーが画面で見ているのは、翻訳され、要約され、分類や投資家向けの見立てが加えられた、分析済みの記事です。にもかかわらず、API で外部に見せる ID や、一覧を取得する DB クエリの起点は、まだ分析前の記事になっていました。ウォッチリストの参照先も同じでした。

不思議なことに、レポジトリのメソッドも API の設計も、これまで何度もレビューし、何度も手を入れてきた場所でした。

設計としては不自然だったにもかかわらず、コードを綺麗にする、責務を分ける、構造を整える、という見方だけでは気づくことができなかったのです。

そこで、ユーザーが画面で見るもの、API が外へ見せる ID、DB クエリが最初に辿る対象を、実際に価値を持つ分析済みの記事へ揃え直しました。

小さな改善だったと思います。けれどこれは、「価値のあるものは何か」という視点を持つことで設計のズレが見えるようになった、今後にもつながる大きな経験でした。

## 4.2 AI 分析を より良いものに

AI 分析が中核なら、それをより良くするにはどうすればよいのか。そう考えるようになりました。

当時、AI 分析で行っていた処理は、次のような流れでした。

・英語の記事タイトルを日本語に翻訳する。
・記事本文を読み、重要な内容を日本語で要約する。
・そのニュースが業界に与える影響や、投資家が注目すべき点を出す。
・記事がどのカテゴリに属するかを判定する。
・記事の内容に合うトピック名を付ける。
・市場への影響度を判定し、なぜその判定にしたのかを日本語で説明する。

しかし、この構成では分析結果の品質に問題がありました。
日本語で出力してほしい箇所に英語が混ざったり、カテゴリ分類でも「なぜこの記事がここに入るのか」と疑問を感じたりすることがありました。

なぜこのような問題が起きるのか？
そこで、タスクの性質を捉え直してみることにしました。
考えてみると、翻訳、要約、分類、投資家向けの見立てという性質の異なるタスクを、1 回の AI 呼び出しに詰め込みすぎているのではと感じました。

分析を２段に分ける、まず取得した記事から重要な情報を正確に取り出し、そのあとで、その情報をもとに投資家向けの判断を加えていくという流れがよいのではないかと考えました。

最初の工程では、記事本文を読み、タイトルや要約を日本語で整える。ここでは、記事に何が書かれているのかを取り出し、次の工程が使える材料にすることに集中します。

次の工程では、その整理済みの内容をもとに、カテゴリを判定し、投資家向けの見立てを加えます。

この整理で、出力を改善することができました。以前のように日本語で出してほしい箇所に英語が混ざることはほとんどなくなり、AI 分析はプロンプトの文言だけでなく、処理の分け方そのものでも品質が変わるのだと実感しました。

ただし、この時点で今の形まで見えていたわけではありません。現在のように、最初の工程で記事の内容を十分に整理し、その密度の高い情報から `key_points` や登場した企業名などを取り出す、という発想まではまだありませんでした。

それでも、この分割は大きな前進でした。責務を分けると綺麗だからではなく、AI 分析の品質を上げるにはどうすればよいかを真剣に考え、実際に工程を見直した最初の経験だったからです。

## 4.3 ユーザーが本当に求めているものを考える

次に、「ユーザーは何を求めているのか」という視点で考えることにしました。

ニュースの一覧画面を見ると、分析結果の中に、明らかに投資文脈に関係のない記事が混ざっていました。ニュースとしては成立していても、投資判断の材料にはならない記事があります。

ここで考えたのは、投資に関係のない記事をどう扱うか、ということでした。

それまで私は、AI の処理が終わった記事を、そのまま「分析済みの記事」として扱っていました。けれど、ユーザーに見せる価値という観点で考えると、それだけでは粗すぎました。

分析が終わった記事の中にも、投資判断の材料になるものと、ニュースとしては成立していてもこのサービスで見せる対象ではないものがあります。そこで、「分析済みの記事」を、単に処理が完了した記事ではなく、投資文脈に入るかどうかまで判定された記事として捉え直しました。

そのために、AI がカテゴリを判定するときに `out_of_scope` を選べるようにしました。投資判断に資する具体的な事象がある記事は投資文脈の記事として扱い、そうでない記事は対象外として扱う。そうすることで、分析済みの記事の中にある違いを、後から曖昧に処理するのではなく、概念として分けられるようにしました。

この変更では、プロンプトの見直しもかなり効きました。単にカテゴリ一覧に `out_of_scope` を追加するだけではなく、「投資判断に資する具体的な事象がない場合は対象外にする」「技術用語が出ているだけで投資価値があると判断しない」という基準を明確にしました。

これによって、AI に無理やり既存カテゴリへ分類させるのではなく、ユーザーに見せる価値がない記事を、正しく対象外として扱えるようになりました。

この境界は、後から型にも反映していきました。投資判断に寄与すると判定された結果には、`out_of_scope` を入れられないようにする。つまり、「対象範囲内なのに対象外」という矛盾を、あとから条件分岐で防ぐのではなく、型の形として書けないようにしていきました。

この頃から、AI に何を出力させるべきかも考え直すようになりました。

それまでは、AI に `impact_level` という市場への影響度を判定するラベルを出力させていました。
クリティカル、ハイ、ミディアム、ローの中から選択させる設計で、一見すると読む優先順位を判断する手がかりになるように見えていました。
けれど実際には、重要ではないニュースにも high が付いたり、全体的に medium に寄ったりして、役に立つ指標にはなっていませんでした。

他にも、新しい技術トレンドが出てきたときにすぐ反映できるように、AI に記事ごとのトピック名を生成させ、その内容で記事を絞り込めるようにしていました。
固定されたカテゴリだけでは、新しい動きを拾いきれないと考えていたからです。

けれど実際には、AI が抽出するトピックは粒度や表現が揺れやすく、制御が難しいものになっていました。同じような記事でも別の名前が付いたり、逆に広すぎる名前が付いたりして、ユーザーにとっても、後続の集計にとっても扱いづらくなっていました。

名寄せの中間テーブルを用意する、定期的にトピックを整理するなどのアイデアはありました。けれど、ここに時間をかけ続けるよりも、投資家が読むときに本当に役立つ情報へ絞るべきだと考え、トピック名を中心にした設計は削っていきました。

エンティティも見直しました。企業名・製品名・技術名を一覧として抜き出すだけでは、なぜその名前が重要なのか、記事のどの文脈で出てきたのかが見えにくい。投資家が知りたいのは、単に特定の企業や技術名が登場したことではなく、その名前がどの重要な出来事の中で出てきたのかでした。

そこで出力の中心を、記事で押さえるべき重要な内容に寄せていきました。固有名を独立した一覧として扱うのではなく、重要な内容の文脈に結びつける方向へ変わっていきました。

AI が生成できる情報を並べるのではなく、投資家が読むときに本当に有効な情報へ絞っていく。ここから、AI 分析の出力を「生成できる情報」ではなく、「ユーザーにとって価値のある情報」として設計し直すようになっていきました。

## 4.4 「分析する価値のある記事」とは何か、という問い

主役・処理・出力を価値から測り直していくと、もう一つの問いが開いてきました。そもそも「分析する価値のある記事」とは何か、という問いです。

きっかけは、ニュースの公開日時でした。

本来見たいのは、そのニュースがいつ公開されたのかです。けれど当時は、公開日時を取得できなかったときに、分析を行なった時刻で補完していました。
これは、単に値が欠けているよりも悪い状態でした。欠けているなら「分からない」と扱えます。けれど、処理時刻が公開日時として入ってしまうと、正しくない値が正しい情報のように見えてしまいます。公開日時は、投資家がニュースの鮮度を判断するための重要な情報だということを捉えることができていませんでした。

このことをきっかけに、「分析をする前の記事」について考えるようになります。

当時、分析をする前の記事の選別は、取得した記事に本文があるかどうかを中心にした、かなりシンプルなものでした。その記事が分析の入力として十分なデータを備えているかまでは考えきれていませんでした。

公開日時の問題は、その見方を変えるきっかけでした。本文が取れていても、いつ公開されたニュースなのかが分からなければ、投資家に見せる情報としては不十分です。

ここで、「取得できた記事」と「分析に進める記事」は同じではない、と捉え直すようになりました。取得できた記事とは、RSS や URL から存在を見つけた記事です。一方で、分析に進める記事とは、本文・タイトル・公開日時のように、分析の入力として必要なデータを備えた記事です。

ただし、この時点で条件が完成していたわけではありません。現在のように型で保証する設計でもなく、どこまでを必須にするかは後から何度も整理し直すことになります。それでも、同じ「記事」でもアプリケーションの中で概念が異なることを学びました。

この過程で、発見した時点では足りない情報があっても、後から補って分析に進める品質まで完成させる、という考えも生まれました。取得するソースによって、提供しているデータが異なるためです。

RSS に本文がほとんど入っていないものもあれば、公開日時が欠けているものもあります。HTML 側のメタデータに日付が残っているものもあれば、日付形式やタイトルの出方にもソースごとの癖があります。

それまでは、取得した記事に必要な情報が足りなければ、分析に進めないものとして扱っていました。けれど、この頃から、足りないなら止めるのではなく、足りない情報を補ってから分析に回す、という発想に変わっていきます。

そのために、HTML から本文や公開日時を補うようにしました。その後は、タイトルも HTML から取り直すようになります。ソースが最初に渡してきた情報をそのまま最終形として扱うのではなく、発見した記事を分析に耐える記事へ整えてから次の工程へ渡す。そういう工程として、取得と分析の間を考えるようになりました。

まだ粗い形ではありましたが、この時期に学んだ大きなことは、同じ「記事」でもアプリケーションの中では別の概念になりうる、ということでした。外部ソースから見つけた記事と、分析に進める記事は同じではありません。前者は発見された情報であり、後者はユーザーに価値を届けるための入力として必要な材料を備えたものです。

これは、ユーザーが求めているものは何かを考えた結果、工程そのものを見直すことにつながった経験でもありました。ユーザーが必要としているのは、ただ取得された記事ではなく、投資判断に使える状態まで整えられた記事です。その視点に立ったことで、取得と分析の間に、足りない情報を補い、分析に進める品質まで整える工程が必要だと考えるようになりました。

## 4.5 第4幕の終わりに

「このアプリで価値を生むのは何か」「どうすればより良くなるのか」。この考えを持つようになってから、設計に対する考え方は大きく変わりました。

ここに設計を投資する価値があるか。ユーザーにとって価値があるものは何か。そう考えるようになってから、手を入れるべき場所が見えやすくなり、開発の進み方も変わりました。それも大きな経験でした。

この考え方は、現在の AI 分析の内容にもつながっています。
今は、AI 分析の両方の工程で、投資文脈に関係のない記事を弾く工夫をしています。
一段目では、明らかに関係のないものを `noise` として止める。
二段目では、カテゴリを分けるときに、どのカテゴリにも紐づかないものを `out_of_scope` として扱う。

出力の形も、同じ考え方から変わっています。
今は、記事の重要な内容を `key_points` として整理し、その中から企業名・製品名・技術名などのメンションを抽出するようにしています。
固有名を独立した一覧として並べるのではなく、どの重要な内容の中で登場したのかまで見えるようにするためです。

ここでの考え方がなければ、このような設計を思いつくこともなかったと思います。

ただ、価値の中心に手を入れたことで、新たな問題も見えてきました。
このアプリでは、非同期パイプラインの中で記事の取得から AI 分析までを行います。あるとき、記事が更新されていないことに気づきました。
取得で落ちたのか、本文補完で止まったのか、AI 分析で失敗したのか。どのステージで止まったのかが、まったく分からない設計になっていました。
そこを見えるようにしようとして監査を入れ始めたことが、次の第5幕につながります。
