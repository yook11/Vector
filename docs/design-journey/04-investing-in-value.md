# 第4幕 — 価値の中核に、設計を集中する

第3幕までで、値・振る舞い・構造に意味を持たせる、少しずつ、設計の手段が揃ってきていました。

けれど、この頃の私は、それらをまだ本質的には使えていませんでした。

「いい設計にしたい」
「コードを綺麗にしたい」
「ベストプラクティスの形に寄せたい」

そうやって手は動かしていたものの、その設計がこのアプリケーションにどのような価値を与えるのかを、まだ十分には掴めていなかったのです。

もちろん、漠然とした構想はありました。海外の先端テックニュースを集め、翻訳し、AI で分析して、日本の投資家が読める形にする。けれど、その中のどの部分が本当に使ってもらう理由になるのか。ユーザーが価値を感じるものはなんなのか？そこを正面から考えきれていませんでした。
当時の開発メモにも、その癖が残っています——「どんなことを実現するための機能なのかを考えずに実装したものは、大体リファクタリングの対象になる」。

第4幕は、その問いが変わった時期の記録です。「どうすれば綺麗になるか」ではなく、「このアプリで価値を生んでいるのは何か」「それをどう良くするか」を先に考えるようになりました。価値の中心は、AI 分析でした。海外ニュースをただ翻訳することではなく、投資家が読むべき情報として整理し、判断材料に変えること。そこが価値の中核だと見て、逆算して設計していきました。

設計は目的ではなく、価値を高めるためのもの。不思議なもので、このように意識を変えてからの方が、漠然と綺麗にしようとしていた頃よりも、
開発ははるかに前に進むようになりました。

ここでやったことは、大きく4つの動きに分けられます。

## 4.1 小さなきっかけ

最初の大きな転換は、このアプリケーションの価値とは何かを考えてみたことでした。

中核は、AI を使って翻訳・要約・分析した記事を提供することだと考えました。

この視点を置いてから、今まで見えていなかったズレが見えるようになりました。

ユーザーが画面で見ているのは、翻訳され、要約され、分類や投資家向けの見立てが加えられた、分析済みの記事です。にもかかわらず、API で外部に見せる ID や、一覧を取得する DB クエリの起点は、まだ分析前の記事になっていました。ウォッチリストの参照先も同じです。

不思議なことに、レポジトリのメソッドも、API の設計も、これまで何度もレビューし、何度も手を入れてきた場所でした。

とても不自然な流れになっていたにも関わらず、コードを綺麗にする、責務を分ける、構造を整える、という見方だけでは気づくことができなかったのです。

ユーザーが画面で見るもの、API が外へ見せる ID、DB クエリが最初に辿る対象を、実際に価値を持つ分析済み記事へ揃え直しました。

これが、価値のあるものは何かという視点を置いたことで、設計のズレが見えるようになった最初の経験でした。
とても小さい改善だったと思います。ただこれは、今後にもつながるとても大きな経験だったと考えています。

## 4.2 AI 分析を より良いものに

AI 分析が中核なら、どのようにすればより良いものになるのか。そう考えるようになりました。

当時、AI 分析で行っていた処理は、次のような流れでした。

・英語の記事タイトルを日本語に翻訳する。
・記事本文を読み、重要な内容を日本語で要約する。
・そのニュースが業界に与える影響や、投資家が注目すべき点を出す。
・記事がどのカテゴリに属するかを判定する。
・記事の内容に合うトピック名を付ける。
・市場への影響度を判定し、なぜその判定にしたのかを日本語で説明する。

この時点では、AI 分析は複数の工程に分かれていませんでした。翻訳、要約、分類、評価までを、1 回の AI 呼び出しでまとめて完結させる流れになっていたのです。

しかし、この構成では分析結果の品質に違和感がありました。
日本語で出力してほしいのに英語が混ざる、カテゴリ分類でも「なぜこの記事がここに入るのか」と感じる、ということが起きていました。

そこで、一回の処理で性質の異なるタスクをやらせすぎているのではないか、工程を分けるべきではないか、と考えるようになりました。

タスクの性質を捉え直すと、まず取得した記事から重要な情報を正確に取り出し、そのあとで、その情報をもとに付加価値をつけていく ―― そういう二段の流れがいいのではないか、と考えました。

最初の工程では、記事本文を読み、タイトルや本文を日本語にし、事実ベースの要約を作る。ここでは、記事に何が書かれているのかを取り出し、次の工程が使える材料に整えることに集中します。

次の工程では、その整理済みの内容をもとに、カテゴリや投資家向けの見立てを判断します。

この整理で、出力は改善しました。以前のように日本語で出してほしい箇所に英語が混ざることはほとんどなくなり、AI 分析はプロンプトの文言だけでなく、処理の分け方そのものでも品質が変わるのだと実感しました。

ただし、この時点で今の形まで見えていたわけではありません。現在のように、最初の工程で記事の内容を十分に整理し、その密度の高い情報から key points や登場した固有名を取り出す、という発想まではまだありませんでした。

それでも、この分割は大きな前進でした。責務を分けると綺麗だからではなく、AI 分析の品質を上げるにはどうすればよいかを真剣に考え、実際に工程を見直した最初の経験だったからです。

## 4.3 ユーザーが本当に求めているものを考える

分析の工程を分けただけでは、まだ十分ではありませんでした。次に考える必要があったのは、そもそもユーザーに見せるべき分析とは何か、ということでした。

同じ視点で見ると、分析結果の中に、明らかに投資文脈に関係のない記事が混ざっていることも問題でした。ニュースとしては成立していても、投資判断の材料にはならない記事があります。そうした記事まで完成した分析として扱うと、ユーザーが見る一覧にも、後続の検索や推薦にもノイズが混ざってしまいます。

ここで必要だったのは、処理に失敗した記事を落とすことではありませんでした。記事として壊れているのではなく、ユーザーに見せる価値がない。そういう記事を、失敗ではなく「対象外」として扱うことでした。

そこで、AI の判定結果に `out_of_scope` を持たせ、投資判断に寄与しない記事は、完成した分析とは別に扱うようにしました。投資判断に寄与すると判定された記事だけをカテゴリや見立てを持つ分析結果として保存し、対象外の記事は「処理失敗」ではなく「見せる対象ではない」という結果として分けて残すことにしました。

この境界は、後から型にも反映していきます。投資判断に寄与すると判定された結果には、`out_of_scope` を入れられないようにする。つまり、「対象範囲内なのに対象外」という矛盾を、あとから条件分岐で防ぐのではなく、型の形として書けないようにしていきました。

AI の出力そのものも、「ユーザーにとって有益な情報とは何か」から測り直しました。象徴的だったのが `impact_level` です。最初は、読む優先順位を決める手がかりになると思っていました。けれど実際に見ていくと、重要ではないニュースにも high が付いたり、全体的に medium に寄ったりして、ユーザーから見て役に立たないものになっていました。

そこで、`impact_level` は削る判断をしました。情報を増やすこと自体に価値があるのではなく、ユーザーの判断を助ける情報だけを残すべきだと考えたからです。

一方で、残すべき情報も測り直していきました。最初は、記事にトピック名を付け、登場した企業・製品・技術名をエンティティとして抜き出していました。トピックは分類や集計の軸になり、エンティティはトレンド検出の材料になると考えていたからです。

けれど、トピックは粒度が揺れやすく、カテゴリ体系の変更にも引っ張られました。エンティティも、固有名の一覧として持つだけでは、なぜその名前が重要なのか、記事のどの文脈で出てきたのかが見えにくい。投資家が知りたいのは、単に特定の企業や技術名が登場したことではなく、その名前がどの重要な出来事の中で出てきたのかでした。

そこで出力の中心を、記事で押さえるべき重要な内容に寄せていきました。整理された記事内容から投資判断に関係する `key_points` を取り出し、その中に登場する固有名を紐づける。固有名を独立した一覧として扱うのではなく、重要な内容の文脈に結びつける方向へ変わっていきました。

`investor_take` は記事全体を投資家の視点でどう捉えるか、`key_points` はその判断を支える具体的な重要点として育っていった情報です。

AI に生成できる情報を並べるのではなく、投資家が読むときに本当に有効な情報へ絞っていく。ここから、AI 分析の出力を「生成できる情報」ではなく、「ユーザーにとって価値のある情報」として設計し直すようになっていきました。

## 4.4 「分析する価値のある記事」とは何か、という問い

主役・処理・出力を価値から測り直していくと、もう一つの問いが開いてきました。そもそも「分析する価値のある記事」とは何か、という問いです。

ここには複数の意味が重なります。本文が揃っていて分析が成立する記事、という技術的な意味もあれば、投資家にとって読む価値がある記事、という価値の意味もあります。

AI 分析の価値を考えるほど、分析そのものだけでなく、AI に渡す前の記事の品質も問題になっていきました。RSS から取れた情報だけでは、本文が足りない記事もあれば、公開日時が欠けている記事もあります。材料となる記事が不十分なままでは、どれだけ分析工程を整えても、出てくる結果の品質には限界があります。

そこで、取得した記事をそのまま分析に流すのではなく、足りない情報を HTML から補う方向へ進みました。最初は本文を補い、すぐに公開日時も補うようにしています。公開日時は、投資家にとって記事の鮮度を判断するための重要な情報だったからです。

この過程で、それまで一つに見えていた「取得した記事」という概念も分かれていきました。一つは、RSS や URL から最初に見つかった記事。もう一つは、本文やタイトルなどが揃い、分析に進める品質を満たした記事です。つまり、取得できたものすべてを分析対象にするのではなく、分析に耐える状態まで整えたものだけを次の工程へ渡す、という考え方が出てきました。

ただし、この時点ではまだ、現在の `AnalyzableArticle` のように型で条件を閉じた設計ではありませんでした。条件の整理も粗く、後から何度も作り直すことになります。それでも、取得と分析の間に「分析に進める品質」という境界が必要だと気づき始めたのは、この時期でした。

## 4.5 第4幕の終わりに

この章で変わったのは、技法ではなく問いの立て方でした。「いい設計にしよう」「綺麗にしよう」と形を目指していたところから、「このアプリで価値を生むのは何か」「それをどう良くするか」を先に問うようになり、設計はその答えを出すための手段になりました。

主役を分析済み記事へ移したのも、分析を分けたのも、対象外の記事を分けたのも、出力を測り直して `impact_level` を削ったのも、すべて同じ一つの問いから出ています。ここに設計を投資する価値があるか。ユーザーにとって価値があるものは何か。その問いを、型や構造ではなくプロダクトの価値に向けた結果でした。

最後には、価値の実体に名前も合わせています。Stage 4 を、狭い「分類」から、対象かどうかの判定・分類・投資家向けの見立てまでを含む「評価」へと改名しました。名前もまた、価値の置き場所を表すものになっていきました。

ただ、この時点で何もかも分かっていたわけではありません。価値の中心に手を入れるほど、今度は「記事がなぜ次の工程へ進まなかったのか」を追えないことが問題になっていきました。取得で落ちたのか、本文補完で止まったのか、AI 分析で失敗したのか。そこを見えるようにしようとして監査を入れ始めたことが、次の第5幕につながります。

次の第5幕では、監査を入れようとしたことで、それまで「分けられている」と思っていた責任の境界が、まだかなり甘かったことに気づいていきます。
