# 第4幕 — 価値の中核に、設計を集中する

第3幕までで、値・振る舞い・構造に意味を載せる道具は一通り手に入りました。ただ、振り返るとこの頃の私は、設計そのものを目的にしていたところがありました。「いい設計にしたい」「コードを綺麗にしたい」「ベストプラクティスの形に寄せたい」——そう手を動かしてはいたのですが、このアプリケーションのどこに価値があるのかを、まだ十分には掴めていませんでした。

もちろん、漠然とした構想はありました。海外の先端テックニュースを集め、翻訳し、AI で分析して、日本の投資家が読める形にする。けれど、その中のどの部分が本当に使ってもらう理由になるのか。ユーザーが価値を感じるものはなんなのか？そこを正面から考えきれていませんでした。当時の開発メモにも、その癖が残っています——「どんなことを実現するための機能なのかを考えずに実装したものは、大体リファクタリングの対象になる」。

第4幕は、その問いが変わった時期の記録です。「どうすれば綺麗になるか」ではなく、「このアプリで価値を生んでいるのは何か」「それをどう良くするか」を先に問うようになりました。価値の中心は、AI 分析でした。海外ニュースをただ翻訳することではなく、投資家が読むべき情報として整理し、判断材料に変えること。そこが価値の中核だと見て、価値から逆算して測り直していきました。

設計は目的ではなく、価値を高めるための手段になりました。不思議なもので、問いを変えてからの方が、漠然と綺麗にしようとしていた頃よりも、
開発ははるかに前に進むようになりました。

この章では、主に三つの動きを振り返ります。

## 4.1 ユーザーが見るものを、設計の起点に戻す

当時 backlog に書いた設計メモには、こう残しています——
「Vector において価値を生んでいるのは 「AI 分析済み記事」であり、「取得した生記事」ではない。ユーザーが画面で見る・操作する対象は全て分析済み記事」(2026-04-08)。

この視点で改めて見ると、既存の設計にはズレがありました。
API で外部に見せる ID や、一覧を取得する DB クエリの起点は、まだ分析済み記事ではなく生記事になっていました。ウォッチリストの参照先も同じです。
つまり、ユーザーに見せたい価値は分析済み記事にあるのに、API やデータアクセスの設計では、取得した記事が主役のままだったのです。

これは、単にコードレビューをしていただけでは気付かなかった、「このアプリで価値を生んでいるのは何か」と考えたからこそ見えた違和感でした。

そこで、外部に見せる ID、クエリの起点を、分析済み記事の側へ寄せていきました(2026-04-08〜09)。
たとえばキーワードの参照先を生記事から分析結果へ付け替えたコミットでは、「キーワードは分析の結果なのだから、分析結果を参照すべき」と整理しています。

この変更で揃えたかったのは、ユーザーが触る入口です。ユーザーが画面で見るもの、API が外へ見せる ID、DB クエリが最初に辿る対象を、実際に価値を持つ分析済み記事へ揃え直しました。

## 4.2 分析の価値を高めるために、構造を変える

価値の中心が AI 分析だと分かると、次の問いは「いまの分析を、どう設計し直せばもっと役に立つものにできるか」になりました。

当時は、翻訳・要約・カテゴリ分類・トピック抽出・投資家向けの見立てまでを、1 回の AI 呼び出しに詰め込んでいました。記事から重要な情報を取り出して整理することと、その情報をもとに投資文脈で分析・判断することは、性質の違うタスクです。にもかかわらず、それらを同じプロンプトで同時にやらせていたのです。

実際、分析結果の品質にも違和感が出ていました。日本語で出力してほしいのに英語が混ざったり、カテゴリ分類も「なぜこの記事がここに入るのか」と感じることがありました。一度に多くのことをやらせていたため、どこを直せば品質が上がるのかも見えにくくなっていました。

そこで、まず記事内容を整理する工程と、その後に投資文脈で判断する工程を分けました。これは「責務を分けると綺麗だから」ではなく、分析の質を上げるための分割でした。

工程を分けてみると、それまで「分析済み記事」と一括りにしていたものの中に、実は二つの概念が混ざっていたことに気づきました。一つは、取得した記事を翻訳し、要約して、日本語で読める形に整える途中成果です。もう一つは、その要約をもとにカテゴリや投資家向けの見立てまで加えた、ユーザーに見せるための完成した分析です。

最初は、翻訳・要約の途中成果と、分類や見立てまで終わった完成形を、同じ保存先で管理していました。翻訳・要約が終わった時点で行を作り、あとから分類や見立てを書き足す形です。そのため、分析が完了するまでは後続工程の項目が空のまま残り、「データがある」こと自体が「ユーザーに見せられる分析が完成している」保証になっていませんでした。完成済みかどうかを、空欄の有無で判断するしかなかったのです。

そこで、途中成果と完成した分析を別の保存先に分けました。翻訳・要約の結果は次の分析工程へ渡すための成果物として残し、完成した分析の保存先には、カテゴリや見立てまで揃った、ユーザーに見せられるものだけを置くようにしました。

AI の出力も「ユーザーにとって有益な情報とは何か」から測り直しました。象徴的だったのが `impact_level` です。最初は、読む優先順位を決める手がかりになると思っていました。けれど実際に見ていくと、重要ではないニュースにも high が付いたり、全体的に medium に寄ったりして、ユーザーから見て役に立たないものになっていました。

そこで、`impact_level` は削る判断をしました。情報を増やすこと自体に価値があるのではなく、ユーザーの判断を助ける情報だけを残すべきだと考えたからです。

同じ視点で見ると、分析結果の中に、明らかに投資文脈に関係のない記事が混ざっていることも問題でした。ニュースとしては成立していても、投資判断の材料にはならない記事があります。そうした記事まで完成した分析として扱うと、ユーザーが見る一覧にも、後続の検索や推薦にもノイズが混ざってしまいます。

ここで必要だったのは、処理に失敗した記事を落とすことではなく、「ユーザーに見せる価値がある記事かどうか」を判断することでした。そこで、AI の判定結果に `out_of_scope` を持たせ、投資判断に寄与しない記事は完成した分析とは別に扱うようにしました。投資判断に寄与すると判定された記事はカテゴリや見立てを持つ分析結果として保存し、対象外の記事は「処理失敗」ではなく「見せる対象ではない」という結果として分けて残すことにしました。

後から、この境界は型にも反映していきます。投資判断に寄与すると判定された結果を表す型には、`out_of_scope` を入れられないようにする。つまり「対象範囲内なのに対象外」という矛盾を、あとから条件分岐で防ぐのではなく、型の形として書けないようにしました。

この考え方は、その後の開発にもつながっていきます。投資家の視点からこのニュースをどう捉えるか、この記事で押さえるべき重要なポイントは何か。`investor_take` や、後の `key_points` は、その問いから残して育てていった情報です。AI に多くを語らせるのではなく、投資家が読むときに本当に有効な情報へ絞っていく。ここから、AI 分析の出力を「生成できる情報」ではなく、「ユーザーにとって価値のある情報」として設計し直すようになっていきました。

## 4.3 取得した記事を、分析に進める記事へ整える

AI 分析の価値を考えるほど、分析そのものだけでなく、AI に渡す前の記事の品質も問題になっていきました。RSS から取れた情報だけでは、本文が足りない記事もあれば、公開日時が欠けている記事もあります。材料となる記事が不十分なままでは、どれだけ分析工程を整えても、出てくる結果の品質には限界があります。

そこで、取得した記事をそのまま分析に流すのではなく、足りない情報を HTML から補う方向へ進みました。最初は本文を補い、すぐに公開日時も補うようにしています。公開日時は、投資家にとって記事の鮮度を判断するための重要な情報だったからです。

この過程で、それまで一つに見えていた「取得した記事」という概念も分かれていきました。一つは、RSS や URL から最初に見つかった記事。もう一つは、本文やタイトルなどが揃い、分析に進める品質を満たした記事です。つまり、取得できたものすべてを分析対象にするのではなく、分析に耐える状態まで整えたものだけを次の工程へ渡す、という考え方が出てきました。

ただし、この時点ではまだ、現在の `AnalyzableArticle` のように型で条件を閉じた設計ではありませんでした。条件の整理も粗く、後から何度も作り直すことになります。それでも、取得と分析の間に「分析に進める品質」という境界が必要だと気づき始めたのは、この時期でした。

## 4.4 第4幕の終わりに

この章で変わったのは、技法ではなく問いの立て方でした。「いい設計にしよう」「綺麗にしよう」と形を目指していたところから、「このアプリで価値を生むのは何か」「それをどう良くするか」を先に問うようになりました。

その問いから、設計の見え方も変わっていきました。ユーザーが見ているのは取得した生記事ではなく分析済み記事だと捉え直し、API やクエリの起点をそこへ寄せた。AI 分析も、ただ多くの情報を出させるのではなく、ユーザーの判断に役立つ形へ絞り直した。さらに、取得した記事をそのまま分析へ流すのではなく、本文や公開日時など、分析に進める品質を満たしているかを見るようになりました。

ただ、この時点で何もかも分かっていたわけではありません。価値の中心に手を入れるほど、今度は「記事がなぜ次の工程へ進まなかったのか」を追えないことが問題になっていきました。取得で落ちたのか、本文補完で止まったのか、AI 分析で失敗したのか。そこを見えるようにしようとして監査を入れ始めたことが、次の第5幕につながります。

次の第5幕では、監査を入れようとしたことで、それまで「分けられている」と思っていた責任の境界が、まだかなり甘かったことに気づいていきます。
