# ADR-003: BFF プロキシパターンによる認証

> 日付: 2026-03 / ステータス: Accepted

## Context

Better Auth は Cookie ベースのセッション管理を行う。
一方 FastAPI バックエンドはステートレスで、セッションストアを持たない。
ブラウザからバックエンドに安全にユーザー情報を伝達する仕組みが必要。

## Alternatives

| 案 | 評価 |
|---|---|
| JWT をバックエンドで検証 | 署名鍵の共有、失効遅延、クロック同期の問題。不採用 |
| バックエンドが auth DB を直接参照 | ステートレス原則に反する。密結合。不採用 |
| ブラウザから直接 API コール + CORS | シークレット漏洩リスク、認証の一元管理不可。不採用 |
| **BFF プロキシ** | Next.js がセッション検証→ヘッダ注入→バックエンド転送。**採用** |

## Decision

Next.js を唯一の公開エントリポイントとし、BFF プロキシパターンを採用。

**フロー**:
1. ブラウザ → Next.js (`/api/proxy/[...path]`)
2. Next.js が Better Auth セッションを検証
3. 信頼済みヘッダを付与してバックエンドへ転送:
   - `X-User-ID` (UUID)
   - `X-User-Role` (admin/user)
   - `X-Internal-Secret` (共有シークレット)
4. FastAPI は `X-Internal-Secret` を検証し、ヘッダからユーザー情報を取得

## Rationale

- **ステートレス維持**: バックエンドにセッション管理が不要。水平スケール可能
- **シークレット保護**: API シークレットはサーバーサイドのみ。ブラウザに露出しない
- **多層防御**: httpOnly Cookie + サーバー限定シークレット + Docker ネットワーク分離
- **監査容易性**: 全 API コールが単一のプロキシルートを通過

## Consequences

- フロントエンド→バックエンド間に1ホップ追加（内部ネットワークなので <1ms）
- プロキシ障害時は全 API が停止（Docker Compose の再起動ポリシーで緩和）
- Docker Compose でバックエンドを `internal` ネットワークのみに接続し、外部アクセスを遮断する必要あり
