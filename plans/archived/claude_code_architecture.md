# Claude Codeのアーキテクチャとコンテキスト最適化
## CLAUDE.mdとSkillsの統合的設計論

---

## 1. エージェント型AIコーディングにおけるステートレス性とコンテキストのジレンマ

Anthropicが提供する**Claude Code**は、ターミナル・デスクトップ・IDE（VS Code / JetBrainsなど）において、自律的にコードベースを探索・編集・テスト・操作する次世代のエージェント型AIコーディングアシスタントである。

しかし、その基盤であるLLMには**「ステートレス（無状態）」** という根本的な性質が存在する。セッションごとに記憶がリセットされるため、毎回プロジェクトへの「オンボーディング」が必要となる。

このオンボーディング情報を継続的に提供する主要な手段が以下の2つである。

| 手段 | 種別 | 役割 |
|------|------|------|
| **CLAUDE.md** | 静的な記憶ファイル | プロジェクト固有の長期記憶 |
| **Skills** | 動的な機能モジュール群 | タスク・ドメイン知識のカプセル化 |

### コンテキストウィンドウとトークンの問題

- すべてのLLMには**コンテキストウィンドウ**という情報処理の上限が存在する
- ソースコードはトークン密度が高く、一回のデバッグで数万トークンを消費する
- コンテキストが飽和すると **「コンテキストの腐敗（Context Rot）」** が発生し、パフォーマンスが著しく低下する
  - 重要な指示の「忘却」
  - 存在しない変数・関数の幻覚（ハルシネーション）
  - APIコストの指数関数的増加・レイテンシの悪化

> **設計の最重要課題：** 「十分な文脈の提供」と「コンテキストウィンドウの枯渇防止」という相反する要求をいかに調和させるか。

## 2. グローバル・コンテキスト層としてのCLAUDE.mdの設計原則

**CLAUDE.md** はプロジェクトのルートに配置され、セッション開始時に必ず最初に読み込まれるマークダウンファイルである。

### 階層的な設定スコープと優先順位

上位のスコープで定義された設定は、下位のスコープを上書きする。

| スコープレベル | 保存場所 | 適用範囲と主な役割 |
|--------------|---------|------------------|
| **マネージド（組織）** | `/etc/claude-code/` またはMDMプロファイル経由 | IT部門が管理する組織全体で強制されるセキュリティポリシー・規約 |
| **ユーザー（個人）** | `~/.claude/CLAUDE.md` | すべてのプロジェクトに適用される個人的なコーディングスタイル・好み |
| **プロジェクト（共有）** | `./CLAUDE.md` または `./.claude/CLAUDE.md` | Gitでバージョン管理され、チーム全体で共有されるプロジェクト固有の設定 |
| **ローカル（非共有）** | `./CLAUDE.local.md` | Gitの管理から除外される個人向け設定（ローカルDBのURLなど） |

### 「What」より「How」に焦点を当てる

❌ **NG：** README.mdをそのままコピーする（プロジェクトの概要・目的の記述）
✅ **OK：** AIがどのように振る舞い、どのようにコードを書くべきかを記述する

#### 振る舞いのガイドライン（Behavior Guidelines）の例

- **応答スタイルの最適化：** 「挨拶を省略する」「絵文字を使用しない」「結論から記述する」
- **検証プロセスの確立：** 「変更後は必ず `npm run test` を実行して検証すること」
- **ツールの優先順位付け：** 「プロジェクト固有のスキルや内部ツールを優先して使用すること」

### 厳格な容量制限とプログレッシブ・ディスクロージャー

- ファイルの長さ：**300行未満**（理想は60行未満）
- 文字数ベース：**300語程度**に抑えるのが理想

**プログレッシブ・ディスクロージャー（漸進的開示）** の原則：
> すべての情報を最初から一度に与えず、重要な情報の「探し方」や「参照先」だけを教え、必要になった時点で初めて詳細情報を読み込ませる。

- DBスキーマの詳細などはCLAUDE.mdに直接書かず、**ファイルパスや行番号の参照（ポインタ）のみ**を記述する
- 具体的なコードスニペットの記載は避ける

### フォーマット規約の分離とリンターの活用

インデント幅・括弧の位置・命名規則などの細かいフォーマット規則はCLAUDE.mdに書かない。

✅ **推奨：** Biome / Prettier / ESLintなどの専門ツールを導入し、CLAUDE.mdには「コミット前にリンターを実行すること」というプロセスのみを指示する。

> `/init` コマンドを実行すると、Claudeがコードベースを分析してベースとなる CLAUDE.md を自動生成してくれる。

## 3. 動的コンテキスト層を担う「Skills」のアーキテクチャ

**Skills（エージェントスキル）** は、特定のタスク・ドメイン知識・反復的なワークフローをカプセル化した動的な機能モジュールである。

- **動作方式：** ファイルシステムベース（`~/.claude/skills/` または `.claude/skills/` 配下）
- Claude Codeがディレクトリを監視し、自動的にスキルを検出・読み込む

### SKILL.mdの構造

すべてのスキルは専用ディレクトリ内の `SKILL.md` に記述され、以下の2セクションで構成される。

1. **YAMLフロントマター**（上部）：メタデータ
2. **Markdownコンテンツ**（下部）：指示の本体

### YAMLフロントマターの主要フィールド

| フィールド名 | 必須 | 役割と詳細仕様 |
|------------|------|--------------|
| `name` | ✅ | スキルの識別子。`/コマンド` のトリガーとなる（小文字・数字・ハイフンのみ、最大64文字） |
| `description` | ✅ | **スキルの成否を分ける最重要フィールド。** Claudeが「いつ」このスキルを使うかを判断するトリガー条件を記述（最大1024文字、三人称で記述） |
| `disable-model-invocation` | ❌ | `true` にするとAIの自律的な読み込みを禁止し、手動コマンドとしてのみ機能させる（本番デプロイなど危険なタスク向け） |
| `user-invocable` | ❌ | `false` にするとユーザー向けコマンド一覧から非表示にする（内部サブタスク専用） |
| `allowed-tools` | ❌ | AIがユーザー許可なしで使えるツールを明示指定（例：`Read, Grep, Bash(git:*)` で読み取り専用環境を構築） |
| `argument-hint` | ❌ | スラッシュコマンド入力時のオートコンプリートヒント（例：`[issue-number]`） |
| `context` | ❌ | `fork` に設定するとメインの会話履歴から切り離されたサブエージェント環境で実行 |

### 遅延読み込み（Lazy Loading）とトークン最適化

```
Level 1（起動時）：すべてのスキルのYAMLフロントマターのみ事前ロード
                   → 約50トークン/スキル（極めて軽量）

Level 2（必要時）：descriptionとタスクが合致した場合のみ
                   SKILL.mdの本文・関連ファイルをコンテキストに展開
```

> ⚠️ **初心者が陥りやすい罠：** スキルが発火しない原因を本文の指示不足と勘違いして本文を書き直してしまうこと。実際には `description` フィールドが不十分なケースがほとんど。

### 評価駆動開発（Evaluation-Driven Development）によるスキル構築

1. **ギャップの特定：** スキルなしの状態でClaudeにタスクを実行させ、失敗・欠落を観察する
2. **評価シナリオの作成：** 特定されたギャップをテストする3つの具体的なシナリオを作成する
3. **ベースラインの確立：** スキルなしの状態でパフォーマンスを測定する
4. **最小限の指示の記述：** ギャップを埋めるために必要な「最小限の指示」だけを記述する
5. **反復と改善：**
   - **過剰トリガー（Overtriggering）：** `description` に否定的なトリガーを追加して範囲を狭める
   - **過少トリガー（Undertriggering）：** 技術的なキーワード・専門用語を追加して発見可能性を高める

## 4. コマンド前処理と高度なサブエージェント機能

### `!command` による動的コンテキストの注入（前処理）

SKILL.md内に感嘆符＋バッククォートでシェルコマンドを記述すると、AIへのプロンプト送信前にシステムがコマンドを実行し、結果を自動注入する。

```markdown
!`gh pr diff`
```

- AIが「コマンドを実行すべきか考え、実行し、結果を待つ」プロセスをスキップ
- **無駄な思考プロセスとトークン・実行時間を大幅に節約**
- 引数の置換もサポート（`$ARGUMENTS` や `$0` で動的に埋め込み可能）

### `context: fork` によるサブエージェントの生成と隔離

`context: fork` を指定すると、現在の会話履歴を一切持たない独立した「サブエージェント」として実行される。

**活用例：**
- コードベース全体の脆弱性スキャン
- 複雑なテストスイートの実行とエラーログ解析

→ 膨大なトークン消費をメインセッションから隔離し、**最終的な結論のみをメインセッションに返す**ことができる。

## 5. 階層型コンテキスト・アーキテクチャの構築

すべての情報を単一のCLAUDE.mdに詰め込んだ場合：
- 毎回のセッション開始時に **2,000トークン以上**が強制ロードされる
- 実際にその情報が関連している割合は **わずか14〜28%**
- 5人の開発者チームで **年間数千ドルの無駄な出費**につながる可能性

### 情報の3ティア管理

| 階層 | 管理レイヤー | 内容の特性と保存場所 | 容量の目安 |
|------|------------|------------------|----------|
| **Tier 1：必須コンテキスト** | グローバル層（CLAUDE.md） | プロジェクトの目的、絶対的な禁止ルール、緊急時のトラブルシューティング | **800トークン未満** |
| **Tier 2：文脈依存コンテキスト** | オンデマンド層（skillsディレクトリ） | コンポーネント固有のドキュメント、外部APIリファレンス、デプロイ・テスト手順 | **500〜1,500トークン**（タスク発生時に動的ロード） |
| **Tier 3：外部・一時コンテキスト** | 外部アクセス層（MCPインデクサー） | 具体的なコードの詳細、過去のPRの内容、外部のWiki情報 | **必要な数行のスニペットのみ**（検索クエリ経由で取得） |

> このアーキテクチャにより、**セッションごとの無駄なコンテキスト消費を最大60%削減**できる。

## 6. コンテキストウィンドウの最適化とリソース管理の高度な技法

### 6.1 能動的なメモリ管理とコンパクション（圧縮）

Claude Codeには自動コンパクション機能があるが、開発者自身が能動的に管理することが推奨される。

#### 主要なコマンド

| コマンド | 用途 |
|--------|------|
| `/context` | 現在のウィンドウ内のトークン消費を可視化 |
| `/cost` | リアルタイムの消費コストを確認（API課金ユーザー向け） |
| `/compact [指示]` | コンテキスト使用率が50%程度に達した時点で手動圧縮（例：`/compact Focus on the API changes`） |
| `/clear` | 一つの論理的なタスクが完了したら履歴を完全消去 |

> CLAUDE.md内に「圧縮時には、常に変更されたファイルの完全なリストとテストコマンドを保持すること」といったメタ指示を記述しておくと、要約の挙動をカスタマイズできる。

### 6.2 探索コストの削減とインデックスベースの検索（MCPの活用）

**コンテキスト浪費の隠れた最大の原因：** AIがコードベースを「探索」するプロセス

```
Glob（ファイル一覧）→ Grep（キーワード検索）→ 3〜4ファイルの全文読み込み
→ 本来の作業前にトークン予算の80〜90%が「情報の検索」だけで浪費される
```

**解決策：** MCPを活用したインデックス検索システムの導入

- `qmd` のようなローカルドキュメントインデクサーをMCPサーバーとして接続
- 事前構築されたインデックスに検索クエリを投げ、必要な10行程度のスニペットのみを瞬時に取得
- **探索に伴うトークン消費を最大95%削減**

> 複数のMCPサーバー（Figma・Jira・Slackなど）を導入する場合、「Tool search（ツール検索）」機能を有効にするとオンデマンドで必要なツールのみを遅延ロードでき、MCP関連のトークン使用量を **14%以下**に削減可能。

### 6.3 フック（Hooks）による自動化と推論のオフロード

フックは、Claude Codeのライフサイクルの特定のポイントでシェルコマンド・HTTPリクエスト・LLMプロンプトを自動実行する機能。

| フックのイベント | 活用例とコンテキスト最適化への寄与 |
|---------------|-------------------------------|
| **SessionStart** | `git log` を実行して直近のコミット履歴を自動注入。AIにGit履歴を検索させる手間とトークンを省く |
| **PreToolUse** | `.env` や保護された設定ファイルの変更をシェルスクリプトで自動ブロック。誤作動による無駄なリカバリー作業を防ぐ |
| **PostToolUse** | ファイル変更後に自動的にコードフォーマッターを実行。AIにフォーマットの責任を負わせない |

> `type: "prompt"` のプロンプトフックは、バックグラウンドで軽量モデル（Haiku）に評価を依頼し、結果のみを受け取る仕組み。重いモデルのコンテキストを消費せずにサブスクリプト的な意思決定が可能。

### 6.4 設定チューニングと物理的な限界値の拡張

#### 出力トークンの拡張

```bash
CLAUDE_CODE_MAX_OUTPUT_TOKENS=64000
```

または `settings.json` の `env` ブロックに永続化することで、デフォルトの32Kキャップを突破できる。

#### 100万（1M）トークンコンテキストモデルの選択

- APIキーを用いた従量課金設定（Consoleアカウント）の場合のみ利用可能
- `/model` コマンドから「Sonnet 1M context」を選択
- 手動での `/compact` 不要で、プロジェクトの全貌を維持したまま開発を進められる
- 100以上のサブエージェントを並列動作させる高度なワークフローも実現可能

#### 適応型推論（Adaptive Reasoning）について

| バージョン | 挙動 |
|----------|------|
| 旧バージョン | `ultrathink` キーワードで思考予算を強制最大化できた |
| 現在 | 拡張思考がデフォルト有効化。Opus 4.6などでは「エフォートレベル」に基づいて動的に思考予算を割り当てる「適応型推論」が導入 |

> ⚠️ 現在、`think harder` や `ultrathink` といった指示は通常のテキストとして処理され、特別なトークン割り当ては発生しない。

#### タスク難易度に基づくモデルの使い分け

| タスク種別 | 推奨モデル |
|----------|----------|
| 複雑なビジネスロジックの設計・実装計画（Plan mode） | **Sonnet / Opus** |
| 確立されたスキルの実行・単純な読み書き・小規模バグ修正（Execution mode） | **Haiku** |

## 7. エージェントワークフローの実践的パラダイム

従来の「コードを書いて、AIに質問する」という受動的な利用から、**「AIに計画を立てさせ、自律的に実行させる」** というエージェント指向のパラダイムへの移行が必要。

### 推奨ワークフロー

```
① Plan（計画）モードから開始
   └─ AIに PLAN.md を生成させ、ビジネスロジック・将来の要件を整理

② IMPLEMENTATION.md を作成
   └─ 開発フェーズの全体像と必要な技術ドキュメントのURLを網羅

③ フェーズごとに実装を開始
   └─ コンテキストが圧迫されたら /clear で履歴を消去
      └─ IMPLEMENTATION.md を読み込ませれば同じ文脈から再開可能
```

> ステートレスなAIの弱点を、ファイルシステムという**永続的な状態（ステート）**を利用して克服する、極めて合理的な運用方法。

## まとめ

> コンテキストウィンドウは**共有の公共財（Public good）**であるという認識を持つべきである。

| 役割 | 手段 | 原則 |
|------|------|------|
| プロジェクトの憲法 | **CLAUDE.md** | 振る舞いの基準（How）を定義。不要なノイズで膨張させない |
| 動的な知識カプセル | **Skills（SKILL.md）** | プログレッシブ・ディスクロージャーにより必要な瞬間にのみ動的呼び出し |
| 探索コストの削減 | **MCPインデクサー** | 不必要なファイル探索を防ぎ、トークン消費を最大95%削減 |
| 定型処理のオフロード | **フック（Hooks）** | AIに考えさせる必要のない処理を自動化 |
| 能動的なクリーニング | **`/compact` / `/clear`** | 定期的なメモリ管理でAIのパフォーマンス劣化を防ぐ |

これらの統合的な設計と運用技術を駆使することでのみ、**長期間にわたって正確かつ高速な自律的コーディング**を維持し、エージェント型AIの真の可能性を引き出すことが可能となる。
